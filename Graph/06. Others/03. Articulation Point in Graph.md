## 1. Problem Statement

Given an **undirected graph**, find all **articulation points** (also called cut vertices).

An **articulation point** is a vertex whose **removal** (along with all its edges) **increases** the number of connected components in the graph.

In simple words: if you delete this city from the map, the map breaks into two or more separate islands.

## Key Insight

```
Observation 1: What makes a vertex an articulation point?

A vertex is an articulation point if it sits on EVERY path
between two different parts of the graph.

Visual (articulation point):

    Group A ─── City X ─── Group B

    Every road from Group A to Group B passes through City X.
    If City X closes, A and B are separated! ✗

Visual (NOT an articulation point):

    Group A ─── City X ─── Group B
         \                 /
          \─── City Y ────/

    There is a bypass through City Y!
    Close City X, traffic still flows! ✓

──────────────────────────────────────────────────────────────

Observation 2: Two Rules for Finding APs

Rule 1 — Root of DFS Tree:
  If the starting vertex has 2 or more children in the DFS tree,
  it is an articulation point.

  Why? Each child subtree is disconnected from the others.
  The root is the ONLY connection between them.

  Visual:

        Root (start)
       /    |    \
      /     |     \
   Subtree Subtree Subtree

  Remove Root → 3 separate pieces!

Rule 2 — Non-root Vertex:
  For an edge parent → child in the DFS tree:

    low[child] >= disc[parent]  →  parent is an AP!

  Why? If the child cannot reach back to the parent or above,
  the parent is the ONLY gateway to that subtree.

  Visual:

        Parent (disc=1)
        │
        │ tree edge
        ↓
        Child (disc=2)
       / \
      /   \
   (subtree)

  If low[child] = 5, and disc[parent] = 1:
  5 >= 1 → child cannot reach back to parent or above!
  → Parent is the ONLY entrance to that region.
  → Parent is an ARTICULATION POINT!

──────────────────────────────────────────────────────────────

Observation 3: Difference from Bridges

  Bridge = critical ROAD (edge)
  Articulation Point = critical CITY (vertex)

  Visual:

      0 ─── 1 ─── 2
            │
            3

  Edge (1,3) is a bridge.
  Vertex 1 is an articulation point.

  Remove edge (1,3): 3 is isolated.
  Remove vertex 1: 0, 2, and 3 are all separated.

  A bridge always connects two articulation points (or one AP
  and a leaf), but not every AP edge is a bridge!
```

## Intuition

```
Imagine a network of islands connected by bridges.

Some islands are CRITICAL — if they sink, other islands become
inaccessible from each other.

Some islands are SAFE — if they sink, boats can still reach
every other island by taking a different route.

──────────────────────────────────────────────────────────────

Critical Island (Articulation Point):

    Island A ─── Island X ─── Island B

    Every route from A to B passes through X.
    If X sinks, A and B are separated! ✗

──────────────────────────────────────────────────────────────

Safe Island (NOT an Articulation Point):

    Island A ─── Island X ─── Island B
         \                     /
          \─── Island Y ─────/

    If X sinks, boats can still go A → Y → B!
    The islands stay connected! ✓

──────────────────────────────────────────────────────────────

The DFS Discovery Times:

Think of disc[] as a timestamp when the explorer first lands
on an island.

    Time 0: Land on Island 0
    Time 1: Land on Island 1
    Time 2: Land on Island 2
    Time 3: Land on Island 3

low[] tells you: "What is the earliest island I can reach from here,
including by sending scouts back through alternate routes?"

If from Island 2 (and everything beyond it), the earliest you can
reach is Island 1 (disc[1] = 1), then:
  low[2] = 1, and disc[parent] might also be 1.
  low[2] >= disc[parent]? 1 >= 1? YES!
  → The parent is an articulation point!

Why? Because Island 2's entire region has NO back door to anything
before the parent. The parent is the ONLY gatekeeper!

──────────────────────────────────────────────────────────────

Root Special Case:

The explorer starts at Island 0. From there, they discover three
separate island chains that never connect to each other except
through Island 0.

        Island 0
       /    |    \
      /     |     \
   Chain A Chain B Chain C

If Island 0 sinks, the three chains are completely separated!
So Island 0 is an articulation point.

We detect this by counting how many separate DFS branches
start directly from the root. If 2 or more, root is an AP.
```

## Algorithm

```
Step 1: Build adjacency list

For each edge (u, v):
  adj[u].push_back(v)
  adj[v].push_back(u)

──────────────────────────────────────────────────────────────

Step 2: Initialize arrays

visited[V] = {false, false, ...}
disc[V]    = {0, 0, ...}     // discovery times
low[V]     = {0, 0, ...}     // lowest reachable discovery time
artpoint[V] = {false, ...}   // articulation point flags
timer = 0

──────────────────────────────────────────────────────────────

Step 3: DFS from every unvisited vertex

For i = 0 to V-1:
  If not visited[i]:
    DFS(i, -1, adj, visited, disc, low, artpoint, timer)

(This handles disconnected graphs!)

──────────────────────────────────────────────────────────────

Step 4: DFS function

Function DFS(node, parent, adj, visited, disc, low, artpoint, timer):
  visited[node] = true
  disc[node] = low[node] = timer++
  children = 0

  For each neighbor of node:
    If neighbor == parent:
      continue  // don't go back to parent

    If neighbor is visited:
      // Back edge found!
      low[node] = min(low[node], disc[neighbor])

    Else:
      // Tree edge - recurse
      children++
      DFS(neighbor, node, adj, visited, disc, low, artpoint, timer)

      // After recursion, update low[node]
      low[node] = min(low[node], low[neighbor])

      // Check articulation point condition
      If parent != -1 AND low[neighbor] >= disc[node]:
        artpoint[node] = true

  // Root special case
  If parent == -1 AND children >= 2:
    artpoint[node] = true

──────────────────────────────────────────────────────────────

Step 5: Collect results

For i = 0 to V-1:
  If artpoint[i] is true:
    add i to answer

Return answer
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

void dfs(int node, int parent, vector<vector<int>> &adjList, vector<int> &disc,
         vector<int> &low, vector<int> &visit, int &timer, vector<int> &ap)
{
    disc[node] = low[node] = timer++;
    visit[node] = 1;
    int child = 0;
    for (auto &&neigh : adjList[node])
    {
        if (neigh == parent)
        {
            continue;
        }
        if (visit[neigh])
        {
            low[node] = min(low[node], disc[neigh]);
        }
        else
        {
            child++;
            dfs(neigh, node, adjList, disc, low, visit, timer, ap);
            low[node] = min(low[node], low[neigh]);
            if (parent != -1 && low[neigh] >= disc[node])
            {
                ap[node] = 1;
            }
        }
    }
    if (parent == -1 && child > 1)
    {
        ap[node] = 1;
    }
}

// ==================== DRIVER CODE ====================
int main(int argc, char const *argv[])
{
    int V = 5;
    vector<vector<int>> edges = {{0, 1}, {1, 2}, {0, 3}, {1, 4}};

    vector<vector<int>> adjList(V);

    for (auto &&i : edges)
    {
        int u = i[0];
        int v = i[1];
        adjList[u].push_back(v);
        adjList[v].push_back(u);
    }

    vector<int> disc(V), low(V), visit(V, 0), ap(V, 0);
    int timer = 0;

    // FIX: Handle disconnected graphs by starting DFS from every unvisited vertex
    for (int i = 0; i < V; i++) {
        if (!visit[i]) {
            dfs(i, -1, adjList, disc, low, visit, timer, ap);
        }
    }

    for (int i = 0; i < V; i++)
    {
        if (ap[i])
            cout << i << endl;
    }

    return 0;
}
```

**Output:**

```
0
1
```

## Step-by-Step Trace

```
Example: V = 5, edges = [[0,1], [1,2], [0,3], [1,4]]

Graph:

    0 ───── 1 ───── 2
    │       │
    │       │
    3       4

Adjacency list:
  adj[0] = [1, 3]
  adj[1] = [0, 2, 4]
  adj[2] = [1]
  adj[3] = [0]
  adj[4] = [1]

── Initialization ──

visited:  [0, 0, 0, 0, 0]
disc:     [0, 0, 0, 0, 0]
low:      [0, 0, 0, 0, 0]
artpoint: [0, 0, 0, 0, 0]
timer = 0

── DFS(0, parent=-1) ──

  visited[0] = 1
  disc[0] = low[0] = 0, timer = 1
  children = 0

  neighbors of 0: [1, 3]

  neighbor 1: not visited, not parent → recurse
    children = 1

    ── DFS(1, parent=0) ──

    visited[1] = 1
    disc[1] = low[1] = 1, timer = 2
    children = 0

    neighbors of 1: [0, 2, 4]

    neighbor 0: is parent → skip!

    neighbor 2: not visited → recurse
      children = 1

      ── DFS(2, parent=1) ──

      visited[2] = 1
      disc[2] = low[2] = 2, timer = 3
      children = 0

      neighbors of 2: [1]

      neighbor 1: is parent → skip!

      No more neighbors. Return from DFS(2).

      Back to DFS(1):
        low[1] = min(low[1], low[2]) = min(1, 2) = 1

        Check AP condition:
          parent(0) != -1? YES
          low[2] = 2 >= disc[1] = 1? YES! ✓
          → artpoint[1] = true!

    neighbor 4: not visited → recurse
      children = 2

      ── DFS(4, parent=1) ──

      visited[4] = 1
      disc[4] = low[4] = 3, timer = 4
      children = 0

      neighbors of 4: [1]

      neighbor 1: is parent → skip!

      No more neighbors. Return from DFS(4).

      Back to DFS(1):
        low[1] = min(low[1], low[4]) = min(1, 3) = 1

        Check AP condition:
          parent(0) != -1? YES
          low[4] = 3 >= disc[1] = 1? YES! ✓
          → artpoint[1] = true (already true)

    No more neighbors. Return from DFS(1).

    Back to DFS(0):
      low[0] = min(low[0], low[1]) = min(0, 1) = 0

      Check AP condition:
        parent == -1 → skip non-root check

  neighbor 3: not visited → recurse
    children = 2

    ── DFS(3, parent=0) ──

    visited[3] = 1
    disc[3] = low[3] = 4, timer = 5
    children = 0

    neighbors of 3: [0]

    neighbor 0: is parent → skip!

    No more neighbors. Return from DFS(3).

    Back to DFS(0):
      low[0] = min(low[0], low[3]) = min(0, 4) = 0

      Check AP condition:
        parent == -1 → skip non-root check

  No more neighbors.

  Root check: parent == -1 AND children = 2 >= 2? YES! ✓
  → artpoint[0] = true!

  Return from DFS(0).

── Final Result ──

disc:  [0, 1, 2, 4, 3]
low:   [0, 1, 2, 4, 3]

artpoint: [1, 1, 0, 0, 0]

Answer: [0, 1]
```

## Edge Cases

```

Graph: 0

Graph: 0 ─── 1

Standard definition: no articulation points for edge graph.

──────────────────────────────────────────────────────────────

Complete graph (no APs):
V = 4, every vertex connected to every other

Graph:
      0
     /|\
    / | \
   1──2──3

Every vertex has multiple alternate routes.
Removing any vertex: rest still fully connected.

Output: []

──────────────────────────────────────────────────────────────

Tree graph (all non-leaves are APs):
V = 5, edges = [[0,1], [0,2], [0,3], [0,4]]

         1
         │
    2 ───0─── 3
         │
         4
```

### Common Mistakes

```
❌ Mistake 1: Forgetting root special case
  Only checking low[child] >= disc[parent] for all vertices.
  The root needs CHILD COUNT >= 2!

  Visual:

        0 (root)
       / \
      1   2

  Remove 0: 1 and 2 are disconnected!
  But low[1] >= disc[0]? 1 >= 0? YES — this might catch it.
  Wait, parent == -1, so the non-root check is skipped!
  Without root check, 0 is MISSED!

✓ Correct:
  if (parent == -1 && children >= 2)
      artpoint[node] = true;

──────────────────────────────────────────────────────────────

❌ Mistake 2: Not handling disconnected graphs
  Only calling dfs(0, -1, adjList, disc, low, visit, timer, ap);
  If vertex 0 is not in the same component as others,
  those vertices are never visited!

  Visual:

    Component 1:    Component 2:
    0 ─── 1         2 ─── 3

    dfs(0) only visits 0 and 1.
    Vertices 2 and 3 are never checked!

✓ Correct:
  for (int i = 0; i < V; i++) {
      if (!visit[i]) {
          dfs(i, -1, adjList, disc, low, visit, timer, ap);
      }
  }

──────────────────────────────────────────────────────────────

❌ Mistake 3: Using low[neib] instead of disc[neib] for visited neighbor
  low[node] = min(low[node], low[neib]);  // WRONG for back edge!

  For a visited neighbor (back edge), we use disc[neib],
  not low[neib]. Because the back edge connects directly
  to neib, not to whatever neib can reach.

✓ Correct:
  if (visited[neib]):
      low[node] = min(low[node], disc[neib]);

──────────────────────────────────────────────────────────────

❌ Mistake 4: Wrong AP condition direction
  if (low[node] >= disc[neib])  // WRONG!

  The condition is low[CHILD] >= disc[PARENT].
  Not the other way around!

✓ Correct:
  if (low[neib] >= disc[node])
      artpoint[node] = true;

──────────────────────────────────────────────────────────────

❌ Mistake 5: Counting children incorrectly for root
  Counting ALL neighbors of root as children.
  But some neighbors might be reached via back edges from
  other subtrees!

  Actually, in DFS tree, children are only the unvisited
  neighbors that trigger recursion. The code correctly
  increments children only before DFS(neib) call.

✓ Correct:
  children++ only inside the "else" (not visited) branch.

──────────────────────────────────────────────────────────────

❌ Mistake 6: Confusing articulation points with bridges
  Bridge = critical edge (road)
  Articulation point = critical vertex (city)

  They are related but NOT the same!
  Every bridge connects two APs (or an AP and a leaf),
  but not every AP edge is a bridge.

✓ Correct: Remember — AP = vertex, Bridge = edge!
```

## Complexity Analysis

**Time Complexity:** `O(V + E)`

- One DFS visits every vertex and edge once.
- All operations inside DFS are O(1).
- Total: O(V + E).

**Space Complexity:** `O(V + E)`

- Adjacency list: O(V + E)
- disc[], low[], visited[], artpoint[]: O(V)
- Recursion stack: O(V)

```
Visual breakdown:

Time:
  DFS visits each vertex once → O(V)
  DFS examines each edge twice (undirected) → O(E)
  Total: O(V + E)

  For V=1000, E=1000: ~2000 operations (instant!)

Space:
  adjacency list: V + E entries
  arrays: 4 × V integers/booleans
  recursion: at most V levels

  Overall: O(V + E)
```

## Interview Questions

```
Q1: What is an articulation point in a graph?

A: An articulation point is like a critical city on a map. If that
   city disappears, the map breaks into two or more separate pieces
   that cannot reach each other.

   Visual:

   Town A ─── City X ─── Town B

   If City X is the ONLY way from A to B:
   → City X is an articulation point! ✓

   Town A ─── City X ─── Town B
        \                 /
         \─── City Y ────/

   If City Y provides a bypass:
   → City X is NOT an articulation point! ✓

──────────────────────────────────────────────────────────────

Q2: What is the difference between an articulation point and a bridge?

A: A bridge is a critical ROAD. An articulation point is a critical CITY.

   Visual:

       0 ─── 1 ─── 2
             │
             3

   Edge (1,3) is a BRIDGE — removing it isolates 3.
   Vertex 1 is an ARTICULATION POINT — removing it separates
   0, 2, and 3 from each other.

   Every bridge connects two critical cities (or a critical city
   and a leaf), but not every road from a critical city is a bridge!

──────────────────────────────────────────────────────────────

Q3: Why is the root of the DFS tree treated specially?

A: The root has no parent above it. So the normal rule
   (low[child] >= disc[parent]) does not apply.

   Instead, we count how many separate branches start from the root.
   If 2 or more, the root is the ONLY connection between those
   branches. Remove the root, and the branches are separated!

   Visual:

        Root
       / | \
      /  |  \
    A    B    C

   Root connects three separate regions.
   Remove Root → A, B, and C are isolated!

──────────────────────────────────────────────────────────────

Q4: What is the condition for a non-root vertex to be an AP?

A: For a tree edge parent → child:

   low[child] >= disc[parent] → parent is an AP!

   In simple words: "From the child's region, the earliest city
   we can reach is the parent itself or later."

   So there is NO back door from the child's region to anything
   before the parent. The parent is the ONLY gate!

   Visual:

        Parent (disc=1)
        │
        │
        Child (disc=2)
       / \
      /   \
   (region)

   If low[child] = 2, and disc[parent] = 1:
   2 >= 1 → no back door! Parent is an AP! ✓

──────────────────────────────────────────────────────────────

Q5: What are disc[] and low[] arrays?

A: disc[] is like a ticket number — the order in which DFS first
   visits each city.

   low[] is like asking: "What is the smallest ticket number I can
   reach from here, including by taking alternate routes?"

   Visual:

   DFS order:
     City 0: ticket 0
     City 1: ticket 1
     City 2: ticket 2

   disc = [0, 1, 2]

   If City 2 has a back road to City 0:
     low[2] = min(disc[2], disc[0]) = min(2, 0) = 0

   So from City 2, we can reach ticket 0!

──────────────────────────────────────────────────────────────

Q6: Can a leaf vertex be an articulation point?

A: No! A leaf has only one road. Removing it does not disconnect
   anything else because the leaf was only connected to one city.

   Visual:

   0 ─── 1 ─── 2
         │
         3  ← leaf

   Remove 3: 0-1-2 still connected.
   Remove 2: 0-1-3 still connected.

   Leaves cannot be articulation points!

──────────────────────────────────────────────────────────────

Q7: Can a graph have no articulation points?

A: Yes! Any graph where every vertex has at least two independent
   routes to the rest of the graph has no articulation points.

   Cycles and complete graphs have no articulation points.

   Visual (no APs):

       0 ─── 1
       │     │
       │     │
       3 ─── 2

   Remove any vertex: the rest still form a connected path!

──────────────────────────────────────────────────────────────

Q8: Can all non-leaf vertices be articulation points?

A: Yes! In a tree (no cycles), every non-leaf vertex is an
   articulation point. Removing it splits the tree into separate
   branches.

   Visual (all non-leaves are APs):

         1
         │
    2 ───0─── 3
         │
         4

   0 is an AP (removing it isolates all leaves).
   Leaves 1, 2, 3, 4 are NOT APs.

──────────────────────────────────────────────────────────────

Q9: What are real-world uses of articulation points?

A: Real-world examples:

   Network security:
   → Find critical servers that would split the network if hacked
   → Add backup connections around them

   Road infrastructure:
   → Identify cities that are the only connection between regions
   → Build bypass highways to reduce dependency

   Power grids:
   → Find substations that would cause blackouts if they fail
   → Design redundant power lines

   Social networks:
   → Find people who are the sole connection between two communities
   → If they leave, information stops flowing between groups

──────────────────────────────────────────────────────────────

Q10: What is the most common mistake in the algorithm?

A: Forgetting the root special case! Many people only check
   low[child] >= disc[parent] and miss that the root needs
   child counting.

   Visual:

        0 (root)
       / \
      1   2

   Remove 0: 1 and 2 are disconnected!
   But parent == -1, so the normal check is skipped.
   Without counting children, 0 is MISSED!

   Always add:
   if (parent == -1 && children >= 2) artpoint[node] = true;

──────────────────────────────────────────────────────────────

Q11: How does the algorithm handle disconnected graphs?

A: We run DFS from every unvisited vertex. Each disconnected
   piece is processed separately, and articulation points in
   ALL pieces are found.

   Visual:

   Piece 1: 0 ─── 1 ─── 2
   Piece 2: 3 ─── 4 ─── 5

   DFS from 0 finds APs in Piece 1.
   Then DFS from 3 finds APs in Piece 2.
   Result combines both!

──────────────────────────────────────────────────────────────

Q12: Can we use BFS instead of DFS?

A: No! BFS does not build a clear parent-child tree, so it is
   very hard to track low[] values and back edges. DFS naturally
   creates a tree structure that makes Tarjan's algorithm work.

   Think of DFS as exploring one tunnel all the way to the end
   before trying another. BFS explores all tunnels one step at a
   time — it does not reveal the tree structure we need.

──────────────────────────────────────────────────────────────

Q13: What happens if the graph has self-loops?

A: A self-loop (a road from a city to itself) does not affect
   articulation points. Removing the city removes the self-loop
   anyway, and the self-loop does not connect to any other city.

   Visual:

   0 ───┐
        │ (self loop)
   0 ───┘

   The self-loop is irrelevant for connectivity.

──────────────────────────────────────────────────────────────

Q14: What happens if the graph has parallel edges?

A: Parallel edges between two cities mean neither city is an
   articulation point JUST because of those edges. But the cities
   might still be APs for other reasons.

   Visual:

   0 ═════ 1   (two parallel edges)
   │
   2

   0 and 1 have two direct connections, so neither is an AP
   because of each other. But 0 might still be an AP because
   of vertex 2!

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

   Mistake 1: Forgetting root child count
   → Fix: if (parent == -1 && child > 1) mark AP

   Mistake 2: Not handling disconnected graphs
   → Fix: Loop through all vertices, dfs if unvisited

   Mistake 3: Using low[] instead of disc[] for back edges
   → Fix: low[node] = min(low[node], disc[neigh]);

   Mistake 4: Wrong condition direction
   → Fix: if (low[neigh] >= disc[node])

   Mistake 5: Confusing APs with bridges
   → Fix: AP = vertex, Bridge = edge

   Mistake 6: Counting all neighbors as root children
   → Fix: Only count unvisited neighbors that trigger DFS
```

## Google-Style Follow-Up Questions

```
Q16: How would you find articulation points in a dynamic graph
     where edges are added and removed over time?

A: Imagine a road network that changes every week. When a new road
   opens, it might create a bypass that turns a critical city into
   a safe city. When a road closes, a safe city might become critical.

   A practical system tracks the DFS tree and updates only the
   cities near the changed road. For big changes, it recomputes
   everything from scratch.

   Visual:

   Before: 0 ─── 1 ─── 2
                  ↑
                  1 is AP

   Add road 0──2:
   Before: 0 ─── 1 ─── 2
           └─────────┘
   Now 1 is NO LONGER an AP! ✓

──────────────────────────────────────────────────────────────

Q17: Can you find both bridges and articulation points in a
     single DFS pass?

A: Yes! The conditions are almost the same:

   Bridge:  low[child] >  disc[parent]  (strictly greater)
   AP:      low[child] >= disc[parent]  (greater or equal)

   For the root: count children >= 2 for AP.
   For bridges: no special root rule.

   Visual:

   During one walk through the map:
     Mark critical roads (bridges) as you go.
     Mark critical cities (APs) as you go.

   Both answers come from the same DFS!

──────────────────────────────────────────────────────────────

Q18: What is the minimum number of vertices to remove to
     completely disconnect a graph?

A: This is the "vertex connectivity" problem. It asks: how many
   cities must close before the map breaks into pieces?

   For a tree: removing the center might disconnect everything.
   For a cycle: you must remove at least 2 cities.
   For a complete graph: you must remove all but one city!

   Visual:

   Cycle: 0 ─── 1 ─── 2 ─── 3 ─── 0
   Remove 0 and 2: 1 and 3 are disconnected.
   Minimum = 2 for this cycle.

   This is related to Menger's Theorem — a deep topic!

──────────────────────────────────────────────────────────────

Q19: How would you find articulation points in a directed graph?

A: In directed graphs (one-way roads), the idea is similar but
   harder. We look for "strong articulation points" whose removal
   breaks strongly connected components (SCCs).

   Visual:

   One-way ring: 0 → 1 → 2 → 0
   Remove 0: 1 and 2 can still reach each other? No, because
   roads are one-way! This is much more complex.

   Algorithm: Use SCC decomposition first, then analyze the
   component graph.

──────────────────────────────────────────────────────────────

Q20: What is the relationship between articulation points and
     biconnected components?

A: A biconnected component is a group of cities with NO articulation
   points inside it. If you remove all articulation points, each
   remaining connected piece is a biconnected component.

   Visual:

   [Component A] ── City X ── [Component B] ── City Y ── [Component C]

   City X and City Y are articulation points.
   A, B, and C are biconnected components.
   Inside each component, there are at least two routes between
   any two cities.

──────────────────────────────────────────────────────────────

Q21: How would you handle a graph with 10^6 vertices and edges?

A: Tarjan's algorithm is O(V + E), so it handles millions easily!
   The only concern is recursion depth — 10^6 recursive calls might
   crash the stack. Use an iterative DFS with an explicit stack
   instead of recursion.

   Visual:

   Recursive DFS: stack managed by computer (limited size)
   Iterative DFS: stack managed by you (unlimited size)

   For 10^6 vertices, iterative is the safe choice!

──────────────────────────────────────────────────────────────

Q22: Can you find articulation points without using recursion?

A: Yes! Use an explicit stack to simulate DFS. Keep track of
   parent, children, and which neighbors have been processed.
   After processing a child, update low[] just like in recursive DFS.

   Visual:

   Stack entry: [city, parent, next_neighbor_index, children_count]

   This is more complex to code but avoids stack overflow for
   very large graphs.

──────────────────────────────────────────────────────────────

Q23: What if the graph has multi-edges (parallel edges)?

A: Parallel edges between two cities mean neither city is an AP
   because of those specific edges — if one road closes, the other
   still connects them. However, the cities might still be APs for
   other reasons (connecting to other parts of the graph).

   Visual:

   0 ═════ 1   (two roads)
   │
   2

   0 and 1 are safe from each other, but 0 is still an AP
   because of vertex 2!

──────────────────────────────────────────────────────────────

Q24: How would you find the "most critical" articulation point?

A: The most critical city is the one whose removal breaks the graph
   into the MOST pieces. During DFS, count how many separate regions
   each city connects. The city with the highest count is most critical.

   Visual:

        0
       /|\
      1 2 3
      | | |
      4 5 6

   Remove 0: three separate pieces (1-4, 2-5, 3-6).
   Remove 1: one piece broken off (4).
   City 0 is most critical!

──────────────────────────────────────────────────────────────

Q25: Design a system to monitor critical cities in a real-time
     road network (like Google Maps).

A: Imagine a live map that updates as roads open and close.

   1. Precompute articulation points using Tarjan's.
   2. When a road opens: check if it creates a bypass for any AP.
      Only recompute for cities near the new road.
   3. When a road closes: check if any safe city becomes critical.
      Only recompute for cities near the closed road.
   4. For major changes, run Tarjan's on the affected region.

   Visual:

   Before: 0 ─── 1 ─── 2
                  ↑
                  AP

   Open road 0──2:
   Now 0 and 2 have a bypass!
   City 1 is NO LONGER an AP.
   Only cities 0, 1, 2 need rechecking.
```
