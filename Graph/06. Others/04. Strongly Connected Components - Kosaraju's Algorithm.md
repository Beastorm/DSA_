## 1. Problem Statement

Given a **Directed Graph** with `V` vertices and `E` edges, find all **Strongly Connected Components (SCCs)**.

A **Strongly Connected Component** is a maximal group of vertices where **every vertex can reach every other vertex** in that group.

In simple words: if you pick any two cities in the same SCC, you can drive from one to the other following the one-way roads.

## Key Insight

```
Observation 1: What is a Strongly Connected Component?

In a directed graph, an SCC is a group of cities where you can
travel from ANY city to ANY OTHER city in the group.

Visual (one SCC):

    ┌─────────────────┐
    │  0 → 1 → 2 → 0  │
    └─────────────────┘

    0 reaches 1, 2
    1 reaches 2, 0
    2 reaches 0, 1
    Everyone reaches everyone! ✓

Visual (NOT one SCC):

    0 ───→ 1 ───→ 2

    0 reaches 1 and 2.
    1 reaches 2.
    2 reaches NO ONE.
    Not strongly connected! ✗

──────────────────────────────────────────────────────────────

Observation 2: Condensation Graph

If we shrink each SCC into a single "super city," the result is
a DAG (Directed Acyclic Graph) called the condensation graph.

Visual:

    Original:
      0 → 1 → 2 → 0  (SCC A)
      2 → 3
      3 → 4 → 5 → 3  (SCC B)

    Condensation:
      ┌───┐
      │ A │ ───→ ┌───┐
      └───┘      │ B │
                 └───┘

    No cycles between super cities! This is always a DAG.

──────────────────────────────────────────────────────────────

Observation 3: Kosaraju's Key Trick

If you reverse ALL roads and start DFS from the city that finished
LAST in the original graph, you will explore exactly one SCC!

Why? The last city to finish in original DFS is in a "sink" SCC
(an SCC with no outgoing roads to other SCCs). When you reverse
roads, that sink SCC becomes a "source" SCC — easy to reach!

Visual:

    Original roads:
      A ───→ B ───→ C

    SCCs: {A}, {B}, {C}
    C finishes last (deepest in DFS).

    Reverse roads:
      A ←─── B ←─── C

    Start from C: DFS reaches only C (its SCC)!
    Then B: reaches only B.
    Then A: reaches only A.

──────────────────────────────────────────────────────────────

Observation 4: Tarjan's Key Trick

During one DFS, if a city's discovery time equals its lowest
reachable time (disc == low), that city is the ROOT of an SCC.
All cities above it in the stack belong to that SCC!

Why? disc == low means: "From this city and everything below it,
we cannot reach any city discovered EARLIER." So this city is
the earliest entry point to its SCC.

Visual:

    DFS stack:
      Bottom: [0, 1, 2, 3] :Top

    At city 2: disc[2] == low[2]
    → 2 is the root of an SCC!
    → Pop until 2: SCC = {3, 2}

    Stack now: [0, 1]

    At city 0: disc[0] == low[0]
    → 0 is the root of an SCC!
    → Pop until 0: SCC = {1, 0}
```

## Intuition

```
Imagine a city with one-way streets only.

Some blocks form a "roundabout" — you can drive from any house
to any other house in that block. That block is ONE SCC.

Some houses sit on a one-way street that never loops back. Each
such house is its own tiny SCC.

──────────────────────────────────────────────────────────────

The Roundabout Test

Pick any two houses in the same SCC. Can you drive from House A
to House B AND from House B to House A? If YES, they are in the
same SCC. If NO, they are in different SCCs.

Visual:

  Roundabout (ONE SCC):

      0 ──→ 1
      ↑     │
      └──── 2

      0 reaches 1 and 2. ✓
      1 reaches 2 and 0. ✓
      2 reaches 0 and 1. ✓
      Everyone reaches everyone!

  One-way street (THREE SCCs):

      0 ──→ 1 ──→ 2

      0 reaches 1 and 2. ✓
      1 reaches 2.       ✓
      2 reaches NO ONE.  ✗
      Three separate SCCs: {0}, {1}, {2}

──────────────────────────────────────────────────────────────

Kosaraju's Trick: Walk Forward, Then Walk Backward

Think of a postman delivering letters on one-way streets.

Step 1 — Walk Forward:
  The postman walks every street once, writing down the order he
  FINISHES each house. Houses at dead ends get written first.
  Houses deep inside roundabouts get written last.

Step 2 — Flip All Street Signs:
  Every one-way street is reversed! Now the postman walks the
  exact same streets but in the opposite direction.

Step 3 — Walk Backward from the Last House:
  The postman starts from the house he finished LAST in Step 1.
  Because the streets are flipped, he can only wander inside ONE
  roundabout. He cannot accidentally cross into another!

  He marks all houses he visits as ONE SCC.
  Then he picks the next unvisited house from his list and repeats.

Why does this work?

The house finished LAST sits in a roundabout with NO exits.
When you flip the streets, that roundabout becomes the easiest
to find — all roads now point INTO it, not out of it!

Visual:

  Original streets:
    Block A ──→ Block B ──→ Block C

  Postman finishes C last (no exit from C).

  Flipped streets:
    Block A ←── Block B ←── Block C

  Start from C: can only reach C!   → SCC = {C}
  Then B: can only reach B!         → SCC = {B}
  Then A: can only reach A!         → SCC = {A}

──────────────────────────────────────────────────────────────

Tarjan's Trick: One Walk with a Stack of Plates

Tarjan does everything in ONE walk. No need to flip streets!

Imagine the explorer carries a stack of plates. Every time he
enters a new house, he puts a plate on the stack with the house
number on it.

As he walks, he secretly asks: "What is the EARLIEST house I can
reach from here, using back alleys?"

When he realizes he CANNOT reach any house EARLIER than the one
he is standing in, he shouts: "I am the bottom plate of this
roundabout!" He then pops every plate above his — all those
houses belong to the SAME SCC.

Visual:

  Explorer enters houses: 0, then 1, then 2, then 3

  Stack of plates: [0, 1, 2, 3]
                    ↑ bottom    ↑ top

  At house 3, he finds a secret alley back to house 1.
  → "From 3, I can reach 1."

  At house 2, he learns: "From 2, I can reach 1 (through 3)."

  At house 1, he learns: "From 1, I can reach... myself."
  → "I cannot reach anything earlier than me!"
  → POP! Houses 3, 2, and 1 form one SCC.

  Stack now: [0]

  At house 0, he learns: "From 0, I can reach... myself."
  → "I cannot reach anything earlier than me!"
  → POP! House 0 forms its own SCC.

The Bottom-Plate Rule:

  When a house cannot reach any house discovered BEFORE it,
  that house is the "bottom plate" of a new SCC.
  Every plate above it is in the same group!
```

## Algorithm

### Kosaraju's Algorithm

```
Step 1: DFS on original graph to get finishing order

For each unvisited vertex:
  DFS(node):
    mark visited
    for each neighbor:
      if not visited:
        DFS(neighbor)
    push node onto stack  ← finish order

Visual:

  Graph: 0 → 1 → 2 → 0, 1 → 3 → 4

  DFS(0):
    visit 0
    → DFS(1):
      visit 1
      → DFS(2):
        visit 2
        → DFS(0): visited, return
        push 2
      → DFS(3):
        visit 3
        → DFS(4):
          visit 4
          push 4
        push 3
      push 1
    push 0

  Stack (top to bottom): 0, 1, 3, 4, 2

──────────────────────────────────────────────────────────────

Step 2: Build transposed graph (reverse all edges)

For each edge (u → v) in original:
  add edge (v → u) in transpose

Visual:

  Original:  0 → 1,  1 → 2,  2 → 0,  1 → 3,  3 → 4
  Transpose: 1 → 0,  2 → 1,  0 → 2,  3 → 1,  4 → 3

──────────────────────────────────────────────────────────────

Step 3: DFS on transposed graph in stack order

While stack not empty:
  node = stack.pop()
  if node not visited:
    sccCount++
    DFS on transpose from node
    (all reachable vertices form one SCC)

Visual:

  Pop 0: unvisited → DFS on transpose from 0
    0 → 2 → 1 (all reachable)
    SCC 1 = {0, 1, 2}

  Pop 1: visited, skip

  Pop 3: unvisited → DFS on transpose from 3
    3 → 1 (visited), so only 3
    SCC 2 = {3}

  Pop 4: unvisited → DFS on transpose from 4
    4 → 3 (visited), so only 4
    SCC 3 = {4}

  Total: 3 SCCs
```

### Tarjan's Algorithm

```
Step 1: Initialize

disc[V] = {-1, -1, ...}
low[V] = {-1, -1, ...}
visited[V] = {false, ...}
inStack[V] = {false, ...}
stack = empty
timer = 0

──────────────────────────────────────────────────────────────

Step 2: DFS from every unvisited vertex

For i = 0 to V-1:
  If not visited[i]:
    DFS(i)

──────────────────────────────────────────────────────────────

Step 3: DFS function

Function DFS(node):
  visited[node] = true
  disc[node] = low[node] = timer++
  push node onto stack
  inStack[node] = true

  For each neighbor of node:
    If neighbor not visited:
      DFS(neighbor)
      low[node] = min(low[node], low[neighbor])

    Else if neighbor is inStack:
      low[node] = min(low[node], disc[neighbor])

  If disc[node] == low[node]:
    // node is the root of an SCC!
    pop from stack until node
    all popped vertices form one SCC

Visual:

  DFS(0): disc=0, low=0, push 0
    neighbor 1: unvisited → DFS(1)
      disc=1, low=1, push 1
      neighbor 2: unvisited → DFS(2)
        disc=2, low=2, push 2
        neighbor 0: visited, inStack → low[2] = min(2, 0) = 0
        disc[2]=2 != low[2]=0 → no SCC yet
      low[1] = min(1, 0) = 0
      neighbor 3: unvisited → DFS(3)
        disc=3, low=3, push 3
        neighbor 4: unvisited → DFS(4)
          disc=4, low=4, push 4
          no neighbors
          disc[4]==low[4] → SCC = {4}
        low[3] = min(3, 4) = 3
        no more neighbors
        disc[3]==low[3] → SCC = {3}
      low[1] = min(0, 3) = 0
      no more neighbors
      disc[1]=1 != low[1]=0 → no SCC yet
    low[0] = min(0, 0) = 0
    no more neighbors
    disc[0]==low[0] → SCC = {0, 1, 2}
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

// ==================== KOSARAJU'S ALGORITHM ====================

class Kosaraju {
private:
    void dfs(int node, vector<int>& vis, vector<int> adj[], stack<int>& st) {
        vis[node] = 1;
        for (int neighbor : adj[node]) {
            if (!vis[neighbor]) {
                dfs(neighbor, vis, adj, st);
            }
        }
        st.push(node);
    }

    void dfsTranspose(int node, vector<int>& vis, vector<int> adjT[]) {
        vis[node] = 1;
        for (int neighbor : adjT[node]) {
            if (!vis[neighbor]) {
                dfsTranspose(neighbor, vis, adjT);
            }
        }
    }

public:
    int kosaraju(int V, vector<int> adj[]) {
        vector<int> vis(V, 0);
        stack<int> st;

        // Step 1: DFS on original graph to get finishing order
        for (int i = 0; i < V; i++) {
            if (!vis[i]) {
                dfs(i, vis, adj, st);
            }
        }

        // Step 2: Build transposed graph
        vector<int> adjT[V];
        for (int u = 0; u < V; u++) {
            for (int v : adj[u]) {
                adjT[v].push_back(u);
            }
        }

        // Step 3: DFS on transposed graph in reverse finish order
        fill(vis.begin(), vis.end(), 0);
        int sccCount = 0;

        while (!st.empty()) {
            int node = st.top();
            st.pop();

            if (!vis[node]) {
                sccCount++;
                dfsTranspose(node, vis, adjT);
            }
        }

        return sccCount;
    }
};

// ==================== TARJAN'S ALGORITHM ====================

void DFS(int node, vector<int> adj[], vector<bool>& visited, vector<int>& disc,
         vector<int>& low, stack<int>& st, vector<bool>& inStack,
         vector<vector<int>>& ans, int& timer) {

    visited[node] = true;
    disc[node] = low[node] = timer++;

    st.push(node);
    inStack[node] = true;

    for (int j = 0; j < adj[node].size(); j++) {
        int neighbor = adj[node][j];

        if (!visited[neighbor]) {
            DFS(neighbor, adj, visited, disc, low, st, inStack, ans, timer);
            low[node] = min(low[node], low[neighbor]);
        }
        else if (inStack[neighbor]) {
            low[node] = min(low[node], disc[neighbor]);
        }
    }

    if (disc[node] == low[node]) {
        vector<int> temp;

        while (st.top() != node) {
            temp.push_back(st.top());
            inStack[st.top()] = false;
            st.pop();
        }

        temp.push_back(st.top());
        inStack[st.top()] = false;
        st.pop();

        sort(temp.begin(), temp.end());
        ans.push_back(temp);
    }
}

vector<vector<int>> tarjans(int v, vector<int> adj[]) {
    vector<vector<int>> ans;
    vector<int> disc(v, -1), low(v, -1);
    vector<bool> visited(v, false), inStack(v, false);
    stack<int> st;
    int timer = 0;

    for (int i = 0; i < v; i++) {
        if (!visited[i]) {
            DFS(i, adj, visited, disc, low, st, inStack, ans, timer);
        }
    }

    sort(ans.begin(), ans.end());
    return ans;
}

// ==================== DRIVER CODE ====================
int main() {
    int V = 5;
    vector<int> adj[V];

    adj[0].push_back(1);
    adj[1].push_back(2);
    adj[2].push_back(0);
    adj[1].push_back(3);
    adj[3].push_back(4);

    // Kosaraju
    Kosaraju obj;
    int sccCount = obj.kosaraju(V, adj);
    cout << "Number of SCCs (Kosaraju): " << sccCount << endl;

    // Tarjan
    vector<vector<int>> result = tarjans(V, adj);
    cout << "\nStrongly Connected Components (Tarjan):" << endl;
    for (auto& comp : result) {
        for (int node : comp) cout << node << " ";
        cout << endl;
    }

    return 0;
}
```

## Step-by-Step Trace

### Kosaraju's Trace

```
Graph: 0 → 1 → 2 → 0,  1 → 3 → 4

Adjacency list:
  adj[0] = [1]
  adj[1] = [2, 3]
  adj[2] = [0]
  adj[3] = [4]
  adj[4] = []

── Step 1: DFS on original graph ──

vis: [0, 0, 0, 0, 0]

DFS(0):
  vis[0] = 1
  neighbor 1: unvisited → DFS(1)
    vis[1] = 1
    neighbor 2: unvisited → DFS(2)
      vis[2] = 1
      neighbor 0: visited → return
      push 2 → stack: [2]
    neighbor 3: unvisited → DFS(3)
      vis[3] = 1
      neighbor 4: unvisited → DFS(4)
        vis[4] = 1
        no neighbors → return
        push 4 → stack: [2, 4]
      push 3 → stack: [2, 4, 3]
    push 1 → stack: [2, 4, 3, 1]
  push 0 → stack: [2, 4, 3, 1, 0]

── Step 2: Build transpose ──

Original edges: 0→1, 1→2, 2→0, 1→3, 3→4

adjT[0] = [2]
adjT[1] = [0]
adjT[2] = [1]
adjT[3] = [1]
adjT[4] = [3]

── Step 3: DFS on transpose ──

vis: [0, 0, 0, 0, 0]
sccCount = 0

Pop 0: unvisited!
  sccCount = 1
  DFS(0) on transpose:
    vis[0] = 1
    neighbor 2: unvisited → DFS(2)
      vis[2] = 1
      neighbor 1: unvisited → DFS(1)
        vis[1] = 1
        neighbor 0: visited → return

    return from 2
  return from 0
  SCC 1 = {0, 1, 2} ✓

    neighbor 1: visited → return

  DFS(4) on transpose:
    vis[4] = 1
    neighbor 3: visited → return
  return from 4
  SCC 3 = {4} ✓

Result: 3 SCCs
```

```
        inStack[4] = 1

        disc[4] == low[4]? 4 == 4? YES! ✓

          temp = [4]
          inStack[4] = 0, pop 4

        low[3] = min(low[3], low[4]) = min(3, 4) = 3

          temp = [3]

    Back to DFS(1):
      low[1] = min(low[1], low[3]) = min(0, 3) = 0

      no more neighbors

      disc[1] == low[1]? 1 == 0? NO → no SCC yet.

    Return from DFS(1).

  Back to DFS(0):
    low[0] = min(low[0], low[1]) = min(0, 0) = 0

    disc[0] == low[0]? 0 == 0? YES! ✓

      temp = [2, 1, 0]

      inStack[0] = 0, pop 0

── Final Result ──

disc:  [0, 1, 2, 3, 4]
low:   [0, 0, 0, 3, 4]
```

```
                4(d=4,l=4) ← SCC root!

SCC pops:
  At 4: pop {4}
  At 3: pop {3}
  At 0: pop {0, 1, 2}

─────────────────────────────────────

Condensation DAG:

    ┌─────────┐
    │ 0, 1, 2 │ ───→ ┌───┐ ───→ ┌───┐
    └─────────┘      │ 3 │      │ 4 │
                     └───┘      └───┘

No cycles between SCCs! Always a DAG.

─────────────────────────────────────

Single Big SCC (cycle):

    0 ───→ 1
    ↑      │
    │      ↓
    3 ←─── 2

Tarjan:
  DFS(0)→1→2→3→0 (back edge)
  low[3]=0, low[2]=0, low[1]=0, low[0]=0
  At 0: disc=0, low=0 → pop ALL: {0,1,2,3}

One SCC containing everyone!

─────────────────────────────────────

No cycles (each vertex is its own SCC):

    0 ───→ 1 ───→ 2
           │       │
           ↓       ↓
           3 ───→ 4

Tarjan:
  At 2: pop {2}
  At 4: pop {4}
  At 3: pop {3}
  At 1: pop {1}
  At 0: pop {0}

Five SCCs, each with one vertex.
```

### Common Mistakes

```
❌ Mistake 1: Forgetting to reverse ALL edges in Kosaraju's transpose
  Only reversing some edges or missing edges.

  Visual:
    Original: 0 → 1, 1 → 2
    Wrong transpose: 1 → 0 only (forgot 2 → 1)
    Correct transpose: 1 → 0, 2 → 1

✓ Correct: For every edge u→v, add v→u in transpose.

──────────────────────────────────────────────────────────────

❌ Mistake 2: Using low[neighbor] instead of disc[neighbor] for
  back edges in Tarjan's algorithm

  low[node] = min(low[node], low[neighbor]);  // WRONG!

  For a back edge to a node in the stack, we use disc[neighbor],
  not low[neighbor]. The back edge connects directly to neighbor.

✓ Correct:
  else if (inStack[neighbor]) {
      low[node] = min(low[node], disc[neighbor]);
  }

──────────────────────────────────────────────────────────────

❌ Mistake 3: Not checking inStack in Tarjan's back edge condition
  if (visited[neighbor]) {  // WRONG!
      low[node] = min(low[node], disc[neighbor]);
  }

  A visited neighbor that is NOT in the stack belongs to a
  different SCC already processed. Do not use it!

✓ Correct:
  else if (inStack[neighbor]) {
      low[node] = min(low[node], disc[neighbor]);
  }

──────────────────────────────────────────────────────────────

❌ Mistake 4: Forgetting to mark inStack = false when popping SCC
  After popping nodes from the stack, they must be marked as
  no longer in the stack. Otherwise, future back edges might
  incorrectly reference them.

✓ Correct:
  inStack[st.top()] = false;
  st.pop();

──────────────────────────────────────────────────────────────

❌ Mistake 5: Not handling disconnected graphs
  Only calling DFS(0) or kosaraju from vertex 0.
  Other components are never visited!

✓ Correct:
  for (int i = 0; i < V; i++)
      if (!visited[i])
          DFS(i, ...);

──────────────────────────────────────────────────────────────

❌ Mistake 6: Confusing SCCs with connected components
  In undirected graphs, connected components are simple.
  In directed graphs, SCCs require TWO-WAY reachability!

  Visual:
    0 ───→ 1

    Connected component (undirected): {0, 1}
    Strongly connected component (directed): {0}, {1}

✓ Correct: SCC requires paths in BOTH directions!
```

## Complexity Analysis

**Kosaraju's Algorithm:**

**Time Complexity:** `O(V + E)`

- First DFS: O(V + E)
- Building transpose: O(V + E)
- Second DFS: O(V + E)
- Total: O(V + E)

**Space Complexity:** `O(V + E)`

- Original adjacency list: O(V + E)
- Transposed adjacency list: O(V + E)
- Visited arrays: O(V)
- Stack: O(V)

**Tarjan's Algorithm:**

**Time Complexity:** `O(V + E)`

- One DFS visits every vertex and edge once
- Stack operations are O(1)
- Total: O(V + E)

**Space Complexity:** `O(V + E)`

- Adjacency list: O(V + E)
- disc[], low[], visited[], inStack[]: O(V)
- Stack: O(V)
- SCC list: O(V)

```
Visual comparison:

Kosaraju:
  2 DFS passes + transpose build
  = 3 × O(V + E) ≈ O(V + E)

Tarjan:
  1 DFS pass
  = O(V + E)

Both are linear time — very fast!
```

## Interview Questions

```
Q1: What is a Strongly Connected Component?

A: An SCC is like a group of cities connected by a network of
   one-way roads where you can drive from ANY city to ANY OTHER
   city in the group.

   Visual:

   SCC:
     ┌─────────────────┐
     │  0 → 1 → 2 → 0  │
     └─────────────────┘
     From 0: can reach 1 and 2.
     From 1: can reach 2 and 0.
     From 2: can reach 0 and 1.
     Everyone reaches everyone! ✓

   NOT an SCC:
     0 ───→ 1 ───→ 2
     0 reaches 1 and 2, but 2 cannot reach anyone.
     Not strongly connected! ✗

──────────────────────────────────────────────────────────────

Q2: What is the difference between a connected component and an SCC?

A: Connected components are for undirected graphs (roads go both ways).
   SCCs are for directed graphs (roads are one-way).

   Visual:

   Undirected: 0 ─── 1
   Connected component: {0, 1} ✓

   Directed: 0 ───→ 1
   SCCs: {0}, {1} (0 cannot be reached from 1) ✓

   In an SCC, you need a path in BOTH directions between every pair!

──────────────────────────────────────────────────────────────

Q3: Why does Kosaraju's algorithm need two DFS passes?

A: The first pass records the order in which cities are fully explored.
   The city that finishes LAST is in a "sink" SCC (no roads leaving it).
   The second pass uses the REVERSE map and starts from that last city.
   Because the map is reversed, the sink SCC becomes a "source" —
   an explorer starting there cannot wander into other SCCs!

   Visual:

   Original:
     A ───→ B ───→ C
     C finishes last (sink).

   Reverse:
     A ←─── B ←─── C
     Start from C: can only reach C's SCC! ✓

──────────────────────────────────────────────────────────────

Q4: What is the key idea behind Tarjan's algorithm?

A: Tarjan's uses a single DFS with a stack. When a city's discovery
time equals its lowest reachable time (disc == low), that city is the
"leader" or root of an SCC. All cities above it on the stack belong
to the same SCC.

   Visual:

   Stack: [0, 1, 2, 3]

   At city 2: disc[2] == low[2]
   → "I am the leader!"
   → Pop until 2: SCC = {3, 2}

   Stack: [0, 1]

   disc == low means: "I cannot reach any city discovered BEFORE me."
   So I am the earliest entry point to my SCC!

──────────────────────────────────────────────────────────────

Q5: What is a condensation graph?

A: If you shrink each SCC into a single "super city," the result is
   a DAG called the condensation graph. There are no cycles between
   super cities!

   Visual:

   Original:
     0→1→2→0 (SCC A), 2→3, 3→4→5→3 (SCC B)

   Condensation:
     ┌───┐ ───→ ┌───┐
     │ A │      │ B │
     └───┘      └───┘

   This DAG is useful for solving problems on directed graphs!

──────────────────────────────────────────────────────────────

Q6: When should you use Kosaraju vs Tarjan?

A: Kosaraju is easier to understand and code. It is great for
   interviews because the logic is clear. Tarjan is slightly more
   efficient (one pass instead of two) but has trickier stack logic.

   Visual:

   Kosaraju: "Explore, flip the map, explore again." (Simple!)
   Tarjan: "Explore and pop SCCs on the fly." (Slightly complex!)

   Both run in O(V + E) time. For most problems, either works fine.

──────────────────────────────────────────────────────────────

Q7: Can a single vertex be an SCC?

A: Yes! If a vertex has no self-loop and no cycle back to itself,
   it forms an SCC of size 1.

   Visual:

   0 ───→ 1 ───→ 2

   SCCs: {0}, {1}, {2}
   Each vertex is alone in its SCC.

──────────────────────────────────────────────────────────────

Q8: Can the entire graph be one SCC?

A: Yes! If the graph is a single big cycle (or has cycles connecting
   everything), the entire graph is one SCC.

   Visual:

   0 ───→ 1
   ↑      │
   │      ↓
   3 ←─── 2

   Everyone reaches everyone!
   One SCC = {0, 1, 2, 3}

──────────────────────────────────────────────────────────────

Q9: What are real-world uses of SCCs?

A: Real-world examples:

   Web page navigation:
   → Find groups of pages where you can click from any page to any other
   → These are "strongly connected" parts of a website

   Social networks:
   → Find tight-knit communities where everyone follows everyone back
   → Unlike one-way celebrity followings

   Compiler design:
   → Find loops in code (control flow graphs)
   → Optimize by analyzing strongly connected basic blocks

   Road networks:
   → Find city districts with two-way access to each other
   → Plan emergency routes that work in both directions

──────────────────────────────────────────────────────────────

Q10: What is the most common mistake in Tarjan's algorithm?

A: Forgetting to check inStack when processing back edges!

   Visual:

   Stack: [0, 1, 2]
   Node 3 was already popped (belongs to another SCC).

   If we see edge 2 → 3 and use disc[3] to update low[2]:
   low[2] becomes smaller, but 3 is NOT in the same SCC!
   This gives WRONG results!

   Always check: if (inStack[neighbor]) before using disc[neighbor].

──────────────────────────────────────────────────────────────

Q11: How does Kosaraju's finishing order work?

A: Think of DFS as exploring a maze. When you back out of a room
   for the last time, you write its name on a list. Rooms in dead
   ends are written first. Rooms at the entrance are written last.

   In SCC terms: the last room written is in a sink SCC — a group
   with no exits to other groups. When you reverse the maze, this
   group becomes the easiest to isolate.

──────────────────────────────────────────────────────────────

Q12: Can we find SCCs using BFS?

A: Not directly. BFS does not naturally reveal the tree structure
   and back edges needed for SCC detection. DFS is the standard
   approach for both Kosaraju's and Tarjan's algorithms.

   Think of BFS as exploring all tunnels one step at a time.
   DFS explores one tunnel all the way to the end before trying
   another. This deep exploration is what reveals cycles and SCCs.

──────────────────────────────────────────────────────────────

Q13: What happens with self-loops in SCC detection?

A: A self-loop (0 → 0) means vertex 0 can reach itself. But it does
   not automatically make 0 part of a larger SCC. It just means
   {0} is a valid SCC by itself.

   Visual:

   0 ──→ 0 (self loop)
   0 ──→ 1

   SCCs: {0}, {1}
   The self-loop does not connect 0 and 1.

──────────────────────────────────────────────────────────────

Q14: What is the difference between a bridge and an SCC?

A: A bridge is a critical edge in an UNDIRECTED graph.
   An SCC is a group of vertices in a DIRECTED graph.

   They solve completely different problems!

   Visual:

   Bridge (undirected):
     0 ─── 1 ─── 2
     Edge (1,2) is a bridge.

   SCC (directed):
     0 → 1 → 2 → 0
     {0, 1, 2} is an SCC.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

   Mistake 1: Not reversing all edges in Kosaraju's transpose
   → Fix: For every u→v, add v→u

   Mistake 2: Using low[] instead of disc[] for back edges in Tarjan's
   → Fix: low[node] = min(low[node], disc[neighbor])

   Mistake 3: Not checking inStack in Tarjan's back edge condition
   → Fix: Only update if neighbor is currently in stack

   Mistake 4: Forgetting to mark inStack = false when popping
   → Fix: Mark false before every pop

   Mistake 5: Not handling disconnected graphs
   → Fix: Loop through all vertices

   Mistake 6: Confusing SCCs with connected components
   → Fix: SCC = two-way reachability in directed graphs
```

## Google-Style Follow-Up Questions

```
Q16: How would you find the size of the largest SCC in a graph?

A: Both Kosaraju's and Tarjan's algorithms naturally group vertices
   into SCCs. Simply count the vertices in each SCC and track the
   maximum!

   Visual:

   SCCs: {0,1,2} (size 3), {3} (size 1), {4,5} (size 2)
   Largest SCC size = 3

   In Kosaraju's: count nodes visited in each transpose DFS.
   In Tarjan's: count nodes popped in each SCC extraction.

──────────────────────────────────────────────────────────────

Q17: Can you find SCCs in a dynamic graph where edges are added
     and removed over time?

A: Imagine a road network that changes every day. Adding or removing
   one road can split an SCC or merge two SCCs together.

   A practical system:
   1. Maintain the current SCC decomposition.
   2. When a new road is added: check if it creates a cycle between
      two previously separate SCCs. If yes, merge them!
   3. When a road is removed: check if it breaks a cycle. If yes,
      recompute SCCs for the affected region.
   4. For major changes, rerun Kosaraju's or Tarjan's from scratch.

   Visual:

   Before: {0,1} and {2,3} (separate SCCs)
   Add road 1→2 and 3→0:
   Now: {0,1,2,3} (one big SCC!)

──────────────────────────────────────────────────────────────

Q18: What is the minimum number of edges to add to make the entire
     graph one SCC?

A: Look at the condensation DAG. If it has K SCCs, you need at least
   K edges to connect them into one cycle. Connect the sinks to the
   sources in a circular pattern!

   Visual:

   Condensation:
     A → B → C

   Add edge C → A:
     A → B → C → A
   Now everyone reaches everyone! Only 1 edge needed.

   For K SCCs: add K edges (source of one to sink of next, circularly).

──────────────────────────────────────────────────────────────

Q19: How would you find SCCs in a graph with 10^6 vertices?

A: Both algorithms are O(V + E), so they handle millions easily!
   The concern is recursion depth. For 10^6 vertices, recursive DFS
   might crash the stack. Use iterative DFS with an explicit stack.

   Visual:

   Recursive DFS: computer manages stack (limited size)
   Iterative DFS: you manage stack (can handle millions)

   Also, use adjacency lists (not matrices) to save memory.

──────────────────────────────────────────────────────────────

Q20: What is the relationship between SCCs and topological sorting?

A: The condensation graph (SCCs shrunk to super-cities) is always a
   DAG. You can topologically sort this DAG! This is very useful for
   solving problems on directed graphs.

   Visual:

   Original:
     0→1→2→0, 2→3, 3→4→5→3

   Condensation:
     SCC_A → SCC_B

   Topological order: [SCC_A, SCC_B]

   This lets you process SCCs in dependency order!

──────────────────────────────────────────────────────────────

Q21: Can you find SCCs using Union-Find (Disjoint Set)?

A: Union-Find works for undirected graphs but NOT directed graphs!
   In directed graphs, reachability is not symmetric — A can reach B
   but B might not reach A. Union-Find assumes symmetry.

   Visual:

   0 → 1
   Union-Find would merge 0 and 1.
   But they are NOT in the same SCC (1 cannot reach 0)!

   For SCCs, stick to Kosaraju's or Tarjan's.

──────────────────────────────────────────────────────────────

Q22: How would you find the "most central" SCC in a graph?

A: The most central SCC is the one that reaches the most other SCCs
   and is reached by the most other SCCs. In the condensation DAG,
   compute reachability between all super-cities (using Floyd-Warshall
   on the DAG or BFS from each super-city).

   Visual:

   Condensation:
        A
       / \
      B   C
       \ /
        D

   D is reached by everyone (most "central").
   But D reaches no one else.

   A reaches everyone (most "influential").
   Choose based on your definition of "central"!

──────────────────────────────────────────────────────────────

Q23: How would you detect if a directed graph is strongly connected?

A: A graph is strongly connected if it has exactly ONE SCC containing
   ALL vertices. Run Kosaraju's or Tarjan's and check if the result
   is a single SCC of size V.

   Visual:

   One SCC of size 5:
     ┌─────────────────┐
     │  0→1→2→3→4→0   │
     └─────────────────┘
   Strongly connected! ✓

   Two SCCs:
     {0,1,2} and {3,4}
   Not strongly connected! ✗

──────────────────────────────────────────────────────────────

Q24: What if the graph has parallel edges (multiple edges between
     the same pair of vertices)?

A: Parallel edges do not change SCCs! If u can reach v through one
   edge, adding more edges between them does not create new paths.
   Both algorithms handle parallel edges naturally.

   Visual:

   0 ═════ 1 (two parallel edges)
   1 → 2
   2 → 0

   SCC: {0, 1, 2} — same as with single edges!

──────────────────────────────────────────────────────────────

Q25: Design a system to find SCCs in a real-time social network
     where users follow each other.

A: Imagine Twitter where users follow each other. An SCC is a group
   where everyone follows everyone back — a tight-knit community.

   System design:
   1. Build a directed graph: user → user they follow.
   2. Run Tarjan's periodically (e.g., daily) to find SCCs.
   3. For new follow events: check if the new edge creates a cycle
      between two previously separate SCCs. If yes, merge them.
   4. For unfollow events: check if the removed edge breaks a cycle.
      If yes, recompute for the affected SCC.
   5. Use the SCCs to recommend "close friends" (same SCC) and
      "influencers" (SCCs with many incoming edges).

   Visual:

   User A follows B, B follows C, C follows A → {A,B,C} is an SCC!
   Recommend these users to each other as "close friends."
```
