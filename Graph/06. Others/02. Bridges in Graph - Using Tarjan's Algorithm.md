## 1. Problem Statement

Given an **undirected connected graph**, find all **bridges** (also called critical connections).

A **bridge** is an edge whose **removal increases** the number of connected components in the graph.

In other words: if you delete a bridge, the graph splits into two or more disconnected pieces.

## Key Insight

```
Observation 1: What makes an edge a bridge?

An edge (u, v) is a bridge if there is NO other way
to get from u to v without using that edge.

Visual (bridge):

    Component A ───(u,v)─── Component B

    Any path from A to B MUST use edge (u,v).
    No alternate route exists!

Visual (not a bridge):

    Component A ───(u,v)─── Component B
         \                 /
          \──────(x,y)────/

    There is another path from A to B!
    Edge (u,v) is NOT a bridge.

──────────────────────────────────────────────────────────────

Observation 2: Cycle edges are never bridges

If an edge is part of a cycle, there is always an alternate
route around the cycle. So it cannot be a bridge.

Visual:

    0 ───── 1
    │       │
    │       │   ← edge (0,1) is part of cycle
    3 ───── 2

    Remove (0,1): still connected via 0→3→2→1 ✓

    But in a line:

    0 ───── 1 ───── 2

    Remove (0,1): 0 is isolated! ✗
    No alternate path → bridge!

──────────────────────────────────────────────────────────────

Observation 3: Tarjan's Bridge Condition

For an edge (u, v) where v is a child of u in DFS tree:

    low[v] > disc[u]  →  (u, v) is a BRIDGE!

What do these mean?
  disc[u] = discovery time of u (when DFS first visits it)
  low[v]  = earliest discovery time reachable from v
             (including through back edges)

Why does this work?

If low[v] > disc[u], it means:
  "From vertex v and everything below it, the earliest vertex
   we can reach is AFTER u."

So there is NO back edge from v's subtree to u or above.
The only connection from v's subtree to the rest of the graph
is through edge (u, v)!

Therefore, removing (u, v) disconnects v's subtree!
```

### Brute Force vs Optimized

```
Brute Force Approach:

For every edge:
  1. Remove the edge
  2. Run BFS/DFS to check if graph is still connected
  3. If disconnected → it's a bridge!
  4. Add the edge back

Visual:

  Edges: e1, e2, e3, e4

  Remove e1, check connectivity, restore e1
  Remove e2, check connectivity, restore e2
  Remove e3, check connectivity, restore e3
  Remove e4, check connectivity, restore e4

  Time: O(E × (V + E)) → too slow!

──────────────────────────────────────────────────────────────

Optimized Approach (Tarjan's Algorithm):

Run ONE DFS. During the DFS, compute disc[] and low[] arrays.
Whenever we find low[child] > disc[parent], we found a bridge!

Visual:

  One DFS traversal:

    0 ──→ 1 ──→ 3
    │     │
    ↓     ↓
    2     4

  During DFS, check each tree edge:
    low[2] vs disc[0] → not a bridge (back edge exists)
    low[3] vs disc[1] → bridge! (no back edge)
    low[4] vs disc[1] → bridge! (no back edge)

  Time: O(V + E) → much faster!
```

## Intuition

```
Imagine a network of islands connected by bridges.

Some bridges are CRITICAL — if they break, some islands become
inaccessible from others.

Some bridges are REDUNDANT — if they break, you can still reach
every island by taking a different route.

──────────────────────────────────────────────────────────────

Critical Bridge:

    Island A ─── Bridge 1 ─── Island B

    There is NO other way from A to B.
    If Bridge 1 breaks, A and B are separated! ✗

──────────────────────────────────────────────────────────────

Redundant Bridge:

    Island A ─── Bridge 1 ─── Island B
         \                     /
          \─── Bridge 2 ─────/

    If Bridge 1 breaks, use Bridge 2!
    If Bridge 2 breaks, use Bridge 1!
    The islands stay connected! ✓

──────────────────────────────────────────────────────────────

The DFS Discovery Times:

Think of disc[] as a timestamp when you first arrive at an island.

    Time 0: Arrive at Island 0
    Time 1: Arrive at Island 1
    Time 2: Arrive at Island 2
    Time 3: Arrive at Island 3

low[] tells you: "What is the earliest island I can reach from here,
including by backtracking through alternate routes?"

If from Island 2 (and everything beyond it), the earliest you can
reach is Island 1 (disc[1] = 1), and disc[1] < low[2]... wait,
let's think again.

Actually: if low[child] > disc[parent], it means:
  "Everything reachable from the child was discovered AFTER the parent.
   There is no secret back door from the child to the parent or above."

So the edge parent→child is the ONLY entrance to that region.
Remove it, and the region is cut off!
```

## Algorithm

```
Step 1: Build adjacency list

For each edge (u, v):
  adj[u].push_back(v)
  adj[v].push_back(u)

──────────────────────────────────────────────────────────────

Step 2: Initialize arrays

visited[V] = {false, false, ...}
disc[V]    = {0, 0, ...}     // discovery times
low[V]     = {0, 0, ...}     // lowest reachable discovery time
timer = 0

──────────────────────────────────────────────────────────────

Step 3: DFS from vertex 0

Function DFS(node, parent):
  visited[node] = true
  disc[node] = low[node] = timer++

  For each neighbor of node:
    If neighbor == parent:
      continue  // don't go back to parent

    If neighbor is visited:
      // Back edge found!
      low[node] = min(low[node], disc[neighbor])

    Else:
      // Tree edge - recurse
      DFS(neighbor, node)

      // After recursion, update low[node]
      low[node] = min(low[node], low[neighbor])

      // Check bridge condition
      If low[neighbor] > disc[node]:
        // neighbor cannot reach back to node or above
        // edge (node, neighbor) is a BRIDGE!
        bridges.push_back({node, neighbor})

──────────────────────────────────────────────────────────────

Step 4: Return bridges
```

## Code

### Brute Force

```cpp
#include <bits/stdc++.h>
using namespace std;

bool isConnected(int V, vector<int> adj[], int skipU, int skipV) {
    vector<bool> visited(V, false);
    queue<int> q;
    q.push(0);
    visited[0] = true;
    int count = 1;

    while (!q.empty()) {
        int node = q.front(); q.pop();
        for (int neigh : adj[node]) {
            if ((node == skipU && neigh == skipV) ||
                (node == skipV && neigh == skipU))
                continue;
            if (!visited[neigh]) {
                visited[neigh] = true;
                count++;
                q.push(neigh);
            }
        }
    }
    return count == V;
}

vector<vector<int>> findBridgesBruteForce(int V, vector<vector<int>>& connections) {
    vector<int> adj[V];
    for (auto& e : connections) {
        adj[e[0]].push_back(e[1]);
        adj[e[1]].push_back(e[0]);
    }

    vector<vector<int>> bridges;
    for (auto& edge : connections) {
        int u = edge[0], v = edge[1];
        if (!isConnected(V, adj, u, v)) {
            bridges.push_back({u, v});
        }
    }
    return bridges;
}
```

### Optimized (Tarjan's Algorithm)

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    void DFS(int node, int parent, vector<int> adj[], vector<bool>& visited,
             vector<int>& disc, vector<int>& low, vector<vector<int>>& bridges, int& timer) {

        visited[node] = true;
        disc[node] = low[node] = timer++;

        for (int j = 0; j < adj[node].size(); j++) {
            int neib = adj[node][j];

            if (neib == parent) continue;

            else if (visited[neib]) {
                low[node] = min(low[node], disc[neib]);
            }
            else {
                DFS(neib, node, adj, visited, disc, low, bridges, timer);
                low[node] = min(low[node], low[neib]);

                if (low[neib] > disc[node]) {
                    bridges.push_back({node, neib});
                }
            }
        }
    }

    vector<vector<int>> criticalConnections(int V, vector<vector<int>>& connections) {
        vector<int> adj[V];
        for (auto& e : connections) {
            adj[e[0]].push_back(e[1]);
            adj[e[1]].push_back(e[0]);
        }

        vector<bool> visited(V, false);
        vector<int> disc(V);
        vector<int> low(V);
        vector<vector<int>> bridges;
        int timer = 0;

        DFS(0, -1, adj, visited, disc, low, bridges, timer);
        return bridges;
    }
};

// ==================== DRIVER CODE ====================
int main() {
    Solution sol;
    vector<vector<int>> conn1 = {{0,1}, {1,2}, {2,0}, {1,3}};
    vector<vector<int>> res1 = sol.criticalConnections(4, conn1);

    cout << "Bridges: ";
    for (auto& e : res1) cout << "[" << e[0] << "," << e[1] << "] ";
    cout << endl;

    return 0;
}
```

**Output:**

```
Bridges: [1,3]
```

## Step-by-Step Trace

```
Example: n = 4, connections = [[0,1], [1,2], [2,0], [1,3]]

Graph:

    0 ───── 1 ───── 3
    │       │
    │       │
    └───────2

Adjacency list:
  adj[0] = [1, 2]
  adj[1] = [0, 2, 3]
  adj[2] = [1, 0]
  adj[3] = [1]

── Initialization ──

visited: [false, false, false, false]
disc:    [0, 0, 0, 0]
low:     [0, 0, 0, 0]
timer = 0

── DFS(0, parent=-1) ──

  visited[0] = true
  disc[0] = low[0] = 0, timer = 1

  neighbors of 0: [1, 2]

  neighbor 1: not visited, not parent → DFS(1, 0)

    ── DFS(1, parent=0) ──

    visited[1] = true
    disc[1] = low[1] = 1, timer = 2

    neighbors of 1: [0, 2, 3]

    neighbor 0: is parent → skip!

    neighbor 2: not visited → DFS(2, 1)

      ── DFS(2, parent=1) ──

      visited[2] = true
      disc[2] = low[2] = 2, timer = 3

      neighbors of 2: [1, 0]

      neighbor 1: is parent → skip!

      neighbor 0: visited, not parent → back edge!
        low[2] = min(low[2], disc[0]) = min(2, 0) = 0

      No more neighbors. Return from DFS(2).

      Back to DFS(1):
        low[1] = min(low[1], low[2]) = min(1, 0) = 0

        Check bridge condition:
          low[2] = 0, disc[1] = 1
          Is low[2] > disc[1]? 0 > 1? NO!
          → Edge (1, 2) is NOT a bridge! ✓

    neighbor 3: not visited → DFS(3, 1)

      ── DFS(3, parent=1) ──

      visited[3] = true
      disc[3] = low[3] = 3, timer = 4

      neighbors of 3: [1]

      neighbor 1: is parent → skip!

      No more neighbors. Return from DFS(3).

      Back to DFS(1):
        low[1] = min(low[1], low[3]) = min(0, 3) = 0

        Check bridge condition:
          low[3] = 3, disc[1] = 1
          Is low[3] > disc[1]? 3 > 1? YES! ✓
          → Edge (1, 3) is a BRIDGE! ✓

          bridges = [[1, 3]]

    No more neighbors. Return from DFS(1).

    Back to DFS(0):
      low[0] = min(low[0], low[1]) = min(0, 0) = 0

      Check bridge condition:
        low[1] = 0, disc[0] = 0
        Is low[1] > disc[0]? 0 > 0? NO!
        → Edge (0, 1) is NOT a bridge! ✓

  neighbor 2: already visited → skip

  No more neighbors. Return from DFS(0).

── Final Result ──

disc:  [0, 1, 2, 3]
low:   [0, 0, 0, 3]

bridges: [[1, 3]]
```

### Common Mistakes

```
❌ Mistake 1: Forgetting to skip parent
  if (neib == parent)  // must skip!
      continue;

  Without this, you would treat the edge back to parent
  as a back edge, giving wrong low values.

  Visual:
    0 ─── 1
    Going from 1 back to 0 is NOT a back edge!
    It is the tree edge we just came from.

✓ Correct: Always skip parent!

──────────────────────────────────────────────────────────────

❌ Mistake 2: Using low[neib] instead of disc[neib] for visited neighbor
  low[node] = min(low[node], low[neib]);  // WRONG for visited!

  For a visited neighbor (back edge), we use disc[neib],
  not low[neib]. Because the back edge connects directly
  to neib, not to whatever neib can reach.

✓ Correct:
  if (visited[neib]):
      low[node] = min(low[node], disc[neib]);

──────────────────────────────────────────────────────────────

❌ Mistake 3: Wrong bridge condition direction
  if (low[node] > disc[neib])  // WRONG!

  The condition is low[CHILD] > disc[PARENT].
  Not the other way around!

✓ Correct:
  if (low[neib] > disc[node])
      bridges.push_back({node, neib});

──────────────────────────────────────────────────────────────

❌ Mistake 4: Not handling disconnected graph
  The problem says the graph is connected, but if it is not,
  DFS from 0 won't visit all vertices.

✓ Correct: Run DFS from all unvisited vertices:
  for (int i = 0; i < V; i++)
      if (!visited[i])
          DFS(i, -1, ...);

──────────────────────────────────────────────────────────────

❌ Mistake 5: Using BFS instead of DFS
  BFS cannot easily compute low[] values because it does not
  build a clear parent-child tree structure.

✓ Correct: Use DFS for Tarjan's algorithm!
```

## Complexity Analysis

**Brute Force:**

**Time Complexity:** `O(E × (V + E))`

- For each edge: remove it, run BFS/DFS → O(V + E)
- Do this for all E edges
- Total: O(E × (V + E))

**Space Complexity:** `O(V + E)`

- Adjacency list: O(V + E)
- Visited array: O(V)
- Queue: O(V)

```
Visual:

  For each of E edges:
    Remove edge → O(1)
    BFS/DFS → O(V + E)
    Restore edge → O(1)

  Total: E × O(V + E) = O(E × (V + E))
```

**Optimized (Tarjan's):**

**Time Complexity:** `O(V + E)`

- One DFS visits every vertex and edge once
- All operations inside DFS are O(1)
- Total: O(V + E)

**Space Complexity:** `O(V + E)`

- Adjacency list: O(V + E)
- disc[], low[], visited[]: O(V)
- Recursion stack: O(V)
- Bridges list: O(E) worst case

```
Visual comparison:

Brute Force:
  E edges × BFS each
  = O(E × (V + E))
  For V=1000, E=1000: ~1,000,000 operations

Tarjan's:
  One DFS
  = O(V + E)
  For V=1000, E=1000: ~2,000 operations

  500x faster! ✓
```

## Interview Questions

```
Q1: What is a bridge in a graph?

A: A bridge is like a single road connecting two towns. If that
   road closes, the two towns are completely cut off from each
   other. There is no other way to travel between them.

   Visual:

   Town A ─── Road X ─── Town B

   If Road X is the ONLY connection:
   → Road X is a bridge! ✓

   Town A ─── Road X ─── Town B
      \                 /
       \── Road Y ────/

   If Road Y also exists:
   → Road X is NOT a bridge! ✓

──────────────────────────────────────────────────────────────

Q2: Why are cycle edges never bridges?

A: If an edge is part of a cycle, there is always another way
   around the cycle. Think of it like a ring road — if one
   section is closed, you can drive around the other way.

   Visual:

      0 ──── 1
      │      │
      │      │   ← edge (0,1) has alternate path 0→3→2→1
      3 ──── 2

   Remove (0,1): still connected via 0→3→2→1 ✓

   But a line has no alternate:

      0 ──── 1 ──── 2

   Remove (0,1): 0 is isolated! ✗

──────────────────────────────────────────────────────────────

Q3: What is the bridge condition in Tarjan's algorithm?

A: For an edge parent → child in the DFS tree:

   low[child] > disc[parent] → BRIDGE!

   What this means in simple words:
   "From the child and everything below it, the earliest vertex
    we can reach is AFTER the parent."

   So there is no secret back door from the child's region
   back to the parent. The only way in or out is through
   the parent→child edge!

   Visual:

        Parent (disc=1)
        │
        │ tree edge
        ↓
        Child (disc=2)
       / \
      /   \
     /     \
   (subtree)

   If low[child] = 5, and disc[parent] = 1:
   5 > 1 → no back edge from subtree to parent or above!
   → The tree edge is a BRIDGE!

──────────────────────────────────────────────────────────────

Q4: What is the difference between disc[] and low[]?

A: disc[] is like a ticket number — the order in which DFS
   first visits each vertex.

   low[] is like asking: "What is the smallest ticket number
   I can reach from here, including by taking back edges?"

   Visual:

   DFS order:
     Vertex 0: ticket 0
     Vertex 1: ticket 1
     Vertex 2: ticket 2
     Vertex 3: ticket 3

   disc = [0, 1, 2, 3]

   If vertex 2 has a back edge to vertex 0:
     low[2] = min(disc[2], disc[0]) = min(2, 0) = 0

   So from vertex 2, we can reach ticket 0!

──────────────────────────────────────────────────────────────

Q5: Why do we skip the parent in DFS?

A: The edge back to the parent is the tree edge we just came
   from. If we counted it as a back edge, we would think there
   is an alternate route when there isn't.

   Visual:

   0 ─── 1

   DFS goes 0 → 1. From 1, neighbor 0 is the parent.
   If we don't skip parent:
     low[1] = min(low[1], disc[0]) = 0
     This makes it look like there is a back edge!
   But (0,1) is just the tree edge, not an alternate path.

   Skip parent → correct low values! ✓

──────────────────────────────────────────────────────────────

Q6: When would you use brute force vs Tarjan's?

A: Brute force is simple to code but very slow. Use it only for
   very small graphs or when you have plenty of time.

   Tarjan's is the standard solution. It finds all bridges in
   one pass — much faster and elegant.

   Visual comparison:

   Brute Force:
     For each edge: remove, check, restore
     Like testing every bridge by closing it one at a time

   Tarjan's:
     Walk through once, mark critical bridges as you go
     Like a smart inspector who spots weak bridges instantly

──────────────────────────────────────────────────────────────

Q7: Can a graph have no bridges?

A: Yes! Any graph where every edge is part of at least one cycle
   has no bridges. Complete graphs and cycles have no bridges.

   Visual (no bridges):

       0 ──── 1
       │      │
       │      │
       3 ──── 2

   Every edge is in a cycle. Remove any edge → still connected! ✓

Q8: Can every edge be a bridge?

A: Yes! In a tree, every edge is a bridge. Removing any edge
   splits the tree into two parts.

   Visual (all edges are bridges):

         1
         │
    2 ───0─── 3
         │
         4

   Remove (0,1): subtree with 1 is disconnected!
   Remove (0,2): subtree with 2 is disconnected!
   And so on...

Q9: What are real-world uses of finding bridges?

A: Real-world examples:

   Network reliability:
   → Find weak points in computer networks
   → If a bridge router fails, network splits!

   Road infrastructure:
   → Identify critical roads that have no alternate routes
   → Prioritize maintenance on these roads

   Power grids:
   → Find transmission lines that would cause blackouts if they fail
   → Build redundant connections around them

   Social networks:
   → Find people who are the only connection between two groups
   → If they leave, the groups become isolated

Q10: What is the most common mistake in Tarjan's algorithm?

A: Forgetting to skip the parent or using the wrong value for
    back edges.

    Visual (wrong — not skipping parent):

    0 ─── 1
    DFS: 0 → 1
    From 1, neighbor 0 is parent.
    If not skipped: low[1] = min(low[1], disc[0]) = 0
    Then check: low[1] > disc[0]? 0 > 0? NO
    → Correctly not a bridge, but for wrong reason!

    Actually, this happens to work here, but in general it
    causes incorrect low values throughout the tree.

    Another common mistake: using low[neib] instead of disc[neib]
    for visited neighbors. This gives wrong back edge values!

──────────────────────────────────────────────────────────────

Q11: How does Tarjan's algorithm handle multiple connected components?

A: Run DFS from every unvisited vertex.

    for (int i = 0; i < V; i++) {
        if (!visited[i]) {
            DFS(i, -1, ...);
        }
    }

    Each component is processed independently.
    Bridges within each component are found correctly.

──────────────────────────────────────────────────────────────

Q12: Can self-loops be bridges?

A: No! A self-loop is an edge from a vertex to itself.
   Removing it does not disconnect anything.

    Visual:

    0 ───┐
         │ (self loop)
    0 ───┘

    The vertex stays connected to itself (trivially).
    And other vertices are unaffected.

──────────────────────────────────────────────────────────────

Q13: Can parallel edges be bridges?

A: If there are two or more edges between the same pair of
    vertices, NONE of them are bridges. Because if one fails,
    the other still connects those vertices.

    Visual:

    0 ═════ 1   (two parallel edges)

    Remove one edge: still connected via the other! ✓
    So neither edge is a bridge.

──────────────────────────────────────────────────────────────

Q14: What is the difference between a bridge and an articulation point?

A: A bridge is an EDGE whose removal disconnects the graph.
   An articulation point is a VERTEX whose removal disconnects
   the graph.

    Visual:

        0 ─── 1 ─── 2
              │
              3

    Edge (1,2) is a bridge.
    Vertex 1 is an articulation point.

    Remove edge (1,2): 2 is disconnected. ✓
    Remove vertex 1: 0, 2, and 3 are all separated. ✓

    They are related but different concepts!

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

    Mistake 1: Not skipping parent in DFS
    → Fix: if (neib == parent) continue;

    Mistake 2: Using low[] instead of disc[] for back edges
    → Fix: low[node] = min(low[node], disc[neib]);

    Mistake 3: Wrong bridge condition direction
    → Fix: if (low[child] > disc[parent])

    Mistake 4: Forgetting to handle multiple components
    → Fix: Loop through all vertices, DFS if unvisited

    Mistake 5: Confusing bridges with articulation points
    → Fix: Bridge = edge, Articulation point = vertex
```

Q17: What if the graph has parallel edges? How does your

   But our adjacency list stores both edges separately!
   DFS might treat one as a tree edge and the other as a back edge.

   Visual:

   0 ═════ 1   (two edges: e1 and e2)

   The other parallel edge is treated as a back edge — correct!

Q18: Can you find bridges in a directed graph?

A: The concept is different in directed graphs. We look for
   "strong bridges" — edges whose removal increases the number

   Edge (1,2) might be a strong bridge if removing it breaks
   the cycle and creates separate SCCs.

   the component graph (which is a DAG). Bridges in the DAG

Q19: How would you find bridges in a graph with 10^5 vertices
     and 10^5 edges? Can brute force work?

A: Brute force is O(E × (V + E)) = 10^5 × (2 × 10^5)
   = 2 × 10^10 operations — way too slow!

   Tarjan's is O(V + E) = 2 × 10^5 operations — instant!

   Visual comparison:

   Brute Force:
     2 × 10^10 operations
     At 10^9 ops/sec → 20 seconds (too slow!)

   Tarjan's:
     2 × 10^5 operations
     At 10^9 ops/sec → 0.0002 seconds (instant!)

   For large graphs, Tarjan's is the ONLY practical choice.

Q20: Given all bridges, how many connected components will

   Visual:

          Bridges: (0,1), (1,2), (2,3) → 3 bridges

     0     1     2     3

   each remaining connected piece is a 2-edge-connected component.

   Visual:

             │     │
             └─────┘

Q23: Can you modify Tarjan's algorithm to also find
     articulation points at the same time?

A: Yes! The conditions are very similar:

   Bridge: low[child] > disc[parent]
   Articulation point: low[child] >= disc[parent]

   Note: >= for articulation point, > for bridge!

   Also, the root is an articulation point if it has 2+
   children in the DFS tree.

   Visual:

        0 (root)
       / \
      /   \
     1     2
     │
     3

   For edge (0,1): low[1] >= disc[0]?
   If yes → 0 is an articulation point!
   And if low[1] > disc[0] → (0,1) is also a bridge!

   We can compute both in a single DFS pass!

Q25: Design a system to monitor bridges in a dynamic network
     where edges are added and removed over time.

A: This is a hard problem! Full recomputation with Tarjan's
   on every change is O(V + E) per update.

   Better approaches:

   1. For edge addition:
      - If the new edge creates a cycle, it might eliminate bridges
      - Only bridges on the new cycle are affected

   2. For edge removal:
      - If the removed edge was NOT a bridge, no change
      - If it WAS a bridge, new bridges might appear in the
        newly separated components

   3. Use dynamic graph algorithms:
      - Maintain a spanning tree
      - Track non-tree edges that form cycles
      - Update bridge status incrementally

   Visual:

   Initial graph: 0 ─── 1 ─── 2
                  Bridges: (0,1), (1,2)

   Add edge (0,2):
      New cycle: 0-1-2-0
      Bridges (0,1) and (1,2) are NO LONGER bridges! ✓

   The update only affects edges on the new cycle.
