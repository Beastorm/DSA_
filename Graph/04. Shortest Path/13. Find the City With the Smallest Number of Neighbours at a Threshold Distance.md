## 1. Problem Statement

There are `n` cities numbered from `0` to `n-1`. Given an array `edges` where `edges[i] = [fromi, toi, weighti]` represents a **bidirectional weighted edge** between cities `fromi` and `toi`, and given integer `distanceThreshold`:

**Find** a city with the **smallest number** of cities reachable within `distanceThreshold` distance.

If multiple such cities exist, return the one with the **greatest number** (largest index).

> **Note:** Distance of a path = sum of edge weights along that path.

**Input Format:**

- `n` = number of cities
- `m` = number of edges
- `edges` = list of bidirectional weighted edges `[u, v, wt]`
- `distanceThreshold` = maximum allowed distance

**Output:**

- City number with smallest reachable neighbor count (largest index if tie)

## Key Insight

```
Why Floyd-Warshall for this problem?

Need: Shortest distances from EVERY city to EVERY other city

→ All-pairs shortest path problem
→ Floyd-Warshall is perfect! ✓

Alternative approaches:

Approach 1: Run Dijkstra from each city
→ Time: O(N × E log N)
→ For dense graph: O(N³ log N)
→ More complex code

Approach 2: Floyd-Warshall
→ Time: O(N³)
→ Simple implementation
→ Perfect for small N (≤ 100-400) ✓

Approach 3: Bellman-Ford from each city
→ Time: O(N² × E)
→ Even slower than Floyd-Warshall

For this problem:
→ N is typically small (constraints often N ≤ 100)
→ Need all-pairs distances
→ Floyd-Warshall is the natural choice! ✓

Strategy:
1. Build all-pairs shortest path matrix (Floyd-Warshall)
2. For each city, count neighbors within threshold
3. Find city with minimum count (largest index if tie)
```

## Intuition

```
Imagine you are evaluating locations for a new facility:

→ Cities = potential locations
→ Edges = roads with travel time
→ Threshold = maximum acceptable commute time
→ Goal: Find location that serves FEWEST nearby cities

Why fewest?
→ Less crowded
→ More isolated/premium
→ Lower infrastructure load
→ Better for specialized services

How to solve:

Step 1: Compute shortest travel times between ALL city pairs
        (Floyd-Warshall gives us this in one go!)

Step 2: For each city, count how many cities are within
        the acceptable commute time

Step 3: Pick the city with the smallest count
        (if tie, pick the one with highest number)

The tie-breaker (largest index) ensures a deterministic
answer when multiple cities are equally isolated.
```

## Algorithm

```
findCity(n, edges, distanceThreshold):

Step 1: Initialize distance matrix

dist[n][n] = infinity for all pairs

For each edge (u, v, weight):
  dist[u][v] = weight
  dist[v][u] = weight (bidirectional)

For each city i:
  dist[i][i] = 0

Step 2: Floyd-Warshall (All-pairs shortest paths)

for k = 0 to n-1:
  for i = 0 to n-1:
    for j = 0 to n-1:
      dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Step 3: Count reachable cities for each city

for each city from 0 to n-1:
  count = 0
  for each otherCity from 0 to n-1:
    if dist[city][otherCity] <= threshold:
      count++
  
  Track city with minimum count
  (if tie, prefer larger city number)

Step 4: Return city with smallest neighbor count
```

### Key Considerations

```
Counting strategy:

→ Count includes the city itself (dist[i][i] = 0 ≤ threshold)
→ So minimum count is always ≥ 1

Tie-breaking:

→ If multiple cities have same minimum count
→ Choose city with LARGEST number
→ Use cnt <= cntCity (not <) to prefer later cities

Edge case handling:

→ If dist[i][k] or dist[k][j] is infinity
→ Skip update (avoid overflow)
→ Check before adding in Floyd-Warshall
```

### Why Use `<=` for Tie-Breaking?

```
Critical line:

if (cnt <= cntCity) {  ← Uses <=, not <
    cntCity = cnt;
    cityNo = city;
}

Why <= instead of <?

Requirement: "If multiple cities, choose largest number"

Example: Cities [0, 1, 2] all have count = 3

With <:
city=0: cnt=3 < ∞ → update cityNo=0
city=1: cnt=3 < 3? NO → skip
city=2: cnt=3 < 3? NO → skip
Result: cityNo = 0 ✗ (WRONG!)

With <=:
city=0: cnt=3 <= ∞ → update cityNo=0
city=1: cnt=3 <= 3 → update cityNo=1 ✓
city=2: cnt=3 <= 3 → update cityNo=2 ✓
Result: cityNo = 2 ✓ (CORRECT!)

Tie-breaking mechanism:
→ Loop processes cities in order: 0, 1, 2, ..., n-1
→ Using <= keeps updating to latest city
→ Last city with minimum count wins! ✓

Alternative (explicit):
if (cnt < cntCity || (cnt == cntCity && city > cityNo)) {
    cntCity = cnt;
    cityNo = city;
}

Both work, but cnt <= cntCity is cleaner! ✓
```

### Overflow Prevention in Floyd-Warshall

```
Standard update:
dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]);

Problem: If dist[i][k] or dist[k][j] is INT_MAX,
their sum overflows!

INT_MAX + INT_MAX = undefined behavior (integer overflow)

Fix: Check before adding

if (dist[i][k] == INT_MAX || dist[k][j] == INT_MAX)
    continue;

dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]);

Or use a large sentinel instead of INT_MAX:

const int INF = 1e9;
// INF + INF = 2e9 < INT_MAX (2.147e9) ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int findCity(int n, int m, vector<vector<int>>& edges, 
                 int distanceThreshold) {
        
        // Initialize distance matrix with large value
        vector<vector<int>> dist(n, vector<int>(n, INT_MAX));
        
        // Build initial distance matrix from edges
        for (auto& edge : edges) {
            int u = edge[0], v = edge[1], wt = edge[2];
            dist[u][v] = wt;
            dist[v][u] = wt;  // Bidirectional
        }
        
        // Distance from city to itself is 0
        for (int i = 0; i < n; i++) {
            dist[i][i] = 0;
        }
        
        // Floyd-Warshall: Find all-pairs shortest paths
        for (int k = 0; k < n; k++) {
            for (int i = 0; i < n; i++) {
                for (int j = 0; j < n; j++) {
                    // Avoid overflow
                    if (dist[i][k] == INT_MAX || dist[k][j] == INT_MAX)
                        continue;
                    
                    dist[i][j] = min(dist[i][j], 
                                     dist[i][k] + dist[k][j]);
                }
            }
        }
        
        // Find city with minimum reachable neighbors
        int minCount = n;
        int resultCity = -1;
        
        for (int city = 0; city < n; city++) {
            int count = 0;
            
            // Count cities within threshold
            for (int other = 0; other < n; other++) {
                if (dist[city][other] <= distanceThreshold) {
                    count++;
                }
            }
            
            // Update if better (or equal but larger city number)
            if (count <= minCount) {
                minCount = count;
                resultCity = city;
            }
        }
        
        return resultCity;
    }
};

int main() {
    int n = 4, m = 4;
    vector<vector<int>> edges = {
        {0, 1, 3}, {1, 2, 1}, {1, 3, 4}, {2, 3, 1}
    };
    int distanceThreshold = 4;
    
    Solution obj;
    int result = obj.findCity(n, m, edges, distanceThreshold);
    
    cout << "City with smallest neighbors: " << result << endl;
    
    return 0;
}
```

### Common Mistakes

```
❌ Mistake 1: Counting excludes self

count = 0;
for (int other = 0; other < n; other++) {
    if (other != city && dist[city][other] <= threshold)
        count++;  // WRONG! Should include self
}

✓ Correct: Include self (dist[city][city] = 0)

────────────────────────────────────────

❌ Mistake 2: Using < instead of <=

if (cnt < cntCity) {  // WRONG! Doesn't prefer larger city
    cntCity = cnt;
    cityNo = city;
}

✓ Correct: Use <= for tie-breaking

────────────────────────────────────────

❌ Mistake 3: Forgetting bidirectional edges

dist[u][v] = weight;  // WRONG! Missing reverse edge

✓ Correct:
dist[u][v] = weight;
dist[v][u] = weight;

────────────────────────────────────────

❌ Mistake 4: Not checking overflow

dist[i][j] = dist[i][k] + dist[k][j];  // Can overflow!

✓ Correct:
if (dist[i][k] == INT_MAX || dist[k][j] == INT_MAX)
    continue;

────────────────────────────────────────

❌ Mistake 5: Wrong initialization

int cntCity = 0;  // WRONG! Should be n or INT_MAX

✓ Correct:
int cntCity = n;  // Maximum possible count
```

### Optimization Opportunities

```
Current: O(N³) Floyd-Warshall

Could we optimize?

Observation 1: Early termination
→ Floyd-Warshall must complete
→ Can't exit early, need all-pairs ✗

Observation 2: Sparse graph
→ If E << N², Dijkstra N times might be faster
→ Dijkstra: O(N × E log N)
→ Dense: O(N³ log N) - worse
→ Sparse: O(N² log N) - better ✓

When to use Dijkstra N times:
E < N²/log(N) approximately

Observation 3: Limited threshold
→ Can use BFS if edges are unweighted
→ But edges are weighted here ✗

Observation 4: Bit manipulation
→ If threshold is small, could track reachable set
→ But doesn't improve asymptotic complexity ✗

Conclusion:
→ For small N (≤ 100): Floyd-Warshall is simplest
→ For larger N with sparse graph: Dijkstra N times
→ Trade-off between code simplicity and efficiency

Practical note:
→ Floyd-Warshall has great cache locality
→ Simple nested loops → CPU-friendly
→ Often faster in practice than theoretical suggests ✓
```

## Complexity Analysis

**Time Complexity:** `O(N³ + N²)`

- Floyd-Warshall: O(N³)
- Counting for each city: O(N²)
- Total: **O(N³)**

**Space Complexity:** `O(N²)`

- Distance matrix: O(N²)
- No additional significant space
- Total: **O(N²)**

```
Detailed breakdown:

Floyd-Warshall phase:
→ 3 nested loops: k, i, j (each N iterations)
→ O(N³) operations

Counting phase:
→ For each city (N iterations)
→ Check each other city (N iterations)
→ O(N²) operations

Total: O(N³) + O(N²) = O(N³)

Space:
→ dist matrix: N × N = O(N²)
→ Constant extra variables
→ No recursion, no stack
→ Total: O(N²)

Comparison with alternatives:

Approach              Time            Space
─────────────────────────────────────────────
Floyd-Warshall        O(N³)           O(N²)
Dijkstra N times      O(N²log N)      O(N²)  [sparse]
                      O(N³log N)      O(N²)  [dense]

For small N (typical in this problem):
→ Floyd-Warshall is preferred ✓
→ Simpler code
→ Better constants (no log factor overhead)
```

## Interview Questions

```
Q1: Why is Floyd-Warshall a good choice for this problem?

A: We need shortest distances from every city to every other city
   (all-pairs shortest path). Floyd-Warshall computes this in
   O(N³) with just three nested loops. Since typical constraints
   have N ≤ 100, N³ = 10⁶ operations is very fast. It also handles
   the dense graph case well and requires minimal code compared
   to running Dijkstra N times.

──────────────────────────────────────────────────────────────

Q2: Why use <= instead of < when comparing counts?

A: The problem requires returning the city with the LARGEST index
   when multiple cities have the same minimum count. By iterating
   cities in order 0, 1, 2, ... and using <=, we keep updating
   the result whenever we see an equal count. This ensures the
   last (largest-index) city with the minimum count is retained.
   Using < would stop at the first minimum and ignore later ties.

──────────────────────────────────────────────────────────────

Q3: How do you prevent integer overflow in Floyd-Warshall?

A: Before adding dist[i][k] + dist[k][j], check if either value
   is INT_MAX (or your chosen infinity sentinel). Adding two
   large values causes undefined integer overflow. The check:
   
   if (dist[i][k] == INT_MAX || dist[k][j] == INT_MAX) continue;
   
   Alternatively, use a sentinel like 1e9 that is large enough
   to represent infinity but small enough that 1e9 + 1e9 still
   fits in a 32-bit signed integer.

──────────────────────────────────────────────────────────────

Q4: When would Dijkstra from each city be better than
    Floyd-Warshall?

A: Dijkstra N times runs in O(N × E log N). For sparse graphs
   where E ≈ N, this becomes O(N² log N), which beats Floyd-
   Warshall's O(N³). For dense graphs where E ≈ N², Dijkstra
   becomes O(N³ log N), which is worse. Use Dijkstra when the
   graph is sparse and N is large (e.g., N > 500).

──────────────────────────────────────────────────────────────

Q5: Does the count include the city itself? Why or why not?

A: The standard interpretation includes the city itself because
   dist[i][i] = 0, which is always ≤ distanceThreshold. However,
   the relative ranking of cities doesn't change whether you
   include self or not, because every city adds exactly 1 to its
   count. The tie-breaking result remains the same. In code, we
   simply count all j where dist[i][j] <= threshold, including
   j = i.

──────────────────────────────────────────────────────────────

Q6: How would you modify the solution if the graph were directed?

A: For a directed graph, only set dist[u][v] = weight (not the
   reverse). Floyd-Warshall naturally handles directed edges.
   The counting phase remains identical. The main change is in
   the initialization: process each edge exactly as given without
   adding the reverse direction.

──────────────────────────────────────────────────────────────

Q7: What if there are multiple edges between the same pair of
    cities?

A: Before running Floyd-Warshall, keep only the edge with the
   minimum weight. A heavier edge between the same pair can never
   be part of a shortest path. During initialization:
   
   dist[u][v] = min(dist[u][v], weight);
   dist[v][u] = min(dist[v][u], weight);

──────────────────────────────────────────────────────────────

Q8: Can this problem be solved with BFS instead?

A: Only if all edge weights are equal (unweighted graph). BFS
   finds shortest paths in O(V + E) for unweighted graphs. Since
   this problem has weighted edges with varying weights, BFS
   would give incorrect distances. For weighted graphs, we need
   Dijkstra (non-negative) or Bellman-Ford (negative allowed).

──────────────────────────────────────────────────────────────

Q9: What does it mean if all cities have the same count?

A: It means every city can reach the same number of other cities
   within the threshold. This typically happens in highly
   symmetric graphs like complete graphs or regular graphs. The
   tie-breaker then simply returns the largest city index (n-1).

──────────────────────────────────────────────────────────────

Q10: How would you solve this if the distanceThreshold were
     different for each city?

A: If each city i has its own threshold T[i], you would still
    compute all-pairs shortest paths with Floyd-Warshall. Then
    during counting, use city-specific thresholds:
    
    if (dist[city][other] <= threshold[city])
        count++;
    
    The overall complexity remains O(N³).

──────────────────────────────────────────────────────────────

Q11: What if we need the K cities with the smallest neighbor
     counts?

A: After computing counts for all cities, use a min-heap or
    sort the cities by (count, -index) to get the K cities
    with smallest counts. If counts tie, the larger index is
    considered "smaller" in the ordering (since we prefer
    larger indices). Time: O(N log N) for sorting.

──────────────────────────────────────────────────────────────

Q12: Why initialize the distance matrix with INT_MAX instead of
     a smaller value?

A: INT_MAX clearly represents "unreachable" or "infinity" and
    is large enough that any valid path sum will be smaller.
    However, we must be careful not to add two INT_MAX values.
    An alternative is 1e9, which is large enough for typical
    constraints (max path < 1e9) and safe to add (1e9 + 1e9 =
    2e9 < INT_MAX ≈ 2.147e9).

──────────────────────────────────────────────────────────────

Q13: What are real-world applications of this problem?

A: Key applications include:
    → Network hub selection: Find data center with fewest
      reachable nodes within latency threshold (isolation)
    → Hospital placement: Minimize patient overlap regions
    → Warehouse location: Find spot with limited delivery
      coverage for specialized goods
    → Emergency services: Identify areas with least access
      for resource allocation
    → Social networks: Find most isolated communities

──────────────────────────────────────────────────────────────

Q14: How would you handle dynamic edge updates (edges added or
     removed after initial computation)?

A: For dynamic graphs, Floyd-Warshall is expensive to re-run.
    Alternatives include:
    → Re-run Floyd-Warshall: O(N³) per update
    → Use dynamic shortest path algorithms
    → For few updates, adjust affected rows/columns only
    → For many updates, consider Johnson's or maintain
      a distance oracle

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes in implementing this
     solution?

A: Common mistakes and fixes:

    Mistake 1: Using < instead of <= for tie-breaking
    → Use <= to prefer larger city indices

    Mistake 2: Forgetting bidirectional edges
    → Add both dist[u][v] and dist[v][u]

    Mistake 3: Integer overflow in Floyd-Warshall
    → Check for INT_MAX before addition

    Mistake 4: Not initializing diagonal to 0
    → Set dist[i][i] = 0 for all i

    Mistake 5: Wrong minCount initialization
    → Initialize to n (max possible), not 0

    Mistake 6: Confusing city index with count
    → Track both minCount and resultCity separately

    Mistake 7: Using BFS for weighted graph
    → Use Floyd-Warshall or Dijkstra for weighted edges
```
