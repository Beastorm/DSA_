## 1. Problem Statement

There are `n` cities numbered from `0` to `n-1`. Given an array `edges` where `edges[i] = [fromi, toi, weighti]` represents a **bidirectional weighted edge** between cities `fromi` and `toi`, and given integer `distanceThreshold`:

**Find** a city with the **smallest number** of cities reachable within `distanceThreshold` distance.

If multiple such cities exist, return the one with the **greatest number** (largest index).

> **Note:** Distance of a path = sum of edge weights along that path.

**Input Format:**

- `n` = number of cities
- `m` = number of edges
- `edges` = list of bidirectional weighted edges `[u, v, wt]`
- `distanceThreshold` = maximum allowed distance

**Output:**

- City number with smallest reachable neighbor count (largest index if tie)

  ```
  City 2: dist to [0]=4, [1]=1, [3]=1 → 3 cities reachable
  City 3: dist to [1]=2, [2]=1 → 2 cities reachable

  Minimum count = 2 (cities 0 and 3)
  Choose largest index → City 3 ✓
  ```

**Example 2:**

- **Input:**

  ```
  N = 3, M = 2
  edges = [[0,1,3], [1,2,1]]
  ```

- **Output:** `2`

  ```
  Reachable cities within threshold=2 (excluding self):

  City 1: dist to [2]=1 → 1 city

  Minimum count = 0 (city 0)

  City 0: [0] = 1 city

  City 0: 0 reachable → count 0
  City 1: 1 reachable → count 1
  City 2: 1 reachable → count 1

  Multiple with min=1 (cities 1 and 2)
  Choose largest index → City 2 ✓
  ```

## Key Insight

```
Why Floyd-Warshall for this problem?

Need: Shortest distances from EVERY city to EVERY other city

→ All-pairs shortest path problem
→ Floyd-Warshall is perfect! ✓

Alternative approaches:

Approach 1: Run Dijkstra from each city
→ Time: O(N × E log N)
→ For dense graph: O(N³ log N)
→ More complex code

Approach 2: Floyd-Warshall
→ Time: O(N³)
→ Simple implementation
→ Perfect for small N (≤ 100-400) ✓

Approach 3: Bellman-Ford from each city
→ Time: O(N² × E)
→ Even slower than Floyd-Warshall

For this problem:
→ N is typically small (constraints often N ≤ 100)
→ Need all-pairs distances
→ Floyd-Warshall is the natural choice! ✓

Strategy:
1. Build all-pairs shortest path matrix (Floyd-Warshall)
2. For each city, count neighbors within threshold
3. Find city with minimum count (largest index if tie)
```

## Intuition

```
Imagine you are evaluating locations for a new facility:

→ Cities = potential locations
→ Edges = roads with travel time
→ Threshold = maximum acceptable commute time
→ Goal: Find location that serves FEWEST nearby cities

Why fewest?
→ Less crowded
→ More isolated/premium
→ Lower infrastructure load
→ Better for specialized services

How to solve:

Step 1: Compute shortest travel times between ALL city pairs
        (Floyd-Warshall gives us this in one go!)

Step 2: For each city, count how many cities are within
        the acceptable commute time

Step 3: Pick the city with the smallest count
        (if tie, pick the one with highest number)

The tie-breaker (largest index) ensures a deterministic
answer when multiple cities are equally isolated.
```

## Algorithm

```
findCity(n, edges, distanceThreshold):

Step 1: Initialize distance matrix

dist[n][n] = infinity for all pairs

For each edge (u, v, weight):
  dist[u][v] = weight
  dist[v][u] = weight (bidirectional)

For each city i:
  dist[i][i] = 0

Step 2: Floyd-Warshall (All-pairs shortest paths)

for k = 0 to n-1:
  for i = 0 to n-1:
    for j = 0 to n-1:
      dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Step 3: Count reachable cities for each city

for each city from 0 to n-1:
  count = 0
  for each otherCity from 0 to n-1:
    if dist[city][otherCity] <= threshold:
      count++
  
  Track city with minimum count
  (if tie, prefer larger city number)

Step 4: Return city with smallest neighbor count
```

### Key Considerations

```
Counting strategy:

→ Count includes the city itself (dist[i][i] = 0 ≤ threshold)
→ So minimum count is always ≥ 1

Tie-breaking:

→ If multiple cities have same minimum count
→ Choose city with LARGEST number
→ Use cnt <= cntCity (not <) to prefer later cities

Edge case handling:

→ If dist[i][k] or dist[k][j] is infinity
→ Skip update (avoid overflow)
→ Check before adding in Floyd-Warshall
```

### Why Use `<=` for Tie-Breaking?

```
Critical line:

if (cnt <= cntCity) {  ← Uses <=, not <
    cntCity = cnt;
    cityNo = city;
}

Why <= instead of <?

Requirement: "If multiple cities, choose largest number"

Example: Cities [0, 1, 2] all have count = 3

With <:
city=0: cnt=3 < ∞ → update cityNo=0
city=1: cnt=3 < 3? NO → skip
city=2: cnt=3 < 3? NO → skip
Result: cityNo = 0 ✗ (WRONG!)

With <=:
city=0: cnt=3 <= ∞ → update cityNo=0
city=1: cnt=3 <= 3 → update cityNo=1 ✓
city=2: cnt=3 <= 3 → update cityNo=2 ✓
Result: cityNo = 2 ✓ (CORRECT!)

Tie-breaking mechanism:
→ Loop processes cities in order: 0, 1, 2, ..., n-1
→ Using <= keeps updating to latest city
→ Last city with minimum count wins! ✓

Alternative (explicit):
if (cnt < cntCity || (cnt == cntCity && city > cityNo)) {
    cntCity = cnt;
    cityNo = city;
}

Both work, but cnt <= cntCity is cleaner! ✓
```

### Overflow Prevention in Floyd-Warshall

```
Standard update:
dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]);

Problem: If dist[i][k] or dist[k][j] is INT_MAX,
their sum overflows!

INT_MAX + INT_MAX = undefined behavior (integer overflow)

Fix: Check before adding

if (dist[i][k] == INT_MAX || dist[k][j] == INT_MAX)
    continue;

dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]);

Or use a large sentinel instead of INT_MAX:

const int INF = 1e9;
// INF + INF = 2e9 < INT_MAX (2.147e9) ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int findCity(int n, int m, vector<vector<int>>& edges, 
                 int distanceThreshold) {
        
        // Initialize distance matrix with large value
        vector<vector<int>> dist(n, vector<int>(n, INT_MAX));
        
        // Build initial distance matrix from edges
        for (auto& edge : edges) {
            int u = edge[0], v = edge[1], wt = edge[2];
            dist[u][v] = wt;
            dist[v][u] = wt;  // Bidirectional
        }
        
        // Distance from city to itself is 0
        for (int i = 0; i < n; i++) {
            dist[i][i] = 0;
        }
        
        // Floyd-Warshall: Find all-pairs shortest paths
        for (int k = 0; k < n; k++) {
            for (int i = 0; i < n; i++) {
                for (int j = 0; j < n; j++) {
                    // Avoid overflow
                    if (dist[i][k] == INT_MAX || dist[k][j] == INT_MAX)
                        continue;
                    
                    dist[i][j] = min(dist[i][j], 
                                     dist[i][k] + dist[k][j]);
                }
            }
        }
        
        // Find city with minimum reachable neighbors
        int minCount = n;
        int resultCity = -1;
        
        for (int city = 0; city < n; city++) {
            int count = 0;
            
            // Count cities within threshold
            for (int other = 0; other < n; other++) {
                if (dist[city][other] <= distanceThreshold) {
                    count++;
                }
            }
            
            // Update if better (or equal but larger city number)
            if (count <= minCount) {
                minCount = count;
                resultCity = city;
            }
        }
        
        return resultCity;
    }
};

int main() {
    int n = 4, m = 4;
    vector<vector<int>> edges = {
        {0, 1, 3}, {1, 2, 1}, {1, 3, 4}, {2, 3, 1}
    };
    int distanceThreshold = 4;
    
    Solution obj;
    int result = obj.findCity(n, m, edges, distanceThreshold);
    
    cout << "City with smallest neighbors: " << result << endl;
    
    return 0;
}
```

### Common Mistakes

```
❌ Mistake 1: Counting excludes self

count = 0;
for (int other = 0; other < n; other++) {
    if (other != city && dist[city][other] <= threshold)
        count++;  // WRONG! Should include self
}

✓ Correct: Include self (dist[city][city] = 0)

────────────────────────────────────────

❌ Mistake 2: Using < instead of <=

if (cnt < cntCity) {  // WRONG! Doesn't prefer larger city
    cntCity = cnt;
    cityNo = city;
}

✓ Correct: Use <= for tie-breaking

────────────────────────────────────────

❌ Mistake 3: Forgetting bidirectional edges

dist[u][v] = weight;  // WRONG! Missing reverse edge

✓ Correct:
dist[u][v] = weight;
dist[v][u] = weight;

────────────────────────────────────────

❌ Mistake 4: Not checking overflow

dist[i][j] = dist[i][k] + dist[k][j];  // Can overflow!

✓ Correct:
if (dist[i][k] == INT_MAX || dist[k][j] == INT_MAX)
    continue;

────────────────────────────────────────

❌ Mistake 5: Wrong initialization

int cntCity = 0;  // WRONG! Should be n or INT_MAX

✓ Correct:
int cntCity = n;  // Maximum possible count
```

### Optimization Opportunities

```
Current: O(N³) Floyd-Warshall

Could we optimize?

Observation 1: Early termination
→ Floyd-Warshall must complete
→ Can't exit early, need all-pairs ✗

Observation 2: Sparse graph
→ If E << N², Dijkstra N times might be faster
→ Dijkstra: O(N × E log N)
→ Dense: O(N³ log N) - worse
→ Sparse: O(N² log N) - better ✓

When to use Dijkstra N times:
E < N²/log(N) approximately

Observation 3: Limited threshold
→ Can use BFS if edges are unweighted
→ But edges are weighted here ✗

Observation 4: Bit manipulation
→ If threshold is small, could track reachable set
→ But doesn't improve asymptotic complexity ✗

Conclusion:
→ For small N (≤ 100): Floyd-Warshall is simplest
→ For larger N with sparse graph: Dijkstra N times
→ Trade-off between code simplicity and efficiency

Practical note:
→ Floyd-Warshall has great cache locality
→ Simple nested loops → CPU-friendly
→ Often faster in practice than theoretical suggests ✓
```

## Complexity Analysis

**Time Complexity:** `O(N³ + N²)`

- Floyd-Warshall: O(N³)
- Counting for each city: O(N²)
- Total: **O(N³)**

**Space Complexity:** `O(N²)`

- Distance matrix: O(N²)
- No additional significant space
- Total: **O(N²)**

```
Detailed breakdown:

Floyd-Warshall phase:
→ 3 nested loops: k, i, j (each N iterations)
→ O(N³) operations

Counting phase:
→ For each city (N iterations)
→ Check each other city (N iterations)
→ O(N²) operations

Total: O(N³) + O(N²) = O(N³)

Space:
→ dist matrix: N × N = O(N²)
→ Constant extra variables
→ No recursion, no stack
→ Total: O(N²)

Comparison with alternatives:

Approach              Time            Space
─────────────────────────────────────────────
Floyd-Warshall        O(N³)           O(N²)
Dijkstra N times      O(N²log N)      O(N²)  [sparse]
                      O(N³log N)      O(N²)  [dense]

For small N (typical in this problem):
→ Floyd-Warshall is preferred ✓
→ Simpler code
→ Better constants (no log factor overhead)
```

## Interview Questions

```
Q1: Why use Floyd-Warshall for this problem?

A: We need shortest distances from every city to every other city.
   Floyd-Warshall gives all of them in one go with just three
   nested loops. It's like photocopying all pages at once instead
   of copying them one by one. For small N, it's the simplest way.

──────────────────────────────────────────────────────────────

Q2: Why use <= instead of <?

A: We want the BIGGEST city number when there's a tie. Using <=
   means "if same count, update anyway." Since we check cities
   0, 1, 2... in order, the last city with the minimum count wins.
   Using < would stop at the first one and miss later ties.

──────────────────────────────────────────────────────────────

Q3: How do we prevent overflow?

A: Before adding two distances, check if either one is infinity.
   Adding two huge numbers breaks the calculator. We skip those
   pairs to stay safe.

──────────────────────────────────────────────────────────────

Q4: When is Dijkstra N times better?

A: When the graph has very few roads (sparse) and N is large.
   For small dense graphs, Floyd-Warshall is simpler and often
   faster in practice. It's a trade-off between code simplicity
   and speed.

──────────────────────────────────────────────────────────────

Q5: Does the count include the city itself?

A: Usually yes, because distance from a city to itself is 0, which
   is always within the threshold. But it doesn't matter for the
   final answer since every city adds the same +1. The winner is
   the same either way.

──────────────────────────────────────────────────────────────

Q6: What if the graph were directed?

A: Only add the road in one direction during setup. Floyd-Warshall
   handles directed graphs naturally. The rest of the code stays
   exactly the same.

──────────────────────────────────────────────────────────────

Q7: Multiple roads between the same cities?

A: Keep only the shortest road. The longer one can never be part
   of a shortest path. Like having two flights — you always pick
   the cheaper one.

──────────────────────────────────────────────────────────────

Q8: Can we use BFS?

A: Only if all roads have the same length. BFS can't handle
   different weights. Here roads have different lengths, so we
   need Floyd-Warshall or Dijkstra.

──────────────────────────────────────────────────────────────

Q9: What if all cities have the same count?

A: This happens in very symmetric graphs where every city can
   reach the same number of neighbors. The tie-breaker then
   simply returns the largest city number (n-1).

──────────────────────────────────────────────────────────────

Q10: What if each city had its own threshold?

A: We would still run Floyd-Warshall once to get all distances.
   Then during counting, we use each city's own threshold instead
   of one global number. The overall work stays the same.

──────────────────────────────────────────────────────────────

Q11: How to find K cities with smallest counts?

A: After counting, sort the cities by count. If two have the same
    count, put the larger city number first. Then pick the first
    K from the sorted list.

──────────────────────────────────────────────────────────────

Q12: Why use INT_MAX for infinity?

A: It means "unreachable." It must be bigger than any real path.
   But adding two INT_MAXs overflows, so we check before adding.
   An alternative is 1e9, which is big but safe to add.

──────────────────────────────────────────────────────────────

Q13: Real-world uses?

A: Common uses include:
    → Finding an isolated data center (fewest nearby servers)
    → Placing a hospital where it covers the fewest other towns
    → Choosing a warehouse location with limited delivery range
    → Finding the quietest spot in a social network

──────────────────────────────────────────────────────────────

Q14: What if roads change after we compute?

A: Floyd-Warshall is expensive to re-run. For small changes, you
    might update only the affected parts. For many changes, use
    more advanced algorithms designed for dynamic graphs.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

    Mistake 1: Using < instead of <=
    → Fix: Use <= so larger city numbers win ties

    Mistake 2: Forgetting roads go both ways
    → Fix: Add dist[u][v] and dist[v][u]

    Mistake 3: Overflow when adding big numbers
    → Fix: Check for infinity before adding

    Mistake 4: Forgetting diagonal = 0
    → Fix: Set dist[i][i] = 0

    Mistake 5: Starting minCount at 0
    → Fix: Start at n (the maximum possible)

    Mistake 6: Mixing up city index and count
    → Fix: Track minCount and resultCity separately

    Mistake 7: Using BFS for weighted roads
    → Fix: Use Floyd-Warshall or Dijkstra
```
