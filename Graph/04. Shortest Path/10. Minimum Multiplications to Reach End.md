## 1. Problem Statement

Given `start`, `end`, and an array `arr` of `n` numbers. At each step, `start` is multiplied by any number in the array, then a **mod 100000** operation is performed to get the new start.

**Find:** The minimum number of steps to reach `end` from `start`. If impossible, return `-1`.

**Input Format:**

- `arr` = array of multipliers
- `start` = starting number
- `end` = target number

**Output:**

- Minimum number of multiplications to reach `end`, or `-1` if unreachable

## Key Insight

```
How is this a graph problem?

Think of each number (0 to 99999) as a NODE.

From any node 'num', we can reach:

→ (num × arr[0]) % 100000
→ (num × arr[1]) % 100000
→ ...
→ (num × arr[k]) % 100000

These are the EDGES!

Graph properties:

→ Nodes: All numbers 0 to 99999
→ Edges: num → (num × arr[i]) % 100000
→ Edge weight: 1 (each multiplication = 1 step)
→ Start node: start
→ End node: end

Goal: Find shortest path from start to end!

Since all edge weights = 1:

→ Use BFS (not Dijkstra) ✓
→ BFS finds shortest path in unweighted graphs
→ Simpler and faster than Dijkstra!
```

---

## Intuition

```
Imagine a combination lock with 100000 possible states (0 to 99999):

→ Current state: start
→ Available operations: multiply by any number in arr, then mod 100000
→ Target state: end
→ Find minimum number of operations

Why BFS?

BFS explores states level by level:
→ Level 0: start (0 steps)
→ Level 1: all numbers reachable in 1 step
→ Level 2: all numbers reachable in 2 steps
→ ...

The first time we reach 'end', we have used the minimum
number of steps! ✓

Why mod 100000?

Without mod, numbers grow exponentially:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ... → ∞

With mod, numbers wrap around in range [0, 99999]:
→ Finite search space!
→ Only 100000 possible nodes
→ BFS guaranteed to terminate ✓
```

## Algorithm

```
minimumMultiplications(arr, start, end):

1. Initialize:
   - Queue: {node, steps}
   - Distance array: dist[100000] = infinity
   - dist[start] = 0
   - mod = 100000

2. Push start into queue: {start, 0}

3. BFS traversal:
   While queue is not empty:
     a. Pop {node, steps} from front
     
     b. For each multiplier in arr:
        newNum = (node × multiplier) % mod
        
        If steps + 1 < dist[newNum]:
          → Update: dist[newNum] = steps + 1
          
          → If newNum == end:
            RETURN steps + 1 ✓ (found answer!)
          
          → Else: Push {newNum, steps + 1}

4. If queue empties without reaching end:
   RETURN -1 (unreachable)
```

### Why BFS Works Here

```
All edges have weight = 1 (each step = 1 multiplication)

→ BFS explores level by level
→ Level k contains all nodes reachable in k steps
→ First time we reach 'end' = minimum steps! ✓

If edges had different weights:
→ Would need Dijkstra with priority queue
→ But here, simple queue is sufficient! ✓
```

### Early Termination Optimization

```
Check INSIDE the relaxation loop:

for (int multiplier : arr) {
    int num = (1LL * node * multiplier) % mod;
    if (steps + 1 < dist[num]) {
        dist[num] = steps + 1;
        if (num == end) return steps + 1;  ← HERE!
        q.push({num, steps + 1});
    }
}

Why check here instead of at queue pop?

→ Return IMMEDIATELY when end is reached
→ Don't wait for it to be popped from queue
→ Saves unnecessary iterations! ✓
```

### Distance Array as Smart Visited Set

```
Why check steps + 1 < dist[num]?

Prevents:
1. Re-processing same number
2. Processing worse paths
3. Cycles in the graph

Example:
3 → 6 (steps=1)
3 → 15 → ... → 6 (steps=5)

When we reach 6 second time:
steps=5, but dist[6]=1
5 + 1 < 1? NO → Skip! ✓

Acts like a "visited" set but smarter!
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int minimumMultiplications(vector<int> &arr, int start, int end) {
        
        // Special case: already at end
        if (start == end) return 0;
        
        // Queue: {number, steps}
        queue<pair<int, int>> q;
        q.push({start, 0});
        
        // Distance array (min steps to reach each number)
        vector<int> dist(100000, 1e9);
        dist[start] = 0;
        
        int mod = 100000;
        
        // BFS traversal
        while (!q.empty()) {
            int node = q.front().first;
            int steps = q.front().second;
            q.pop();
            
            // Try all multipliers
            for (int multiplier : arr) {
                int num = (1LL * node * multiplier) % mod;
                
                // If found shorter path to num
                if (steps + 1 < dist[num]) {
                    dist[num] = steps + 1;
                    
                    // Reached end?
                    if (num == end) return steps + 1;
                    
                    // Continue exploring
                    q.push({num, steps + 1});
                }
            }
        }
        
        // End unreachable
        return -1;
    }
};

int main() {
    int start = 3, end = 30;
    vector<int> arr = {2, 5, 7};
    
    Solution obj;
    int ans = obj.minimumMultiplications(arr, start, end);
    
    cout << "Minimum steps: " << ans << endl;
    
    return 0;
}
```

### Modulo Operation Explained

```
Why mod 100000?

Without mod:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ... → ∞
Numbers grow exponentially, infinite search space!

With mod:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ...
After mod: numbers wrap around in range [0, 99999]

Example:
123456 % 100000 = 23456 (wraps around)
7 × 20000 = 140000 % 100000 = 40000

Search space:
→ Only 100000 possible numbers
→ Finite graph!
→ BFS guaranteed to terminate ✓
```

## Edge Cases

```
Start equals end:          Single multiplier:
start = 5, end = 5         arr = [1]
Return: 0                  Start can never change!
(already there)            Only reachable if start==end
                           Otherwise: -1

All multipliers = 1:       Large multipliers:
arr = [1, 1, 1]           arr = [99999]
Number never changes!      Creates large jumps
Return: 0 if start==end    Still works with mod ✓
       -1 otherwise

Cycle detection:           GCD considerations:
3 → 6 → 12 → 24 → ...     arr = [2, 4]
Eventually cycles back     start = 3, end = 7
due to mod 100000         Both 2 and 4 are even
dist[] prevents infinite  → All products are even
loop ✓                    → 7 (odd) unreachable! ✗

Zero in array:             Coprime to 100000:
arr = [0, 2, 5]          If gcd(multiplier, 100000) = 1
0 × anything = 0         Can potentially reach all numbers
Stuck at 0!              Example: arr = [3, 7, 11]
Might make end           (All coprime to 100000)
unreachable ✗
```

## Complexity Analysis

**Time Complexity:** `O(100000 × N)`

- Worst case: visit all 100000 numbers
- For each number, try N multipliers
- Total: **O(100000 × N)**
- In practice: much faster (early termination)

**Space Complexity:** `O(100000)`

- Distance array: O(100000)
- Queue: O(100000) in worst case
- Total: **O(100000)**

```
Detailed breakdown:

V = 100000 (total possible nodes/numbers)
E = V × N (each node has N outgoing edges)
N = |arr| (number of multipliers)

BFS time: O(V + E) = O(100000 + 100000×N) = O(100000×N)
BFS space: O(V) = O(100000)

Comparison with Dijkstra:

Dijkstra: O(E log V) = O(100000×N × log 100000)
BFS:      O(E)       = O(100000×N)

BFS is faster! ✓ (no log factor)

Why is 100000 acceptable?
→ Modern computers handle 10^5 operations easily
→ BFS is very efficient (simple queue operations)
→ Early termination makes it even faster in practice ✓
```

## Interview Questions

```
Q1: Why use BFS instead of Dijkstra for this problem?

A: All edges have equal weight (each multiplication costs 1 step).
   BFS is optimal for unweighted graphs and runs in O(V + E) time
   using a simple queue, whereas Dijkstra would require a priority
   queue with O(E log V) complexity. BFS is both simpler and faster
   when edge weights are uniform.

──────────────────────────────────────────────────────────────

Q2: Why do we need the modulo operation?

A: Without mod, numbers grow exponentially (e.g., 3 × 2^k → ∞),
   creating an infinite state space. Mod 100000 constrains the
   search space to exactly 100000 nodes (0 to 99999), making BFS
   guaranteed to terminate and keeping memory bounded.

──────────────────────────────────────────────────────────────

Q3: Why check steps + 1 < dist[num] instead of a simple visited set?

A: A simple visited set would mark a node as visited the first time
   we see it. But BFS explores level by level, so the first visit is
   already optimal. However, using dist[] is more robust: it handles
   cases where the same node might be reached through different paths
   and clearly prevents cycles by rejecting longer paths to already-
   visited nodes. It also makes the code pattern reusable for problems
   where edge weights might vary.

──────────────────────────────────────────────────────────────

Q4: Why use 1LL in the multiplication (1LL * node * multiplier)?

A: node and multiplier can each be up to 100000. Their product can
   reach 10^10, which exceeds INT_MAX (≈ 2.1 × 10^9). Using 1LL
   promotes the expression to long long before multiplication,
   preventing integer overflow. The modulo then brings the result
   safely back into the int range.

──────────────────────────────────────────────────────────────

Q5: When should we return -1?

A: Return -1 when the queue becomes empty without ever reaching the
   end node. This means all reachable states from start have been
   explored, and end is not among them. The finite state space
   (100000 nodes) guarantees BFS will terminate.

──────────────────────────────────────────────────────────────

Q6: Why is early termination (checking num == end inside the loop)
    valid? Could there be a shorter path discovered later?

A: No. BFS explores nodes in strictly increasing order of steps.
   Level k is fully processed before level k+1. When we first
   generate 'end' at steps + 1, no other path can have fewer steps
   because all paths with fewer steps have already been processed.
   Therefore, immediate return is correct and optimal.

──────────────────────────────────────────────────────────────

Q7: How does this problem relate to state-space search?

A: Each number modulo 100000 is a "state." Each multiplier defines
   a state transition. The problem asks for the shortest sequence
   of transitions from start state to end state. This is identical
   to finding the shortest path in an unweighted graph, which is
   exactly what BFS does.

──────────────────────────────────────────────────────────────

Q8: What happens if the array contains 0?

A: Multiplying by 0 always yields 0. Once we reach state 0, all
   subsequent states remain 0. If end != 0 and 0 is reached before
   end, that branch becomes a dead end. The dist[] array prevents
   reprocessing 0, so the algorithm still works correctly, but 0
   in the array can reduce reachability.

──────────────────────────────────────────────────────────────

Q9: Can we use a boolean visited array instead of dist[]?

A: Yes, for this specific problem we could use a boolean visited[]
   array because BFS guarantees the first visit is the shortest.
   However, dist[] is preferred because:
   1. It explicitly stores the distance (useful for debugging)
   2. It generalizes to weighted graphs (Dijkstra)
   3. It clearly shows the "relaxation" pattern common in shortest
      path algorithms

──────────────────────────────────────────────────────────────

Q10: What is the maximum number of steps in the worst case?

A: In the worst case, we might visit all 100000 states before
    reaching end or determining it's unreachable. The path length
    is at most 99999 steps (since we can't revisit a state with
    a shorter or equal distance).

──────────────────────────────────────────────────────────────

Q11: How would you solve this if each multiplier had a different cost?

A: If multipliers had different costs, BFS would no longer be
    sufficient. We would need Dijkstra's algorithm with a priority
    queue, where the edge weight from node u to v depends on which
    multiplier was used. The dist[] update logic would remain
    similar, but we'd use a min-heap ordered by total cost.

──────────────────────────────────────────────────────────────

Q12: Why is the state space exactly 100000 and not larger?

A: The modulo operation (mod 100000) maps every integer result
    into the range [0, 99999]. There are exactly 100000 distinct
    residues modulo 100000. Once two operations produce the same
    residue, they represent the same state, and further operations
    from that state will follow identical paths.

──────────────────────────────────────────────────────────────

Q13: What if start and end are the same? Should we return 0 or
     perform at least one multiplication?

A: Return 0. Zero steps are needed because we are already at the
    target. This is handled by the early check: if (start == end)
    return 0.

──────────────────────────────────────────────────────────────

Q14: How does this algorithm handle cycles?

A: Cycles are naturally handled by the dist[] array. If we reach
    a node that we've already visited with fewer or equal steps,
    we skip it. Since the state space is finite (100000 nodes),
    and we never revisit a node with a better or equal distance,
    the algorithm cannot loop infinitely.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes in implementing this?

A: Common mistakes and fixes:

    Mistake 1: Integer overflow in multiplication
    → Use (1LL * node * multiplier) % mod

    Mistake 2: Forgetting to handle start == end
    → Add early check: if (start == end) return 0

    Mistake 3: Using a visited set and skipping valid re-visits
    → Use dist[] with < comparison instead

    Mistake 4: Not applying modulo immediately
    → Apply % mod right after multiplication

    Mistake 5: Returning from queue pop instead of generation
    → Check inside the loop for early termination

    Mistake 6: Wrong array size (99999 instead of 100000)
    → Use size 100000 for indices 0 through 99999
```
