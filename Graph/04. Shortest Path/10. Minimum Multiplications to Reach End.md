## 1. Problem Statement

Given `start`, `end`, and an array `arr` of `n` numbers. At each step, `start` is multiplied by any number in the array, then a **mod 100000** operation is performed to get the new start.

**Find:** The minimum number of steps to reach `end` from `start`. If impossible, return `-1`.

**Input Format:**

- `arr` = array of multipliers
- `start` = starting number
- `end` = target number

**Output:**

- Minimum number of multiplications to reach `end`, or `-1` if unreachable

## Key Insight

```
How is this a graph problem?

Think of each number (0 to 99999) as a NODE.

From any node 'num', we can reach:

→ (num × arr[0]) % 100000
→ (num × arr[1]) % 100000
→ ...
→ (num × arr[k]) % 100000

These are the EDGES!

Graph properties:

→ Nodes: All numbers 0 to 99999
→ Edges: num → (num × arr[i]) % 100000
→ Edge weight: 1 (each multiplication = 1 step)
→ Start node: start
→ End node: end

Goal: Find shortest path from start to end!

Since all edge weights = 1:

→ Use BFS (not Dijkstra) ✓
→ BFS finds shortest path in unweighted graphs
→ Simpler and faster than Dijkstra!
```

---

## Intuition

```
Imagine a combination lock with 100000 possible states (0 to 99999):

→ Current state: start
→ Available operations: multiply by any number in arr, then mod 100000
→ Target state: end
→ Find minimum number of operations

Why BFS?

BFS explores states level by level:
→ Level 0: start (0 steps)
→ Level 1: all numbers reachable in 1 step
→ Level 2: all numbers reachable in 2 steps
→ ...

The first time we reach 'end', we have used the minimum
number of steps! ✓

Why mod 100000?

Without mod, numbers grow exponentially:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ... → ∞

With mod, numbers wrap around in range [0, 99999]:
→ Finite search space!
→ Only 100000 possible nodes
→ BFS guaranteed to terminate ✓
```

## Algorithm

```
minimumMultiplications(arr, start, end):

1. Initialize:
   - Queue: {node, steps}
   - Distance array: dist[100000] = infinity
   - dist[start] = 0
   - mod = 100000

2. Push start into queue: {start, 0}

3. BFS traversal:
   While queue is not empty:
     a. Pop {node, steps} from front
     
     b. For each multiplier in arr:
        newNum = (node × multiplier) % mod
        
        If steps + 1 < dist[newNum]:
          → Update: dist[newNum] = steps + 1
          
          → If newNum == end:
            RETURN steps + 1 ✓ (found answer!)
          
          → Else: Push {newNum, steps + 1}

4. If queue empties without reaching end:
   RETURN -1 (unreachable)
```

### Why BFS Works Here

```
All edges have weight = 1 (each step = 1 multiplication)

→ BFS explores level by level
→ Level k contains all nodes reachable in k steps
→ First time we reach 'end' = minimum steps! ✓

If edges had different weights:
→ Would need Dijkstra with priority queue
→ But here, simple queue is sufficient! ✓
```

### Early Termination Optimization

```
Check INSIDE the relaxation loop:

for (int multiplier : arr) {
    int num = (1LL * node * multiplier) % mod;
    if (steps + 1 < dist[num]) {
        dist[num] = steps + 1;
        if (num == end) return steps + 1;  ← HERE!
        q.push({num, steps + 1});
    }
}

Why check here instead of at queue pop?

→ Return IMMEDIATELY when end is reached
→ Don't wait for it to be popped from queue
→ Saves unnecessary iterations! ✓
```

### Distance Array as Smart Visited Set

```
Why check steps + 1 < dist[num]?

Prevents:
1. Re-processing same number
2. Processing worse paths
3. Cycles in the graph

Example:
3 → 6 (steps=1)
3 → 15 → ... → 6 (steps=5)

When we reach 6 second time:
steps=5, but dist[6]=1
5 + 1 < 1? NO → Skip! ✓

Acts like a "visited" set but smarter!
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int minimumMultiplications(vector<int> &arr, int start, int end) {
        
        // Special case: already at end
        if (start == end) return 0;
        
        // Queue: {number, steps}
        queue<pair<int, int>> q;
        q.push({start, 0});
        
        // Distance array (min steps to reach each number)
        vector<int> dist(100000, 1e9);
        dist[start] = 0;
        
        int mod = 100000;
        
        // BFS traversal
        while (!q.empty()) {
            int node = q.front().first;
            int steps = q.front().second;
            q.pop();
            
            // Try all multipliers
            for (int multiplier : arr) {
                int num = (1LL * node * multiplier) % mod;
                
                // If found shorter path to num
                if (steps + 1 < dist[num]) {
                    dist[num] = steps + 1;
                    
                    // Reached end?
                    if (num == end) return steps + 1;
                    
                    // Continue exploring
                    q.push({num, steps + 1});
                }
            }
        }
        
        // End unreachable
        return -1;
    }
};

int main() {
    int start = 3, end = 30;
    vector<int> arr = {2, 5, 7};
    
    Solution obj;
    int ans = obj.minimumMultiplications(arr, start, end);
    
    cout << "Minimum steps: " << ans << endl;
    
    return 0;
}
```

### Modulo Operation Explained

```
Why mod 100000?

Without mod:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ... → ∞
Numbers grow exponentially, infinite search space!

With mod:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ...
After mod: numbers wrap around in range [0, 99999]

Example:
123456 % 100000 = 23456 (wraps around)
7 × 20000 = 140000 % 100000 = 40000

Search space:
→ Only 100000 possible numbers
→ Finite graph!
→ BFS guaranteed to terminate ✓
```

## Edge Cases

```
Start equals end:          Single multiplier:
start = 5, end = 5         arr = [1]
Return: 0                  Start can never change!
(already there)            Only reachable if start==end
                           Otherwise: -1

All multipliers = 1:       Large multipliers:
arr = [1, 1, 1]           arr = [99999]
Number never changes!      Creates large jumps
Return: 0 if start==end    Still works with mod ✓
       -1 otherwise

Cycle detection:           GCD considerations:
3 → 6 → 12 → 24 → ...     arr = [2, 4]
Eventually cycles back     start = 3, end = 7
due to mod 100000         Both 2 and 4 are even
dist[] prevents infinite  → All products are even
loop ✓                    → 7 (odd) unreachable! ✗

Zero in array:             Coprime to 100000:
arr = [0, 2, 5]          If gcd(multiplier, 100000) = 1
0 × anything = 0         Can potentially reach all numbers
Stuck at 0!              Example: arr = [3, 7, 11]
Might make end           (All coprime to 100000)
unreachable ✗
```

## Complexity Analysis

**Time Complexity:** `O(100000 × N)`

- Worst case: visit all 100000 numbers
- For each number, try N multipliers
- Total: **O(100000 × N)**
- In practice: much faster (early termination)

**Space Complexity:** `O(100000)`

- Distance array: O(100000)
- Queue: O(100000) in worst case
- Total: **O(100000)**

```
Detailed breakdown:

V = 100000 (total possible nodes/numbers)
E = V × N (each node has N outgoing edges)
N = |arr| (number of multipliers)

BFS time: O(V + E) = O(100000 + 100000×N) = O(100000×N)
BFS space: O(V) = O(100000)

Comparison with Dijkstra:

Dijkstra: O(E log V) = O(100000×N × log 100000)
BFS:      O(E)       = O(100000×N)

BFS is faster! ✓ (no log factor)

Why is 100000 acceptable?
→ Modern computers handle 10^5 operations easily
→ BFS is very efficient (simple queue operations)
→ Early termination makes it even faster in practice ✓
```

## Interview Questions

```
Q1: Why use BFS instead of Dijkstra?

A: Every multiplication costs exactly 1 step. BFS explores level by
   level — level 0 is start, level 1 is 1 step away, level 2 is
   2 steps away, etc. The first time we reach the end, we know it
   is the shortest path. Dijkstra is like using a calculator for
   1+1 — it works but is unnecessary.

──────────────────────────────────────────────────────────────

Q2: Why do we need mod 100000?

A: Without mod, numbers grow forever (3 → 6 → 12 → 24 → ...).
   We would have infinite states to search. Mod keeps numbers
   between 0 and 99999, so we only have 100000 possible states.
   Think of it like a clock — after 12, it goes back to 1.

──────────────────────────────────────────────────────────────

Q3: Why use dist[] instead of a simple visited set?

A: A visited set says "never come back." But dist[] says "only
   come back if you found a shorter path." In BFS, the first
   visit is already the shortest, so both work here. But dist[]
   is better practice and helps when you learn harder problems.

──────────────────────────────────────────────────────────────

Q4: Why write 1LL * node * multiplier?

A: node and multiplier can be as big as 100000. Their product can
   reach 10 billion, which does not fit in a normal int. 1LL turns
   the calculation into "long long" first, so we don't overflow.
   The mod then brings it back to a safe range.

──────────────────────────────────────────────────────────────

Q5: When do we return -1?

A: When the queue becomes empty and we never reached the end.
   This means we searched every possible state and the target is
   unreachable. Like checking every room in a house and not finding
   the key.

──────────────────────────────────────────────────────────────

Q6: Why can we return immediately when we find the end?

A: BFS checks all 1-step paths before any 2-step path, all 2-step
   before 3-step, and so on. So the first time we reach the end,
   no shorter path can exist. We can stop right away.

──────────────────────────────────────────────────────────────

Q7: How is this a graph problem?

A: Think of each number (0 to 99999) as a room. Each multiplier is
   a door that takes you to another room. We want the shortest
   route from the start room to the end room. That is exactly what
   BFS does — it finds the shortest path in an unweighted graph.

──────────────────────────────────────────────────────────────

Q8: What happens if the array contains 0?

A: Multiplying by 0 always gives 0. Once you reach 0, you are stuck
   there forever. It is like a black hole — once you fall in, you
   can never leave. If the end is not 0, then 0 is a dead end.

──────────────────────────────────────────────────────────────

Q9: Can we use a boolean visited array?

A: Yes, for this problem it would work fine. But dist[] is better
   because it also remembers how many steps each state took. This
   is useful for debugging and for learning more advanced algorithms.

──────────────────────────────────────────────────────────────

Q10: What is the maximum number of steps in the worst case?

A: At most 99999 steps. There are only 100000 states, and we never
   revisit a state with a shorter or equal distance. So we can't
   loop forever.

──────────────────────────────────────────────────────────────

Q11: What if each multiplier had a different cost?

A: Then we would need Dijkstra instead of BFS. BFS only works when
   every move costs the same. If some moves cost 2 and others cost
   5, we need Dijkstra's priority queue to always pick the cheapest
   next move.

──────────────────────────────────────────────────────────────

Q12: Why is the state space exactly 100000?

A: Mod 100000 gives remainders from 0 to 99999. That is exactly
   100000 possible numbers. No matter how big the multiplication
   result is, mod wraps it into this range. So we only ever need
   to track 100000 states.

──────────────────────────────────────────────────────────────

Q13: What if start and end are the same?

A: Return 0. You are already at the target, so you need zero steps.
   Always handle this case first to avoid unnecessary work.

──────────────────────────────────────────────────────────────

Q14: How does the algorithm handle cycles?

A: The dist[] array prevents infinite loops. If we return to the
   same number with more steps than before, we simply skip it.
   Since there are only 100000 states and we don't revisit with
   worse paths, the algorithm always finishes.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

    Mistake 1: Forgetting 1LL and getting overflow
    → Fix: Use (1LL * node * multiplier) % mod

    Mistake 2: Not handling start == end
    → Fix: Add if (start == end) return 0;

    Mistake 3: Using a visited set and skipping valid paths
    → Fix: Use dist[] with < comparison

    Mistake 4: Forgetting to apply mod
    → Fix: Apply % mod right after multiplication

    Mistake 5: Wrong array size (99999 instead of 100000)
    → Fix: Use size 100000 for indices 0 through 99999

    Mistake 6: Checking for end only when popping from queue
    → Fix: Check immediately when generating the new state
```
