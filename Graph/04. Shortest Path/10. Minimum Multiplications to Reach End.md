## Minimum Multiplications to Reach End

### Problem Statement:

Given `start`, `end`, and an array `arr` of `n` numbers. At each step, `start` is multiplied by any number in the array, then a **mod 100000** operation is performed to get the new start.

**Find:** The minimum number of steps to reach `end` from `start`. If impossible, return `-1`.

---

## Examples

**Example 1:**

**Input:**
```
arr = [2, 5, 7]
start = 3
end = 30
```

**Output:** `2`

```
Step-by-step:
Step 0: start = 3
Step 1: 3 × 2 = 6 % 100000 = 6
Step 2: 6 × 5 = 30 % 100000 = 30 ✓ (reached end!)

Minimum steps = 2 ✓
```

**Example 2:**

**Input:**
```
arr = [3, 4, 65]
start = 7
end = 66175
```

**Output:** `4`

```
Step-by-step:
Step 0: start = 7
Step 1: 7 × 3 = 21 % 100000 = 21
Step 2: 21 × 3 = 63 % 100000 = 63
Step 3: 63 × 65 = 4095 % 100000 = 4095
Step 4: 4095 × 65 = 266175 % 100000 = 66175 ✓

Minimum steps = 4 ✓
```

**Example 3:**

**Input:**
```
arr = [2]
start = 3
end = 7
```

**Output:** `-1`

```
Possible values from 3:
3 → 6 → 12 → 24 → 48 → 96 → ...
All are even numbers (multiples of 2)
7 is odd → unreachable!

Return -1 ✗
```

## Key Insight

```
How is this a graph problem?

Think of each number (0 to 99999) as a NODE.
From any node 'num', we can reach:
→ (num × arr[0]) % 100000
→ (num × arr[1]) % 100000
→ ...
→ (num × arr[k]) % 100000

These are the EDGES!

Graph properties:
→ Nodes: All numbers 0 to 99999
→ Edges: num → (num × arr[i]) % 100000
→ Edge weight: 1 (each multiplication = 1 step)
→ Start node: start
→ End node: end

Goal: Find shortest path from start to end!

Since all edge weights = 1:
→ Use BFS (not Dijkstra) ✓
→ BFS finds shortest path in unweighted graphs
→ Simpler and faster than Dijkstra!
```

---

## Algorithm

```
1. Initialize:
   - Queue: {node, steps}
   - Distance array: dist[100000] = infinity
   - dist[start] = 0
   - mod = 100000

2. Push start into queue: {start, 0}

3. BFS traversal:
   While queue is not empty:
     a. Pop {node, steps} from front
     
     b. For each multiplier in arr:
        newNum = (node × multiplier) % mod
        
        If steps + 1 < dist[newNum]:
          → Update: dist[newNum] = steps + 1
          
          → If newNum == end:
            RETURN steps + 1 ✓ (found answer!)
          
          → Else: Push {newNum, steps + 1}

4. If queue empties without reaching end:
   RETURN -1 (unreachable)
```

```
Why BFS works here:

All edges have weight = 1 (each step = 1 multiplication)
→ BFS explores level by level
→ Level k contains all nodes reachable in k steps
→ First time we reach 'end' = minimum steps! ✓

If edges had different weights:
→ Would need Dijkstra with priority queue
→ But here, simple queue is sufficient! ✓

Why mod 100000?
→ Without mod, numbers grow exponentially
→ 3 × 2^k grows to infinity quickly
→ Mod constrains search space to 100000 nodes
→ Makes problem tractable! ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int minimumMultiplications(vector<int> &arr, int start, int end) {
        
        // Special case: already at end
        if (start == end) return 0;
        
        // Queue: {number, steps}
        queue<pair<int, int>> q;
        q.push({start, 0});
        
        // Distance array (min steps to reach each number)
        vector<int> dist(100000, 1e9);
        dist[start] = 0;
        
        int mod = 100000;
        
        // BFS traversal
        while (!q.empty()) {
            int node = q.front().first;
            int steps = q.front().second;
            q.pop();
            
            // Try all multipliers
            for (int multiplier : arr) {
                int num = (1LL * node * multiplier) % mod;
                
                // If found shorter path to num
                if (steps + 1 < dist[num]) {
                    dist[num] = steps + 1;
                    
                    // Reached end?
                    if (num == end) return steps + 1;
                    
                    // Continue exploring
                    q.push({num, steps + 1});
                }
            }
        }
        
        // End unreachable
        return -1;
    }
};

int main() {
    int start = 3, end = 30;
    vector<int> arr = {2, 5, 7};
    
    Solution obj;
    int ans = obj.minimumMultiplications(arr, start, end);
    
    cout << "Minimum steps: " << ans << endl;
    
    return 0;
}
```

## Modulo Operation Explained

```
Why mod 100000?

Without mod:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ... → ∞
Numbers grow exponentially, infinite search space!

With mod:
3 → 6 → 12 → 24 → 48 → 96 → 192 → ...
After mod: numbers wrap around in range [0, 99999]

Example:
123456 % 100000 = 23456 (wraps around)
7 × 20000 = 140000 % 100000 = 40000

Search space:
→ Only 100000 possible numbers
→ Finite graph!
→ BFS guaranteed to terminate ✓

Edge case consideration:
num = (1LL * node * multiplier) % mod
     ↑
Uses long long to prevent overflow before mod!
node × multiplier can exceed int range
```

## Distance Array Updates

```
Initial:         After {3,0}:      After {6,1}:
dist[3] = 0      dist[3] = 0       dist[3] = 0
all others = ∞   dist[6] = 1       dist[6] = 1
                 dist[15] = 1      dist[12] = 2
                 dist[21] = 1      dist[15] = 1
                                   dist[21] = 1
                                   dist[30] = 2 ← FOUND!

Why check steps + 1 < dist[num]?

Prevents:
1. Re-processing same number
2. Processing worse paths
3. Cycles in the graph

Example:
3 → 6 (steps=1)
3 → 15 → ... → 6 (steps=5)

When we reach 6 second time:
steps=5, but dist[6]=1
5 + 1 < 1? NO → Skip! ✓

Acts like a "visited" set but smarter!
```

## Why Not Dijkstra?

```
Dijkstra vs BFS for this problem:

BFS (used here):
→ Simple queue
→ O(V + E) time
→ Works because edge weight = 1
→ Processes nodes level by level
→ Simpler code ✓

Dijkstra (unnecessary):
→ Priority queue (min-heap)
→ O(E log V) time
→ Designed for varying edge weights
→ Overkill for unit weights
→ More complex

Example showing equivalence:
All edges weight = 1
BFS level k = Dijkstra distance k
Same result, BFS is faster! ✓

When to use Dijkstra instead?
→ If different multipliers had different "costs"
→ e.g., arr = [(2, cost=1), (5, cost=3), ...]
→ Then Dijkstra would be needed
```

## Early Termination Optimization

```
Code includes important optimization:

if (num == end) return steps + 1;

This is INSIDE the loop:
for (int multiplier : arr) {
    int num = (1LL * node * multiplier) % mod;
    if (steps + 1 < dist[num]) {
        dist[num] = steps + 1;
        if (num == end) return steps + 1;  ← HERE!
        q.push({num, steps + 1});
    }
}

Why check here instead of at queue pop?

Benefit: Return IMMEDIATELY when end is reached
→ Don't wait for it to be popped from queue
→ Saves unnecessary iterations! ✓

Example:
Queue: [{6,1}, {15,1}, {21,1}]
Processing 6, discover 30 at steps=2
→ RETURN 2 immediately ✓
→ Don't process {15,1} and {21,1} ✗

Alternative (less optimal):
Check at queue pop:
while (!q.empty()) {
    auto [node, steps] = q.front();
    q.pop();
    if (node == end) return steps;  ← Later
    ...
}

Both correct, but first is faster! ✓
```

## Overflow Prevention

```
Critical line:
int num = (1LL * node * multiplier) % mod;
          ↑
          Why 1LL?

Without 1LL:
int num = (node * multiplier) % mod;  ✗

Problem:
node = 50000, multiplier = 50000
node × multiplier = 2,500,000,000
→ Exceeds INT_MAX (2,147,483,647) ✗
→ Integer overflow → undefined behavior!

With 1LL:
1LL × node × multiplier
→ Promotes to long long before multiplication
→ 2,500,000,000 fits in long long ✓
→ Then apply % mod
→ Result fits back in int ✓

Alternative approaches:
long long num = ((long long)node * multiplier) % mod;
long long num = (1LL * node * multiplier) % mod;  ← Cleaner
```

## Complexity Analysis

**Time Complexity:** `O(100000 × N)`
- Worst case: visit all 100000 numbers
- For each number, try N multipliers
- Total: **O(100000 × N)**
- In practice: much faster (early termination)

**Space Complexity:** `O(100000)`
- Distance array: O(100000)
- Queue: O(100000) in worst case
- Total: **O(100000)**

```
Detailed breakdown:

V = 100000 (total possible nodes/numbers)
E = V × N (each node has N outgoing edges)
N = |arr| (number of multipliers)

BFS time: O(V + E) = O(100000 + 100000×N) = O(100000×N)
BFS space: O(V) = O(100000)

Comparison with Dijkstra:
Dijkstra: O(E log V) = O(100000×N × log 100000)
BFS:      O(E)       = O(100000×N)

BFS is faster! ✓ (no log factor)

Why is 100000 acceptable?
→ Modern computers handle 10^5 operations easily
→ BFS is very efficient (simple queue operations)
→ Early termination makes it even faster in practice ✓
```

## When to Use This Pattern

```
✓ Use BFS with modulo state space when:
  → Operations generate numbers with modulo
  → Need minimum steps to reach target
  → State space is finite (due to modulo)
  → All operations have equal cost (weight = 1)

✗ Don't use when:
  → No modulo constraint → Use standard BFS/Dijkstra
  → Operations have different costs → Use Dijkstra
  → Need all paths, not just shortest → Use DFS/backtracking

Similar problems:
→ Minimum Multiplications to Reach End (this problem) ✓
→ Word Ladder (transform word to another)
→ Open the Lock (minimum rotations)
→ Perfect Squares (minimum squares to sum to n)
→ Coin Change (minimum coins to make amount)

Key pattern:
"Minimum steps with state transformation + constraint" = BFS! ✓

Pattern variations:
→ With modulo → Finite state space
→ Without modulo → Might need pruning
→ 2D states → Grid BFS
→ Multiple constraints → Multi-dimensional state
```
