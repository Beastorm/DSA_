## 1. Problem Statement

Given a **weighted, undirected graph** with `V` vertices and `E` edges, find the **shortest distance from source node `src` to ALL other vertices**.

**Input Format:**
- `V` = number of vertices
- `E` = number of edges
- `src` = source vertex
- Edge list (u, v, w) for each edge

**Output:**
- Array of shortest distances from source to all vertices
- `-1` for unreachable vertices

---

## Examples

**Example 1:**

- **Input:** `V=3, E=3, src=2`
- **Edges:** `0-1(1), 0-2(6), 1-2(3)`
- **Output:** `[6, 3, 0]`

```

       1
  0 ──────── 1
  │ ╲         │
  6   3      3
  │     ╲     │
  └────── 2 ─┘

  Dist from 2:
  → 0: 6 (direct) ✓
  → 1: 3 (direct) ✓
  → 2: 0 (itself) ✓

  Result: [6, 3, 0]

```

**Example 2:**

- **Input:** `V=5, E=6, src=0`
- **Output:** `[0, 2, 4, 1, 5]`

```

  0 ──2── 1
  │      │
  1      5
  │      │
  4 ──3── 2
  │    ╱
  └───1───

  Dist from 0:
  → 0: 0 (itself) ✓
  → 1: 2 (direct) ✓
  → 2: 4 (via 4→2) ✓
  → 3: 1 (direct) ✓
  → 4: 5 (via 0→1→3→4 or 0→3→4) ✓

  Result: [0, 2, 4, 1, 5]

```

## Key Insight

```

  Problem: Find SHORTEST DISTANCE from src to ALL vertices.

  BFS works for UNIT weights only (counts hops).

  For weighted graphs: use DIJKSTRA with MIN-HEAP!

  Core idea:
  Always process the node with the SMALLEST current distance.
  → Greedy approach: once a node's shortest distance is found, it's FINAL.

  Min-heap (priority queue) efficiently gives us the
  minimum distance node in O(log V) time.

  Relaxation:
  If dist[u] + weight(u→v) < dist[v]:
      dist[v] ← dist[u] + weight(u→v)  (better path found!)
      Push (dist[v], v) to heap

```

---

## Intuition

```

  Imagine you're a radio signal spreading from a source:

  1. Signal starts at source with strength 0
  2. The signal ALWAYS travels to the NEAREST unvisited city first
  3. From that city, it tries to reach new cities — if going through
     this city gives a faster route, update that city's distance!
  4. Keep spreading until ALL cities are reached

  Why it works:
  When a node is removed from the min-heap, no shorter path to it
  can exist through any unprocessed node because all remaining nodes
  have distance at least as large. So its distance is LOCKED IN!

```

---

## Algorithm

```

  dijkstra(V, adj, src):

  1. dist[] = all infinity
     dist[src] = 0

  2. Min-heap pq = {(0, src)}

  3. While pq not empty:
       (d, node) = pq.top(); pop

       If d > dist[node]: SKIP  ← outdated/stale entry

       For each (neighbor, wt) in adj[node]:
           If dist[node] + wt < dist[neighbor]:
               dist[neighbor] = dist[node] + wt
               pq.push({dist[neighbor], neighbor})

  4. Return dist[]  ← Contains shortest distance from src to ALL nodes

```

```

  Why skip outdated entries?

  We might push (5, A) then later find shorter path (3, A).
  Both entries exist in the heap.

  When we pop (3, A): process it, dist[A]=3
  Later pop (5, A): 5 > dist[A]=3 → SKIP!

  This "lazy deletion" avoids complex heap update operations.

```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

vector<int> shortest(vector<vector<int>> adj[], int V, int S) {
    // Min-heap priority queue storing {distance, node}
    priority_queue<pair<int, int>,
                   vector<pair<int, int>>,
                   greater<pair<int, int>>> minPQ;

    minPQ.push({0, S});  // Start from source with distance 0

    vector<int> dist(V, 1e9);  // Initialize all distances to infinity
    dist[S] = 0;               // Distance to source is 0

    while (!minPQ.empty()) {
        auto [d, node] = minPQ.top();
        minPQ.pop();

        // Skip outdated/stale entries (lazy deletion)
        if (d > dist[node])
            continue;

        // Explore all neighbors of current node
        for (auto &&i : adj[node]) {
            int wt = i[1];    // Edge weight
            int neigh = i[0]; // Neighbor node

            // Relaxation: found a shorter path!
            if (dist[node] + wt < dist[neigh]) {
                dist[neigh] = dist[node] + wt;
                minPQ.push({dist[neigh], neigh});
            }
        }
    }

    return dist;
}

int main(int argc, char const *argv[]) {
    int V = 3, E = 3, S = 2;

    // Create adjacency list
    vector<vector<int>> adj[V];

    // Add undirected edges: {neighbor, weight}
    adj[0].push_back({1, 1});
    adj[0].push_back({2, 6});
    adj[1].push_back({2, 3});
    adj[1].push_back({0, 1});
    adj[2].push_back({1, 3});
    adj[2].push_back({0, 6});

    vector<int> ans = shortest(adj, V, S);

    // Print shortest distances
    for (auto &&i : ans) {
        if (i == 1e9)
            cout << -1 << endl;  // Unreachable
        else
            cout << i << endl;   // Shortest distance
    }

    return 0;
}
```

## Step-by-Step Trace

```

  Graph: V=3, E=3
  Edges: 0-1(1), 0-2(6), 1-2(3)
  Source=2

  dist = [∞, ∞, 0]
  pq   = [(0, 2)]

  ── Pop (0, 2): process ──
  neighbor 1: 0+3=3 < ∞ → dist[1]=3, push (3,1)
  neighbor 0: 0+6=6 < ∞ → dist[0]=6, push (6,0)

  dist=[6,3,0], pq=[(3,1),(6,0)]

  ── Pop (3, 1): process ──
  neighbor 2: 3+3=6 > dist[0]=6 → skip (not <)
  neighbor 0: 3+1=4 < dist[0]=6 → dist[0]=4, push (4,0)

  dist=[4,3,0], pq=[(4,0),(6,0)]

  ── Pop (4, 0): process ──
  neighbor 1: 4+1=5 > dist[1]=3 → skip
  neighbor 2: 4+6=10 > dist[2]=0 → skip

  pq=[(6,0)]

  ── Pop (6, 0): d=6 > dist[0]=4 → SKIP (outdated)

  Final: dist=[4, 3, 0]

  Output: [6, 3, 0]  ← 0→6, 1→3, 2→0

```

## Edge Cases

```

  Source is isolated:
  Source has no edges → only dist[src]=0, rest = -1 ✓

  Disconnected graph:
  Nodes in different components remain at infinity
  → return -1 for unreachable nodes ✓

  Negative weights:
  → Use Bellman-Ford instead!
  Dijkstra assumes distances only decrease → fails with negative! ✓

  Single node, no edges:
  dist[src] = 0 ✓

  All edges have same weight:
  Dijkstra works, but BFS is faster (O(V+E) vs O((V+E) log V)) ✓

```

## Complexity Analysis

**Time Complexity:** O((V + E) log V)

- Each vertex's shortest distance is **finalized once**: **O(V log V)**
- Each successful relaxation inserts one heap entry: **O(E log V)**
  (at most O(E) successful relaxations possible)
- Every edge is scanned once from adjacency lists: **O(E)**
- Heap may contain stale entries: still O((V+E) log V)
- Total: **O((V + E) log V)**

**Space Complexity:** O(V + E)

- Adjacency list: **O(V + E)**
- Distance array: **O(V)**
- Priority queue:
  - Typical/optimized: **O(V)** (when removing old entries)
  - With lazy deletion: **O(E)** worst case (stale entries allowed)
- Total: **O(V + E)**

```

  Why O(E) for priority queue?
  - Each successful relaxation inserts ONE entry
  - Each edge can be relaxed at most once (when better path found)
  - Stale entries may be popped but don't add new entries
  - So at most O(E) successful insertions

  Comparison with other shortest path algorithms:

  | Algorithm        | Graph Type           | Time Complexity      |
  |------------------|----------------------|----------------------|
  | BFS              | Unit weight          | O(V + E)             |
  | Topo Sort        | DAG                  | O(V + E)             |
  | Dijkstra         | Non-negative wt      | O((V+E) log V)       |
  | Bellman-Ford     | Any weights          | O(V × E)             |
  | SPFA             | Any weights          | O(V × E) avg         |

  Dijkstra is optimal for non-negative weighted graphs ✓

```

## Interview Questions

```

  Q1: Why does Dijkstra's algorithm use a Min-Heap?

  A: To always extract the node with the smallest distance efficiently.
     Without a heap, we'd need O(V) to find minimum each time.
     With min-heap: O(log V) per extraction → total O((V+E) log V).


  ──────────────────────────────────────────────────────────────

  Q2: What happens if all edge weights are 1?

  A: BFS is the PREFERRED algorithm, but Dijkstra still works!

     BFS advantage:
     - Uses simple queue → O(V + E) time
     - No heap overhead

     Dijkstra with heap:
     - Still works correctly!
     - But complexity remains O((V+E) log V)
     - Overkill for unit weights

     Only if we replace heap with queue does Dijkstra become BFS.


  ──────────────────────────────────────────────────────────────

  Q3: Why can't Dijkstra handle negative weight edges?

  A: Because it assumes once a node's distance is finalized,
     it can never be improved. Classic counterexample:

       S → A (5)
       S → B (2)
       B → A (-4)

     Initially from S:
     dist[A] = 5 (direct)
     dist[B] = 2 (direct)

     Suppose A is popped first (dist=5). A is "finalized".
     Later, processing B (dist=2):
     dist[A] = 2 + (-4) = -2 < 5 ✗

     But A was already finalized! No way to update it.

     Negative weights can create SHORTER paths AFTER a node
     is processed → breaks Dijkstra's greedy correctness.


  ──────────────────────────────────────────────────────────────

  Q4: What is "relaxation" in Dijkstra's algorithm?

  A: Relaxation is the process of updating a node's distance:

       Before: dist[v] = 5
       Check:  dist[u] + weight(u→v) = 3 + 1 = 4
       After:  dist[v] = 4  (if 4 < 5)

     Think of it as "relaxing a tight rope" — we found a
     shorter path, so we loosen the distance value.


  ──────────────────────────────────────────────────────────────

  Q5: Why do we skip outdated/stale entries from the heap?

  A: Due to "lazy deletion" — when we find a shorter path to
     a node already in heap, we push new entry but don't
     remove the old one.

     Example:
     1. Push (5, A) to heap
     2. Later find shorter path: dist[A] = 3, push (3, A)
     3. Pop (3, A): process it (correct)
     4. Pop (5, A): 5 > dist[A]=3 → SKIP!

     This avoids expensive heap deletions at cost of
     extra entries (still O((V+E) log V) overall).


  ──────────────────────────────────────────────────────────────

  Q6: What is the space complexity?

  A: O(V + E) where:
     - Adjacency list: O(V + E)
     - Distance array: O(V)
     - Priority queue: O(E) with lazy deletion

     We use O(V + E) because:
     - Graph stored as adjacency list takes O(V + E)
     - PQ can have up to O(E) entries (one per successful relaxation)


  ──────────────────────────────────────────────────────────────

  Q7: Can Dijkstra's algorithm detect negative cycles?

  A: NO! Dijkstra cannot detect negative cycles.
     If a negative cycle exists, distances keep decreasing,
     but Dijkstra stops when all nodes are "finalized".

     Use Bellman-Ford to detect negative cycles:
     - Run V-1 iterations
     - If any distance updates in V-th iteration → negative cycle!


  ──────────────────────────────────────────────────────────────

  Q8: Why don't we use a visited[] array?

  A: Many textbook implementations use visited[].

     With lazy deletion approach, we don't need it:

       if (d > dist[node]) continue;

     This already guarantees outdated entries are ignored.

     Key difference:
     - visited[] implementations: never insert a node twice
     - Lazy deletion implementations: may insert duplicates
       (which are ignored later when popped)

     Both approaches are correct!
     - visited[]: Explicit tracking, cleaner conceptually
     - Lazy deletion: Simpler code, widely used in contests


  ──────────────────────────────────────────────────────────────

  Q9: Why use "dis > dist[node]" instead of "dis != dist[node]"?

  A: Because dist[node] is updated BEFORE pushing into the heap.

     When we find a better path:
     1. Update dist[node] = new_distance
     2. Push (dist[node], node) to heap

     A heap entry can only become outdated by becoming larger,
     never smaller.

     So:
     - dis > dist[node] → SKIP (stale entry, outdated)
     - dis == dist[node] → Process (correct entry)
     - dis < dist[node] → Impossible!

     Using > is sufficient and more precise than !=.


  ──────────────────────────────────────────────────────────────

  Q10: Why doesn't Dijkstra revisit finalized nodes?

  A: Because of the greedy-choice property!

     When a node leaves the min-heap (finalized):
     - It has the minimum distance among ALL unprocessed nodes
     - Any alternative path to this node must first go through
       an unprocessed node
     - But all unprocessed nodes have distance ≥ this node's distance
     - Therefore, no shorter path can exist!

     Once finalized, distance is locked in forever.


  ──────────────────────────────────────────────────────────────

  Q11: Can Dijkstra work on directed graphs?

  A: YES! Nothing changes in the algorithm!

     Implementation difference:
     - Undirected: Add edge in both directions
       adj[u].push_back({v, w});
       adj[v].push_back({u, w});

     - Directed: Add edge only in specified direction
       adj[u].push_back({v, w});  // only this way

     The greedy property and correctness proof still apply.


  ──────────────────────────────────────────────────────────────

  Q12: What's the difference between Dijkstra and Bellman-Ford?

  | Aspect           | Dijkstra        | Bellman-Ford           |
  |------------------|-----------------|------------------------|
  | Edge weights     | Non-negative    | Any (no negative cycle)|
  | Time complexity  | O((V+E) log V) | O(V × E)               |
  | Detects neg cycle| NO              | YES                    |
  | Iterations       | 1 pass (greedy) | V-1 passes             |
  | Use when         | Fast, no neg    | Negative weights        |

  Note: Bellman-Ford works even with negative weights,
  provided there is no reachable negative cycle.


  ──────────────────────────────────────────────────────────────

  Q13: What is Multi-Source Dijkstra?

  A: What if there are multiple source nodes?

     Solution: Simply initialize all sources with distance = 0
               and push all of them into the priority queue!

     Algorithm:
     1. Initialize dist[all_sources] = 0
     2. Push all sources into pq: {(0, src1), (0, src2), ...}
     3. Run normal Dijkstra

     Result:
     - Automatically computes shortest distance from the NEAREST source
     - Equivalent to adding a virtual super-source connected to all
       sources with edges of weight 0

     Use cases:
     - Multiple pickup locations (find nearest taxi)
     - Multiple warehouses (find nearest delivery center)


  ──────────────────────────────────────────────────────────────

  Q14: Why is Dijkstra considered greedy?

  A: At every step, Dijkstra permanently chooses the vertex with
     the smallest tentative distance.

     This greedy choice is safe because edge weights are non-negative.
     Once the smallest-distance vertex is chosen, no future path
     can improve it.

     Greedy correctness proof:
     - Suppose we pick vertex u with smallest distance dist[u]
     - Any alternate path to u must go through some other vertex v
     - dist[v] ≥ dist[u] (since u has minimum distance)
     - Therefore, any alternate path has length ≥ dist[u]
     - dist[u] is already optimal!


  ──────────────────────────────────────────────────────────────

  Q15: What's the relationship between Dijkstra and Prim's MST?

  A: They look almost identical! Compare:

     | Aspect       | Dijkstra           | Prim's MST         |
     |--------------|--------------------|--------------------|
     | Goal         | Shortest path      | Minimum spanning   |
     | Minimizes     | Path cost          | Edge cost          |
     | Selection    | Min distance to src| Min edge to tree   |
     | Relaxation   | dist[u] + w < dist | w < key[v]        |
     | Result       | Shortest-path tree | MST                |

     Famous one-liner to remember:
     "Prim minimizes edge weight. Dijkstra minimizes path weight."


  ──────────────────────────────────────────────────────────────

  Q16: How would you debug a failing Dijkstra implementation?

  A: Common bugs and fixes:

     Bug 1: Not skipping outdated entries
     → Add: if (d > dist[node]) continue;

     Bug 2: Forgetting to initialize source distance
     → dist[src] = 0;

     Bug 3: Off-by-one in 0-indexed vs 1-indexed
     → Be consistent! Use 0-indexed internally.

     Bug 4: Using wrong comparison (>= instead of <)
     → Relax only when strictly less: dist[u] + w < dist[v]

     Bug 5: Not replacing infinity with -1 for output
     → After algorithm: if (dist[i] == 1e9) dist[i] = -1;


  ──────────────────────────────────────────────────────────────

  Q17: Can we optimize Dijkstra for dense graphs?

  A: Yes! For dense graphs (E ≈ V²):

     Instead of adjacency list + heap, use a simple array:
     - O(V²) — no heap overhead
     - For each node, scan all nodes to find minimum distance
     - Beats heap version when E is very large

     | Graph Type | Best Implementation | Complexity |
     |------------|--------------------| -----------|
     | Sparse (E ≈ V) | Min-Heap | O((V+E) log V) |
     | Dense (E ≈ V²) | Array Scan | O(V²) |

```
