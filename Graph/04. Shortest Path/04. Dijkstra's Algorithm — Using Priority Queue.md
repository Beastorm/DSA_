### 1. Source to Destination

### Problem Statement {#problem-statement}

Given a **weighted, undirected, connected graph** with `V` vertices and `E` edges, find the **shortest path from source node `1` to destination node `n`**, along with the total distance.

---

## Examples {#examples-1}

**Example 1:**

- **Input:** `V=5, E=6`, edges with weights, source=1, destination=5
- **Output:** `[4, 1, 4, 3, 5]`

```

  1 ──2── 2
  │      │
  1      5
  │      │
  4 ──3── 3
  │    ╱
  └───1───

  Path: 1 → 4 → 3 → 5 = 1 + 3 + 1 = 4 ✓

```

**Example 2:**

- **Input:** `V=4, E=4`, edges: `1─2─2, 2─3─3, 1─4─5, 4─3─1`, src=1, dest=3
- **Output:** `[3, 1, 2, 3]`

```

       2
  1 ────── 2
  │ ╲       │
  5  1      3
  │    ╲   │
  4 ────── 3

  Path 1→2→3: 2+3=5
  Path 1→4→3: 5+1=6
  Shortest: 1→2→3 = 3 ✓

```

---

## Key Insight

```

  Problem: Find SHORTEST PATH from node 1 to node n.

  BFS works for UNIT weights only (counts hops).

  For weighted graphs: use DIJKSTRA with MIN-HEAP!

  Core idea:
  Always process the node with the SMALLEST current distance.
  → Greedy approach: once a node's shortest distance is found, it's FINAL.

  Min-heap (priority queue) efficiently gives us the
  minimum distance node in O(log V) time.

  Relaxation:
  If dist[u] + weight(u→v) < dist[v]:
      dist[v] ← dist[u] + weight(u→v)  (better path found!)
      Push (dist[v], v) to heap
      Update parent[v] = u  (for path reconstruction)

```

---

## Intuition

```

  Think of Dijkstra like finding the shortest route on a map:

  1. Start from source, mark all roads as "very long"
  2. Pick the CLOSEST unvisited city (using a priority queue)
  3. From that city, check if going through it gives shorter routes
     to nearby cities — if yes, update those distances!
  4. Repeat step 2-3 until you reach destination

  Why it works:
  The closest city can't be reached faster any other way —
  any alternative path must first go through a city at least
  as far away. So its distance is FINAL when we pick it!

```

---

## Algorithm

```

  shortestPath(V, adj, src=1, dest=n):

  1. dist[] = all infinity, parent[] = self
     dist[1] = 0

  2. Min-heap pq = {(0, 1)}

  3. While pq not empty:
       (d, node) = pq.top(); pop

       If d > dist[node]: SKIP  ← outdated/stale entry

       For each (neighbor, wt) in adj[node]:
           If dist[node] + wt < dist[neighbor]:
               dist[neighbor] = dist[node] + wt
               pq.push({dist[neighbor], neighbor})
               parent[neighbor] = node  ← record path

  4. If dist[n] == infinity: return {-1}

  5. Reconstruct path by backtracking from n using parent[]
     Prepend dist[n] at the beginning

  6. Return [weight, 1, ..., n]

```

```

  Why skip outdated entries?

  We might push (5, A) then later find shorter path (3, A).
  Both entries exist in the heap.

  When we pop (3, A): process it, dist[A]=3
  Later pop (5, A): 5 > dist[A]=3 → SKIP!

  This "lazy deletion" avoids complex heap update operations.

```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    vector<int> shortestPath(int n, int m, vector<vector<int>>& edges) {
        // Create an adjacency list: node -> {adjacent_node, weight}
        vector<pair<int, int>> adj[n + 1];
        for (auto it : edges) {
            adj[it[0]].push_back({it[1], it[2]});
            adj[it[1]].push_back({it[0], it[2]});
        }

        // Min-heap priority queue storing pair of {dist, node}
        priority_queue<pair<int, int>,
                       vector<pair<int, int>>,
                       greater<pair<int, int>>> pq;

        vector<int> dist(n + 1, 1e9);  // Initialize distances to infinity
        vector<int> parent(n + 1);     // Array to reconstruct path

        // Initialize parent of each node as itself
        for (int i = 1; i <= n; i++) {
            parent[i] = i;
        }

        dist[1] = 0;       // Distance to source node 1 is 0
        pq.push({0, 1});

        while (!pq.empty()) {
            auto it = pq.top();
            int dis = it.first;
            int node = it.second;
            pq.pop();

            // Skip processing if a shorter path already resolved
            if (dis > dist[node]) continue;

            for (auto it : adj[node]) {
                int adjNode = it.first;
                int wt = it.second;

                // Relaxation step
                if (dis + wt < dist[adjNode]) {
                    dist[adjNode] = dis + wt;
                    pq.push({dist[adjNode], adjNode});
                    parent[adjNode] = node;  // Record path
                }
            }
        }

        // If destination node n is unreachable, return {-1}
        if (dist[n] == 1e9) return {-1};

        // Reconstruct the path backwards from n to 1
        vector<int> ans;
        int node = n;
        while (parent[node] != node) {
            ans.push_back(node);
            node = parent[node];
        }
        ans.push_back(1);

        // Prepend the total shortest path weight
        ans.push_back(dist[n]);

        // Reverse to get: [weight, source, ..., destination]
        reverse(ans.begin(), ans.end());
        return ans;
    }
};

// --- DRIVER CODE ---
int main() {
    int n = 5;
    int m = 6;

    vector<vector<int>> edges = {
        {1, 2, 2},
        {2, 5, 5},
        {1, 4, 1},
        {4, 3, 3},
        {3, 5, 1},
        {2, 3, 4}
    };

    Solution solver;
    vector<int> result = solver.shortestPath(n, m, edges);

    if (result.size() == 1 && result[0] == -1) {
        cout << "No path exists from node 1 to node " << n << endl;
    } else {
        int totalWeight = result[0];
        cout << "Total Shortest Path Weight: " << totalWeight << endl;
        cout << "Shortest Path Sequence: ";
        for (size_t i = 1; i < result.size(); ++i) {
            cout << result[i] << (i == result.size() - 1 ? "" : " -> ");
        }
        cout << endl;
    }

    return 0;
}
```

## Why Not BFS for Weighted Graphs? {#why-not-bfs-for-weighted-graphs}

```

  BFS (unit weight):                    Dijkstra (weighted):
  
  1 ──1── 2 ──1── 3                    1 ──2── 2
      │                                 │      │
      1                                 1      5
      │                                 │      │
  4 ──1── 5                             4 ──1── 3

  BFS: dist[5]=2 (2 hops) ✓              BFS: dist[3]=2 (2 hops) ✗
                                              Should be 1+5=6 via 4!

  BFS counts HOPS, not WEIGHTS
  → Fails when edge weights differ

  Dijkstra respects actual WEIGHTS
  → Always finds true shortest path ✓

```

## Edge Cases {#edge-cases}

```

  Source is destination (n=1):         Disconnected graph:
  dist[1] = 0, path = [0, 1] ✓         Node 5 unreachable
                                       → return {-1} ✓

  Multiple equal-weight paths:         Negative weights:
  1─3─5 (weight 4)                     → Use Bellman-Ford!
  1─2─5 (weight 4)                      Dijkstra assumes distances
  First found is returned ✓            only decrease → fails!

  Single node, no edges:
  path = [0, 1] ✓

```

## Complexity Analysis {#complexity-analysis}

**Time Complexity:** O((V + E) log V)

- Each vertex extracted from heap once: **O(V log V)**
- Each edge relaxation may push to heap: **O(E log V)**
- Path reconstruction: **O(V)**
- Total: **O((V + E) log V)**

**Space Complexity:** O(V + E)

- Adjacency list: **O(V + E)**
- Distance array: **O(V)**
- Parent array: **O(V)**
- Priority queue (worst case with stale entries): **O(V + E)**
- Total: **O(V + E)**

```

  Comparison with other shortest path algorithms:

  Algorithm        Graph Type           Time Complexity
  ──────────────────────────────────────────────────────
  BFS              Unit weight          O(V + E)
  Topo Sort        DAG                  O(V + E)
  Dijkstra         Non-negative wt      O((V+E) log V)
  Bellman-Ford     Any weights          O(V × E)
  SPFA             Any weights          O(V × E) avg

  Dijkstra is optimal for non-negative weighted graphs ✓

```

## Path Reconstruction Explained {#path-reconstruction-explained}

```

  The parent array tracks where we came from:

  After algorithm completes:
  parent = [—, 1, 1, 4, 1, 3]
           index 1 2 3 4 5 6

  Reconstruct path from 5 to 1:
  5 → parent[5] = 3
  3 → parent[3] = 4
  4 → parent[4] = 1
  1 → parent[1] = 1 (stop!)

  Collected backwards: [5, 3, 4, 1]
  Reverse: [1, 4, 3, 5]
  Prepend weight: [5, 1, 4, 3, 5]

  Output format: [weight, source, ..., destination]

```

## Interview Questions {#interview-questions}

```

  Q1: Why does Dijkstra's algorithm use a Min-Heap?

  A: To always extract the node with the smallest distance efficiently.
     Without a heap, we'd need O(V) to find minimum each time.
     With min-heap: O(log V) per extraction → total O((V+E) log V).


  ──────────────────────────────────────────────────────────────

  Q2: What happens if all edge weights are 1?

  A: Dijkstra becomes equivalent to BFS!
     Both will run in O(V + E) since:
     - Each extraction from heap is O(log V)
     - But with equal weights, we could use a queue (BFS)
     - In practice, heap has same complexity as BFS for this case


  ──────────────────────────────────────────────────────────────

  Q3: Why can't Dijkstra handle negative weight edges?

  A: Because it assumes once a node's distance is finalized,
     it can never be improved. With negative weights:

       1 ─── 2 ─── 3
       │           │
       └─── -4 ────┘

     Start at 1: dist[2]=2, dist[3]=4
     Process 2: dist[3] = 2 + (-4) = -2 < 4 ✗
     But 3 might already have been "finalized"!

     Negative weights can create shorter paths AFTER a node
     is processed → breaks Dijkstra's greedy correctness.


  ──────────────────────────────────────────────────────────────

  Q4: Can we stop early once destination is reached?

  A: YES! For "source to destination" variant:
     - When destination is popped from heap, its distance is FINAL
     - No shorter path can exist (same greedy proof)
     - Saves time if we only need one destination
     - For "source to all", we must process all nodes


  ──────────────────────────────────────────────────────────────

  Q5: What is the difference between Dijkstra and BFS?

  | Aspect          | BFS              | Dijkstra           |
  |-----------------|------------------|---------------------|
  | Graph type      | Unweighted       | Weighted           |
  | Edge weights    | All = 1          | Any non-negative   |
  | Data structure  | Queue            | Priority Queue      |
  | Time complexity | O(V + E)         | O((V+E) log V)     |
  | What it finds   | Fewest HOPS      | Shortest path      |


  ──────────────────────────────────────────────────────────────

  Q6: What is "relaxation" in Dijkstra's algorithm?

  A: Relaxation is the process of updating a node's distance:

       Before: dist[v] = 5
       Check:  dist[u] + weight(u→v) = 3 + 1 = 4
       After:  dist[v] = 4  (if 4 < 5)

     Think of it as "relaxing a tight rope" — we found a
     shorter path, so we loosen the distance value.


  ──────────────────────────────────────────────────────────────

  Q7: Why do we skip outdated/stale entries from the heap?

  A: Due to "lazy deletion" — when we find a shorter path to
     a node already in heap, we push new entry but don't
     remove the old one.

     Example:
     1. Push (5, A) to heap
     2. Later find shorter path: dist[A] = 3, push (3, A)
     3. Pop (3, A): process it (correct)
     4. Pop (5, A): 5 > dist[A]=3 → SKIP!

     This avoids expensive heap deletions at cost of
     extra entries (still O((V+E) log V) overall).


  ──────────────────────────────────────────────────────────────

  Q8: What is the space complexity and why?

  A: O(V + E) where:
     - Adjacency list: O(V + E)
     - Distance array: O(V)
     - Parent array: O(V) [only for path reconstruction]
     - Priority Queue: O(V + E) worst case [stale entries]

     We use O(V + E) because:
     - Graph stored as adjacency list takes O(V + E)
     - PQ can have up to O(V + E) entries
     - Everything else is O(V)


  ──────────────────────────────────────────────────────────────

  Q9: Can Dijkstra's algorithm detect negative cycles?

  A: NO! Dijkstra cannot detect negative cycles.
     If a negative cycle exists, distances keep decreasing,
     but Dijkstra stops when all nodes are "finalized".

     Use Bellman-Ford to detect negative cycles:
     - Run V-1 iterations
     - If any distance updates in V-th iteration → negative cycle!


  ──────────────────────────────────────────────────────────────

  Q10: How would you handle the case when multiple shortest
       paths exist?

  A: Dijkstra will return whichever path is found first based
     on the order of relaxation. If you need a specific path:

     Option 1: Keep all parent choices in a map
     Option 2: Modify relaxation to prefer certain criteria
     Option 3: Post-process to find all equal-weight paths

     For most interview purposes: "First found, first returned"
     is the expected answer.


  ──────────────────────────────────────────────────────────────

  Q11: What is the time complexity of Dijkstra's algorithm?

  A: O((V + E) log V) for typical implementation.

     Breakdown:
     - Each vertex extracted once: O(V log V)
     - Each edge can cause one relaxation: O(E log V)
       (each push to heap is O(log V))
     - Total: O((V + E) log V)

     Special case: If all edges have same weight,
     can use bucket queue → O(V + E)


  ──────────────────────────────────────────────────────────────

  Q12: Is Dijkstra a dynamic programming algorithm?

  A: Yes, it's often considered both greedy AND DP!

     Greedy aspect:
     - Always picks the minimum distance node next
     - Never reconsiders a "finalized" node

     DP aspect:
     - Optimal substructure: shortest path to v consists of
       shortest path to some u + edge u→v
     - Overlapping subproblems: distances are computed once

     The "greedy choice property" makes it work like a DP
     with optimal substructure.


  ──────────────────────────────────────────────────────────────

  Q13: How do you reconstruct the shortest path?

  A: Using a parent array:
     1. While processing, set parent[v] = u when we update dist[v]
     2. After algorithm completes, backtrack from destination
     3. Follow parent pointers: dest → ... → source
     4. Reverse the path to get source → ... → dest
     5. Prepend the total distance

     Time: O(V) for path reconstruction


  ──────────────────────────────────────────────────────────────

  Q14: Can we use max-heap instead of min-heap?

  A: NO! That would completely break the algorithm.

     Min-heap: Always gives us the UNVISITED node with
               SMALLEST distance → correct greedy choice

     Max-heap: Would give us the node with LARGEST distance
               → We might process nodes in wrong order
               → Distances won't be finalized correctly
               → Shortest path guarantee is lost!

```

## Examples {#examples-2}

**Example 1:**

**Example 2:**

       4
  0 ──────── 2
  │           │
  1           2
  │           │
  3 ──────────1

  Dist from 0:
  → 0: 0 (source) ✓
  → 1: 3 (via 3: 1+2) ✓
  → 2: 4 (direct) ✓
  → 3: 1 (direct) ✓

```

## Key Insight {#key-insight-2}

```

  Problem: Find SHORTEST DISTANCE from src to ALL vertices.

  Same greedy approach as before:
  Always process the node with the SMALLEST current distance.

  Difference from source→destination:
  ✓ Continue until heap is EMPTY (all vertices processed)
  ✗ No early termination when destination is reached
  ✗ No path reconstruction needed (only distances)

  The algorithm naturally computes all shortest paths!

```

---

## Intuition {#intuition-2}

```

  Imagine you're a radio signal spreading from a source:

  1. Signal starts at source with strength 0
  2. The signal ALWAYS travels to the NEAREST unvisited city first
  3. From that city, it tries to reach new cities — if going through
     this city gives a faster route, update that city's distance!
  4. Keep spreading until ALL cities are reached

  Why it works:
  Once a city's signal reaches you at distance X, no other path
  can be shorter — because any alternative must first pass through
  a city at least distance X away. So distance is LOCKED IN!

```

---

## Algorithm {#algorithm-2}

```

  dijkstraAll(V, adj, src):

  1. dist[] = all infinity
     dist[src] = 0

  2. Min-heap pq = {(0, src)}

  3. While pq not empty:
       (d, node) = pq.top(); pop

       If d > dist[node]: SKIP  ← outdated entry

       For each (neighbor, wt) in adj[node]:
           If dist[node] + wt < dist[neighbor]:
               dist[neighbor] = dist[node] + wt
               pq.push({dist[neighbor], neighbor})

  4. Return dist[]  ← Contains shortest distance from src to ALL nodes

```

```

  Key Differences from Source→Destination:

  | Aspect              | Src→Dst       | Src→All        |
  |---------------------|---------------|----------------|
  | Termination         | When dst popped | When pq empty  |
  | Early exit          | Yes           | No             |
  | Parent array        | Needed        | Not needed     |
  | Return value        | Path + weight | Array of dists |
  | Process all nodes   | No            | Yes            |

```

## Code {#code-2}

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    vector<int> dijkstra(int V, vector<vector<pair<int, int>>>& adj, int src) {
        // Step 1: Initialize distances to infinity
        vector<int> dist(V, 1e9);
        dist[src] = 0;

        // Step 2: Min-heap: {distance, node}
        priority_queue<pair<int, int>,
                       vector<pair<int, int>>,
                       greater<pair<int, int>>> pq;

        pq.push({0, src});

        // Step 3: Process until all nodes are settled
        while (!pq.empty()) {
            int d = pq.top().first;
            int node = pq.top().second;
            pq.pop();

            // Skip outdated (stale) entries
            if (d > dist[node]) continue;

            // Relax all edges from current node
            for (auto [neighbor, wt] : adj[node]) {
                if (dist[node] + wt < dist[neighbor]) {
                    dist[neighbor] = dist[node] + wt;
                    pq.push({dist[neighbor], neighbor});
                }
            }
        }

        // Replace unreachable nodes with -1 (optional)
        for (int i = 0; i < V; i++) {
            if (dist[i] == 1e9) dist[i] = -1;
        }

        return dist;
    }
};

// --- DRIVER CODE ---
int main() {
    int V = 5;
    vector<vector<pair<int, int>>> adj(V);

    // Add undirected weighted edges: {neighbor, weight}
    adj[0].push_back({1, 2});  adj[1].push_back({0, 2});   // 0─2─1
    adj[0].push_back({3, 1});  adj[3].push_back({0, 1});   // 0─1─3
    adj[1].push_back({2, 4});  adj[2].push_back({1, 4});   // 1─4─2
    adj[2].push_back({4, 3});  adj[4].push_back({2, 3});   // 2─3─4
    adj[3].push_back({4, 1});  adj[4].push_back({3, 1});   // 3─1─4
    adj[2].push_back({3, 5});  adj[3].push_back({2, 5});   // 2─5─3

    Solution obj;
    vector<int> distances = obj.dijkstra(V, adj, 0);

    cout << "Shortest distances from node 0:" << endl;
    for (int i = 0; i < V; i++) {
        cout << "To " << i << ": " << distances[i] << endl;
    }

    return 0;
}
```

## Comparing Both Variants {#comparing-both-variants}

```

  ┌─────────────────────────────────────────────────────────────────┐
  │                    Source → Destination                         │
  ├─────────────────────────────────────────────────────────────────┤
  │ Goal:      Find ONE shortest path to specific destination       │
  │ Output:    [weight, path_nodes...]                              │
  │ Stop:      When destination node is popped from heap             │
  │ Parent:    Needed to reconstruct path                            │
  │ Use when:  You only care about one destination                   │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │                    Source → ALL Vertices                         │
  ├─────────────────────────────────────────────────────────────────┤
  │ Goal:      Find shortest distance to EVERY vertex                │
  │ Output:    [dist_0, dist_1, dist_2, ..., dist_{V-1}]            │
  │ Stop:      When heap is empty (all vertices processed)           │
  │ Parent:    Not needed (just distances)                           │
  │ Use when:  Need distances to multiple destinations               │
  └─────────────────────────────────────────────────────────────────┘

  Same algorithm, different stopping condition!

  Both run in O((V+E) log V) time complexity.

```

## Complexity Analysis {#complexity-analysis-2}

**Time Complexity:** O((V + E) log V)

- Each vertex extracted once: **O(V log V)**
- Each edge may cause relaxation: **O(E log V)**
- Total: **O((V + E) log V)**

**Space Complexity:** O(V + E)

- Adjacency list: **O(V + E)**
- Distance array: **O(V)**
- Priority queue: **O(V + E)** worst case
- Total: **O(V + E)**

## Interview Questions {#interview-questions-2}



  Q1: When should I use Source→All instead of Source→Destination?

  A: Use Source→All when:
     - You need distances to multiple destinations
     - Building systems like GPS (find nearest anything)
     - Network routing tables need all paths
     - Precomputing all pairs of paths

     Use Source→Destination when:
     - Only ONE destination matters
     - Memory/time is limited
     - Early termination saves work


  ──────────────────────────────────────────────────────────────

  Q2: Can we optimize Dijkstra for dense graphs?

  A: Yes! For dense graphs (E ≈ V²):

     Instead of adjacency list + heap, use a simple array:
     - O(V²) — no heap overhead
     - For each node, scan all nodes to find minimum distance
     - Beats heap version when E is very large

     | Graph Type | Best Implementation | Complexity |
     |------------|--------------------| -----------|
     | Sparse (E ≈ V) | Min-Heap | O((V+E) log V) |
     | Dense (E ≈ V²) | Array Scan | O(V²) |


  ──────────────────────────────────────────────────────────────

  Q3: How does Dijkstra handle disconnected components?

  A: Nodes in different components remain at infinity (unreachable).

     Example: Graph with 3 components
     Component 1: nodes 1-2-3 (source here)
     Component 2: nodes 4-5
     Component 3: node 6

     Result: dist[1]=0, dist[2]=3, dist[3]=5
             dist[4]=∞, dist[5]=∞, dist[6]=∞

     Return -1 (or infinity) for unreachable nodes.


  ──────────────────────────────────────────────────────────────

  Q4: Is Dijkstra used in real-world applications?

  A: ABSOLUTELY! Common real-world uses:

     1. Google Maps / GPS Navigation
        - Find shortest route from your location
        - Calculate ETA, alternate routes

     2. Network Routing (OSPF Protocol)
        - Routers use Dijkstra to find best path
        - Internet backbone routing

     3. Flight Scheduling
        - Find cheapest routes between airports
        - Airlines use variations for ticket pricing

     4. Video Game Pathfinding
        - Units navigate game worlds
        - Real-time strategy games

     5. Robotics / Drone Navigation
        - Path planning with energy costs


  ──────────────────────────────────────────────────────────────

  Q5: What if the source node doesn't exist in the graph?

  A: Handle edge cases:

     - If source = null/invalid: return empty or error
     - If source has no edges: only dist[source]=0, rest = -1
     - If graph is empty (V=0): return empty result

     Always validate inputs before running algorithm!


  ──────────────────────────────────────────────────────────────

  Q6: How would you modify Dijkstra for 0-1 BFS?

  A: When edge weights are only 0 or 1, use Deque instead of heap:

       if weight == 0: push_front
       if weight == 1: push_back

     This gives O(V + E) time — faster than O((V+E) log V)!

     Useful for:
     - Grid graphs with "blocked/unblocked" cells
     - Graphs with two types of edges
     - State machines with binary transitions


  ──────────────────────────────────────────────────────────────

  Q7: Can we parallelize Dijkstra's algorithm?

  A: Yes, but with care! Challenges:

     1. Heap operations are hard to parallelize
     2. Race conditions when updating distances
     3. Graph structure affects parallel efficiency

     Approaches:
     - Multi-source Dijkstra (parallel BFS from multiple sources)
     - Delta-stepping algorithm (partition edges by weight)
     - Johnson's algorithm for all-pairs (parallelize across sources)

     For interview: Mention it's "inherently sequential"
     due to greedy choice property.


  ──────────────────────────────────────────────────────────────

  Q8: What's the relationship between Dijkstra and Prim's MST?

  A: They look almost identical! Compare:

     | Aspect       | Dijkstra           | Prim's MST         |
     |--------------|--------------------|--------------------|
     | Goal         | Shortest path      | Minimum spanning   |
     | Selection    | Min distance to src| Min edge to tree   |
     | Relaxation   | dist[u] + w < dist | w < key[v]        |
     | Result       | Single path        | Whole tree         |

     The data structures are literally the same!
     Dijkstra finds shortest paths from one source.
     Prim finds minimum spanning tree from one root.


  ──────────────────────────────────────────────────────────────

  Q9: How do you handle very large graphs?

  A: Strategies for massive graphs:

     1. Bidirectional Dijkstra
        - Run from source AND destination
        - Meet in middle → ~2x faster

     2. A* Search
        - Use heuristic to guide search
        - Much faster if good heuristic exists

     3. Contraction Hierarchies
        - Preprocess shortcuts
        - Speed up repeated queries

     4. ALT (A* + Landmarks + Triangle inequality)
        - Pick landmarks, use as heuristic
        - Used in Google Maps!


  ──────────────────────────────────────────────────────────────

  Q10: What is the "lazy deletion" we mentioned earlier?

  A: A technique to avoid expensive heap deletions:

     Normal heap: When distance updates, delete old entry
     → O(log V) for deletion = expensive!

     Lazy deletion: Just insert new entry, keep old one
     → When popped, check if it's outdated (dist[node] < d)
     → If outdated, skip it!

     Trade-off:
     - Heap size grows (more entries)
     - But no expensive deletions
     - Still O((V+E) log V) overall


  ──────────────────────────────────────────────────────────────

  Q11: How would you debug a failing Dijkstra implementation?

  A: Common bugs and fixes:

     Bug 1: Not skipping outdated entries
     → Add: if (d > dist[node]) continue;

     Bug 2: Forgetting to initialize source distance
     → dist[src] = 0;

     Bug 3: Off-by-one in 0-indexed vs 1-indexed
     → Be consistent! Use 0-indexed internally.

     Bug 4: Not reversing path reconstruction
     → ans.reverse() before returning

     Bug 5: Using wrong comparison (>= instead of <)
     → Relax only when strictly less: dist[u] + w < dist[v]


  ──────────────────────────────────────────────────────────────

  Q12: What's the difference between Dijkstra and Bellman-Ford?

  | Aspect           | Dijkstra        | Bellman-Ford      |
  |------------------|-----------------|-------------------|
  | Edge weights     | Non-negative    | Any (incl. neg)   |
  | Time complexity  | O((V+E) log V)  | O(V × E)          |
  | Detects neg cycle| NO              | YES               |
  | Iterations       | 1 pass (greedy) | V-1 passes        |
  | Use when         | Fast, no neg     | Negative weights  |
