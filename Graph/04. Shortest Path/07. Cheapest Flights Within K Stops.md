## 1. Problem Statement

There are `n` cities connected by `m` flights. You are given an array `flights` where `flights[i] = [fromi, toi, pricei]` indicates a flight from city `fromi` to city `toi` with cost `pricei`.

Given three integers `src`, `dst`, and `k`, find the **cheapest price** from `src` to `dst` with **at most k stops**. If no such route exists, return `-1`.

**Input Format:**

- `n` = number of cities
- `flights` = list of directed flights `[from, to, price]`
- `src` = source city
- `dst` = destination city
- `k` = maximum number of stops allowed

**Output:**

- Cheapest price from `src` to `dst` with at most `k` stops
- `-1` if no valid route exists

## Key Insight

```
  Why NOT Standard Dijkstra?

  Standard Dijkstra:
  → Always processes node with minimum distance
  → Guarantees shortest path globally
  → But IGNORES the stops constraint! ✗

  This problem:
  → Shortest path within K stops
  → A longer (costlier) path with fewer stops
    might be valid, while a cheaper path with
    more stops is INVALID!

  Example:
  Path A: cost=400, stops=2 ✗ (exceeds k=1)
  Path B: cost=700, stops=1 ✓ (valid!)

  Path B is ANSWER despite higher cost!
  Standard Dijkstra would pick Path A → WRONG! ✗

  Solution: Modified BFS
  → Track BOTH cost AND stops
  → Process level-by-level (stops = BFS levels)
  → Update distance only if stops ≤ k
  → Queue: {stops, {node, cost}}
```

---

## Intuition

```
  Imagine booking flights with a layover limit:

  1. You start at your source city with 0 cost and 0 stops.
  2. Each flight you take increases your stop count by 1.
  3. You cannot exceed k stops (layovers).
  4. Among all valid itineraries, you want the cheapest total price.

  Why BFS works:
  BFS naturally processes flights by number of stops.
  Level 0 = direct flights (0 stops), Level 1 = one layover,
  Level 2 = two layovers, and so on.

  By capping the BFS at level k, we only consider valid itineraries.
  The distance array keeps track of the cheapest price found so far
  for each city at or before the current stop level.
```

## Algorithm

```
  CheapestFlight(n, flights, src, dst, K):

  1. Build adjacency list from flights:
     adj[u] = {(v, price), ...}

  2. Initialize:
     - Queue: {stops, {node, cost}}
     - Distance array: dist[n] = infinity
     - dist[src] = 0

  3. Push source into queue: {0, {src, 0}}
     (0 stops, source node, 0 cost)

  4. BFS traversal:
     While queue is not empty:
       a. Pop {stops, {node, cost}} from front

       b. If stops > k: continue (exceeded limit)

       c. For each neighbor (adjNode, edgeWeight):
          newCost = cost + edgeWeight

          If newCost < dist[adjNode] AND stops ≤ k:
            → Update: dist[adjNode] = newCost
            → Push: {stops+1, {adjNode, newCost}}

            Note: We check stops ≤ k to allow one more hop
                  (stops+1 in queue might reach k+1 total)

  5. After BFS:
     If dist[dst] == infinity: return -1
     Else: return dist[dst]
```

```
  Why BFS instead of Dijkstra?

  BFS property:
  → Processes nodes level by level
  → Level = number of stops!
  → Level 0: src (0 stops)
  → Level 1: nodes 1 hop away (1 stop)
  → Level 2: nodes 2 hops away (2 stops)
  → ...
  → Level k+1: nodes at most k stops away

  Stop constraint naturally enforced by BFS levels! ✓

  Why not use priority queue (Dijkstra)?
  → PQ processes by cost (not stops)
  → Might process expensive nodes with few stops
    before cheap nodes with many stops
  → Doesn't leverage the "at most k stops" constraint
  → BFS is simpler and sufficient! ✓
```

### Why Queue, Not Priority Queue?

```
  Standard Dijkstra uses Priority Queue:
  → Always process node with minimum cost
  → Guarantees globally shortest path
  → But ignores stop constraint!

  This problem uses Queue (BFS):
  → Process nodes level by level (by stops)
  → Respects stop constraint naturally
  → May not give globally shortest path
  → But gives shortest path within k stops! ✓

  Example scenario:
  Queue state: [{1,{1,100}}, {1,{2,50}}]

  Priority Queue would process {1,{2,50}} first (smaller cost)
  Regular Queue processes {1,{1,100}} first (FIFO)

  Both are valid because:
  → Both have same number of stops (1)
  → Within same level, order doesn't matter
  → BFS explores all level k before level k+1

  Simple queue is SUFFICIENT and FASTER! ✓
  Time: O(E) vs O(E log V) with PQ
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int CheapestFLight(int n, vector<vector<int>>& flights, int src, int dst, int K) {

        // Build adjacency list
        vector<vector<pair<int, int>>> adj(n);
        for (auto it : flights) {
            adj[it[0]].push_back({it[1], it[2]});
        }

        // Queue: {stops, {node, cost}}
        queue<pair<int, pair<int, int>>> q;
        q.push({0, {src, 0}});

        // Distance array (minimum cost to reach each node)
        vector<int> dist(n, 1e9);
        dist[src] = 0;

        // BFS traversal
        while (!q.empty()) {
            auto it = q.front();
            q.pop();

            int stops = it.first;
            int node = it.second.first;
            int cost = it.second.second;

            // Exceeded stop limit
            if (stops > K) continue;

            // Explore neighbors
            for (auto iter : adj[node]) {
                int adjNode = iter.first;
                int edW = iter.second;

                // If a cheaper path is found within the stop limit
                if (cost + edW < dist[adjNode] && stops <= K) {
                    dist[adjNode] = cost + edW;
                    q.push({stops + 1, {adjNode, cost + edW}});
                }
            }
        }

        // Return result
        return (dist[dst] == 1e9) ? -1 : dist[dst];
    }
};

int main() {
    int n = 4, src = 0, dst = 3, K = 1;

    vector<vector<int>> flights = {
        {0, 1, 100}, {1, 2, 100}, {2, 0, 100},
        {1, 3, 600}, {2, 3, 200}
    };

    Solution obj;
    int ans = obj.CheapestFLight(n, flights, src, dst, K);

    cout << "Cheapest price: " << ans << endl;

    return 0;
}
```

## Step-by-Step Trace

```
  Input:
  n = 4, src = 0, dst = 3, k = 1
  flights = [[0,1,100], [1,2,100], [2,0,100], [1,3,600], [2,3,200]]

  ── Build Adjacency List ──

  adj[0] = {(1, 100)}
  adj[1] = {(2, 100), (3, 600)}
  adj[2] = {(0, 100), (3, 200)}
  adj[3] = {}

  ── Initialization ──

  dist = [0, ∞, ∞, ∞]
  queue = [{0, {0, 0}}]  // {stops, {node, cost}}

  ── BFS Iterations ──

  Iteration 1: Pop {0, {0, 0}}
      stops=0, node=0, cost=0
      stops ≤ k=1 ✓

      Neighbor (1, 100):
        newCost = 0 + 100 = 100 < dist[1]=∞
        Update: dist[1] = 100
        Push: {1, {1, 100}}

      dist = [0, 100, ∞, ∞]
      queue = [{1, {1, 100}}]

  Iteration 2: Pop {1, {1, 100}}
      stops=1, node=1, cost=100
      stops ≤ k=1 ✓

      Neighbor (2, 100):
        newCost = 100 + 100 = 200 < dist[2]=∞
        Update: dist[2] = 200
        Push: {2, {2, 200}}

      Neighbor (3, 600):
        newCost = 100 + 600 = 700 < dist[3]=∞
        Update: dist[3] = 700
        Push: {2, {3, 700}}

      dist = [0, 100, 200, 700]
      queue = [{2, {2, 200}}, {2, {3, 700}}]

  Iteration 3: Pop {2, {2, 200}}
      stops=2, node=2, cost=200
      stops > k=1 ✗ → CONTINUE (skip this iteration)

  Iteration 4: Pop {2, {3, 700}}
      stops=2, node=3, cost=700
      stops > k=1 ✗ → CONTINUE

  Queue empty → DONE!

  Final: dist[3] = 700
  Return: 700 ✓
```

    0 → 1 → [3]  ← Actually this uses 1 stop!
            cost=700 ✓

  Clarification:
  "At most k stops" = at most k intermediate nodes
  NOT including src and dst!

    Intermediate node: 1 (count=1)
    Stops = 1 ✓ (valid for k=1)

  0 → 1 → 2 → 3:
    Intermediate nodes: 1, 2 (count=2)

  Source = 0, Destination = 3, K = 1

  Step      Pop entry        dist[] state        Queue state
  ─────────────────────────────────────────────────────────────────
  Init      —                [0, ∞, ∞, ∞]        [(0,(0,0))]

  Pop(0,0)  process          [0, 100, ∞, ∞]      [(1,(1,100))]

  Pop(1,100) process         [0, 100, 200, 700]  [(2,(2,200)),(2,(3,700))]

  Pop(2,200) skip            unchanged           [(2,(3,700))]
                              (stops > K)

  Pop(3,700) skip            unchanged           []
                              (stops > K)

  Final: dist[3] = 700
  Return: 700 ✓

### Why Path [0→1→2→3] is Invalid

```
  Path: 0 → 1 → 2 → 3
  Cost: 100 + 100 + 200 = 400
  Stops: 2 (visiting nodes 1 and 2 before the destination)

  But k = 1 (at most 1 stop allowed)!

  When we pop {2, {2, 200}} from queue:
    stops=2 > k=1 → SKIP!

    Never explore the neighbors of node 2
    Never update dist[3] via path 0→1→2→3

  BFS naturally enforces the stop constraint:
  → Level 0: src (0 stops)
  → Level 1: 1 hop away (1 stop) ✓
  → Level 2: 2 hops away (2 stops) ✗ (exceeds k=1)

  Paths exceeding k stops are automatically pruned! ✓
```

### Distance Array Updates

```
  Initial:        After {0,{0,0}}:    After {1,{1,100}}:

  [0, ∞, ∞, ∞]    [0, 100, ∞, ∞]      [0, 100, 200, 700]

  Key observation:
  → dist[3] = 700 from path 0→1→3 (stops=1)
  → Path 0→1→2→3 would give cost=400 but stops=2
  → Since stops=2 > k=1, we never explore it
  → dist[3] stays 700 ✓

  If k=2 instead:
  → Path 0→1→2→3 with stops=2 would be allowed
  → dist[3] would update from 700 to 400
  → Answer would be 400 ✓

  The stop constraint changes the optimal path! ✓
```

## Edge Cases

```
  Direct flight exists:
  flights = [[0,3,200]]
  k = 0
  Output: 200 ✓
  (0 stops = direct flight)

  No path exists:
  flights = [[0,1,100]]
  dst = 3
  Output: -1 ✓

  All paths exceed k:
  flights = [[0,1,100],
             [1,2,100],
             [2,3,100]]
  k = 1, dst = 3
  Output: -1
  (shortest needs 2 stops)

  Multiple paths, same cost:
  flights = [[0,1,100],
             [0,2,200],
             [1,2,100]]
  k = 1, dst = 2
  Both cost 200, either valid
  Output: 200 ✓

  k = n-1 (no real constraint):
  All paths valid
  Reduces to standard shortest path

  Self-loop:
  flights = [[0,0,100]]
  Typically skipped (no practical use)

  Negative prices:
  Problem states prices ≥ 0
  No need to handle negatives

  Very large k:
  If k ≥ n-1, constraint effectively removed
```

### Stops vs Edges Clarification

```
  Common confusion: "stops" vs "edges"

  Stops = intermediate nodes (NOT including src and dst)
  Edges = number of flights taken

  Example path: 0 → 1 → 2 → 3

  Nodes: 0, 1, 2, 3 (4 nodes)
  Intermediate nodes: 1, 2 (2 nodes)
  Stops: 2 ✓
  Edges/Flights: 3 (0→1, 1→2, 2→3)

  "At most k stops" = at most k intermediate nodes

  For k=1:
  0 → 1 → 3: intermediate=[1], stops=1 ✓
  0 → 1 → 2 → 3: intermediate=[1,2], stops=2 ✗

  In code:
  {stops, {node, cost}}

  stops starts at 0 (source has 0 intermediate nodes)
  stops increments each time we move to next node
  When stops > k, we skip (too many intermediate nodes)
```

## Complexity Analysis

**Time Complexity:** O(E × K)

- Each edge can be relaxed at most K times
- In worst case, we might visit same edge multiple times
  with different stop counts (0, 1, 2, ..., K)
- Total: **O(E × K)** where E = number of flights
- For K << V, this is effectively O(E)

**Space Complexity:** O(V + E)

- Adjacency list: O(E)
- Distance array: O(V)
- Queue: O(V × K) in worst case (same node with different stops)
- Total: **O(V + E)**

```
  Comparison with other approaches:

  Approach                Time              Space     Handles K?
  ──────────────────────────────────────────────────────────────────
  BFS (this solution)     O(E × K)          O(V+E)    YES ✓
  Standard Dijkstra       O(E log V)        O(V+E)    NO ✗
  Modified Dijkstra       O(E × K log V)    O(V+E)    YES ✓
    with (cost, stops)

  BFS is simpler and sufficient for this problem! ✓

  Why BFS works here:
  → K is usually small (constraint: K ≤ V-1)
  → Stop levels naturally map to BFS levels
  → No need for PQ overhead
  → Cleaner, faster implementation
```

### Alternative: Bellman-Ford Approach

```
  Idea: Run modified Bellman-Ford for K+1 iterations

  Bellman-Ford property:
  → After i iterations, dist[v] = shortest path with ≤ i edges

  Modified for this problem:
  → After i iterations, dist[v] = cheapest with ≤ i stops

  Pseudocode:

  dist[src] = 0, all others = ∞

  for (i = 0; i <= K; i++):
      temp = dist.copy()  // Important: use previous iteration

      for each flight (u, v, price):
          if temp[u] != ∞:
              dist[v] = min(dist[v], temp[u] + price)

  return dist[dst]

  Time: O(K × E)
  Space: O(V)

  Both BFS and Bellman-Ford work! Choose based on preference.
  BFS is more intuitive for "stops" constraint. ✓
```

## Interview Questions

```
  Q1: Why can't we use standard Dijkstra for this problem?

  A: Standard Dijkstra minimizes total cost without any constraint
     on the number of edges (stops). This problem requires the
     cheapest cost WITHIN at most k stops. A cheaper path with more
     than k stops is invalid, so Dijkstra's globally optimal answer
     might violate the constraint.

  ──────────────────────────────────────────────────────────────

  Q2: Why does BFS work here instead of Dijkstra?

  A: BFS processes nodes level by level, where each level corresponds
     to the number of stops used. By limiting BFS to level k, we
     naturally enforce the stop constraint. Within each level, we
     update the cheapest cost found so far for each city.

  ──────────────────────────────────────────────────────────────

  Q3: What does the queue store in this BFS?

  A: The queue stores {stops, {node, cost}}:
     - stops = number of intermediate nodes used so far
     - node = current city
     - cost = total price to reach this city

  ──────────────────────────────────────────────────────────────

  Q4: What does "stops" mean exactly?

  A: Stops refer to intermediate cities visited between src and dst,
     NOT including src and dst themselves.

     Example: 0 → 1 → 2 → 3
     Intermediate cities: 1 and 2
     Stops = 2

  ──────────────────────────────────────────────────────────────

  Q5: Why do we push neighbors even when stops == k?

  A: Because the current entry has stops = k, meaning we have used
     k intermediate nodes to reach the current node. Taking one more
     flight to reach the destination adds 0 intermediate nodes if
     the neighbor is the destination. Wait — actually in our code,
     stops increments after every flight, so if stops == k, the
     next node would have stops = k+1, which we then skip.

     The check "if (stops > K) continue" ensures we don't explore
     beyond k stops. The check "stops <= K" before relaxing allows
     the last valid relaxation at level k.

  ──────────────────────────────────────────────────────────────

  Q6: Can we use a priority queue here?

  A: Yes, but it doesn't improve the time complexity and adds
     overhead. A modified Dijkstra that tracks both cost and stops
     would be O(E × K log V). BFS gives O(E × K) with simpler code.

     Since the constraint is on stops (levels), not cost, BFS is
     the natural choice.

  ──────────────────────────────────────────────────────────────

  Q7: What is the Bellman-Ford approach to this problem?

  A: Run K+1 iterations of relaxation using all flights. After each
     iteration, the distance array represents the cheapest cost
     with at most that many stops. Use a temporary copy of distances
     each iteration to avoid using updated values within the same
     iteration.

  ──────────────────────────────────────────────────────────────

  Q8: Why is the time complexity O(E × K)?

  A: In the worst case, each edge can be relaxed up to K+1 times,
     once for each possible stop count. Hence the total work is
     proportional to E × K.

  ──────────────────────────────────────────────────────────────

  Q9: What if k is very large (e.g., k ≥ n-1)?

  A: The constraint becomes irrelevant because any simple path in
     a graph with n nodes has at most n-1 edges. The problem then
     reduces to standard shortest path, and the BFS solution still
     works correctly.

  ──────────────────────────────────────────────────────────────

  Q10: What if there are multiple flights between the same cities?

  A: The adjacency list will store all of them. During relaxation,
      we consider each flight and keep the minimum cost. This is
      handled naturally by the algorithm.

  ──────────────────────────────────────────────────────────────

  Q11: How do you reconstruct the actual flight path?

  A: Maintain a parent array. When updating dist[adjNode], also set
     parent[adjNode] = node. After BFS completes, trace back from
     dst to src using parent pointers.

  ──────────────────────────────────────────────────────────────

  Q12: What are common bugs in this implementation?

  A: Common bugs and fixes:

      Bug 1: Checking stops > k after relaxing neighbors
      → Check stops > k immediately after popping from queue

      Bug 2: Not initializing dist[src] = 0
      → dist[src] = 0

      Bug 3: Using priority queue thinking it's faster
      → Use simple queue for level-order BFS

      Bug 4: Confusing stops with edges
      → Stops = intermediate nodes, not total flights

      Bug 5: Not using a copy in Bellman-Ford approach
      → Use temp = dist before each iteration

  ──────────────────────────────────────────────────────────────

  Q13: When would you prefer modified Dijkstra over BFS?

  A: If the problem had additional constraints or costs that make
     the stop level not the primary ordering factor, modified
     Dijkstra might be needed. But for pure "cheapest within k stops",
     BFS is simpler and optimal.

  ──────────────────────────────────────────────────────────────

  Q14: Can this algorithm handle negative flight prices?

  A: The problem guarantees non-negative prices. With negative prices,
     this BFS approach might not work because cheaper paths could
     keep being discovered. Bellman-Ford handles negative weights
     but still needs care with the stop constraint.

  ──────────────────────────────────────────────────────────────

  Q15: What are similar problems using this pattern?

  A: Similar problems:
      - Bus Routes (minimize transfers = stops)
      - Word Ladder (unweighted, BFS directly)
      - Minimum Cost to Reach Destination in Time
      - Network Delay Time (K = ∞, use Dijkstra)

     Key pattern: "Minimize cost with constraint on path length"
     → BFS with levels or modified Bellman-Ford.
```
