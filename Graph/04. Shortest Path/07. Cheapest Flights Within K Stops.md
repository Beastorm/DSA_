## Cheapest Flights Within K Stops

### Problem Statement:

There are `n` cities connected by `m` flights. You are given an array `flights` where `flights[i] = [fromi, toi, pricei]` indicates a flight from city `fromi` to city `toi` with cost `pricei`.

Given three integers `src`, `dst`, and `k`, find the **cheapest price** from `src` to `dst` with **at most k stops**. If no such route exists, return `-1`.

---

## Examples

**Example 1:**

**Input:**
```
n = 4
flights = [[0,1,100], [1,2,100], [2,0,100], [1,3,600], [2,3,200]]
src = 0, dst = 3, k = 1
```

**Output:** `700`

```
Graph:
    100        600
0 ─────→ 1 ─────────→ 3
↑        |             ↑
|        | 100         | 200
| 100    ↓             |
└─────── 2 ────────────┘

Paths from 0 to 3:
Path 1: 0→1→3 (stops=1, cost=100+600=700) ✓
Path 2: 0→1→2→3 (stops=2, cost=100+100+200=400) ✗ (exceeds k=1)

Answer: 700 (cheapest within k=1 stops)
```

**Example 2:**

**Input:**
```
n = 3
flights = [[0,1,100], [1,2,100], [0,2,500]]
src = 0, dst = 2, k = 1
```

**Output:** `200`

```
Graph:
    100       100
0 ─────→ 1 ─────→ 2
|                  ↑
└──────500─────────┘

Paths from 0 to 2:
Path 1: 0→2 (stops=0, cost=500)
Path 2: 0→1→2 (stops=1, cost=100+100=200) ✓

Answer: 200 (cheaper within k=1 stops)
```

## Key Insight

```
Why NOT Standard Dijkstra?

Standard Dijkstra:
→ Always processes node with minimum distance
→ Guarantees shortest path globally
→ But IGNORES the stops constraint! ✗

This problem:
→ Shortest path within K stops
→ A longer (costlier) path with fewer stops
  might be valid, while a cheaper path with
  more stops is INVALID!

Example:
Path A: cost=400, stops=2 ✗ (exceeds k=1)
Path B: cost=700, stops=1 ✓ (valid!)
Path B is ANSWER despite higher cost!

Standard Dijkstra would pick Path A → WRONG! ✗

Solution: Modified BFS
→ Track BOTH cost AND stops
→ Process level-by-level (stops = BFS levels)
→ Update distance only if stops ≤ k
→ Queue: {stops, {node, cost}}
```

---

## Algorithm

```
1. Build adjacency list from flights:
   adj[u] = {(v, price), ...}

2. Initialize:
   - Queue: {stops, {node, cost}}
   - Distance array: dist[n] = infinity
   - dist[src] = 0

3. Push source into queue: {0, {src, 0}}
   (0 stops, source node, 0 cost)

4. BFS traversal:
   While queue is not empty:
     a. Pop {stops, {node, cost}} from front
     
     b. If stops > k: continue (exceeded limit)
     
     c. For each neighbor (adjNode, edgeWeight):
        newCost = cost + edgeWeight
        
        If newCost < dist[adjNode] AND stops ≤ k:
          → Update: dist[adjNode] = newCost
          → Push: {stops+1, {adjNode, newCost}}
          
        Note: We check stops ≤ k to allow one more hop
              (stops+1 in queue might reach k+1 total)

5. After BFS:
   If dist[dst] == infinity: return -1
   Else: return dist[dst]
```

```
Why BFS instead of Dijkstra?

BFS property:
→ Processes nodes level by level
→ Level = number of stops!
→ Level 0: src (0 stops)
→ Level 1: nodes 1 hop away (1 stop)
→ Level 2: nodes 2 hops away (2 stops)
→ ...
→ Level k+1: nodes at most k stops away

Stop constraint naturally enforced by BFS levels! ✓

Why not use priority queue (Dijkstra)?
→ PQ processes by cost (not stops)
→ Might process expensive nodes with few stops
  before cheap nodes with many stops
→ Doesn't leverage the "at most k stops" constraint
→ BFS is simpler and sufficient! ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int CheapestFLight(int n, vector<vector<int>> &flights,
                       int src, int dst, int K) {
        
        // Build adjacency list
        vector<pair<int, int>> adj[n];
        for (auto it : flights) {
            adj[it[0]].push_back({it[1], it[2]});
        }

        // Queue: {stops, {node, cost}}
        queue<pair<int, pair<int, int>>> q;
        q.push({0, {src, 0}});

        // Distance array (minimum cost to reach each node)
        vector<int> dist(n, 1e9);
        dist[src] = 0;

        // BFS traversal
        while (!q.empty()) {
            auto it = q.front();
            q.pop();
            
            int stops = it.first;
            int node = it.second.first;
            int cost = it.second.second;

            // Exceeded stop limit
            if (stops > K) continue;

            // Explore neighbors
            for (auto iter : adj[node]) {
                int adjNode = iter.first;
                int edW = iter.second;

                // If cheaper path found within stop limit
                if (cost + edW < dist[adjNode] && stops <= K) {
                    dist[adjNode] = cost + edW;
                    q.push({stops + 1, {adjNode, cost + edW}});
                }
            }
        }

        // Return result
        return (dist[dst] == 1e9) ? -1 : dist[dst];
    }
};

int main() {
    int n = 4, src = 0, dst = 3, K = 1;
    vector<vector<int>> flights = {
        {0, 1, 100}, {1, 2, 100}, {2, 0, 100},
        {1, 3, 600}, {2, 3, 200}
    };

    Solution obj;
    int ans = obj.CheapestFLight(n, flights, src, dst, K);

    cout << "Cheapest price: " << ans << endl;

    return 0;
}
```

  stops=0, node=0, cost=0
  stops ≤ k=1 ✓

    newCost = 0 + 100 = 100 < dist[1]=∞
    Update: dist[1] = 100
    Push: {1, {1, 100}}

  queue = [{1, {1, 100}}]

  stops=1, node=1, cost=100
  stops ≤ k=1 ✓
  
  Neighbor (2, 100):
    newCost = 100 + 100 = 200 < dist[2]=∞
    Update: dist[2] = 200
    Push: {2, {2, 200}}
  
  Neighbor (3, 600):
    newCost = 100 + 600 = 700 < dist[3]=∞

    Push: {2, {3, 700}}

  stops > k=1 ✗ → CONTINUE (skip this iteration)

Iteration 4: Pop {2, {3, 700}}
  stops=2, node=3, cost=700
  stops > k=1 ✗ → CONTINUE

## Why Path [0→1→2→3] is Invalid

```
Path: 0 → 1 → 2 → 3
Cost: 100 + 100 + 200 = 400
Stops: 2 (visiting nodes 1 and 2 before destination)

But k = 1 (at most 1 stop allowed)!

When we pop {2, {2, 200}} from queue:
  stops=2 > k=1 → SKIP!
  Never explore neighbors of node 2
  Never update dist[3] via path 0→1→2→3

BFS naturally enforces stop constraint:
→ Level 0: src (0 stops)
→ Level 1: 1 hop away (1 stop) ✓
→ Level 2: 2 hops away (2 stops) ✗ (exceeds k=1)

Paths exceeding k stops are automatically pruned! ✓
```

## BFS Level Visualization

```
Level 0 (stops=0):
  [0]  ← source

Level 1 (stops=1):
  0 → [1]  ← 1 stop away
      cost=100

Level 2 (stops=2):
  0 → 1 → [2]  ← 2 stops away (EXCEEDS k=1 ✗)
          cost=200
  
  0 → 1 → [3]  ← 2 stops away (EXCEEDS k=1 ✗)
          cost=700

But wait! We already reached [3] in level 1:
  0 → 1 → [3]  ← Actually this uses 1 stop!
          cost=700 ✓

Clarification:
"At most k stops" = at most k intermediate nodes
NOT including src and dst!

0 → 1 → 3:
  Intermediate node: 1 (count=1)
  Stops = 1 ✓ (valid for k=1)

0 → 1 → 2 → 3:
  Intermediate nodes: 1, 2 (count=2)
  Stops = 2 ✗ (exceeds k=1)
```

## Distance Array Updates

```
Initial:    After {0,{0,0}}:    After {1,{1,100}}:
[0, ∞, ∞, ∞]  [0, 100, ∞, ∞]    [0, 100, 200, 700]

Key observation:
→ dist[3] = 700 from path 0→1→3 (stops=1)
→ Path 0→1→2→3 would give cost=400 but stops=2
→ Since stops=2 > k=1, we never explore it
→ dist[3] stays 700 ✓

If k=2 instead:
→ Path 0→1→2→3 with stops=2 would be allowed
→ dist[3] would update from 700 to 400
→ Answer would be 400 ✓

The stop constraint changes the optimal path! ✓
```

## Why Queue, Not Priority Queue?

```
Standard Dijkstra uses Priority Queue:
→ Always process node with minimum cost
→ Guarantees globally shortest path
→ But ignores stop constraint!

This problem uses Queue (BFS):
→ Process nodes level by level (by stops)
→ Respects stop constraint naturally
→ May not give globally shortest path
→ But gives shortest path within k stops! ✓

Example scenario:
Queue state: [{1,{1,100}}, {1,{2,50}}]

Priority Queue would process {1,{2,50}} first (smaller cost)
Regular Queue processes {1,{1,100}} first (FIFO)

Both are valid because:
→ Both have same number of stops (1)
→ Within same level, order doesn't matter
→ BFS explores all level k before level k+1

Simple queue is SUFFICIENT and FASTER! ✓
Time: O(E) vs O(E log V) with PQ
```

## Edge Cases

```
Direct flight exists:          No path exists:
flights = [[0,3,200]]          flights = [[0,1,100]]
k = 0                          dst = 3
Output: 200 ✓                  Output: -1 ✓
(0 stops = direct flight)

All paths exceed k:            Multiple paths, same cost:
flights = [[0,1,100],          flights = [[0,1,100],
           [1,2,100],                    [0,2,200],
           [2,3,100]]                    [1,2,100]]
k = 1, dst = 3                 k = 1, dst = 2
Output: -1                     Both cost 200, either valid
(shortest needs 2 stops)       Output: 200 ✓

k = n-1 (no real constraint):  Self-loop:
All paths valid               flights = [[0,0,100]]
Reduces to standard           Typically skipped
shortest path                 (no practical use)

Negative prices:               Very large k:
Problem states prices ≥ 0     If k ≥ n-1, constraint
No need to handle negatives   effectively removed
```

## Stops vs Edges Clarification

```
Common confusion: "stops" vs "edges"

Stops = intermediate nodes (NOT including src and dst)
Edges = number of flights taken

Example path: 0 → 1 → 2 → 3

Nodes: 0, 1, 2, 3 (4 nodes)
Intermediate nodes: 1, 2 (2 nodes)
Stops: 2 ✓
Edges/Flights: 3 (0→1, 1→2, 2→3)

"At most k stops" = at most k intermediate nodes

For k=1:
0 → 1 → 3: intermediate=[1], stops=1 ✓
0 → 1 → 2 → 3: intermediate=[1,2], stops=2 ✗

In code:
{stops, {node, cost}}
stops starts at 0 (source has 0 intermediate nodes)
stops increments each time we move to next node
When stops > k, we skip (too many intermediate nodes)
```

## Complexity Analysis

**Time Complexity:** `O(E × K)` ≈ `O(E)` for small K
- Each edge can be relaxed at most K times
- In worst case, we might visit same edge multiple times
  with different stop counts (0, 1, 2, ..., K)
- Total: **O(E × K)** where E = number of flights
- For K << V, this is effectively O(E)

**Space Complexity:** `O(V + E)`
- Adjacency list: O(E)
- Distance array: O(V)
- Queue: O(V × K) in worst case (same node with different stops)
- Total: **O(V + E)**

```
Comparison with Dijkstra:

Approach              Time              Space       Handles K?
──────────────────────────────────────────────────────────────────
BFS (this solution)   O(E × K)          O(V+E)      YES ✓
Standard Dijkstra     O(E log V)        O(V+E)      NO ✗
Modified Dijkstra     O(E × K log V)    O(V+E)      YES ✓
  with (cost, stops)

BFS is simpler and sufficient for this problem! ✓

Why BFS works here:
→ K is usually small (constraint: K ≤ V-1)
→ Stop levels naturally map to BFS levels
→ No need for PQ overhead
→ Cleaner, faster implementation
```

## Alternative: Bellman-Ford Approach

```
Idea: Run modified Bellman-Ford for K+1 iterations

Bellman-Ford property:
→ After i iterations, dist[v] = shortest path with ≤ i edges

Modified for this problem:
→ After i iterations, dist[v] = cheapest with ≤ i stops

Pseudocode:
dist[src] = 0, all others = ∞
for (i = 0; i <= K; i++):
    temp = dist.copy()  // Important: use previous iteration
    for each flight (u, v, price):
        if temp[u] != ∞:
            dist[v] = min(dist[v], temp[u] + price)
return dist[dst]

Time: O(K × E)
Space: O(V)

Example trace:
Iteration 0: dist=[0, ∞, ∞, ∞]
Iteration 1: dist=[0, 100, ∞, ∞]  (0→1)
Iteration 2: dist=[0, 100, 200, 700]  (0→1→2, 0→1→3)

Both BFS and Bellman-Ford work! Choose based on preference.
BFS is more intuitive for "stops" constraint. ✓
```

## When to Use This Pattern

```
✓ Use BFS with stops constraint when:
  → Shortest path with limited hops/stops
  → Graph traversal with level/depth limit
  → "At most K" constraint on path length
  → Weighted graph but stops matter more than total cost

✗ Don't use when:
  → Only care about minimum cost (no stop limit) → Dijkstra
  → Need globally optimal path → Dijkstra
  → Negative weights exist → Bellman-Ford

Similar problems:
→ Network Delay Time (K = ∞, use Dijkstra)
→ Bus Routes (minimize transfers = stops)
→ Word Ladder (unweighted, BFS directly)
→ Minimum Cost to Reach Destination in Time

Key pattern:
"Minimize cost with constraint on path length" = BFS with levels! ✓
```
