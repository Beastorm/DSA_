## 1. Problem Statement

You are in a city with `n` intersections numbered from `0` to `n-1` with **bi-directional roads**. The graph is connected and has at most one road between any two intersections.

Given an integer `n` and a 2D array `roads` where `roads[i] = [ui, vi, timei]` means there's a road between intersections `ui` and `vi` that takes `timei` minutes.

**Find:** In how many ways can you travel from intersection `0` to intersection `n-1` in the **shortest amount of time**.

Return the answer modulo `10^9 + 7`.

**Input Format:**

- `n` = number of intersections
- `roads` = list of bi-directional roads `[u, v, time]`

**Output:**

- Number of shortest paths from `0` to `n-1`, modulo `10^9 + 7`

## Key Insight

```
  This problem combines TWO challenges:

  1. Find shortest distance (Dijkstra's)
  2. Count number of shortest paths (DP on shortest path DAG)

  Standard Dijkstra:
  → Finds shortest distance to all nodes ✓
  → But doesn't track HOW MANY paths! ✗

  Modified Dijkstra with Counting:
  → Maintains dist[v] = shortest distance to v
  → ALSO maintains ways[v] = number of shortest paths to v
  → Updates both simultaneously! ✓

  Key insight:

  When relaxing edge u → v:

  1. If dis + wt < dist[v]:
     → Found NEW shorter path
     → ways[v] = ways[u] (inherit from u)

  2. If dis + wt == dist[v]:
     → Found ANOTHER path with same length
     → ways[v] += ways[u] (add more ways)

  This counts ALL shortest paths! ✓
```

## Intuition

```
  Imagine you are mapping all fastest routes to the airport:

  1. First, find the shortest travel time to every intersection.
  2. While doing this, keep a counter at each intersection showing
     how many different fastest routes lead to it.
  3. When you discover a faster route to an intersection, reset its
     counter to match the previous intersection's counter.
  4. When you discover an equally fast route, add the previous
     intersection's counter to the current one.

  Why it works:
  Every shortest path to the destination must pass through some
  predecessor that is also on a shortest path. By counting how many
  shortest paths reach each predecessor, and summing those counts
  whenever they lead to the destination with the same total distance,
  we accumulate the total number of shortest paths.
```

## Algorithm

```
  countPaths(n, roads):

  1. Build adjacency list from edges (bi-directional):
     adj[u] = {(v, time), ...}
     adj[v] = {(u, time), ...}

  2. Initialize:
     - Priority Queue (min-heap): {dist, node}
     - dist[0...n-1] = infinity
     - ways[0...n-1] = 0
     - dist[src] = 0
     - ways[src] = 1 (one way to reach source)

  3. Push source: {0, src} into PQ

  4. Modified Dijkstra:
     While PQ is not empty:
       a. Pop {dis, node} with minimum distance

       b. Skip outdated entries: if dis > dist[node], continue

       c. For each neighbor (adjNode, edW):
          newDist = dis + edW

          Case 1: newDist < dist[adjNode]
            → Found SHORTER path
            → dist[adjNode] = newDist
            → ways[adjNode] = ways[node] (inherit)
            → Push {newDist, adjNode} into PQ

          Case 2: newDist == dist[adjNode]
            → Found ANOTHER shortest path
            → ways[adjNode] = (ways[adjNode] + ways[node]) % mod
            → NO push (already in PQ with same dist)

          Case 3: newDist > dist[adjNode]
            → Longer path, ignore

  5. Return ways[n-1] % MOD
```

```
  Why does counting work?

  Mathematical principle:
  If node u has k shortest paths from source,
  and edge u → v is part of a shortest path to v,
  then v inherits those k paths!

  Example:
  Source → u has 3 shortest paths
  u → v is on shortest path to v
  → v also has (at least) 3 shortest paths ✓

  If multiple nodes lead to v with same shortest distance:
  Source → u1 (2 paths) → v
  Source → u2 (3 paths) → v
  → v has 2 + 3 = 5 shortest paths! ✓

  This is dynamic programming on the shortest path DAG!
```

### Why `long long` for Distance?

```
  Edge weights can be large (up to 10^5)
  Number of edges can be large (up to 10^5)

  Maximum distance = sum of all edges
  Worst case:
  100,000 edges × 100,000 weight = 10^10

  → Exceeds INT_MAX (≈ 2 × 10^9) ✗
  → Requires long long ✓

  But ways[] can be int:
  ways is always taken modulo 10^9+7
  → Never exceeds 10^9+7
  → Fits in int ✓

  Type safety:
  long long dis = pq.top().first;
  Ensures no overflow in distance calculations ✓
```

### Why Modulo 10^9 + 7?

```
  Number of paths can grow exponentially!

  Example: Grid graph
  n × n grid, shortest path from (0,0) to (n,n)
  Number of shortest paths = C(2n, n) ← Binomial coefficient

  For n=50:
  C(100, 50) ≈ 10^29 → Exceeds long long! ✗

  Solution: Take modulo at each step
  ways[adjNode] = (ways[adjNode] + ways[node]) % mod

  Properties of modulo:
  (a + b) % m = ((a % m) + (b % m)) % m
  → Safe to apply modulo incrementally ✓

  Final answer:
  return ways[n-1] % mod;

  Even though we've been applying modulo throughout,
  applying it one more time ensures correctness ✓
```

### Difference from Standard Dijkstra

```
  Standard Dijkstra:

  while (!pq.empty()) {
      auto [dis, node] = pq.top();
      pq.pop();

      for (auto [adjNode, edW] : adj[node]) {
          if (dis + edW < dist[adjNode]) {
              dist[adjNode] = dis + edW;
              pq.push({dist[adjNode], adjNode});
          }
      }
  }

  Modified Dijkstra (counting paths):

  while (!pq.empty()) {
      auto [dis, node] = pq.top();
      pq.pop();

      if (dis > dist[node]) continue;  ← Skip stale entries

      for (auto [adjNode, edW] : adj[node]) {
          // Shorter path found
          if (dis + edW < dist[adjNode]) {
              dist[adjNode] = dis + edW;
              pq.push({dist[adjNode], adjNode});
              ways[adjNode] = ways[node];  ← NEW!
          }
          // Same distance, another path
          else if (dis + edW == dist[adjNode]) {  ← NEW!
              ways[adjNode] = (ways[adjNode] + ways[node]) % mod;
          }
      }
  }

  Key additions:
  1. ways[] array to count paths
  2. Update ways[] when relaxing
  3. Add to ways[] when same distance found
  4. Skip stale PQ entries
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    int countPaths(int n, vector<vector<int>>& roads) {

        // Build adjacency list (bi-directional)
        vector<vector<pair<int, int>>> adj(n);
        for (auto it : roads) {
            adj[it[0]].push_back({it[1], it[2]});
            adj[it[1]].push_back({it[0], it[2]});
        }

        // Min-heap: {distance, node}
        priority_queue<pair<long long, int>,
                       vector<pair<long long, int>>,
                       greater<pair<long long, int>>> pq;

        // Distance and ways arrays
        vector<long long> dist(n, LLONG_MAX);
        vector<int> ways(n, 0);

        // Initialize source
        dist[0] = 0;
        ways[0] = 1;
        pq.push({0, 0});

        int mod = (int)(1e9 + 7);

        // Modified Dijkstra
        while (!pq.empty()) {
            long long dis = pq.top().first;
            int node = pq.top().second;
            pq.pop();

            // Skip outdated entries
            if (dis > dist[node]) continue;

            // Explore neighbors
            for (auto it : adj[node]) {
                int adjNode = it.first;
                long long edW = it.second;

                // Case 1: Found shorter path
                if (dis + edW < dist[adjNode]) {
                    dist[adjNode] = dis + edW;
                    pq.push({dis + edW, adjNode});
                    ways[adjNode] = ways[node];
                }
                // Case 2: Found another shortest path
                else if (dis + edW == dist[adjNode]) {
                    ways[adjNode] = (ways[adjNode] + ways[node]) % mod;
                }
            }
        }

        return ways[n - 1] % mod;
    }
};

int main() {
    int n = 7;
    vector<vector<int>> edges = {
        {0, 6, 7}, {0, 1, 2}, {1, 2, 3}, {1, 3, 3}, {6, 3, 3},
        {3, 5, 1}, {6, 5, 1}, {2, 5, 1}, {0, 4, 5}, {4, 6, 2}
    };

    Solution obj;
    int ans = obj.countPaths(n, edges);

    cout << "Number of shortest paths: " << ans << endl;

    return 0;
}
```

```
  Simplified example:
  n = 4, edges = [[0,1,1], [1,2,1], [0,2,2], [1,3,1], [2,3,1]]

  Graph:

      1       1
  0 ────→ 1 ────→ 3
  │       │       ↑
  2       1       1
  │       ↓       │
  └─────→ 2 ──────┘

  ── Initialization ──

  dist = [0, ∞, ∞, ∞]
  ways = [1, 0, 0, 0]
  pq = [{0, 0}]

  ── Iterations ──

  Iteration 1: Pop {0, 0}
      node=0, dis=0

      Neighbor 1 (weight=1):
        newDist = 0 + 1 = 1 < dist[1]=∞
        → dist[1] = 1, ways[1] = ways[0] = 1
        → Push {1, 1}

      Neighbor 2 (weight=2):
        newDist = 0 + 2 = 2 < dist[2]=∞
        → dist[2] = 2, ways[2] = ways[0] = 1
        → Push {2, 2}

      dist = [0, 1, 2, ∞]
      ways = [1, 1, 1, 0]
      pq = [{1, 1}, {2, 2}]

  Iteration 2: Pop {1, 1}
      node=1, dis=1

      Neighbor 0 (weight=1):
        newDist = 1 + 1 = 2 > dist[0]=0 → skip

      Neighbor 2 (weight=1):
        newDist = 1 + 1 = 2 == dist[2]=2
        → SAME distance! Another shortest path!
        → ways[2] = (1 + 1) % mod = 2

      Neighbor 3 (weight=1):
        newDist = 1 + 1 = 2 < dist[3]=∞
        → dist[3] = 2, ways[3] = ways[1] = 1
        → Push {2, 3}

      dist = [0, 1, 2, 2]
      ways = [1, 1, 2, 1]
      pq = [{2, 2}, {2, 3}]

  Iteration 3: Pop {2, 2}
      node=2, dis=2

      Neighbor 0: skip (longer)
      Neighbor 1: skip (longer)

      Neighbor 3 (weight=1):
        newDist = 2 + 1 = 3 > dist[3]=2 → skip

      pq = [{2, 3}]

  Iteration 4: Pop {2, 3}
      node=3, dis=2
      No neighbors (destination)

      pq = [] (empty)

  ── Result ──

  dist[3] = 2
  ways[3] = 1 (only via 0→1→3)

  Wait, what about 0→2→3?
  newDist = 0→2(2) + 2→3(1) = 3 > dist[3]=2
  It's LONGER, so it's ignored! ✓

  Paths to node 2:
  1. 0 → 2 (distance=2)
  2. 0 → 1 → 2 (distance=2)
  ways[2] = 2 ✓

  Paths to node 3:
  1. 0 → 1 → 3 (distance=2)
  ways[3] = 1 ✓
```

## Edge Cases

```
  Single node:
  n = 1
  ways[0] = 1
  Return: 1 ✓

  Direct edge to destination:
  0 ──5──→ n-1
  Only 1 path
  ways[n-1] = 1 ✓

  Multiple edges same length:
  0 ──2──→ 1
  |        |
  └──2──→ ─┘
  Both paths same length
  ways[1] = 2 ✓

  Disconnected graph:
  0 ──→ 1    2 ──→ 3
  (destination unreachable)
  dist[n-1] = ∞
  ways[n-1] = 0
  Return: 0 ✓

  All paths same length:
  Diamond graph:
      0
     / \
    1   2
     \ /
      3
  ways[3] = 2 ✓

  Very long path:
  Chain: 0→1→2→...→n-1
  Each edge weight = 1
  Only 1 path
  ways[n-1] = 1 ✓

  Large answer:
  Exponential paths
  Need modulo ✓

  Zero-weight edges:
  Allowed (non-negative)
  Doesn't break Dijkstra ✓
```

## Complexity Analysis

**Time Complexity:** O(E × log V)

- Building adjacency list: O(E)
- Modified Dijkstra:
  - Each vertex processed at most once: O(V)
  - Each edge relaxed at most once: O(E)
  - Each PQ operation: O(log V)
- Total: **O(E × log V)**

**Space Complexity:** O(V + E)

- Adjacency list: O(E)
- Distance array: O(V)
- Ways array: O(V)
- Priority queue: O(V) at most
- Total: **O(V + E)**

```
  Where V = n (number of nodes)
        E = |roads| (number of edges)

  Compared to standard Dijkstra:
  → Same time complexity ✓
  → Only adds O(V) for ways array
  → Minimal overhead for counting!

  The elegance:
  We count paths "for free" during Dijkstra
  without any asymptotic slowdown! ✓
```

## Interview Questions

```
  Q1: Why use Dijkstra instead of BFS for this problem?

  A: Edge weights (road times) can vary. BFS only works for unit
     weights. Dijkstra correctly finds shortest paths in graphs
     with non-negative edge weights, and we can simultaneously
     count the number of such paths.

  ──────────────────────────────────────────────────────────────

  Q2: When do you reset the ways counter, and when do you add to it?

  A: Reset (ways[v] = ways[u]) when a strictly shorter path to v
     is found through u. Add (ways[v] += ways[u]) when a path of
     exactly the same shortest length is found through u.

  ──────────────────────────────────────────────────────────────

  Q3: Why do we skip stale priority queue entries with
      if (dis > dist[node]) continue?

  A: A node may be inserted multiple times with different distances
     as we discover better paths. When we pop an entry with a
     distance larger than the current known shortest distance,
     that entry is outdated and should be ignored.

  ──────────────────────────────────────────────────────────────

  Q4: Why use long long for distances but int for ways?

  A: Distances can sum to values exceeding INT_MAX (e.g., 10^10),
     so long long prevents overflow. Ways are always kept modulo
     10^9+7, so they never exceed that value and fit in int.

  ──────────────────────────────────────────────────────────────

  Q5: Why take modulo 10^9 + 7?

  A: The number of shortest paths can grow exponentially (e.g.,
     binomial coefficients in grid graphs). Taking modulo prevents
     overflow and is a standard requirement in competitive
     programming problems.

  ──────────────────────────────────────────────────────────────

  Q6: What happens if two different predecessors discover the same
      shortest distance to a node?

  A: We add their ways together. For example, if both u1 and u2 can
     reach v with distance equal to dist[v], then ways[v] becomes
     ways[u1] + ways[u2] (modulo applied).

  ──────────────────────────────────────────────────────────────

  Q7: Does the order of processing nodes affect the final ways count?

  A: No. Dijkstra processes nodes in order of increasing shortest
     distance. By the time we process a node, all shortest paths
     to it have already been discovered and counted. The final
     ways count is independent of processing order among same-level
     nodes.

  ──────────────────────────────────────────────────────────────

  Q8: Can this approach work on directed graphs?

  A: Yes. The algorithm is the same. The only change is building
     the adjacency list in only the specified direction for each
     directed edge.

  ──────────────────────────────────────────────────────────────

  Q9: What if the graph has negative edge weights?

  A: Dijkstra fails with negative weights. For counting shortest
     paths with possible negative weights, you would need
     Bellman-Ford or SPFA, with careful handling of negative cycles.

  ──────────────────────────────────────────────────────────────

  Q10: How would you count ALL paths from source to destination,
       not just shortest ones?

  A: If the graph is a DAG, use topological sort and dynamic
      programming: ways[v] += ways[u] for every edge u→v.

      If the graph has cycles, the problem is more complex and may
      have infinitely many paths unless cycles are handled.

  ──────────────────────────────────────────────────────────────

  Q11: What is the time complexity, and why is it the same as
       standard Dijkstra?

  A: O(E log V). The only extra work per edge is a constant-time
     update to the ways array. We do not add any additional heap
     operations, so the asymptotic complexity remains unchanged.

  ──────────────────────────────────────────────────────────────

  Q12: Why don't we push into the priority queue when we find an
       equally short path?

  A: Because the distance to that node hasn't changed. The node is
     already in the queue (or has been processed) with the same
     shortest distance. Pushing again would create a stale entry
     that we'd later skip.

  ──────────────────────────────────────────────────────────────

  Q13: What if there are multiple edges between the same pair of nodes?

  A: The algorithm handles each edge independently during relaxation.
     If two edges have different weights, only the shorter one will
     contribute to shortest paths. If they have the same weight,
     they represent distinct paths and both contribute to the count.

  ──────────────────────────────────────────────────────────────

  Q14: How would you reconstruct one actual shortest path, not just
       the count?

  A: Maintain a parent array. When relaxing an edge u→v that gives
     a strictly shorter distance, set parent[v] = u. When an equal
     distance is found, you can either keep the existing parent or
     track multiple parents for all shortest paths.

  ──────────────────────────────────────────────────────────────

  Q15: What are the most common mistakes in this implementation?

  A: Common mistakes and fixes:

      Mistake 1: Not using modulo
      → ways[v] = (ways[v] + ways[u]) % mod

      Mistake 2: Using int for distances
      → Use long long for dist array and heap keys

      Mistake 3: Initializing ways[src] to 0
      → ways[src] = 1 (exactly one way to be at the source)

      Mistake 4: Updating ways on longer paths
      → Only update when dis + edW <= dist[v]

      Mistake 5: Forgetting bi-directional edges
      → Add both u→v and v→u to adjacency list

      Mistake 6: Not skipping stale heap entries
      → Add if (dis > dist[node]) continue
```
