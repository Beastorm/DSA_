## 1. Problem Statement

Given a **weighted, directed and connected graph** of `V` vertices and `E` edges, find the **shortest distance** of all vertices from source vertex `S`.

> **Note:** If the graph contains a **negative cycle**, return an array consisting of only `-1`.

**Input Format:**

- `V` = number of vertices
- `edges` = list of directed edges `[u, v, wt]`
- `S` = source vertex

**Output:**

- Array of shortest distances from `S` to all vertices, or `[-1]` if negative cycle exists

## Key Insight

```
Why Bellman-Ford when we have Dijkstra?

Dijkstra's Algorithm:

→ Works with non-negative weights only ✓
→ Fails with negative weights ✗
→ Cannot detect negative cycles ✗
→ Faster: O(E log V) with priority queue

Bellman-Ford Algorithm:

→ Works with negative weights ✓
→ Can detect negative cycles ✓
→ Slower: O(V × E)
→ Simpler implementation

When to use Bellman-Ford:

1. Graph has negative edge weights
2. Need to detect negative cycles
3. Simpler code preferred (no PQ needed)

Negative cycle example:

A → B (weight = 1)
B → C (weight = -3)
C → A (weight = 1)

Total cycle weight = 1 + (-3) + 1 = -1 < 0

Going around this cycle repeatedly:
→ Distance keeps decreasing
→ No shortest path exists!
→ Must detect and report this! ✓
```

## Intuition

```
Imagine you are gradually improving your knowledge of shortest paths:

Round 1: You learn all 1-edge paths from source
Round 2: You learn all 2-edge paths (building on round 1)
Round 3: You learn all 3-edge paths (building on round 2)
...
Round V-1: You learn all (V-1)-edge paths

After V-1 rounds, you know every possible shortest path!

Why? Because any simple path has at most V-1 edges.
If a path uses V or more edges, it must revisit a vertex,
meaning it contains a cycle.

Then comes the crucial check:
Round V: Try to improve any distance again.

If you CAN still improve:
→ The improvement must come from a cycle
→ Only a negative cycle can keep improving distances
→ Report that no valid shortest paths exist! ✗

If you CANNOT improve:
→ All distances are truly optimal
→ Return the shortest distances ✓
```

## Algorithm

```
bellman_ford(V, edges, S):

1. Initialize:
   - dist[V] = infinity for all vertices
   - dist[S] = 0 (source)

2. Relax all edges V-1 times:
   for i = 0 to V-2:
     for each edge (u, v, wt):
       if dist[u] + wt < dist[v]:
         dist[v] = dist[u] + wt
   
   Why V-1 iterations?
   → In a graph with V vertices
   → Longest simple path has at most V-1 edges
   → After V-1 relaxations, all shortest paths found! ✓

3. Check for negative cycle (V-th relaxation):
   for each edge (u, v, wt):
     if dist[u] + wt < dist[v]:
       → Negative cycle detected!
       → return [-1]
   
   If any distance still improves after V-1 iterations:
   → Must be due to negative cycle
   → No finite shortest paths exist!

4. Return dist[] if no negative cycle
```

### Why V-1 Iterations Suffice

```
Mathematical proof:

Lemma: After k iterations of Bellman-Ford,
       dist[v] = shortest path using ≤ k edges

Proof by induction:

Base case (k=0):
  Only dist[S] = 0, all others = ∞
  Correct! Source has 0-edge path to itself.

Inductive step:
  Assume true for k iterations.
  In iteration k+1:
    For edge u→v:
      dist[v] = min(dist[v], dist[u] + wt)
    
    By induction, dist[u] = shortest using ≤ k edges
    So dist[u] + wt = shortest to v using ≤ k+1 edges
    Update captures this! ✓

After V-1 iterations:
  dist[v] = shortest path using ≤ V-1 edges
  
  But shortest simple path has ≤ V-1 edges!
  (V vertices, V-1 edges max in simple path)
  
  Therefore, dist[v] = actual shortest distance! ✓

V-th iteration:
  If any update happens:
  → Using V edges in path
  → Must revisit some vertex (only V vertices)
  → Must be a cycle
  → If improvement happens, cycle is negative! ✗
```

### What is "Relaxation"?

```
Edge u → v with weight wt:

Current situation:
dist[u] = 10
dist[v] = 20
wt = 5

Question: Is path through u better?

Path through u: dist[u] + wt = 10 + 5 = 15
Current dist[v]: 20

15 < 20 → YES, better! Update dist[v] = 15 ✓

Visual:

     dist[u]=10
        (u)
         |
      wt=5
         ↓
        (v)
     dist[v]=20 → update to 15 ✓

This is "relaxing" edge u→v!

We "relax" the constraint dist[v] ≥ dist[u] + wt
```

### Why Check dist[u] != 1e8?

```
Code line:

if (dist[u] != 1e8 && dist[u] + wt < dist[v]) {
   ↑ Why this check?

Reason: Prevent updating from unreachable nodes

Example:
dist[u] = ∞ (unreachable from source)
Edge u → v with weight -10

Without check:
dist[u] + wt = ∞ + (-10) = still ∞ (in theory)

But in practice:
1e8 + (-10) = 99999990 < dist[v]=1e8? NO
Still works, but...

What if wt is very negative?
dist[u] = 1e8, wt = -2e8
dist[u] + wt = -1e8 < dist[v]=1e8? YES ✗

WRONG update from unreachable node!

With check:
if (dist[u] != 1e8 && ...)
→ Only relax from REACHABLE nodes ✓
→ Correct semantics
→ Prevents integer overflow issues

Best practice: Always check reachability! ✓
```

### Optimization: Early Termination

```
Standard Bellman-Ford:

Always does V-1 iterations, even if done early

Optimized version:

for (int i = 0; i < V - 1; i++) {
    bool updated = false;  ← Track updates
    
    for (auto it : edges) {
        int u = it[0], v = it[1], wt = it[2];
        if (dist[u] != 1e8 && dist[u] + wt < dist[v]) {
            dist[v] = dist[u] + wt;
            updated = true;  ← Mark update
        }
    }
    
    if (!updated) break;  ← Early exit! ✓
}

Benefits:
→ If no update in iteration i, won't be any in i+1
→ Can stop early and save time! ✓

Example:
V = 100, but shortest paths found in 5 iterations
→ Standard: 99 iterations
→ Optimized: 5 iterations ✓
→ Huge speedup in practice!

Still O(V×E) worst case, but much faster average! ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    vector<int> bellman_ford(int V, vector<vector<int>>& edges, int S) {
        
        // Initialize distances
        vector<int> dist(V, 1e8);
        dist[S] = 0;
        
        // Relax all edges V-1 times
        for (int i = 0; i < V - 1; i++) {
            for (auto it : edges) {
                int u = it[0];   // source
                int v = it[1];   // destination
                int wt = it[2];  // weight
                
                // Relaxation step
                if (dist[u] != 1e8 && dist[u] + wt < dist[v]) {
                    dist[v] = dist[u] + wt;
                }
            }
        }
        
        // V-th relaxation: Check for negative cycle
        for (auto it : edges) {
            int u = it[0];
            int v = it[1];
            int wt = it[2];
            
            // If still relaxable, negative cycle exists
            if (dist[u] != 1e8 && dist[u] + wt < dist[v]) {
                return {-1};  // Negative cycle detected
            }
        }
        
        return dist;
    }
};

int main() {
    int V = 6;
    vector<vector<int>> edges = {
        {3, 2, 6},
        {5, 3, 1},
        {0, 1, 5},
        {1, 5, -3},
        {1, 2, -2},
        {3, 4, -2},
        {2, 4, 3}
    };
    
    int S = 0;
    Solution obj;
    vector<int> dist = obj.bellman_ford(V, edges, S);
    
    cout << "Shortest distances: ";
    for (int d : dist) {
        cout << d << " ";
    }
    cout << endl;
    
    return 0;
}
```

### Bellman-Ford vs Dijkstra

```
Property              Bellman-Ford           Dijkstra
──────────────────────────────────────────────────────────────
Time Complexity       O(V × E)               O(E log V)
Negative Weights      YES ✓                  NO ✗
Negative Cycle Det.   YES ✓                  NO ✗
Data Structure        Array                  Priority Queue
Implementation        Simpler ✓              More complex
Space Complexity      O(V)                   O(V + E)
Use Case              Negative edges         Non-negative edges

Example where Dijkstra fails:

Edges: [0→1 (1), 1→2 (-5), 0→2 (2)]

Dijkstra:
1. Process 0: dist = [0, 1, 2]
2. Process 1 (min): dist = [0, 1, -4]
   But Dijkstra already finalized dist[2]=2!
   Doesn't update to -4 → WRONG! ✗

Bellman-Ford:
1. Iteration 1: dist = [0, 1, 2]
2. Iteration 2: dist = [0, 1, -4] ✓
   Correctly finds shortest path 0→1→2 = -4

When to use which?
→ Non-negative weights + speed needed → Dijkstra
→ Negative weights or cycle detection → Bellman-Ford
```

## Complexity Analysis

**Time Complexity:** `O(V × E)`

- V-1 iterations: O(V)
- Each iteration processes all E edges: O(E)
- Negative cycle check: O(E)
- Total: **O(V × E)**

**Space Complexity:** `O(V)`

- Distance array: O(V)
- No additional data structures needed
- Total: **O(V)**

```
Detailed analysis:

Best case: O(V × E)
  Even if distances stabilize early,
  we still do V-1 iterations (no early exit)

Worst case: O(V × E)
  All iterations make updates

Average case: O(V × E)
  Same (no optimization in standard version)

Optimization possible:
  Track if any update happened in iteration
  If no update, can exit early! ✓
  
  bool updated = false;
  for each edge: if (relax) updated = true;
  if (!updated) break;

Space is minimal:
  Only O(V) for dist array
  No queue, no heap, no visited array! ✓

Comparison with other algorithms:

Algorithm          Time            Space     Neg Weights?
─────────────────────────────────────────────────────────
Bellman-Ford       O(V×E)          O(V)      YES ✓
Dijkstra (PQ)      O(E log V)      O(V+E)    NO ✗
Floyd-Warshall     O(V³)           O(V²)     YES ✓
BFS (unweighted)   O(V+E)          O(V)      N/A
```

## Interview Questions

```
Q1: Why does Bellman-Ford need V-1 iterations?

A: In a graph with V vertices, the longest simple path has at most
   V-1 edges. Each iteration k finds the shortest paths using at
   most k edges. After V-1 iterations, all shortest simple paths
   are discovered. The V-th iteration checks if any distance can
   still improve, which would indicate a negative cycle.

──────────────────────────────────────────────────────────────

Q2: How does Bellman-Ford detect negative cycles?

A: After V-1 iterations, all shortest simple paths should be found.
   If we perform one more (V-th) relaxation pass and any distance
   still improves, that improvement must come from traversing a
   cycle. Since only a negative cycle can reduce the total path
   weight indefinitely, we conclude a negative cycle exists and
   return [-1].

──────────────────────────────────────────────────────────────

Q3: Why does Dijkstra fail with negative weights but Bellman-Ford
    works?

A: Dijkstra greedily finalizes the shortest distance to a node
   when it is popped from the priority queue, assuming that
   non-negative edges cannot produce a shorter path later.
   With negative edges, a longer initial path might later be
   reduced by a negative edge, but Dijkstra has already locked
   in the wrong answer. Bellman-Ford does not finalize any node
   early; it iteratively relaxes all edges, allowing negative
   edges to propagate their effects fully.

──────────────────────────────────────────────────────────────

Q4: What does the "relaxation" step actually do?

A: Relaxation checks if going through an intermediate node u
   provides a shorter path to node v than the currently known
   distance. If dist[u] + wt(u,v) < dist[v], we update dist[v]
   to this smaller value. The term "relax" refers to tightening
   the upper bound on the shortest path distance.

──────────────────────────────────────────────────────────────

Q5: Why do we skip edges from unreachable nodes (dist[u] == INF)?

A: If u is unreachable from the source, its distance should be
   infinity, and any path through u is invalid. Without this
   check, a very negative edge weight from an unreachable node
   could cause an incorrect update (due to finite infinity
   representation). The guard ensures we only propagate valid
   paths from reachable vertices.

──────────────────────────────────────────────────────────────

Q6: Can Bellman-Ford handle undirected graphs?

A: Yes. For an undirected edge (u, v) with weight wt, simply add
   two directed edges: u→v (wt) and v→u (wt). If the weight is
   negative, be aware that u→v and v→u form a cycle of weight
   2*wt, which is negative if wt < 0. So negative-weight
   undirected edges immediately create negative cycles.

──────────────────────────────────────────────────────────────

Q7: What is the early termination optimization?

A: In any iteration, if no distance value changes, then no future
   iteration will change anything either (because the same edges
   will be checked). We can break early. This doesn't improve
   worst-case complexity but often speeds up real-world cases
   dramatically.

──────────────────────────────────────────────────────────────

Q8: How does Bellman-Ford compare to SPFA (Shortest Path Faster
    Algorithm)?

A: SPFA is a queue-based optimization of Bellman-Ford that only
   relaxes edges from nodes whose distances recently changed.
   It is often faster in practice (especially on sparse graphs)
   but has O(V×E) worst-case time. It is also more complex and
   can be slower on graphs crafted to exploit its queue behavior.
   Bellman-Ford is simpler and more predictable.

──────────────────────────────────────────────────────────────

Q9: Can we find the actual shortest path, not just distances?

A: Yes. Maintain a parent array alongside dist[]. Whenever we
   relax an edge u→v (i.e., update dist[v]), set parent[v] = u.
   After the algorithm completes, trace back from any node using
   the parent array to reconstruct the path from the source.
   This works as long as there is no negative cycle.

──────────────────────────────────────────────────────────────

Q10: What happens if there are multiple negative cycles?

A: Bellman-Ford detects the existence of at least one negative
    cycle reachable from the source. It does not identify all
    such cycles or their locations. The V-th relaxation check
    returns [-1] as soon as it finds any edge that can still be
    relaxed, which proves a negative cycle exists somewhere.

──────────────────────────────────────────────────────────────

Q11: Is Bellman-Ford suitable for large graphs?

A: With O(V×E) time, Bellman-Ford is too slow for very large
    dense graphs (e.g., V=10^4, E=10^8). For such cases with
    non-negative weights, Dijkstra is preferred. However, if
    negative weights or cycle detection are required, Bellman-
    Ford or SPFA are the standard choices. For all-pairs
    shortest paths with negative weights, Floyd-Warshall
    (O(V³)) may be used for small dense graphs.

──────────────────────────────────────────────────────────────

Q12: Why use 1e8 as infinity instead of INT_MAX?

A: Using INT_MAX risks integer overflow when adding a weight
    (e.g., INT_MAX + (-5) causes undefined behavior). A large
    finite value like 1e8 provides enough headroom for typical
    competitive programming constraints while avoiding overflow
    in dist[u] + wt. For very large weights, use a larger
    sentinel or long long.

──────────────────────────────────────────────────────────────

Q13: Can Bellman-Ford detect a negative cycle that is NOT
     reachable from the source?

A: No. The algorithm only relaxes edges from reachable nodes.
    A negative cycle in an unreachable component will never be
    detected because dist[] values for those nodes remain at
    infinity and are never updated. If detecting all negative
    cycles is required, run Bellman-Ford from a super-source
    connected to all nodes with zero-weight edges.

──────────────────────────────────────────────────────────────

Q14: What are real-world applications of Bellman-Ford?

A: Key applications include:
    → Currency arbitrage: Detecting profitable exchange cycles
      (negative cycle = arbitrage opportunity)
    → Network routing: Distance-vector protocols like RIP and
      BGP use similar relaxation logic
    → Game theory: Solving shortest path problems in games with
      penalties (negative rewards)
    → Financial modeling: Detecting inconsistencies in pricing
      that allow risk-free profit

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes in implementing
     Bellman-Ford?

A: Common mistakes and fixes:

    Mistake 1: Forgetting the V-th iteration for cycle check
    → Always add the extra pass over all edges

    Mistake 2: Not checking dist[u] != INF before relaxing
    → Guard prevents incorrect updates from unreachable nodes

    Mistake 3: Using the wrong infinity value causing overflow
    → Use a safe sentinel like 1e8 or long long

    Mistake 4: Returning too early on first negative cycle sign
    → Actually correct to return [-1] immediately ✓

    Mistake 5: Processing edges in wrong order or missing edges
    → Process ALL edges in each iteration, every time

    Mistake 6: Confusing directed vs undirected edges
    → For undirected, add both directions explicitly

    Mistake 7: Not initializing dist[S] = 0
    → Source distance must be 0, all others infinity
```
