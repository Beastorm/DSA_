## 1. Problem Statement

Given a graph of `V` vertices numbered from `0` to `V-1`, find the **shortest distances between every pair of vertices** in an edge-weighted directed graph.

The graph is represented as an **adjacency matrix** of size `n × n`:

- `matrix[i][j]` = weight of edge from `i` to `j`
- `matrix[i][j] = -1` means no edge from `i` to `j`

**Input Format:**

- `matrix` = `V × V` adjacency matrix

**Output:**

- Updated matrix where `matrix[i][j]` = shortest distance from `i` to `j`

## Key Insight

```
What is Floyd-Warshall?

All-Pairs Shortest Path algorithm:

→ Finds shortest path between ALL pairs of vertices
→ Works with negative edge weights ✓
→ Can detect negative cycles ✓
→ Uses Dynamic Programming on intermediate vertices

Comparison with other algorithms:

Algorithm          Type            Time        Neg Weights?
─────────────────────────────────────────────────────────────
Dijkstra           Single-source   O(E log V)  NO ✗
Bellman-Ford       Single-source   O(V×E)      YES ✓
Floyd-Warshall     All-pairs       O(V³)       YES ✓

When to use Floyd-Warshall:

1. Need shortest paths between ALL pairs
2. Graph is dense (many edges)
3. Graph is small to medium (V ≤ 400-500)
4. Negative edges present (but no negative cycles)
5. Simple implementation preferred (3 nested loops!)

Core idea:

For each intermediate vertex k:
  For each pair (i, j):
    Is path i→k→j better than direct i→j?
    If yes, update!
```

## Intuition

```
Imagine you are gradually allowed to use more cities as stopovers:

Initially: You can only fly direct (no stopovers)
→ matrix[i][j] = direct flight cost

Phase 1: You may stop at city 0
→ Check if i→0→j is cheaper than direct i→j
→ Update all pairs accordingly

Phase 2: You may also stop at city 1
→ Check if i→1→j is cheaper
→ But now "i→1" and "1→j" might already use city 0!
→ So paths can use cities {0, 1} as intermediates

...

Phase V: You may stop at any city {0, 1, ..., V-1}
→ All possible paths considered!
→ matrix[i][j] = true shortest distance ✓

Why does this work?

Every shortest path is built from shorter subpaths.
If the shortest path from i to j goes through k,
then the i→k part and k→j part are also shortest paths
using only smaller intermediate vertices.

This optimal substructure is what makes DP work! ✓
```

## Algorithm

```
Floyd-Warshall Dynamic Programming:

State: dist[i][j] = shortest distance from i to j
       using vertices {0, 1, ..., k} as intermediates

Recurrence:

dist[i][j] = min(
  dist[i][j],              // Direct path (no k)
  dist[i][k] + dist[k][j]  // Via intermediate k
)

Process:

1. Initialize: dist = adjacency matrix
   dist[i][j] = {
     0           if i == j
     weight(i,j) if edge exists
     ∞ (or -1)   if no edge
   }

2. For each intermediate vertex k (0 to V-1):
     For each source vertex i (0 to V-1):
       For each destination vertex j (0 to V-1):
         if dist[i][k] + dist[k][j] < dist[i][j]:
           dist[i][j] = dist[i][k] + dist[k][j]

3. After all iterations:
   dist[i][j] = shortest path from i to j

Key observation:

→ Order of k loop MUST be outermost! ✓
→ After k iterations, dist[i][j] uses vertices {0,...,k}
→ This is the DP dimension!
```

### Why k Must Be Outermost Loop

```
CRITICAL: Order matters!

Correct order:

for k in 0 to V-1:
  for i in 0 to V-1:
    for j in 0 to V-1:
      dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Why k must be outer?

DP interpretation:

After k iterations:
  dist[i][j] = shortest path using vertices {0,1,...,k}

If k is inner loop (WRONG):

for i in 0 to V-1:
  for j in 0 to V-1:
    for k in 0 to V-1:  ← WRONG!
      dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Problem:

When computing dist[i][j], we try ALL k at once
But dist[i][k] and dist[k][j] aren't finalized yet!
Missing intermediate optimizations!

Example where wrong order fails:

Edges: 0→1 (1), 1→2 (1), 0→2 (10)

Correct (k outer):
k=0: No change
k=1: dist[0][2] = min(10, dist[0][1]+dist[1][2])
                = min(10, 1+1) = 2 ✓

Wrong (k inner):
i=0, j=2:
  k=0: dist[0][2] = min(10, 0+10) = 10
  k=1: dist[0][2] = min(10, 1+1) = 2

Still works? But only by luck!
More complex example shows failure...

Trust the DP order! ✓
```

### Handling -1 (No Edge)

```
Problem representation uses -1 for "no edge"

Two approaches:

Approach 1: Convert -1 to infinity
(Used in provided code)

// Before algorithm
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        if (matrix[i][j] == -1) {
            matrix[i][j] = 1e9;  // infinity
        }
    }
}

// Run Floyd-Warshall with 1e9

// After algorithm
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        if (matrix[i][j] >= 1e9) {
            matrix[i][j] = -1;  // unreachable
        }
    }
}

Approach 2: Check for -1 in loops
(Alternative)

for (int k = 0; k < n; k++) {
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            if (matrix[i][k] == -1 || matrix[k][j] == -1)
                continue;  // Skip if intermediate path invalid
            
            if (matrix[i][j] == -1) {
                matrix[i][j] = matrix[i][k] + matrix[k][j];
            } else {
                matrix[i][j] = min(matrix[i][j],
                                   matrix[i][k] + matrix[k][j]);
            }
        }
    }
}

Both work! First is cleaner. ✓
```

### Negative Cycle Detection

```
Floyd-Warshall can detect negative cycles!

After algorithm completes:

Check diagonal: if dist[i][i] < 0 for any i
→ Negative cycle exists! ✗

Why?

dist[i][i] = shortest path from i to i (cycle)
If < 0, there's a negative cycle involving i!

Code to detect:

for (int i = 0; i < n; i++) {
    if (matrix[i][i] < 0) {
        cout << "Negative cycle detected!" << endl;
        return;
    }
}

Example with negative cycle:

Edges: 0→1 (1), 1→2 (-5), 2→0 (1)
Cycle: 0→1→2→0 = 1+(-5)+1 = -3 < 0

After Floyd-Warshall:
dist[0][0] will become negative!
dist[1][1] will become negative!
dist[2][2] will become negative!

→ Cycle detected ✓

If negative cycle exists:
Shortest paths are undefined (keep getting shorter)
Report the cycle or return error ✗
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    void shortest_distance(vector<vector<int>> &matrix) {
        
        int n = matrix.size();
        
        // Replace -1 with large value for unreachable
        for (int i = 0; i < n; i++) {
            for (int j = 0; j < n; j++) {
                if (matrix[i][j] == -1) {
                    matrix[i][j] = 1e9;
                }
                // Diagonal should be 0
                if (i == j) matrix[i][j] = 0;
            }
        }
        
        // Floyd-Warshall: Try each vertex as intermediate
        for (int k = 0; k < n; k++) {
            for (int i = 0; i < n; i++) {
                for (int j = 0; j < n; j++) {
                    // Relax via k
                    matrix[i][j] = min(
                        matrix[i][j],
                        matrix[i][k] + matrix[k][j]
                    );
                }
            }
        }
        
        // Convert back: large value → -1 (unreachable)
        for (int i = 0; i < n; i++) {
            for (int j = 0; j < n; j++) {
                if (matrix[i][j] >= 1e9) {
                    matrix[i][j] = -1;
                }
            }
        }
    }
};

int main() {
    vector<vector<int>> matrix = {
        {0, 2, -1, -1},
        {1, 0, 3, -1},
        {-1, -1, 0, -1},
        {3, 5, 4, 0}
    };
    
    Solution sol;
    sol.shortest_distance(matrix);
    
    int n = matrix.size();
    cout << "Shortest distance matrix:\n";
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            cout << matrix[i][j] << " ";
        }
        cout << "\n";
    }
    
    return 0;
}
```

### Floyd-Warshall vs Other Algorithms

```
Algorithm          Type        Time        Space    Neg Weights?  All Pairs?
────────────────────────────────────────────────────────────────────────────
Floyd-Warshall     DP          O(V³)       O(V²)    YES ✓         YES ✓
Dijkstra (V times) Greedy      O(V²logV)   O(V²)    NO ✗          YES ✓
Bellman-Ford (V)   DP          O(V²×E)     O(V²)    YES ✓         YES ✓
Johnson's          Both        O(V²logV+VE) O(V²)   YES ✓         YES ✓

When to use Floyd-Warshall:

✓ Small to medium graph (V ≤ 400-500)
✓ Dense graph (many edges)
✓ Need all-pairs shortest paths
✓ Negative weights present
✓ Simple implementation desired

When NOT to use:

✗ Large graph (V > 1000) → Use Johnson's
✗ Sparse graph → Use Dijkstra V times
✗ Only single-source needed → Use Dijkstra/Bellman-Ford
✗ Unweighted graph → Use BFS V times (faster)

Special cases:

→ If only non-negative weights: Dijkstra V times
→ If very large graph: Johnson's algorithm
→ If detecting cycles: Bellman-Ford or Floyd-Warshall
```

## Edge Cases

```
Single vertex:           No edges:
matrix = [[0]]          matrix = [[0, -1],
Output: [[0]]                     [-1, 0]]

                                 [-1, 0]]
                        No paths to update ✓

matrix = [[0, -1],      matrix = [[0, -1],
          [-1, -5]]               [-1, -5]]

All pairs connected     matrix = [[0, -1, -1],

                        No paths between components
                        Stays -1 ✓

matrix = [[0, 1],       Need to ensure infinity value
          [1, 0]]       is larger than sum of any path
Symmetric ✓             Use 1e9 or LLONG_MAX/2 ✓

Negative weights        Zero-weight edges:
(no negative cycle):    Allowed! ✓
Algorithm works ✓       Represents "free" transitions

## Complexity Analysis

**Time Complexity:** `O(V³)`

- Three nested loops: k, i, j
- Each loop runs V times
- Inside: O(1) operation (min and addition)
- Total: **O(V³)**

**Space Complexity:** `O(V²)` or `O(1)`

- Input matrix: O(V²)
- No extra data structures
- In-place algorithm: **O(1)** auxiliary space
- If counting input: **O(V²)**

```
Detailed analysis:

V = number of vertices

Exact operations:

→ V iterations for k
→ V iterations for i
→ V iterations for j
→ Total iterations = V × V × V = V³

Each iteration:
→ 2 array accesses: matrix[i][k], matrix[k][j]
→ 1 addition: matrix[i][k] + matrix[k][j]
→ 1 comparison: min(...)
→ 1 assignment: matrix[i][j] = ...
→ All O(1)

Comparison with running Dijkstra V times:

Dijkstra once: O(E log V)
Dijkstra V times: O(V × E log V)

For dense graph (E ≈ V²):
→ V Dijkstra: O(V³ log V)
→ Floyd-Warshall: O(V³)
→ Floyd-Warshall wins! ✓

For sparse graph (E ≈ V):
→ V Dijkstra: O(V² log V)
→ Floyd-Warshall: O(V³)
→ Dijkstra wins! ✓

Space efficiency:
→ In-place modification of input matrix
→ No queue, heap, or visited arrays
→ Extremely space-efficient! ✓
```

## Interview Questions

```
Q1: Why must the k loop be the outermost loop in Floyd-Warshall?

A: The DP state after k iterations represents shortest paths using
   only vertices {0, 1, ..., k} as intermediates. For this to be
   correct, when we process vertex k, all shortest paths using
   {0, ..., k-1} must already be computed. The outer k loop
   ensures this inductive property. If k were inner, we would
   try to use vertex k before fully exploring paths through
   smaller intermediates, leading to incorrect or suboptimal
   results.

──────────────────────────────────────────────────────────────

Q2: How does Floyd-Warshall detect negative cycles?

A: After the algorithm completes, check the diagonal entries
   dist[i][i]. If any dist[i][i] < 0, a negative cycle exists
   involving vertex i. This is because the shortest path from
   a vertex to itself should be 0 (or the self-loop weight).
   A negative value means we can loop around a cycle and keep
   decreasing the distance indefinitely.

──────────────────────────────────────────────────────────────

Q3: When should I use Floyd-Warshall over running Dijkstra V times?

A: Use Floyd-Warshall when:
   → The graph is dense (E ≈ V²)
   → The graph has negative edge weights
   → V is small (≤ 400-500)
   → You want extremely simple code (3 nested loops)
   → You need the result in a matrix format

   Use V×Dijkstra when:
   → The graph is sparse (E << V²)
   → All weights are non-negative
   → V is large
   → You only need distances from a few sources

──────────────────────────────────────────────────────────────

Q4: Why is Floyd-Warshall considered a dynamic programming algorithm?

A: It satisfies both DP properties:

   Optimal Substructure:
   If the shortest path from i to j goes through k, then the
   i→k portion and k→j portion are also shortest paths using
   only smaller intermediate vertices.

   Overlapping Subproblems:
   The shortest path from i to j might be used as a subpath
   in many other shortest path computations. The algorithm
   stores all-pairs distances in a table to avoid recomputation.

   The state dist[i][j] after k iterations builds on the state
   after k-1 iterations, which is the hallmark of DP.

──────────────────────────────────────────────────────────────

Q5: Can Floyd-Warshall handle unreachable pairs of vertices?

A: Yes. If there is no path from i to j, dist[i][j] remains at
   infinity (or -1 in the original representation). The algorithm
   naturally handles this because no sequence of relaxations will
   produce a finite value for unreachable pairs.

──────────────────────────────────────────────────────────────

Q6: What is the significance of setting matrix[i][i] = 0?

A: The distance from a vertex to itself is 0 (empty path). Even if
   the input has self-loops, the shortest path from i to i is at
   most 0 (just stay at i). Setting the diagonal to 0 ensures the
   algorithm starts with the correct base case. If a negative
   self-loop exists, the algorithm will detect it by producing a
   negative diagonal value.

──────────────────────────────────────────────────────────────

Q7: How would you reconstruct the actual shortest path, not just
    the distance?

A: Maintain a "next" matrix (or parent matrix). Initially,
   next[i][j] = j if there is an edge i→j. During relaxation,
   if dist[i][k] + dist[k][j] improves dist[i][j], set
   next[i][j] = next[i][k]. To reconstruct the path from i to j:
   start at i, repeatedly go to next[current][j] until reaching j.

──────────────────────────────────────────────────────────────

Q8: Why convert -1 to a large value like 1e9 instead of handling
    -1 specially in the loops?

A: Converting -1 to infinity simplifies the inner loop to a
   single min() operation without conditional checks. This makes
   the code cleaner and slightly faster. The conversion back to
   -1 at the end restores the original convention. The large
   value must exceed any possible real path sum to avoid
   interfering with legitimate relaxations.

──────────────────────────────────────────────────────────────

Q9: What happens if the graph has multiple edges between the same
    pair of vertices?

A: In the adjacency matrix representation, only one weight can be
   stored per pair. Before running Floyd-Warshall, you should keep
   only the minimum weight among multiple edges, as the heavier
   edges can never be part of a shortest path.

──────────────────────────────────────────────────────────────

Q10: Can Floyd-Warshall work on undirected graphs?

A: Yes. For an undirected edge between u and v with weight wt,
    set both matrix[u][v] = wt and matrix[v][u] = wt. The
    algorithm then naturally handles the symmetry. Note that
    an undirected negative edge creates a 2-cycle of negative
    weight, which the algorithm will detect as a negative cycle.

──────────────────────────────────────────────────────────────

Q11: What is Johnson's algorithm, and when is it better than
     Floyd-Warshall?

A: Johnson's algorithm combines Bellman-Ford and Dijkstra to
    solve all-pairs shortest paths in O(V² log V + VE) time.
    It is faster than Floyd-Warshall for sparse graphs with
    negative weights. However, it is more complex to implement
    and uses a priority queue. Use Johnson's when V is large
    and E is relatively small.

──────────────────────────────────────────────────────────────

Q12: Why might Floyd-Warshall give wrong answers without detecting
     a negative cycle?

A: If the "infinity" value is too small, a legitimate path sum
    might exceed it, causing incorrect comparisons. Also, integer
    overflow can occur if path sums exceed the data type limit.
    Use a sufficiently large infinity (e.g., 1e9 for int, or
    LLONG_MAX/2 for long long) and consider using long long for
    distance values if weights are large.

──────────────────────────────────────────────────────────────

Q13: What are some real-world applications of Floyd-Warshall?

A: Key applications include:
    → Network routing tables (distance between all routers)
    → Social network analysis (degrees of separation)
    → Game AI (precomputing shortest paths between all map locations)
    → Transportation planning (all-pairs travel times)
    → Transitive closure (reachability, by treating edges as 0/1)
    → Currency arbitrage (detecting profitable exchange cycles)

──────────────────────────────────────────────────────────────

Q14: How does Floyd-Warshall compare to BFS for unweighted graphs?

A: For unweighted graphs, running BFS from each vertex gives
    all-pairs shortest paths in O(V × (V + E)) = O(V² + VE).
    For sparse graphs, this is much faster than Floyd-Warshall's
    O(V³). BFS is preferred for unweighted graphs. Floyd-Warshall
    shines when edges have varying (possibly negative) weights.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes in implementing
     Floyd-Warshall?

A: Common mistakes and fixes:

    Mistake 1: Wrong loop order (k not outermost)
    → Always: for(k) for(i) for(j)

    Mistake 2: Not initializing diagonal to 0
    → Set matrix[i][i] = 0 for all i

    Mistake 3: Using -1 directly without converting to infinity
    → Convert -1 to a large value before the loops

    Mistake 4: Infinity value too small
    → Use 1e9 or larger, ensuring it's > max possible path

    Mistake 5: Integer overflow in addition
    → Use long long if weights are large

    Mistake 6: Not converting infinity back to -1
    → Convert values ≥ 1e9 back to -1 for output

    Mistake 7: Forgetting negative cycle check
    → Check diagonal after algorithm: if (dist[i][i] < 0) cycle!
