## 1. Problem Statement

Given a graph of `V` vertices numbered from `0` to `V-1`, find the **shortest distances between every pair of vertices** in an edge-weighted directed graph.

The graph is represented as an **adjacency matrix** of size `n × n`:

- `matrix[i][j]` = weight of edge from `i` to `j`
- `matrix[i][j] = -1` means no edge from `i` to `j`

**Input Format:**

- `matrix` = `V × V` adjacency matrix

**Output:**

- Updated matrix where `matrix[i][j]` = shortest distance from `i` to `j`

## Key Insight

```
What is Floyd-Warshall?

All-Pairs Shortest Path algorithm:

→ Finds shortest path between ALL pairs of vertices
→ Works with negative edge weights ✓
→ Can detect negative cycles ✓
→ Uses Dynamic Programming on intermediate vertices

Comparison with other algorithms:

Algorithm          Type            Time        Neg Weights?
─────────────────────────────────────────────────────────────
Dijkstra           Single-source   O(E log V)  NO ✗
Bellman-Ford       Single-source   O(V×E)      YES ✓
Floyd-Warshall     All-pairs       O(V³)       YES ✓

When to use Floyd-Warshall:

1. Need shortest paths between ALL pairs
2. Graph is dense (many edges)
3. Graph is small to medium (V ≤ 400-500)
4. Negative edges present (but no negative cycles)
5. Simple implementation preferred (3 nested loops!)

Core idea:

For each intermediate vertex k:
  For each pair (i, j):
    Is path i→k→j better than direct i→j?
    If yes, update!
```

## Intuition

```
Imagine you are gradually allowed to use more cities as stopovers:

Initially: You can only fly direct (no stopovers)
→ matrix[i][j] = direct flight cost

Phase 1: You may stop at city 0
→ Check if i→0→j is cheaper than direct i→j
→ Update all pairs accordingly

Phase 2: You may also stop at city 1
→ Check if i→1→j is cheaper
→ But now "i→1" and "1→j" might already use city 0!
→ So paths can use cities {0, 1} as intermediates

...

Phase V: You may stop at any city {0, 1, ..., V-1}
→ All possible paths considered!
→ matrix[i][j] = true shortest distance ✓

Why does this work?

Every shortest path is built from shorter subpaths.
If the shortest path from i to j goes through k,
then the i→k part and k→j part are also shortest paths
using only smaller intermediate vertices.

This optimal substructure is what makes DP work! ✓
```

## Algorithm

```
Floyd-Warshall Dynamic Programming:

State: dist[i][j] = shortest distance from i to j
       using vertices {0, 1, ..., k} as intermediates

Recurrence:

dist[i][j] = min(
  dist[i][j],              // Direct path (no k)
  dist[i][k] + dist[k][j]  // Via intermediate k
)

Process:

1. Initialize: dist = adjacency matrix
   dist[i][j] = {
     0           if i == j
     weight(i,j) if edge exists
     ∞ (or -1)   if no edge
   }

2. For each intermediate vertex k (0 to V-1):
     For each source vertex i (0 to V-1):
       For each destination vertex j (0 to V-1):
         if dist[i][k] + dist[k][j] < dist[i][j]:
           dist[i][j] = dist[i][k] + dist[k][j]

3. After all iterations:
   dist[i][j] = shortest path from i to j

Key observation:

→ Order of k loop MUST be outermost! ✓
→ After k iterations, dist[i][j] uses vertices {0,...,k}
→ This is the DP dimension!
```

### Why k Must Be Outermost Loop

```
CRITICAL: Order matters!

Correct order:

for k in 0 to V-1:
  for i in 0 to V-1:
    for j in 0 to V-1:
      dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Why k must be outer?

DP interpretation:

After k iterations:
  dist[i][j] = shortest path using vertices {0,1,...,k}

If k is inner loop (WRONG):

for i in 0 to V-1:
  for j in 0 to V-1:
    for k in 0 to V-1:  ← WRONG!
      dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Problem:

When computing dist[i][j], we try ALL k at once
But dist[i][k] and dist[k][j] aren't finalized yet!
Missing intermediate optimizations!

Example where wrong order fails:

Edges: 0→1 (1), 1→2 (1), 0→2 (10)

Correct (k outer):
k=0: No change
k=1: dist[0][2] = min(10, dist[0][1]+dist[1][2])
                = min(10, 1+1) = 2 ✓

Wrong (k inner):
i=0, j=2:
  k=0: dist[0][2] = min(10, 0+10) = 10
  k=1: dist[0][2] = min(10, 1+1) = 2

Still works? But only by luck!
More complex example shows failure...

Trust the DP order! ✓
```

### Handling -1 (No Edge)

```
Problem representation uses -1 for "no edge"

Two approaches:

Approach 1: Convert -1 to infinity
(Used in provided code)

// Before algorithm
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        if (matrix[i][j] == -1) {
            matrix[i][j] = 1e9;  // infinity
        }
    }
}

// Run Floyd-Warshall with 1e9

// After algorithm
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        if (matrix[i][j] >= 1e9) {
            matrix[i][j] = -1;  // unreachable
        }
    }
}

Approach 2: Check for -1 in loops
(Alternative)

for (int k = 0; k < n; k++) {
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            if (matrix[i][k] == -1 || matrix[k][j] == -1)
                continue;  // Skip if intermediate path invalid
            
            if (matrix[i][j] == -1) {
                matrix[i][j] = matrix[i][k] + matrix[k][j];
            } else {
                matrix[i][j] = min(matrix[i][j],
                                   matrix[i][k] + matrix[k][j]);
            }
        }
    }
}

Both work! First is cleaner. ✓
```

### Negative Cycle Detection

```
Floyd-Warshall can detect negative cycles!

After algorithm completes:

Check diagonal: if dist[i][i] < 0 for any i
→ Negative cycle exists! ✗

Why?

dist[i][i] = shortest path from i to i (cycle)
If < 0, there's a negative cycle involving i!

Code to detect:

for (int i = 0; i < n; i++) {
    if (matrix[i][i] < 0) {
        cout << "Negative cycle detected!" << endl;
        return;
    }
}

Example with negative cycle:

Edges: 0→1 (1), 1→2 (-5), 2→0 (1)
Cycle: 0→1→2→0 = 1+(-5)+1 = -3 < 0

After Floyd-Warshall:
dist[0][0] will become negative!
dist[1][1] will become negative!
dist[2][2] will become negative!

→ Cycle detected ✓

If negative cycle exists:
Shortest paths are undefined (keep getting shorter)
Report the cycle or return error ✗
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    void shortest_distance(vector<vector<int>> &matrix) {
        
        int n = matrix.size();
        
        // Replace -1 with large value for unreachable
        for (int i = 0; i < n; i++) {
            for (int j = 0; j < n; j++) {
                if (matrix[i][j] == -1) {
                    matrix[i][j] = 1e9;
                }
                // Diagonal should be 0
                if (i == j) matrix[i][j] = 0;
            }
        }
        
        // Floyd-Warshall: Try each vertex as intermediate
        for (int k = 0; k < n; k++) {
            for (int i = 0; i < n; i++) {
                for (int j = 0; j < n; j++) {
                    // Relax via k
                    matrix[i][j] = min(
                        matrix[i][j],
                        matrix[i][k] + matrix[k][j]
                    );
                }
            }
        }
        
        // Convert back: large value → -1 (unreachable)
        for (int i = 0; i < n; i++) {
            for (int j = 0; j < n; j++) {
                if (matrix[i][j] >= 1e9) {
                    matrix[i][j] = -1;
                }
            }
        }
    }
};

int main() {
    vector<vector<int>> matrix = {
        {0, 2, -1, -1},
        {1, 0, 3, -1},
        {-1, -1, 0, -1},
        {3, 5, 4, 0}
    };
    
    Solution sol;
    sol.shortest_distance(matrix);
    
    int n = matrix.size();
    cout << "Shortest distance matrix:\n";
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            cout << matrix[i][j] << " ";
        }
        cout << "\n";
    }
    
    return 0;
}
```

### Floyd-Warshall vs Other Algorithms

```
Algorithm          Type        Time        Space    Neg Weights?  All Pairs?
────────────────────────────────────────────────────────────────────────────
Floyd-Warshall     DP          O(V³)       O(V²)    YES ✓         YES ✓
Dijkstra (V times) Greedy      O(V²logV)   O(V²)    NO ✗          YES ✓
Bellman-Ford (V)   DP          O(V²×E)     O(V²)    YES ✓         YES ✓
Johnson's          Both        O(V²logV+VE) O(V²)   YES ✓         YES ✓

When to use Floyd-Warshall:

✓ Small to medium graph (V ≤ 400-500)
✓ Dense graph (many edges)
✓ Need all-pairs shortest paths
✓ Negative weights present
✓ Simple implementation desired

When NOT to use:

✗ Large graph (V > 1000) → Use Johnson's
✗ Sparse graph → Use Dijkstra V times
✗ Only single-source needed → Use Dijkstra/Bellman-Ford
✗ Unweighted graph → Use BFS V times (faster)

Special cases:

→ If only non-negative weights: Dijkstra V times
→ If very large graph: Johnson's algorithm
→ If detecting cycles: Bellman-Ford or Floyd-Warshall
```

## Edge Cases

```
Single vertex:           No edges:
matrix = [[0]]          matrix = [[0, -1],
Output: [[0]]                     [-1, 0]]
Trivial ✓               Output: [[0, -1],
                                 [-1, 0]]
                        No paths to update ✓

Self-loop:              Negative self-loop:
matrix = [[0, -1],      matrix = [[0, -1],
          [-1, -5]]               [-1, -5]]
Invalid! (self-loop     Negative cycle!
with weight -5)         dist[1][1] < 0 ✗

Complete graph:         Disconnected components:
All pairs connected     matrix = [[0, -1, -1],
All paths updated ✓               [-1, 0, -1],
                                  [-1, -1, 0]]
                        No paths between components
                        Stays -1 ✓

All weights same:       Very large weights:
matrix = [[0, 1],       Need to ensure infinity value
          [1, 0]]       is larger than sum of any path
Symmetric ✓             Use 1e9 or LLONG_MAX/2 ✓

Negative weights        Zero-weight edges:
(no negative cycle):    Allowed! ✓
Algorithm works ✓       Represents "free" transitions
```

## Complexity Analysis

**Time Complexity:** `O(V³)`

- Three nested loops: k, i, j
- Each loop runs V times
- Inside: O(1) operation (min and addition)
- Total: **O(V³)**

**Space Complexity:** `O(V²)` or `O(1)`

- Input matrix: O(V²)
- No extra data structures
- In-place algorithm: **O(1)** auxiliary space
- If counting input: **O(V²)**

```
Detailed analysis:

V = number of vertices

Exact operations:

→ V iterations for k
→ V iterations for i
→ V iterations for j
→ Total iterations = V × V × V = V³

Each iteration:
→ 2 array accesses: matrix[i][k], matrix[k][j]
→ 1 addition: matrix[i][k] + matrix[k][j]
→ 1 comparison: min(...)
→ 1 assignment: matrix[i][j] = ...
→ All O(1)

Comparison with running Dijkstra V times:

Dijkstra once: O(E log V)
Dijkstra V times: O(V × E log V)

For dense graph (E ≈ V²):
→ V Dijkstra: O(V³ log V)
→ Floyd-Warshall: O(V³)
→ Floyd-Warshall wins! ✓

For sparse graph (E ≈ V):
→ V Dijkstra: O(V² log V)
→ Floyd-Warshall: O(V³)
→ Dijkstra wins! ✓

Space efficiency:
→ In-place modification of input matrix
→ No queue, heap, or visited arrays
→ Extremely space-efficient! ✓
```

## Interview Questions

```
Q1: Why must the k loop be the outermost?

A: k represents "which stopover cities are allowed." After k=0,
   paths can use city 0. After k=1, paths can use cities 0 and 1.
   We must build this up one city at a time. If k were inside,
   we might try using city 5 before we even know paths through
   city 1. It's like learning to walk before you run.

──────────────────────────────────────────────────────────────

Q2: How does Floyd-Warshall detect negative cycles?

A: Check the diagonal of the matrix. dist[i][i] is the cost to
   go from city i back to itself. This should be 0. If it's
   negative, you found a loop that makes money (or costs negative)
   every time you go around it.

──────────────────────────────────────────────────────────────

Q3: Floyd-Warshall vs Dijkstra V times — which to use?

A: Floyd-Warshall is one simple triple loop — great for small
   dense graphs. Dijkstra V times is better for large sparse
   graphs. Think of it like taking a bus (Floyd) vs taxis
   (Dijkstra) — the bus is cheaper for groups, taxis for solo.

──────────────────────────────────────────────────────────────

Q4: Why is this called dynamic programming?

A: It breaks a big problem into smaller pieces. "Shortest path
   using cities 0..k" builds on "shortest path using cities
   0..k-1." Like building a LEGO tower one brick at a time.

──────────────────────────────────────────────────────────────

Q5: Can it handle cities with no path between them?

A: Yes. If there is no path, the value stays at infinity (or -1).
   The algorithm naturally leaves it alone. Like two islands
   with no bridge between them.

──────────────────────────────────────────────────────────────

Q6: Why set the diagonal (i to i) to 0?

A: The cost to stay at the same city is zero. Even if the input
   says something else, staying put costs nothing. If the input
   has a negative self-loop, the algorithm will detect it as a
   negative cycle.

──────────────────────────────────────────────────────────────

Q7: How do we find the actual path, not just the distance?

A: Keep a "next" table. When you find a shorter path through k,
   note that from i you should first go to k. Then follow these
   breadcrumbs from start to destination.

──────────────────────────────────────────────────────────────

Q8: Why convert -1 to a big number like 1e9?

A: -1 means "no road." But in a min() operation, -1 would be
   treated as a very cheap road. We convert it to a huge number
   so min() ignores it. After the algorithm, we convert back to
   -1 for the final answer.

──────────────────────────────────────────────────────────────

Q9: What if there are multiple roads between the same two cities?

A: Keep only the smallest weight. The heavier road can never be
   part of a shortest path. Like having two flights — you always
   pick the cheaper one.

──────────────────────────────────────────────────────────────

Q10: Can it work on undirected graphs?

A: Yes. Add the road in both directions with the same weight.
   But be careful — a negative undirected road instantly creates
   a negative cycle because going back and forth costs negative.

──────────────────────────────────────────────────────────────

Q11: What is Johnson's algorithm?

A: Another all-pairs algorithm that is faster for large sparse
   graphs. It is more complex to code. Think of Floyd-Warshall
   as a bicycle and Johnson's as a sports car — the car is
   faster but harder to drive.

──────────────────────────────────────────────────────────────

Q12: Why might the answer be wrong without detecting a cycle?

A: If your "infinity" is too small, a real path might look like
   infinity. Or numbers might overflow. Use a big enough infinity
   (like 1e9) and use long long if weights are large.

──────────────────────────────────────────────────────────────

Q13: What are real-world uses?

A: Common uses include:
    → GPS systems that precompute all city-to-city distances
    → Social networks finding "degrees of separation"
    → Game AI planning routes between all map locations
    → Flight booking sites with connection databases
    → Network routers tracking latency between all nodes

──────────────────────────────────────────────────────────────

Q14: How does it compare to BFS for unweighted graphs?

A: BFS is faster for unweighted graphs (roads with same length).
   Floyd-Warshall is for weighted graphs where different roads
   have different lengths. Don't use a sledgehammer for a nail.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

    Mistake 1: Putting k inside instead of outside
    → Fix: Always for(k) then for(i) then for(j)

    Mistake 2: Forgetting diagonal = 0
    → Fix: Set matrix[i][i] = 0

    Mistake 3: Using -1 directly in min()
    → Fix: Convert -1 to a big number first

    Mistake 4: Infinity too small
    → Fix: Use 1e9 or bigger

    Mistake 5: Integer overflow
    → Fix: Use long long if weights are large

    Mistake 6: Not converting back to -1
    → Fix: Change big numbers back to -1 at the end

    Mistake 7: Forgetting to check for negative cycles
    → Fix: Check if any diagonal value is negative
```
