## 1. Problem Statement

Given a **weighted, undirected, connected graph** with `V` vertices and `E` edges, find the **shortest distance from source vertex `S` to ALL other vertices**. The graph does **not** contain any negative weight cycle.

**Input Format:**

- `V` = number of vertices
- `E` = number of edges
- `S` = source vertex
- `adj` = adjacency list where `adj[i]` contains pairs `{neighbor, weight}`

**Output:**

- Array of shortest distances from source to all vertices
- `-1` for unreachable vertices

---

## Examples

**Example 1:**

- **Input:** `V=2, S=0`
- **Edges:** `0—1` with weight `9`
- **Output:** `[0, 9]`

```
       Graph:

       0 ──9── 1

       Dist from 0:
       → 0: 0 (itself) ✓
       → 1: 9 (direct) ✓

       Result: [0, 9]
```

**Example 2:**

- **Input:** `V=3, S=2`
- **Edges:** `0—1(1), 0—2(6), 1—2(3)`
- **Output:** `[4, 3, 0]`

```
       Graph:

            1
       0 ─────── 1
       │         │
       6         3
       │         │
       └──── 2 ──┘

       Source = 2

       Dist from 2:
       → 0: 4 (via 2→1→0 = 3+1) ✓
       → 1: 3 (direct)            ✓
       → 2: 0 (itself)            ✓

       Result: [4, 3, 0]
```

## Key Insight

```
  Why Set instead of Priority Queue?

  Priority Queue (min-heap):
  → Cannot delete/update existing entries!
  → Old stale {dist, node} pairs remain in heap.
  → Leads to processing outdated entries.

  Set (ordered):
  → CAN erase old {dist, node} pair! ✓
  → Insert new {updated_dist, node}.
  → Always processes the true minimum distance.
  → Avoids redundant processing!

  Set advantage:
  If node reached via dist=9 first, then dist=4 later:

    Priority Queue: both {9,node} and {4,node} exist
    Set:            erase {9,node}, insert {4,node} → cleaner! ✓

  Both give O(E log V) but set avoids extra iterations.
```

## Intuition

```
  Imagine a delivery driver starting from a warehouse:

  1. The driver maintains a sorted list of all cities he can reach
     next, along with the current best distance to each.

  2. He always picks the city with the smallest known distance.

  3. When he finds a shorter route to a city already on the list,
     he ERASES the old entry and writes the new, better one.

  4. Because all road weights are non-negative, once a city is
     picked from the list, no shorter route to it can ever be found.

  Why it works:
  The set always contains the most up-to-date distance estimates.
  By erasing stale entries, we avoid revisiting outdated states.
  The greedy choice of the minimum node is safe because non-negative
  weights mean distances can only increase as we explore further.
```

## Algorithm

```
  dijkstraSet(V, adj, S):

  1. Build weighted adjacency list
     adj[u] = {(v, weight), ...}

  2. Initialize dist[] = infinity for all nodes
     dist[source] = 0

  3. Insert {0, source} into the set

  4. While set is not empty:
       a. Extract {dist, node} with SMALLEST distance (set.begin())
       b. Erase it from the set
       c. For each neighbor (adjNode, edgeWt) of node:
          If dist[node] + edgeWt < dist[adjNode]:
             → Erase old {dist[adjNode], adjNode} from set (if exists)
             → Update dist[adjNode] = dist[node] + edgeWt
             → Insert {dist[adjNode], adjNode} into set

  5. Return dist[]
```

```
  Why does greedy extraction of minimum work?

  Dijkstra's greedy property:
  The node with the smallest dist in the set
  has its FINAL shortest distance!

  Why? All remaining nodes have dist >= current min.
  No future relaxation can improve current min.

  → Safe to finalize and process it! ✓

  ⚠️ This breaks with NEGATIVE weights!
  A negative edge could later reduce an already-finalized dist.
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    vector<int> dijkstra(int V, vector<vector<pair<int,int>>>& adj, int S) {
        // Set stores {distance, node} in sorted order
        // Smallest distance is always at begin()
        set<pair<int, int>> st;

        // Initialize all distances as infinity
        vector<int> dist(V, 1e9);

        // Source distance = 0
        dist[S] = 0;
        st.insert({0, S});

        // Process until set is empty
        while (!st.empty()) {
            // Extract node with minimum distance
            auto it = *(st.begin());
            int dis = it.first;
            int node = it.second;
            st.erase(it);

            // Explore all neighbors
            for (auto [adjNode, edgW] : adj[node]) {
                // If shorter path found
                if (dis + edgW < dist[adjNode]) {
                    // Erase old entry if it exists
                    if (dist[adjNode] != 1e9)
                        st.erase({dist[adjNode], adjNode});

                    // Update distance
                    dist[adjNode] = dis + edgW;

                    // Insert updated entry
                    st.insert({dist[adjNode], adjNode});
                }
            }
        }

        return dist;
    }
};

int main() {
    int V = 3, E = 3, S = 2;

    vector<vector<pair<int,int>>> adj(V);

    adj[0].push_back({1, 1});
    adj[0].push_back({2, 6});
    adj[1].push_back({2, 3});
    adj[1].push_back({0, 1});
    adj[2].push_back({1, 3});
    adj[2].push_back({0, 6});

    Solution obj;
    vector<int> res = obj.dijkstra(V, adj, S);

    cout << "Shortest distances: ";
    for (int i = 0; i < V; i++)
        cout << res[i] << " ";
    cout << endl;

    return 0;
}
```

## Step-by-Step Trace

```
  Graph: V=3, S=2
  Edges: 0—1(1), 0—2(6), 1—2(3)

  ── Step 1: Adjacency List ──

  adj[0] = {(1,1), (2,6)}
  adj[1] = {(2,3), (0,1)}
  adj[2] = {(1,3), (0,6)}

  ── Step 2: Initialize ──

  dist = [∞, ∞, 0]
  set  = {(0, 2)}

  ── Step 3: Process ──

  Iteration 1: Extract (0, 2) → node=2, dis=0
      set = {}

      neighbor 1: 0+3=3 < ∞ → dist[1]=3, insert (3,1)
      neighbor 0: 0+6=6 < ∞ → dist[0]=6, insert (6,0)

      dist = [6, 3, 0]
      set  = {(3, 1), (6, 0)}

  Iteration 2: Extract (3, 1) → node=1, dis=3
      set = {(6, 0)}

      neighbor 2: 3+3=6 > dist[2]=0 → skip
      neighbor 0: 3+1=4 < dist[0]=6 → erase (6,0), dist[0]=4, insert (4,0)

      dist = [4, 3, 0]
      set  = {(4, 0)}

  Iteration 3: Extract (4, 0) → node=0, dis=4
      set = {}

      neighbor 1: 4+1=5 > dist[1]=3 → skip
      neighbor 2: 4+6=10 > dist[2]=0 → skip

      dist unchanged

  Set empty → DONE!

  Final: dist = [4, 3, 0] ✓
```

## Set vs Priority Queue Comparison

```
  Property            Set                    Priority Queue
  ──────────────────────────────────────────────────────────────
  Data Structure      Balanced BST           Binary Heap
  Extract min         O(log V) ✓             O(log V) ✓
  Insert              O(log V) ✓             O(log V) ✓
  Erase specific      O(log V) ✓             NOT possible ✗
  Stale entries       None ✓                 Can accumulate ✗
  Time Complexity     O(E log V)             O(E log V)
  Practical speed     Slightly slower        Slightly faster
                      (BST overhead)         (simpler structure)

  When to use Set?
  → When you want cleaner code with no stale entries.
  → Easier to debug and reason about.

  When to use PQ?
  → When raw speed matters (lower constant factor).
  → Just skip stale entries with a visited check.
```

## Edge Cases

```
  Single node graph:
  V=1, S=0
  dist = [0] ✓

  Disconnected components:
  Nodes not reachable from S
  dist stays ∞ → return -1 ✓

  All edges same weight:
  Reduces to BFS essentially
  Dijkstra still works ✓

  Self-loops:
  Edge u→u with weight w
  dist[u]+w > dist[u]=0 → skip ✓

  Multiple edges between same pair of nodes:
  Dijkstra naturally picks the shortest one ✓

  Large weights:
  Works fine, just larger values in the set ✓

  ⚠️ NEGATIVE WEIGHTS:
  Dijkstra FAILS! Use Bellman-Ford instead.
  Greedy "finalize min" assumption breaks
  when negative edges can reduce finalized distances.
```

---

## Why Dijkstra Fails with Negative Weights

```
  Example:

       5          -3
  0 ─────→ 1 ───────→ 2
  │                     ↑
  │                     │
  └──────── 4 ──────────┘

  Dijkstra processes:
  1. Extract (0, 0): relax → dist[1]=5, dist[2]=4
  2. Extract (4, 2): dist[2]=4 FINALIZED ✗
  3. Extract (5, 1): relax → dist[2]=5+(-3)=2 < 4

  But dist[2] was already finalized as 4!
  Actual shortest: 0→1→2 = 5+(-3) = 2

  Dijkstra gives 4, correct answer is 2 → WRONG! ✗

  Solution: Use Bellman-Ford for negative weights.
```

## Complexity Analysis

**Time Complexity:** O(E log V)

- Each edge is relaxed at most once: O(E)
- Each relaxation involves set insert/erase: O(log V)
- Total: **O(E log V)**

**Space Complexity:** O(V + E)

- Adjacency list: O(V + E)
- Distance array: O(V)
- Set: O(V) at most
- Total: **O(V + E)**

```
  Comparison across algorithms:

  Algorithm             Time              Space     Neg Weights?
  ──────────────────────────────────────────────────────────────
  Dijkstra (Set)        O(E log V)        O(V+E)    NO ✗
  Dijkstra (PQ)         O(E log V)        O(V+E)    NO ✗
  Bellman-Ford          O(V × E)          O(V)      YES ✓
  DAG (Topo Sort)       O(V + E)          O(V+E)    YES ✓ (DAG only)

  For non-negative weighted graphs:
  Dijkstra is optimal! ✓
```

## Interview Questions

```
  Q1: Why use a set instead of a priority queue for Dijkstra?

  A: A set (balanced BST) allows us to erase existing entries,
     while a priority queue (binary heap) does not.

     When we find a shorter path to a node already in the set,
     we erase the old {dist, node} pair and insert the new one.
     This keeps the set clean with no stale entries.

  ──────────────────────────────────────────────────────────────

  Q2: What is the time complexity of set-based Dijkstra?

  A: O(E log V). Each edge is relaxed at most once, and each
     set operation (insert/erase/extract-min) takes O(log V).

  ──────────────────────────────────────────────────────────────

  Q3: Is set-based Dijkstra faster than priority queue Dijkstra?

  A: Theoretically both are O(E log V). In practice, priority queue
     has a smaller constant factor because heap operations are
     simpler than balanced BST operations.

     However, set-based Dijkstra avoids processing stale entries,
     which can make it faster in graphs with many edge relaxations.

  ──────────────────────────────────────────────────────────────

  Q4: Can Dijkstra's algorithm handle negative weights?

  A: NO. Dijkstra's greedy approach assumes that once a node is
     extracted with minimum distance, its distance is finalized.
     Negative weight edges can create shorter paths later,
     breaking this assumption.

     For negative weights, use Bellman-Ford.

  ──────────────────────────────────────────────────────────────

  Q5: How do you handle unreachable nodes?

  A: Initialize all distances to infinity (e.g., 1e9). After the
     algorithm completes, any node still at infinity is unreachable.

     Convert infinity to -1 when returning or printing the result.

  ──────────────────────────────────────────────────────────────

  Q6: Why don't we need a visited array with the set approach?

  A: Because we erase old entries from the set, there are no stale
     entries to skip. Each node is extracted from the set exactly
     once with its final shortest distance.

     This is unlike the priority queue approach where we might
     need a visited array or distance check to skip stale entries.

  ──────────────────────────────────────────────────────────────

  Q7: What happens if there are multiple edges between the same pair of nodes?

  A: Dijkstra naturally handles this. During relaxation, it will
     consider each edge and keep only the minimum distance.

     If a shorter edge is found later, the set will be updated.

  ──────────────────────────────────────────────────────────────

  Q8: Can Dijkstra work on directed graphs?

  A: YES. The algorithm is the same for directed graphs.

     The only difference is in building the adjacency list:
     - Undirected: add edge in both directions
     - Directed: add edge only in the specified direction

  ──────────────────────────────────────────────────────────────

  Q9: What is the difference between lazy deletion (PQ) and eager deletion (Set)?

  A: Lazy deletion (PQ):
     - Insert new distance without removing old one
     - Skip outdated entries when popped using if (d > dist[node])

     Eager deletion (Set):
     - Erase old distance before inserting new one
     - No outdated entries remain

     Both are correct; set is cleaner, PQ is slightly faster.

  ──────────────────────────────────────────────────────────────

  Q10: What are common bugs in set-based Dijkstra?

  A: Common bugs and fixes:

     Bug 1: Forgetting to erase old entry before inserting new one
     → Add: if (dist[adjNode] != 1e9) st.erase({dist[adjNode], adjNode});

     Bug 2: Using dist[node] instead of dis (extracted distance)
     → Use the distance from the set iterator: int dis = it.first;

     Bug 3: Not initializing source distance
     → dist[S] = 0;

     Bug 4: Returning infinity instead of -1
     → Convert 1e9 to -1 for unreachable nodes

     Bug 5: Using set.erase(value) when iterator is needed
     → Use st.erase(it) for the extracted element

  ──────────────────────────────────────────────────────────────

  Q11: What is the space complexity of set-based Dijkstra?

  A: O(V + E). The adjacency list takes O(V + E), the distance
     array takes O(V), and the set contains at most O(V) entries
     (one per node with its current best distance).

  ──────────────────────────────────────────────────────────────

  Q12: When should you prefer Bellman-Ford over Dijkstra?

  A: Use Bellman-Ford when:
      - Edge weights may be negative
      - You need to detect negative cycles
      - The graph is small enough for O(V × E) time

     Use Dijkstra when:
      - Edge weights are non-negative
      - You want faster O(E log V) performance

  ──────────────────────────────────────────────────────────────

  Q13: Can we use multiset instead of set?

  A: Yes, but it's not necessary if we always erase old entries.

     A multiset allows duplicate {dist, node} pairs, which defeats
     the purpose of eager deletion. Use set to maintain uniqueness.

  ──────────────────────────────────────────────────────────────

  Q14: What is the role of the {distance, node} pair ordering in the set?

  A: std::set sorts pairs lexicographically: first by distance,
     then by node index. This ensures the smallest distance is
     always at set.begin(), regardless of node index.

  ──────────────────────────────────────────────────────────────

  Q15: How would you modify the algorithm for multi-source Dijkstra?

  A: Initialize all source nodes with distance 0 and insert them
     all into the set at the beginning.

     Then run the standard Dijkstra loop. The result gives the
     shortest distance from the nearest source to each node.
```
