## 1. Problem Statement

Given an **undirected, unweighted graph** with `N` vertices and `M` edges, find the **shortest distance from source node `src` to ALL other vertices**. All edges have a weight of `1`.

**Input Format:**

- `N` = number of vertices
- `M` = number of edges
- `src` = source vertex (given as `0`)
- Edge list `(u, v)` for each undirected edge

**Output:**

- Array of shortest distances from source to all vertices
- `-1` for unreachable vertices

## Key Insight

```
  All edges have UNIT WEIGHT = 1.

  Why BFS works perfectly here?

  BFS explores nodes LEVEL BY LEVEL:

  Level 0: source (dist=0)
  Level 1: all nodes 1 hop away (dist=1)
  Level 2: all nodes 2 hops away (dist=2)
  ...

  First time BFS reaches a node = SHORTEST path!

  (No need for Dijkstra's min-heap with unit weights)

  Weighted graph (different weights) → Dijkstra
  Unit weight graph                  → BFS ✓
```

---

## Intuition

```
  Imagine dropping a stone in a still pond:

  1. The ripple starts exactly at the source.
  2. It expands outward one ring at a time.
  3. Every point the ripple touches first is via the shortest route.

  BFS is exactly that ripple:

  - Queue holds the current wavefront.
  - We process the entire wavefront (level) before moving to the next.
  - Since every edge is 1 unit long, each new wave is exactly 1 unit farther.

  Why it works:
  When a node is first visited, we have already reached it through the
  minimum number of edges possible. No later path can be shorter because
  every path to it must contain at least that many edges.
```

## Algorithm

```
  shortestPathUnitWeight(N, edges, src):

  1. Build adjacency list from edges (undirected)

  2. Initialize dist[] = infinity for all nodes
     dist[src] = 0

  3. BFS from src:
       Push src to queue
       While queue not empty:
           node = queue.front(); pop
           For each neighbor of node:
               If dist[node] + 1 < dist[neighbor]:
                   dist[neighbor] = dist[node] + 1
                   push neighbor to queue

  4. Build result:
       If dist[i] != infinity → ans[i] = dist[i]
       Else                   → ans[i] = -1 (unreachable)

  5. Return ans
```

```
  Why check dist[node]+1 < dist[neighbor]?

  Ensures we only update if a shorter path is found.
  For unit weight undirected graph + BFS:
  Each node is processed only ONCE in practice
  (first visit = shortest path for unit weights).
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    vector<int> shortestPath(vector<vector<int>>& edges, int N, int M, int src) {
        // Step 1: Build undirected adjacency list
        vector<vector<int>> adj(N);
        for (auto& edge : edges) {
            adj[edge[0]].push_back(edge[1]);
            adj[edge[1]].push_back(edge[0]); // Undirected
        }

        // Step 2: Initialize distances to infinity
        vector<int> dist(N, 1e9);
        dist[src] = 0;

        // Step 3: BFS from source
        queue<int> q;
        q.push(src);

        while (!q.empty()) {
            int node = q.front();
            q.pop();

            for (int neighbor : adj[node]) {
                // Found shorter path to neighbor
                if (dist[node] + 1 < dist[neighbor]) {
                    dist[neighbor] = dist[node] + 1;
                    q.push(neighbor);
                }
            }
        }

        // Step 4: Build result (-1 for unreachable)
        vector<int> ans(N, -1);
        for (int i = 0; i < N; i++) {
            if (dist[i] != 1e9) {
                ans[i] = dist[i];
            }
        }

        return ans;
    }
};

int main() {
    int N = 9, M = 10;
    vector<vector<int>> edges = {
        {0,1},{0,3},{3,4},{4,5},
        {5,6},{1,2},{2,6},{6,7},
        {7,8},{6,8}
    };

    Solution obj;
    vector<int> ans = obj.shortestPath(edges, N, M, 0);

    cout << "Shortest distances from node 0: ";
    for (int x : ans) cout << x << " ";
    cout << endl;

    return 0;
}
```

## Edge Cases

```
  Source is isolated:
  src=0, adj[0]=[]
  dist[0]=0, rest = -1 ✓

  Disconnected graph:
  0 ─── 1 ─── 2      3 ─── 4  (separate component)

  BFS from 0 never reaches 3, 4.
  dist[3]=∞, dist[4]=∞
  Final result:
  ans[3] = -1 (unreachable)
  ans[4] = -1 (unreachable) ✓

  Single node:
  N=1, no edges
  ans=[0] ✓

  Linear chain 0─1─2─3─4:
  dist=[0,1,2,3,4] ✓

  All nodes connected:
  dist[] all updated ≠ ∞
  No -1 in result ✓
```

## Complexity Analysis

**Time Complexity:** O(N + M)

- Building adjacency list: O(M)
- BFS traversal: O(N + 2M) ← each node once, each edge twice (undirected)
- Building result: O(N)
- Total: **O(N + M)**

**Space Complexity:** O(N + M)

- Adjacency list: O(N + 2M)
- Distance array: O(N)
- BFS queue: O(N)
- Result array: O(N)
- Total: **O(N + M)**

```
  N = vertices, M = edges

  BFS = optimal for unit weight shortest path
  No priority queue needed! ✓
  Simpler and faster than Dijkstra for this case ✓

  Comparison with other shortest path algorithms:

  | Algorithm        | Graph Type           | Time Complexity      |
  |------------------|----------------------|----------------------|
  | BFS              | Unit weight          | O(N + M)             |
  | Dijkstra         | Non-negative weights | O((N+M) log N)       |
  | Bellman-Ford     | Any weights          | O(N × M)             |
```

## Interview Questions

```
  Q1: Why does BFS work for finding shortest paths in unweighted graphs?

  A: BFS explores nodes level by level.
     Level k contains all nodes reachable using exactly k edges.
     The first time a node is visited, it is reached via the
     minimum number of edges possible, which equals the shortest
     path length in an unweighted graph.

  ──────────────────────────────────────────────────────────────

  Q2: Why can't we use BFS for weighted graphs with different edge weights?

  A: BFS assumes every edge contributes equally to the path length.
     In weighted graphs, a path with fewer edges may be longer
     than a path with more edges but smaller total weight.

     Example:
     0 ──10── 1 ──1── 2
     │                  │
     └──────1───────────┘

     BFS from 0 would find 0→1 (1 edge) first, giving dist[1]=10.
     But the actual shortest path 0→2→1 has weight 2.
     Dijkstra is needed for weighted graphs.

  ──────────────────────────────────────────────────────────────

  Q3: Why is Dijkstra not needed for unit weight graphs?

  A: Dijkstra uses a min-heap to always process the closest node.
     With unit weights, BFS's queue naturally orders nodes by distance:

     - All nodes at distance d are enqueued before nodes at distance d+1
     - The FIFO queue acts as a perfect "min-heap" for distances

     So BFS achieves O(N + M) without heap overhead.

  ──────────────────────────────────────────────────────────────

  Q4: What does the condition dist[node] + 1 < dist[neighbor] do?

  A: It checks whether reaching the neighbor through the current
     node gives a shorter path than previously known.

     If true, we update dist[neighbor] and enqueue it.
     With unit weights and BFS, this check rarely triggers after
     the first visit because the first visit is already optimal.

  ──────────────────────────────────────────────────────────────

  Q5: Can a node be enqueued multiple times in this BFS?

  A: In theory yes, but in practice no for unit weights.

     Since the first visit is always the shortest path,
     dist[node] + 1 < dist[neighbor] becomes false on subsequent visits.

     However, the check is kept for correctness and to handle any
     graph where multiple paths exist.

  ──────────────────────────────────────────────────────────────

  Q6: How do you handle unreachable nodes?

  A: Initialize all distances to infinity (a large value like 1e9).
     After BFS, any node still at infinity is unreachable.

     Final conversion:
     if (dist[i] == 1e9) ans[i] = -1;
     else                ans[i] = dist[i];

  ──────────────────────────────────────────────────────────────

  Q7: What is the difference between BFS and DFS for shortest path?

  | Aspect        | BFS                     | DFS                    |
  |---------------|-------------------------|------------------------|
  | Order         | Level by level          | Deep exploration       |
  | Shortest path | Finds shortest in unwt  | Does NOT guarantee     |
  | Use case      | Shortest path, nearest  | Cycle detect, topo     |
  | Memory        | O(N) queue              | O(N) recursion stack   |

  DFS can find A path, but not necessarily the SHORTEST path.

  ──────────────────────────────────────────────────────────────

  Q8: Why use 1e9 as infinity instead of INT_MAX?

  A: To avoid integer overflow.

     When we compute dist[node] + 1, if dist[node] were INT_MAX,
     the addition could overflow. Using a large but safe value
     like 1e9 prevents this while still acting as infinity for
     reasonable input constraints.

  ──────────────────────────────────────────────────────────────

  Q9: Can this algorithm work on directed graphs?

  A: Yes! BFS works for shortest paths in directed unweighted graphs too.

     The only change is building the adjacency list:
     - Undirected: add edge in both directions
     - Directed:   add edge only in the given direction

     The BFS logic remains exactly the same.

  ──────────────────────────────────────────────────────────────

  Q10: How would you find the actual shortest path, not just the distance?

  A: Maintain a parent[] array during BFS.

     When we first visit a neighbor, set parent[neighbor] = node.
     After BFS, trace back from destination to source using parent[]
     to reconstruct the path.

     Example:
     parent[1] = 0, parent[2] = 1, parent[6] = 2
     Path from 0 to 6: 6 ← 2 ← 1 ← 0 → 0→1→2→6

  ──────────────────────────────────────────────────────────────

  Q11: What if the source is not node 0?

  A: The algorithm works for any source node.

     Just pass the desired source to the function.
     Initialize dist[src] = 0 and push src into the queue.
     The rest of the logic remains unchanged.

  ──────────────────────────────────────────────────────────────

  Q12: When should you prefer Dijkstra over BFS?

  A: Use Dijkstra when edge weights are not all equal.

     Use BFS when:
     - All edges have the same weight (typically 1)
     - You only need the number of edges in the shortest path
     - You want O(N + M) time complexity

     Dijkstra with a min-heap costs O((N + M) log N), which is
     overkill for unit weights.

  ──────────────────────────────────────────────────────────────

  Q13: How does multi-source BFS work?

  A: If there are multiple source nodes, initialize all of them
     with distance 0 and push all into the queue initially.

     Algorithm:
     1. dist[source_i] = 0 for every source
     2. Push all sources into queue
     3. Run normal BFS

     Result: shortest distance from the NEAREST source to each node.

     Use cases:
     - Finding nearest gate/water/source in a grid
     - Spread of multiple fires/infections
     - Multiple starting points in a maze

  ──────────────────────────────────────────────────────────────

  Q14: What are common bugs in BFS shortest path implementations?

  A: Common bugs and fixes:

     Bug 1: Forgetting graph is undirected
     → Add reverse edge: adj[v].push_back(u);

     Bug 2: Not initializing source distance
     → dist[src] = 0;

     Bug 3: Not checking if a neighbor was already visited
     → Use if (dist[node] + 1 < dist[neighbor])

     Bug 4: Returning infinity instead of -1
     → Convert 1e9 to -1 before returning

     Bug 5: 0-indexed vs 1-indexed mismatch
     → Be consistent with the problem's indexing

  ──────────────────────────────────────────────────────────────

  Q15: Is BFS greedy? Why or why not?

  A: BFS is not greedy in the same sense as Dijkstra or Prim.

     It processes nodes in FIFO order, expanding the entire frontier
     uniformly. It never makes a permanent "best choice"; instead,
     it guarantees that the first time a node is reached, the path
     is shortest because all edges have equal weight.

     In unit-weight graphs, BFS's level-order expansion is equivalent
     to Dijkstra's greedy choice, but without explicit minimization.
```
