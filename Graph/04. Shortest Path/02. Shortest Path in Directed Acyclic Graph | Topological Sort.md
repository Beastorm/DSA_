## 1. Problem Statement

Given a **weighted Directed Acyclic Graph (DAG)** with `N` vertices and `M` edges, find the **shortest distance from source node `src` to ALL other vertices**. Return `-1` for unreachable vertices.

**Input Format:**

- `N` = number of vertices
- `M` = number of edges
- `src` = source vertex (given as `0`)
- Edge list `(u, v, w)` for each directed edge `u → v` with weight `w`

**Output:**

- Array of shortest distances from source to all vertices
- `-1` for unreachable vertices

## Key Insight

```
  Why Topological Sort for DAG shortest path?

  In a DAG, topological order guarantees:

  When we process node u, ALL predecessors of u
  have already been processed!

  So when we relax edges from u:
  dist[u] is already FINALIZED (shortest possible)

  This avoids needing Dijkstra's priority queue!

  DAG (no cycles) → Topo Sort → O(V+E)
  General graph    → Dijkstra  → O((V+E) log V)

  Topo sort is FASTER for DAGs! ✓
```

**So remember this:**

- *Relaxation* means attempting to improve the distance using an edge.
- *Checking* is always done.
- *Updating* happens only if the new path is shorter.

That's why people often say:

> *"Relax every edge."*

---

## Intuition

```
  Imagine a pipeline of tasks where every task depends on earlier tasks:

  1. Start at the source task with cost 0.
  2. Process tasks only after ALL their dependencies are done.
  3. When a task is processed, its minimum cost is already known.
  4. Propagate that cost to all tasks that depend on it.

  Topological sort gives us that exact order.

  Why it works:
  In a DAG, there are no cycles, so we can always arrange nodes
  linearly such that every edge points forward. By the time we
  reach a node, every possible path to it has already been examined,
  and the minimum cost is locked in.
```

---

## Algorithm

```
  shortestPathDAG(N, edges, src):

  1. Build weighted adjacency list
     adj[u] = {(v, weight), ...}

  2. Topological Sort using DFS → push to stack

  3. Initialize dist[] = infinity
     dist[src] = 0

  4. Process nodes in topological order (pop from stack):

       For each neighbor (v, wt) of current node u:
           If dist[u] + wt < dist[v]:
               dist[v] = dist[u] + wt  ← RELAX

  5. Replace infinity with -1 for unreachable nodes

  6. Return dist[]
```

```
  Why does processing in topo order work?

  Topo order: u always comes before v if u→v exists.

  When we reach u, dist[u] is already finalized.
  Relaxing u's edges: dist[v] = min(dist[v], dist[u]+wt)

  → Guaranteed optimal since all paths to u are settled ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
private:
    // DFS for topological sort
    void dfs(int node, vector<vector<pair<int,int>>>& adj,
             vector<bool>& visited, stack<int>& st) {
        visited[node] = true;

        for (auto [neighbor, wt] : adj[node]) {
            if (!visited[neighbor]) {
                dfs(neighbor, adj, visited, st);
            }
        }

        st.push(node); // Push AFTER exploring all neighbors
    }

public:
    vector<int> shortestPath(int N, int M, vector<vector<int>>& edges) {
        // Step 1: Build weighted adjacency list
        vector<vector<pair<int,int>>> adj(N);
        for (auto& e : edges) {
            adj[e[0]].push_back({e[1], e[2]}); // u → v with weight w
        }

        // Step 2: Topological sort (DFS)
        vector<bool> visited(N, false);
        stack<int> st;

        for (int i = 0; i < N; i++) {
            if (!visited[i]) dfs(i, adj, visited, st);
        }

        // Step 3: Initialize distances
        vector<int> dist(N, 1e9);
        dist[0] = 0; // Source = 0

        // Step 4: Process nodes in topological order
        while (!st.empty()) {
            int node = st.top();
            st.pop();

            // Relax all edges from the current node
            if (dist[node] != 1e9) { // Only if reachable
                for (auto [neighbor, wt] : adj[node]) {
                    if (dist[node] + wt < dist[neighbor]) {
                        dist[neighbor] = dist[node] + wt;
                    }
                }
            }
        }

        // Step 5: Replace infinity with -1
        vector<int> ans(N);
        for (int i = 0; i < N; i++) {
            ans[i] = (dist[i] == 1e9) ? -1 : dist[i];
        }

        return ans;
    }
};

int main() {
    int N = 7, M = 8;
    vector<vector<int>> edges = {
        {0,4,2},{0,5,3},{5,4,1},{4,6,3},
        {4,2,1},{6,1,2},{2,3,3},{1,3,1}
    };

    Solution obj;
    vector<int> ans = obj.shortestPath(N, M, edges);

    cout << "Shortest distances: ";
    for (int x : ans) cout << x << " ";
    cout << endl;

    return 0;
}
```

  ── Step 1: Adjacency List ──

  adj[0] = {(4,2), (5,3)}
  adj[4] = {(6,3), (2,1)}
  adj[5] = {(4,1)}
  adj[6] = {(1,2)}
  adj[2] = {(3,3)}
  adj[1] = {(3,1)}
  adj[3] = {}

  ── Step 2: Topological Sort (DFS) ──

  DFS(0): visits 4 → 6 → 1 → 3 (leaf) → back → 2 → 3 (visited) → back → back → 5 → 4 (visited)

  Topo stack (bottom → top): [3, 1, 6, 2, 4, 5, 0]
  Pop order: 0, 5, 4, 2, 6, 1, 3

  ── Step 3: Initialize distances ──

  dist = [0, ∞, ∞, ∞, ∞, ∞, ∞]

  ── Step 4: Relax in topological order ──

  Pop 0 (dist=0):
      neighbor 4: 0+2=2 < ∞ → dist[4]=2
      neighbor 5: 0+3=3 < ∞ → dist[5]=3

  dist = [0, ∞, ∞, ∞, 2, 3, ∞]

  Pop 5 (dist=3):
      neighbor 4: 3+1=4 > dist[4]=2 → skip

  dist unchanged

  Pop 4 (dist=2):
      neighbor 6: 2+3=5 < ∞ → dist[6]=5
      neighbor 2: 2+1=3 < ∞ → dist[2]=3

  dist = [0, ∞, 3, ∞, 2, 3, 5]

  Pop 2 (dist=3):
      neighbor 3: 3+3=6 < ∞ → dist[3]=6

  dist = [0, ∞, 3, 6, 2, 3, 5]

  Pop 6 (dist=5):
      neighbor 1: 5+2=7 < ∞ → dist[1]=7

  dist = [0, 7, 3, 6, 2, 3, 5]

  Pop 1 (dist=7):
      neighbor 3: 7+1=8 > dist[3]=6 → skip

  Pop 3 (dist=6): no outgoing edges

  Final: dist = [0, 7, 3, 6, 2, 3, 5] ✓

## Why Relaxation Order Matters

```
  Topological order for example 1: 0, 5, 4, 2, 6, 1, 3

  When we process 4 (dist=2):
  We've ALREADY processed 0 and 5 (both point to 4).

  So dist[4]=2 is FINALIZED at this point!
  (5→4 with dist=3+1=4 can't improve dist[4]=2)

  Guarantee: When node u is processed,
  no unprocessed predecessor can improve dist[u].

  → DAG property + topo order = optimal! ✓

  Compare with wrong order (process 4 before 5):
  dist[4] would be 2 initially (from 0).
  Later, 5→4 might offer 3+1=4 > 2 → no improvement.

  Still correct in this case! But topo sort guarantees this naturally.
```

## Edge Cases

```
  Unreachable nodes:
  If some nodes are not reachable from source,
  dist stays at infinity → replaced with -1 ✓

  Single source, no edges:
  dist = [0, -1, -1, ..., -1]

  Disconnected DAG:
  Multiple components; nodes in other component → -1 ✓

  Source with no outgoing edges:
  dist = [0, -1, -1, ..., -1]
  Only source is reachable ✓

  Negative edge weights:
  Topo sort can handle negative weights in DAG! ✓
  (Dijkstra would fail here)

  Empty graph:
  N=1, M=0 → dist = [0] ✓
```

## Complexity Analysis

**Time Complexity:** O(V + E)

- Building adjacency list: O(E)
- DFS for topological sort: O(V + E)
- Processing in topo order + relaxation: O(V + E)
- Total: **O(V + E)**

**Space Complexity:** O(V + E)

- Adjacency list: O(V + E)
- Visited array: O(V)
- Stack: O(V)
- Distance array: O(V)
- Total: **O(V + E)**

```
  Compared to Dijkstra O((V+E) log V):

  For large sparse DAGs, topo sort is significantly faster
  since it avoids priority queue overhead! ✓

  Comparison with other shortest path algorithms:

  | Algorithm        | Graph Type           | Time Complexity       |
  |------------------|----------------------|-----------------------|
  | BFS              | Unit weight          | O(V + E)              |
  | Topo Sort        | DAG (any weights)    | O(V + E)              |
  | Dijkstra         | Non-negative weights | O((V+E) log V)        |
  | Bellman-Ford     | Any weights          | O(V × E)              |
```

## Interview Questions

```
  Q1: Why can we use topological sort for shortest paths in a DAG?

  A: In a DAG, we can order nodes such that every edge goes forward.
     By the time we process a node u, all its predecessors have
     already been processed, so dist[u] is finalized.
     Then relaxing u's outgoing edges gives correct shortest distances.

  ──────────────────────────────────────────────────────────────

  Q2: Can this algorithm handle negative edge weights?

  A: YES! This is a major advantage over Dijkstra.

     Because we process nodes in topological order and each node
     is finalized before its outgoing edges are relaxed, negative
     weights cannot cause later improvements through cycles.
     DAGs have no cycles, so the algorithm remains correct.

  ──────────────────────────────────────────────────────────────

  Q3: Why doesn't topological sort work for graphs with cycles?

  A: Topological sort is only defined for DAGs.

     In a cyclic graph, you cannot arrange nodes linearly such
     that every edge points forward. Without such an order, we
     cannot guarantee a node's distance is finalized before
     relaxing its outgoing edges.

  ──────────────────────────────────────────────────────────────

  Q4: What is the difference between Dijkstra and topological sort
      for shortest paths?

  | Aspect           | Topo Sort (DAG)        | Dijkstra (General)  |
  |------------------|------------------------|---------------------|
  | Graph type       | DAG only               | Any graph (no neg)  |
  | Time complexity  | O(V + E)               | O((V+E) log V)      |
  | Data structure   | Stack                  | Min-heap            |
  | Negative weights | YES ✓                  | NO ✗                |
  | Simpler code?    | YES ✓                  | More complex        |

  ──────────────────────────────────────────────────────────────

  Q5: What does "relaxation" mean in this context?

  A: Relaxation means attempting to improve a neighbor's distance:

       Before: dist[v] = 10
       Check:  dist[u] + weight(u→v) = 3 + 4 = 7
       After:  dist[v] = 7  (if 7 < 10)

     We relax every outgoing edge of a node after its distance
     is finalized.

  ──────────────────────────────────────────────────────────────

  Q6: Why do we skip nodes with dist[node] == infinity?

  A: If dist[node] is still infinity, the node is unreachable
     from the source. Relaxing its outgoing edges would add
     infinity + weight, which is useless and could cause overflow.

  ──────────────────────────────────────────────────────────────

  Q7: How do we get the actual shortest path, not just distances?

  A: Maintain a parent[] array during relaxation.

     When dist[v] is updated from u, set parent[v] = u.
     After processing, trace back from destination to source
     using parent[] to reconstruct the path.

     Example:
     parent[4]=0, parent[2]=4, parent[3]=2
     Path from 0 to 3: 3 ← 2 ← 4 ← 0 → 0→4→2→3

  ──────────────────────────────────────────────────────────────

  Q8: Can we use BFS or DFS instead of topological sort?

  A: - BFS only works for unweighted graphs.
     - DFS alone does not give shortest paths because it explores
       depth-first, not by minimum cost.
     - Topological sort provides the correct processing order for
       weighted DAGs.

  ──────────────────────────────────────────────────────────────

  Q9: What if the source is not node 0?

  A: The algorithm works for any source.

     Set dist[src] = 0 and perform topological sort as usual.
     Nodes unreachable from src will remain at infinity.

  ──────────────────────────────────────────────────────────────

  Q10: When should I prefer topological sort over Dijkstra?

  A: Use topological sort when:
      - The graph is a DAG
      - You want O(V + E) time complexity
      - Edge weights may be negative

     Use Dijkstra when:
      - The graph is not a DAG
      - Edge weights are non-negative
      - O((V+E) log V) is acceptable

  ──────────────────────────────────────────────────────────────

  Q11: Can topological sort detect cycles?

  A: Yes! Kahn's algorithm (BFS-based topo sort) can detect cycles.

     If after processing, not all nodes are visited, the graph
     contains a cycle. DFS-based topo sort can also detect cycles
     using recursion stack tracking.

  ──────────────────────────────────────────────────────────────

  Q12: How is topological sort implemented using DFS?

  A: 1. Run DFS from every unvisited node.
     2. After exploring all neighbors of a node, push it onto a stack.
     3. Pop nodes from the stack to get the topological order.

     The key idea: a node is added to the order only after all
     nodes reachable from it have been added.

  ──────────────────────────────────────────────────────────────

  Q13: What is Kahn's algorithm, and can it be used here?

  A: Kahn's algorithm computes topological order using in-degrees.

     1. Calculate in-degree of every node.
     2. Enqueue all nodes with in-degree 0.
     3. Process nodes from queue, reduce in-degrees of neighbors.
     4. If a neighbor's in-degree becomes 0, enqueue it.

     Yes, Kahn's algorithm can replace DFS-based topo sort here.
     The relaxation step remains the same.

  ──────────────────────────────────────────────────────────────

  Q14: What are common bugs in this implementation?

  A: Common bugs and fixes:

     Bug 1: Forgetting the graph is directed
     → Only add edge u → v, not v → u.

     Bug 2: Not initializing source distance
     → dist[src] = 0.

     Bug 3: Relaxing edges of unreachable nodes
     → Add check: if (dist[node] != 1e9)

     Bug 4: Returning infinity instead of -1
     → Convert 1e9 to -1 before returning.

     Bug 5: Wrong topo sort order (push before DFS completion)
     → Push node to stack AFTER visiting all neighbors.

  ──────────────────────────────────────────────────────────────

  Q15: Why is topological sort called a "linearization" of a DAG?

  A: Because it arranges all nodes in a straight line such that
     every directed edge points from left to right.

     This linear order respects all dependencies and makes
     dynamic-programming-style updates (like shortest path)
     straightforward.
```
