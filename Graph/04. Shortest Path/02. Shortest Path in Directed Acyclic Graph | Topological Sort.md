## Shortest Path in Directed Acyclic Graph | Topological Sort

### Problem Statement:

Given a **DAG** (Directed Acyclic Graph) with `N` vertices and `M` weighted edges, find the shortest path from **source vertex 0** to all other vertices. Return `-1` for unreachable vertices.

---

## Examples

**Example 1:**
- **Input:** `n=7`, edges=`[[0,4,2],[0,5,3],[5,4,1],[4,6,3],[4,2,1],[6,1,2],[2,3,3],[1,3,1]]`
- **Output:** `0 7 3 6 2 3 5`
```
  Graph:
  0 ──2──→ 4 ──1──→ 2 ──3──→ 3
  |         |                  ↑
  3──→ 5──1──┘      6 ──2──→ 1──1──┘
             └──3──→ 6

  Dist: 0=0, 1=7, 2=3, 3=6, 4=2, 5=3, 6=5 ✓
```

**Example 2:**
- **Input:** `n=6`, edges=`[[0,1,2],[0,4,1],[4,5,4],[4,2,2],[1,2,3],[2,3,6],[5,3,1]]`
- **Output:** `0 2 3 6 1 5`

---

## Key Insight
```
  Why Topological Sort for DAG shortest path?

  In a DAG, topological order guarantees:
  When we process node u, ALL predecessors of u
  have already been processed!

  So when we relax edges from u:
  dist[u] is already FINALIZED (shortest possible)

  This avoids needing Dijkstra's priority queue!

  DAG (no cycles) → Topo Sort → O(V+E)
  General graph    → Dijkstra  → O((V+E) log V)

  Topo sort is FASTER for DAGs! ✓
```

---

## Algorithm
```
  1. Build weighted adjacency list
     adj[u] = {(v, weight), ...}

  2. Topological Sort using DFS → push to stack

  3. Initialize dist[] = infinity
     dist[src] = 0

  4. Process nodes in topological order (pop from stack):
     For each neighbor (v, wt) of current node u:
         If dist[u] + wt < dist[v]:
             dist[v] = dist[u] + wt  ← RELAX

  5. Replace infinity with -1 for unreachable nodes

  6. Return dist[]
```
```
  Why does processing in topo order work?

  Topo order: u always comes before v if u→v exists.
  When we reach u, dist[u] is already finalized.
  Relaxing u's edges: dist[v] = min(dist[v], dist[u]+wt)
  → Guaranteed optimal since all paths to u are settled ✓
```

## Code
```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
private:
    // DFS for topological sort
    void dfs(int node, vector<pair<int,int>> adj[], vector<bool>& visited, stack<int>& st) {
        visited[node] = true;

        for (auto [neighbor, wt] : adj[node]) {
            if (!visited[neighbor]) {
                dfs(neighbor, adj, visited, st);
            }
        }

        st.push(node); // Push AFTER exploring all neighbors
    }

public:
    vector<int> shortestPath(int N, int M, vector<vector<int>>& edges) {
        // Step 1: Build weighted adjacency list
        vector<pair<int,int>> adj[N];
        for (auto& e : edges) {
            adj[e[0]].push_back({e[1], e[2]}); // u→v with weight w
        }

        // Step 2: Topological sort (DFS)
        vector<bool> visited(N, false);
        stack<int> st;

        for (int i = 0; i < N; i++) {
            if (!visited[i]) dfs(i, adj, visited, st);
        }

        // Step 3: Initialize distances
        vector<int> dist(N, 1e9);
        dist[0] = 0; // Source = 0

        // Step 4: Process nodes in topological order
        while (!st.empty()) {
            int node = st.top();
            st.pop();

            // Relax all edges from the current node
            if (dist[node] != 1e9) { // Only if reachable
                for (auto [neighbor, wt] : adj[node]) {
                    if (dist[node] + wt < dist[neighbor]) {
                        dist[neighbor] = dist[node] + wt;
                    }
                }
            }
        }

        // Step 5: Replace infinity with -1
        vector<int> ans(N);
        for (int i = 0; i < N; i++) {
            ans[i] = (dist[i] == 1e9) ? -1 : dist[i];
        }

        return ans;
    }
};

int main() {
    int N = 7, M = 8;
    vector<vector<int>> edges = {
        {0,4,2},{0,5,3},{5,4,1},{4,6,3},
        {4,2,1},{6,1,2},{2,3,3},{1,3,1}
    };

    Solution obj;
    vector<int> ans = obj.shortestPath(N, M, edges);

    cout << "Shortest distances: ";
    for (int x : ans) cout << x << " ";
    cout << endl;

    return 0;
}
```

## Step-by-Step Trace
```
  N=7, edges: 0→4(2), 0→5(3), 5→4(1), 4→6(3),
              4→2(1), 6→1(2), 2→3(3), 1→3(1)

  ── Step 1: Adjacency List ──
  adj[0]={(4,2),(5,3)}, adj[4]={(6,3),(2,1)},
  adj[5]={(4,1)},       adj[6]={(1,2)},
  adj[2]={(3,3)},       adj[1]={(3,1)}

  ── Step 2: Topological Sort (DFS) ──
  DFS(0): visits 4→6→1→3(leaf)→back→2→3(vis)→back→back→5→4(vis)
  Topo stack (bottom→top): [3,1,6,2,4,5,0]
  Pop order: 0,5,4,2,6,1,3

  ── Step 3: Init dist ──
  dist=[0,∞,∞,∞,∞,∞,∞]

  ── Step 4: Relax in topo order ──

  Pop 0 (dist=0):
    neighbor 4: 0+2=2 < ∞ → dist[4]=2
    neighbor 5: 0+3=3 < ∞ → dist[5]=3
  dist=[0,∞,∞,∞,2,3,∞]

  Pop 5 (dist=3):
    neighbor 4: 3+1=4 > dist[4]=2 → skip
  dist unchanged

  Pop 4 (dist=2):
    neighbor 6: 2+3=5 < ∞ → dist[6]=5
    neighbor 2: 2+1=3 < ∞ → dist[2]=3
  dist=[0,∞,3,∞,2,3,5]

  Pop 2 (dist=3):
    neighbor 3: 3+3=6 < ∞ → dist[3]=6
  dist=[0,∞,3,6,2,3,5]

  Pop 6 (dist=5):
    neighbor 1: 5+2=7 < ∞ → dist[1]=7
  dist=[0,7,3,6,2,3,5]

  Pop 1 (dist=7):
    neighbor 3: 7+1=8 > dist[3]=6 → skip

  Pop 3 (dist=6): no outgoing edges

  Final: dist=[0,7,3,6,2,3,5] ✓
```

## Why Relaxation Order Matters
```
  Topological order for example 1: 0,5,4,2,6,1,3

  When we process 4 (dist=2):
  We've ALREADY processed 0 and 5 (both point to 4).
  So dist[4]=2 is FINALIZED at this point!
  (5→4 with dist=3+1=4 can't improve dist[4]=2)

  Guarantee: When node u is processed,
  no unprocessed predecessor can improve dist[u].
  → DAG property + topo order = optimal! ✓

  Compare with wrong order (process 4 before 5):
  dist[4] would be 2 initially (from 0).
  Later 5→4 might offer 3+1=4 > 2 → no improvement.
  Still correct! But topo sort ensures this naturally.
```

## Topo Sort vs Dijkstra for DAG
```
  Property          Topo Sort (DAG)        Dijkstra (General)
  ─────────────────────────────────────────────────────────────
  Graph type        DAG only               Any graph (no neg)
  Time complexity   O(V + E)               O((V+E) log V)
  Data structure    Stack (topo order)     Min-heap/Priority Queue
  Handles neg wt?   YES ✓                  NO ✗
  Simpler code?     YES ✓                  More complex

  Topo Sort advantage:
  → Can handle negative edge weights in DAG! ✓
  → Dijkstra fails with negative weights.
  → Much simpler for DAGs.
```

---

## Edge Cases
```
  Unreachable nodes:           Single source, no edges:
  If some nodes not reachable  dist=[0,-1,-1,...,-1]
  from source → dist stays ∞
  → replaced with -1 ✓

  Disconnected DAG:            Source with no outgoing:
  Multiple components           dist=[0,-1,-1,...,-1]
  Nodes in other component      Only source is reachable ✓
  → -1 ✓
```

---

## Complexity Analysis

**Time Complexity:** O(V + E)
- Building adjacency list: O(E)
- DFS for topological sort: O(V + E)
- Processing in topo order + relaxation: O(V + E)
- Total: **O(V + E)**

**Space Complexity:** O(V + E)
- Adjacency list: O(V + E)
- Visited array: O(V)
- Stack: O(V)
- Distance array: O(V)
- Total: **O(V + E)**
```
  Compared to Dijkstra O((V+E) log V):
  For large sparse DAGs, topo sort is significantly faster
  since it avoids priority queue overhead! ✓
```
