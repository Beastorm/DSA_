## 1. Problem Statement

Given a weighted undirected connected graph with **V** vertices and **E** edges, find the **Minimum Spanning Tree (MST)**.

An MST is a subset of edges that:
- Connects all vertices
- Contains exactly V-1 edges
- Has a minimum total weight
- Forms no cycles

**Input Format:**
- `V` = number of vertices
- `edges` = list of edges `[u, v, weight]`

**Output:**
- MST edges and total weight (or just the weight)

## Key Insight

```
Kruskal's Algorithm uses a greedy approach with Union-Find:

1. Sort all edges by weight (cheapest first)
2. Pick the cheapest edge that doesn't form a cycle
3. Repeat until all vertices are connected

Why does this work?

→ Cycle Property: The most expensive edge in any cycle is never
  in the MST. By picking cheap edges first and skipping expensive
  ones that form cycles, we always make the right choice.

→ The cut property also applies: the cheapest edge crossing any
  cut must be in the MST.

Comparison with Prim's:

Property            Kruskal's              Prim's
────────────────────────────────────────────────────────
Approach            Edge-based             Vertex-based
Start               Cheapest edge          Any vertex
Grows               Forest → Tree          Connected tree
Data Structure      Union-Find (DSU)       Priority Queue
Time                O(E log E)             O(E log V)
Best for            Sparse graphs          Dense graphs

Both produce correct MST! ✓
```

## Intuition

```
Imagine you are building roads between cities on a budget:

→ You have a list of all possible roads with their costs
→ You want to connect every city as cheaply as possible
→ You cannot build loops (waste of money)

Kruskal's approach:

Step 1: Sort all roads by cost (cheapest first)
Step 2: Look at the cheapest road
        → Does it connect two separate groups of cities?
        → YES → Build it! It is useful.
        → NO  → Skip it! It would create a loop.
Step 3: Repeat until all cities are in one group

Think of it like merging friend circles:
→ Each city starts alone
→ A road between two cities merges their circles
→ If two cities are already in the same circle, building another
  road between them is pointless — it just creates a loop

The Union-Find data structure helps us quickly check:
→ "Are these two cities already connected?"
→ "Merge these two groups of cities"
```

## Algorithm

```
kruskal(V, edges):

1. Sort all edges by weight in ascending order

2. Initialize Union-Find with V nodes
   (each node is its own component)

3. Initialize:
   mstEdges = empty list
   totalWeight = 0

4. For each edge (u, v, weight) in sorted order:
   
   a. If find(u) != find(v):
      → They are in different components
      → Add edge to MST
      → totalWeight += weight
      → union(u, v)  (merge components)
      
   b. Else:
      → They are already connected
      → Skip this edge (would form cycle)
   
   c. If mstEdges has V-1 edges:
      → MST is complete!
      → Break early

5. Return totalWeight (and mstEdges if needed)
```

### Why Union-Find?

```
Union-Find answers two questions quickly:

1. "Are u and v in the same group?"
   → find(u) == find(v) ?
   → Nearly O(1) with path compression

2. "Merge the groups of u and v"
   → union(u, v)
   → Nearly O(1) with union by size/rank

Without Union-Find, checking for cycles would require:
→ BFS/DFS every time → O(V + E) per check
→ Total: O(E × (V + E)) → Way too slow!

With Union-Find:
→ O(E log E) for sorting
→ O(E × α(V)) for all union/find operations
→ Total: O(E log E) → Very fast! ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

struct Edge {
    int u, v, weight;
    
    bool operator<(const Edge& other) const {
        return weight < other.weight;
    }
};

class DSU {
private:
    vector<int> parent, size;
    
public:
    DSU(int n) {
        parent.resize(n);
        size.resize(n, 1);
        for (int i = 0; i < n; i++) {
            parent[i] = i;
        }
    }
    
    int find(int x) {
        if (parent[x] == x) return x;
        return parent[x] = find(parent[x]);
    }
    
    bool unite(int x, int y) {
        int px = find(x);
        int py = find(y);
        
        if (px == py) return false;
        
        // Union by size
        if (size[px] < size[py]) swap(px, py);
        parent[py] = px;
        size[px] += size[py];
        return true;
    }
};

// Returns MST edges
vector<Edge> kruskal(int n, vector<Edge>& edges) {
    sort(edges.begin(), edges.end());
    
    DSU dsu(n);
    vector<Edge> mst;
    
    for (const auto& e : edges) {
        if (dsu.unite(e.u, e.v)) {
            mst.push_back(e);
            if (mst.size() == n - 1) break;
        }
    }
    
    return mst;
}

// Returns MST weight
int kruskalWeight(int n, vector<Edge>& edges) {
    sort(edges.begin(), edges.end());
    
    DSU dsu(n);
    int totalWeight = 0;
    int edgeCount = 0;
    
    for (const auto& e : edges) {
        if (dsu.unite(e.u, e.v)) {
            totalWeight += e.weight;
            edgeCount++;
            if (edgeCount == n - 1) break;
        }
    }
    
    return (edgeCount == n - 1) ? totalWeight : -1;
}

int main() {
    int n = 4, m = 5;
    vector<Edge> edges = {
        {0, 1, 4},
        {0, 2, 5},
        {0, 3, 3},
        {1, 3, 2},
        {2, 3, 1}
    };
    
    vector<Edge> mst = kruskal(n, edges);
    
    int totalWeight = 0;
    cout << "MST Edges:\n";
    for (const auto& e : mst) {
        cout << e.u << " - " << e.v << " : " << e.weight << "\n";
        totalWeight += e.weight;
    }
    cout << "Total Weight: " << totalWeight << "\n";
    
    return 0;
}
```

**Output:**

```
MST Edges:
2 - 3 : 1
1 - 3 : 2
0 - 3 : 3
Total Weight: 6
```

```

Step 1: Sort edges by weight

Sorted edges:
1. (2,3,1)
2. (1,3,2)
3. (0,3,3)
4. (0,1,4)
5. (0,2,5)

Initial DSU state:
parent = [0, 1, 2, 3]
size   = [1, 1, 1, 1]

Components: {0}, {1}, {2}, {3}

─────────────────────────────────────

Edge 1: (2,3,1) — weight 1

find(2) = 2, find(3) = 3
Different components → unite them!

parent[3] = 2
size[2] = 2

Components: {0}, {1}, {2,3}
MST: {(2,3,1)}, Weight = 1

─────────────────────────────────────

Edge 2: (1,3,2) — weight 2

find(1) = 1
find(3) → parent[3]=2 → find(2)=2
Different components → unite them!

parent[2] = 1
size[1] = 3

Components: {0}, {1,2,3}
MST: {(2,3,1), (1,3,2)}, Weight = 3

─────────────────────────────────────

Edge 3: (0,3,3) — weight 3

find(0) = 0
find(3) → parent[3]=2 → parent[2]=1 → find(1)=1
Different components → unite them!

parent[1] = 0
size[0] = 4

Components: {0,1,2,3}
MST: {(2,3,1), (1,3,2), (0,3,3)}, Weight = 6

All 4 nodes connected! MST complete.

─────────────────────────────────────

Edge 4: (0,1,4) — weight 4

find(0) = 0
find(1) → parent[1]=0 → find(0)=0
Same component → SKIP! (would create cycle)

─────────────────────────────────────

Edge 5: (0,2,5) — weight 5

find(0) = 0
find(2) → parent[2]=1 → parent[1]=0 → find(0)=0
Same component → SKIP! (would create cycle)

─────────────────────────────────────

Final Result:
MST edges: (2,3,1), (1,3,2), (0,3,3)
Total weight: 6 ✓
```

## Edge Cases

```
Disconnected graph:
V = 4, edges only connect 2 pairs
→ MST impossible!
→ Check: if mst.size() != V-1, return -1

Single vertex:
V = 1, no edges
→ MST weight = 0
→ No edges needed

Negative weights:
Edges with negative costs
→ Kruskal handles them fine
→ Just picks the least total (most negative is "cheapest")

Multiple edges between same pair:
(0,1,5) and (0,1,3)
→ Sorting puts (0,1,3) first
→ The cheaper one gets picked
→ The expensive one is skipped if it forms a cycle

Self-loops:
(u,u,weight)
→ find(u) == find(u) always
→ Always skipped automatically
```

## Complexity Analysis

**Time Complexity:** `O(E log E)`

- Sorting edges: O(E log E)
- Union-Find operations: O(E × α(V)) ≈ O(E)
- Total: **O(E log E)**

**Space Complexity:** `O(V + E)`

- Edge list: O(E)
- Union-Find arrays: O(V)
- MST edges: O(V)
- Total: **O(V + E)**

```
Comparison with Prim's:

Algorithm    Time          Space      Best For
─────────────────────────────────────────────────
Kruskal      O(E log E)    O(V+E)     Sparse graphs
Prim         O(E log V)    O(V+E)     Dense graphs

Sparse graph: E ≈ V (few edges)
→ Kruskal is slightly better
→ Sorting is fast when E is small

Dense graph: E ≈ V² (many edges)
→ Prim is slightly better
→ Avoids sorting all edges
```

## Interview Questions

```
Q1: What is Kruskal's algorithm used for?

A: Kruskal's finds the Minimum Spanning Tree — the cheapest set
   of roads that connects all cities. It works by sorting all
   roads by cost and building the cheapest ones first, skipping
   any road that would create a loop.

──────────────────────────────────────────────────────────────

Q2: How is Kruskal's different from Prim's?

A: Kruskal looks at EDGES. It sorts all edges and picks the
   cheapest ones that don't form cycles. It grows a forest of
   small trees that eventually merge into one.

   Prim looks at VERTICES. It starts at one city and grows a
   single tree by always adding the cheapest road to a new city.

   Think of Kruskal as "pick the cheapest road from anywhere"
   and Prim as "grow outward from where you are."

──────────────────────────────────────────────────────────────

Q3: Why does sorting edges first work?

A: The cycle property says the most expensive edge in any loop
   is never in the MST. By checking edges from cheapest to most
   expensive, we always pick useful cheap roads first. When we
   encounter an expensive road, the cheap roads have already
   connected those cities some other way, so we can safely skip
   it.

──────────────────────────────────────────────────────────────

Q4: Why do we need Union-Find?

A: To quickly check if two cities are already connected. Without
   it, we would need BFS/DFS every time, which is very slow.
   Union-Find answers "are they connected?" in nearly O(1) time.

   It is like having a phone book of group leaders — instead of
   asking every person if they know someone, you just check who
   their leader is.

──────────────────────────────────────────────────────────────

Q5: What happens when we find an edge that forms a cycle?

A: We skip it. If two cities are already in the same group,
   adding another road between them just creates a loop. Loops
   are never needed in a spanning tree — you can remove any road
   in a loop and everyone is still connected.

──────────────────────────────────────────────────────────────

Q6: When should I use Kruskal's vs Prim's?

A: Use Kruskal's when the graph has few edges (sparse) and is
   given as a simple list of edges. Sorting is fast when there
   are not many edges.

   Use Prim's when the graph has many edges (dense) and is given
   as an adjacency list or matrix. It grows one connected tree,
   which feels natural for dense graphs.

   Both give the same answer. Pick whichever is easier to code
   for your input format.

──────────────────────────────────────────────────────────────

Q7: Can Kruskal's handle negative edge weights?

A: Yes. Negative weights just mean someone pays you to build that
   road. Kruskal will happily include them because they lower the
   total cost. The algorithm works exactly the same — sort by
   weight and pick the cheapest.

──────────────────────────────────────────────────────────────

Q8: What if the graph is disconnected?

A: Kruskal's will only connect the components it can reach. If
   the graph has multiple disconnected parts, you will end up
   with fewer than V-1 edges in your MST. You should check this
   at the end and report that no spanning tree exists.

──────────────────────────────────────────────────────────────

Q9: Why is the time complexity O(E log E)?

A: Sorting the edges takes O(E log E) time. After that, each
   union and find operation is nearly O(1), and we do at most E
   of them. So the sorting step dominates the total time.

──────────────────────────────────────────────────────────────

Q10: What is the space complexity?

A: O(V + E). You need:
    → The list of edges: O(E)
    → The Union-Find arrays: O(V)
    → The MST edges: O(V)

──────────────────────────────────────────────────────────────

Q11: Does Kruskal's work with directed graphs?

A: No. MST is only defined for undirected graphs where a road
    between A and B works both ways. For directed graphs, you
    need different algorithms.

──────────────────────────────────────────────────────────────

Q12: Can there be multiple MSTs?

A: Yes, when different edges have the same weight. If two roads
    cost the same, Kruskal might pick either one and still get
    the same total cost. The MST is not always unique.

──────────────────────────────────────────────────────────────

Q13: What are real-world uses of Kruskal's?

A: Common uses include:
    → Building the cheapest road network between towns
    → Laying fiber optic cables to connect all buildings
    → Designing water pipe systems with minimum pipe length
    → Clustering data points that are close to each other
    → Any problem where you need to connect everything at the
      lowest possible cost

──────────────────────────────────────────────────────────────

Q14: How does Kruskal's relate to the cycle property?

A: The cycle property says the most expensive edge in any cycle
    is never in the MST. Kruskal's uses this directly — it sorts
    edges cheap to expensive, so when it encounters an edge that
    forms a cycle, that edge must be the most expensive one in
    that cycle (because cheaper edges were already picked). So
    skipping it is always correct.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

    Mistake 1: Forgetting to sort edges first
    → Fix: Always sort by weight before processing

    Mistake 2: Not using Union-Find for cycle detection
    → Fix: Use DSU — BFS/DFS is too slow

    Mistake 3: Not checking if MST is complete
    → Fix: After the loop, check if you have V-1 edges

    Mistake 4: Adding duplicate edges to the MST
    → Fix: DSU.unite() returns false if already connected

    Mistake 5: Confusing Kruskal's with Prim's
    → Fix: Kruskal sorts ALL edges first. Prim uses a priority
           queue and grows from one node.

    Mistake 6: Using union without find first
    → Fix: Always call find() to get the leaders before union()
```
