## Disjoint Set | Union by Rank | Union by Size | Path Compression

### Why Do We Need a Disjoint Set?

```
Problem: Given components of an undirected graph

Component 1:        Component 2:
1 ── 2 ── 3        4 ── 5
                    |
                    6 ── 7

Question: Are node 1 and node 5 in the same component?

Brute Force Approach (BFS/DFS):
→ Traverse from node 1, mark all reachable nodes
→ Check if node 5 is reachable
→ Time: O(N + E)

Disjoint Set Approach:
→ Find ultimate parent of node 1: findPar(1)
→ Find ultimate parent of node 5: findPar(5)
→ If same parent → same component!
→ Time: O(α(N)) ≈ O(1) (amortized constant time!)

Disjoint Set wins for dynamic graphs! ✓
```

## Dynamic Graphs

```
What is a dynamic graph?

A graph that changes its configuration over time
as edges are added or removed.

Example: Adding edges one by one
Edges: {{1,2}, {2,3}, {4,5}, {6,7}, {5,6}, {3,7}}

After adding 4 edges:     After adding all 6 edges:
1 ── 2 ── 3              1 ── 2 ── 3
                              \     \
4 ── 5   6 ── 7         4 ── 5 ── 6 ── 7

Query: Are nodes 1 and 4 in same component?
After 4 edges: NO ✗
After 6 edges: YES ✓

For dynamic graphs, Disjoint Set provides:
→ Efficient union operations (adding edges)
→ Fast component queries (checking connectivity)
→ Nearly constant time per operation! ✓
```

## Disjoint Set Functionalities

```
Two main operations:

1. findPar(node):
   Find the ultimate parent (root) of a node
   → Determines which component node belongs to
   → With path compression: O(α(N)) ≈ O(1)

2. Union(u, v):
   Merge components containing nodes u and v
   → Adds an edge between u and v
   → Two variants:
     a) Union by Rank
     b) Union by Size

Additional features:
→ Path Compression: Optimization for findPar()
→ Makes tree flatter, speeds up future queries
```

---

## Key Terminologies

```
1. Ultimate Parent:
   The root node of a component
   
   Example tree:
        1 (ultimate parent)
       / \
      2   3
     /
    4
   
   ultimate_parent(4) = 1
   ultimate_parent(2) = 1
   ultimate_parent(1) = 1

2. Rank:
   Distance from current node to furthest leaf
   (including all nodes beneath current node)
   
   Example:
        1  rank=3
       / \
      2   3  rank=1,1
     /
    4  rank=0
   
   rank[1] = 3 (furthest leaf is 4)
   rank[2] = 1
   rank[3] = 0
   rank[4] = 0

3. Size:
   Number of nodes in the component rooted at current node
   
   Example (same tree):
   size[1] = 4 (nodes 1,2,3,4)
   size[2] = 2 (nodes 2,4)
   size[3] = 1
   size[4] = 1
```

## Union by Rank

### Data Structures

```cpp
vector<int> rank;    // rank[i] = rank of node i
vector<int> parent;  // parent[i] = parent of node i

Initialization:
rank[i] = 0 for all i
parent[i] = i for all i (each node is its own parent)
```

### Algorithm

```
unionByRank(u, v):

1. Find ultimate parents:
   pu = findPar(u)
   pv = findPar(v)

2. If already in same component: return
   if (pu == pv) return;

3. Compare ranks:
   Case 1: rank[pu] < rank[pv]
     → Attach smaller (pu) to larger (pv)
     → parent[pu] = pv
   
   Case 2: rank[pv] < rank[pu]
     → Attach smaller (pv) to larger (pu)
     → parent[pv] = pu
   
   Case 3: rank[pu] == rank[pv]
     → Attach either to other
     → parent[pv] = pu
     → rank[pu]++  (height increases!)

Why attach smaller to larger?
→ Keeps tree height minimal
→ Reduces path length for findPar()
→ Optimizes future queries! ✓
```

## Path Compression

```
Problem: Without optimization, findPar() takes O(log N)

Tree before path compression:
        1
       /
      2
     /
    3
   /
  4

findPar(4): 4 → 3 → 2 → 1 (3 hops)

Tree after path compression:
        1
      / | \
     2  3  4

findPar(4): 4 → 1 (1 hop!)

How it works:
While finding ultimate parent,
connect every node directly to root!

Code:
int findPar(int node) {
    if (node == parent[node])
        return node;
    return parent[node] = findPar(parent[node]);
                          ↑
                  Path compression happens here!
}

Benefit:
→ Future queries become O(1)
→ Tree becomes very flat
→ Amortized time: O(α(N)) ≈ constant!
```

---

## Why Attach Smaller Rank to Larger?

```
Bad approach: Attach larger to smaller

Before:               After:
  1 (rank=2)            3 (rank=1)
 / \                    |
2   3   4 (rank=1)      4 (rank=1)
   / \    |            / \
  5   6   7           5   6
                     /
                    1 (rank=2)
                   / \
                  2   7

Height increases significantly! ✗
More traversal for findPar()

Good approach: Attach smaller to larger

Before:               After:
  1 (rank=2)            1 (rank=2)
 / \                   /|\
2   3   4 (rank=1)    2 3 4
       / \    |          / \
      5   6   7         5   6
                             \
                              7

Height stays minimal! ✓
Less traversal needed

Conclusion:
Always connect smaller rank to larger rank
→ Minimizes tree height
→ Optimizes query time! ✓
```

## Union by Size

### Why Union by Size?

```
Problem with Union by Rank:
After path compression, rank becomes inaccurate!

Example:
Before compression:      After compression:
      1 (rank=2)              1 (rank=2)
     /                       /|\
    2 (rank=1)              2 3 4
   /                        
  3 (rank=0)                Rank says 2,
 /                          but actual
4 (rank=0)                  height is 1!

Rank is now distorted! ✗

Solution: Use size instead!
→ Size = number of nodes in component
→ Size doesn't get distorted by path compression ✓
→ More intuitive! ✓
```

### Data Structures

```cpp
vector<int> size;    // size[i] = size of component rooted at i
vector<int> parent;  // parent[i] = parent of node i

Initialization:
size[i] = 1 for all i (each node is a component of size 1)
parent[i] = i for all i
```

### Algorithm

```
unionBySize(u, v):

1. Find ultimate parents:
   pu = findPar(u)
   pv = findPar(v)

2. If already in same component: return
   if (pu == pv) return;

3. Compare sizes:
   Case 1: size[pu] < size[pv]
     → Attach smaller (pu) to larger (pv)
     → parent[pu] = pv
     → size[pv] += size[pu]  ← Add sizes!
   
   Case 2: size[pv] <= size[pu]
     → Attach smaller (pv) to larger (pu)
     → parent[pv] = pu
     → size[pu] += size[pv]  ← Add sizes!

Note: We ALWAYS update the size of the parent
      to whom we attached the other component!
```

## Complete Implementation

```cpp
#include <bits/stdc++.h>
using namespace std;

class DisjointSet {
    vector<int> rank, parent, size;
    
public:
    DisjointSet(int n) {
        rank.resize(n + 1, 0);
        parent.resize(n + 1);
        size.resize(n + 1, 1);
        
        for (int i = 0; i <= n; i++) {
            parent[i] = i;
        }
    }
    
    // Find ultimate parent with path compression
    int findUPar(int node) {
        if (node == parent[node])
            return node;
        return parent[node] = findUPar(parent[node]);
    }
    
    // Union by rank
    void unionByRank(int u, int v) {
        int ulp_u = findUPar(u);
        int ulp_v = findUPar(v);
        
        if (ulp_u == ulp_v) return;  // Already same component
        
        // Attach smaller rank to larger rank
        if (rank[ulp_u] < rank[ulp_v]) {
            parent[ulp_u] = ulp_v;
        }
        else if (rank[ulp_v] < rank[ulp_u]) {
            parent[ulp_v] = ulp_u;
        }
        else {
            // Equal ranks: attach either way, increase rank
            parent[ulp_v] = ulp_u;
            rank[ulp_u]++;
        }
    }
    
    // Union by size
    void unionBySize(int u, int v) {
        int ulp_u = findUPar(u);
        int ulp_v = findUPar(v);
        
        if (ulp_u == ulp_v) return;  // Already same component
        
        // Attach smaller size to larger size
        if (size[ulp_u] < size[ulp_v]) {
            parent[ulp_u] = ulp_v;
            size[ulp_v] += size[ulp_u];
        }
        else {
            parent[ulp_v] = ulp_u;
            size[ulp_u] += size[ulp_v];
        }
    }
};

int main() {
    DisjointSet ds(7);
    
    ds.unionBySize(1, 2);
    ds.unionBySize(2, 3);
    ds.unionBySize(4, 5);
    ds.unionBySize(6, 7);
    ds.unionBySize(5, 6);
    
    // Check if 3 and 7 are in same component
    if (ds.findUPar(3) == ds.findUPar(7)) {
        cout << "Same\n";
    }
    else {
        cout << "Not same\n";
    }
    
    // Connect 3 and 7
    ds.unionBySize(3, 7);
    
    // Check again
    if (ds.findUPar(3) == ds.findUPar(7)) {
        cout << "Same\n";
    }
    else {
        cout << "Not same\n";
    }
    
    return 0;
}
```

**Output:**
```
Not same
Same
```

## Path Compression in Action

```
Before path compression (after all unions):
        4
      / | \
     1  5  6
    /|\    |
   2 3     7

findUPar(3):
3 → parent[3]=1 → parent[1]=4 → return 4
Now update: parent[3] = 4

After path compression:
        4
     /||| \
    1 2 3 5 6
            |
            7

Next time findUPar(3):
3 → parent[3]=4 → return 4 (one step!)

This is why time complexity → O(α(N)) ≈ O(1)!
```

## Time Complexity

```
findPar() with path compression:
→ Amortized time: O(α(N))
→ α(N) = Inverse Ackermann function
→ For all practical N, α(N) ≤ 4
→ Effectively constant time! ✓

unionByRank() / unionBySize():
→ 2 × findPar() calls
→ Constant work after that
→ Total: O(α(N)) ≈ O(4) ≈ O(1)

Why O(4)?
Long mathematical derivation involving:
→ Inverse Ackermann function
→ Amortized analysis
→ Not required for interviews!
→ Just know: nearly constant time ✓

Practical values:
N = 10^5: α(N) ≈ 3
N = 10^9: α(N) ≈ 4
N = 10^600: α(N) ≈ 5

For all realistic inputs: α(N) ≤ 4!
```

## Union by Rank vs Union by Size

```
Property              Union by Rank       Union by Size
─────────────────────────────────────────────────────────────
Metric                Rank (height)       Size (# nodes)
Initial value         0                   1
Update on union       Sometimes +1        Always updated
After compression     Distorted ✗         Accurate ✓
Intuitive             Less                More ✓
Practical use         Less common         More common ✓

Recommendation: Use Union by Size! ✓
→ More intuitive (actual node count)
→ Size remains accurate
→ Same time complexity
→ Easier to understand and debug
```

## Common Mistakes

```
❌ Mistake 1: Not using path compression
int findUPar(int node) {
    if (node == parent[node]) return node;
    return findUPar(parent[node]);  // WRONG! No compression
}

✓ Correct:
return parent[node] = findUPar(parent[node]);

❌ Mistake 2: Forgetting to update size in unionBySize
if (size[ulp_u] < size[ulp_v]) {
    parent[ulp_u] = ulp_v;
    // WRONG! Forgot to update size
}

✓ Correct:
parent[ulp_u] = ulp_v;
size[ulp_v] += size[ulp_u];

❌ Mistake 3: Comparing nodes instead of ultimate parents
if (u == v) return;  // WRONG!

✓ Correct:
if (findUPar(u) == findUPar(v)) return;

❌ Mistake 4: Updating rank in both cases
if (rank[ulp_u] < rank[ulp_v]) {
    parent[ulp_u] = ulp_v;
    rank[ulp_v]++;  // WRONG! Only update when equal
}

✓ Correct:
Only increment rank when ranks are equal!

❌ Mistake 5: Wrong initialization
size[i] = 0;  // WRONG! Should be 1
rank[i] = 1;  // WRONG! Should be 0

✓ Correct:
size[i] = 1;  // Each node is component of size 1
rank[i] = 0;  // Initial rank is 0
```

## Summary

```
Disjoint Set (Union-Find):
→ Efficiently manages dynamic connectivity
→ Two main operations: findPar() and union()
→ Nearly constant time with optimizations

Key optimizations:
1. Path Compression:
   Flatten tree during findPar()
   → Makes future queries O(1)

2. Union by Rank/Size:
   Attach smaller component to larger
   → Keeps tree height minimal
   → Reduces query time

Time Complexity: O(α(N)) ≈ O(1) amortized
Space Complexity: O(N)

When to use:
→ Dynamic graph connectivity queries
→ Kruskal's MST algorithm
→ Connected component problems
→ Online edge addition/queries

Recommendation:
Use Union by Size with Path Compression! ✓
→ Most intuitive and practical approach
```
