## 1. Why Do We Need a Disjoint Set?

```
Problem: Given components of an undirected graph

Component 1:        Component 2:

1 ── 2 ── 3         4 ── 5
                    |
                    6 ── 7

Question: Are node 1 and node 5 in the same component?

Brute Force Approach (BFS/DFS):
→ Traverse from node 1, mark all reachable nodes
→ Check if node 5 is reachable
→ Time: O(N + E)

Disjoint Set Approach:
→ Find ultimate parent of node 1: findPar(1)
→ Find ultimate parent of node 5: findPar(5)
→ If same parent → same component!
→ Time: O(α(N)) ≈ O(1) (amortized constant time!)

Disjoint Set wins for dynamic graphs! ✓
```

## Examples

**Example 1: Dynamic Connectivity**

```
Edges added one by one: {{1,2}, {2,3}, {4,5}, {6,7}, {5,6}, {3,7}}

After adding 4 edges:     After adding all 6 edges:

1 ── 2 ── 3              1 ── 2 ── 3
                              \     \
4 ── 5   6 ── 7         4 ── 5 ── 6 ── 7

Query: Are nodes 1 and 4 in same component?

After 4 edges: NO ✗
After 6 edges: YES ✓
```

**Example 2: Union by Size Trace**

```
Operations: union(1,2), union(2,3), union(4,5), union(5,6)

After unions:

  1           4
 / \          |
2   3         5
              |
              6

Query: find(3) == find(6)?
→ find(3) = 1, find(6) = 4
→ Different parents → Not same component ✗

Operation: union(3, 6)
→ Component sizes: 3 (nodes 1,2,3) vs 3 (nodes 4,5,6)
→ Attach either to other
→ Now all connected!

Query: find(3) == find(6)?
→ Same parent → Same component ✓
```

## Key Insight

```
Disjoint Set Functionalities:

Two main operations:

1. findPar(node):
   Find the ultimate parent (root) of a node
   → Determines which component node belongs to
   → With path compression: O(α(N)) ≈ O(1)

2. Union(u, v):
   Merge components containing nodes u and v
   → Adds an edge between u and v
   → Two variants:
     a) Union by Rank
     b) Union by Size

Additional feature:
→ Path Compression: Optimization for findPar()
→ Makes tree flatter, speeds up future queries

Key Terminologies:

1. Ultimate Parent:
   The root node of a component
   
   Example tree:
        1 (ultimate parent)
       / \
      2   3
     /
    4
   
   ultimate_parent(4) = 1
   ultimate_parent(2) = 1
   ultimate_parent(1) = 1

2. Rank:
   Distance from the current node to the furthest leaf
   
   Example:
        1  rank=2
       / \
      2   3  rank=1,0
     /
    4  rank=0

3. Size:
   Number of nodes in the component rooted at the current node
   
   Example (same tree):
   size[1] = 4 (nodes 1,2,3,4)
   size[2] = 2 (nodes 2,4)
   size[3] = 1
   size[4] = 1
```

## Intuition

```
Imagine you are managing friend circles in a social network:

→ Each person starts in their own circle
→ When two people become friends, their circles merge
→ When someone new joins, they are added to the right circle
→ To check if two people know each other, see if they are in
  the same circle

The "ultimate parent" is like the group leader:
→ Everyone in the circle points to the leader
→ The leader points to themselves
→ To find someone's circle, follow the chain to the leader

Path compression is like a shortcut:
→ Instead of A→B→C→Leader, we make A→Leader directly
→ Next time, A finds the leader in one step

Union by size is like merging companies:
→ Always merge the smaller company into the larger one
→ The big company absorbs the small one
→ This keeps the organization flat and efficient
```

## Algorithm

```
Union by Rank:

1. Find ultimate parents:
   pu = findPar(u)
   pv = findPar(v)

2. If already in same component: return

3. Compare ranks:
   → If rank[pu] < rank[pv]: parent[pu] = pv
   → If rank[pv] < rank[pu]: parent[pv] = pu
   → If equal: parent[pv] = pu, rank[pu]++

Why attach smaller to larger?
→ Keeps tree height minimal
→ Reduces path length for findPar()
→ Optimizes future queries! ✓

────────────────────────────────────────

Union by Size:

1. Find ultimate parents:
   pu = findPar(u)
   pv = findPar(v)

2. If already in same component: return

3. Compare sizes:
   → If size[pu] < size[pv]:
       parent[pu] = pv
       size[pv] += size[pu]
   → Else:
       parent[pv] = pu
       size[pu] += size[pv]

Why size instead of rank?
→ Size stays accurate after path compression
→ More intuitive (actual node count)
→ Same time complexity
→ Recommended for most cases! ✓

────────────────────────────────────────

Path Compression:

int findPar(int node) {
    if (node == parent[node])
        return node;
    return parent[node] = findPar(parent[node]);
                          ↑
                  Path compression happens here!
}

While finding ultimate parent,
connect every node directly to root!

Benefit:
→ Future queries become O(1)
→ Tree becomes very flat
→ Amortized time: O(α(N)) ≈ constant!
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class DisjointSet {
    vector<int> rank, parent, size;
    
public:
    DisjointSet(int n) {
        rank.resize(n + 1, 0);
        parent.resize(n + 1);
        size.resize(n + 1, 1);
        
        for (int i = 0; i <= n; i++) {
            parent[i] = i;
        }
    }
    
    // Find ultimate parent with path compression
    int findUPar(int node) {
        if (node == parent[node])
            return node;
        return parent[node] = findUPar(parent[node]);
    }
    
    // Union by rank
    void unionByRank(int u, int v) {
        int ulp_u = findUPar(u);
        int ulp_v = findUPar(v);
        
        if (ulp_u == ulp_v) return;  // Already same component
        
        // Attach smaller rank to larger rank
        if (rank[ulp_u] < rank[ulp_v]) {
            parent[ulp_u] = ulp_v;
        }
        else if (rank[ulp_v] < rank[ulp_u]) {
            parent[ulp_v] = ulp_u;
        }
        else {
            // Equal ranks: attach either way, increase rank
            parent[ulp_v] = ulp_u;
            rank[ulp_u]++;
        }
    }
    
    // Union by size
    void unionBySize(int u, int v) {
        int ulp_u = findUPar(u);
        int ulp_v = findUPar(v);
        
        if (ulp_u == ulp_v) return;  // Already same component
        
        // Attach smaller size to larger size
        if (size[ulp_u] < size[ulp_v]) {
            parent[ulp_u] = ulp_v;
            size[ulp_v] += size[ulp_u];
        }
        else {
            parent[ulp_v] = ulp_u;
            size[ulp_u] += size[ulp_v];
        }
    }
};

int main() {
    DisjointSet ds(7);
    
    ds.unionBySize(1, 2);
    ds.unionBySize(2, 3);
    ds.unionBySize(4, 5);
    ds.unionBySize(6, 7);
    ds.unionBySize(5, 6);
    
    // Check if 3 and 7 are in same component
    if (ds.findUPar(3) == ds.findUPar(7)) {
        cout << "Same\n";
    }
    else {
        cout << "Not same\n";
    }
    
    // Connect 3 and 7
    ds.unionBySize(3, 7);
    
    // Check again
    if (ds.findUPar(3) == ds.findUPar(7)) {
        cout << "Same\n";
    }
    else {
        cout << "Not same\n";
    }
    
    return 0;
}
```

**Output:**

```
Not same
Same
```

## Path Compression in Action

```
Before path compression (after all unions):

        4
      / | \
     1  5  6
    /|\    |
   2 3     7

findUPar(3):
3 → parent[3]=1 → parent[1]=4 → return 4
Now update: parent[3] = 4

After path compression:

        4
     /||| \
    1 2 3 5 6
            |
            7

Next time findUPar(3):
3 → parent[3]=4 → return 4 (one step!)

This is why time complexity → O(α(N)) ≈ O(1)!
```

## Edge Cases

```
Single node:             Self-union:
n = 1                   union(1, 1)
parent[1] = 1           Already same parent
Already its own leader  Nothing happens ✓

Disconnected before:     Already connected:
find(1) != find(2)      union(1, 2) again
Union connects them     find(1) == find(2)
                        Skip, already same ✓

Large N:                 All nodes connected:
N = 10^6                One big component
Works efficiently       size[root] = N
with path compression   find() = O(1) ✓
```

## Complexity Analysis

**Time Complexity:** `O(α(N))` per operation

- α(N) = Inverse Ackermann function
- For all practical N, α(N) ≤ 4
- Effectively **constant time!** ✓

**Space Complexity:** `O(N)`

- Parent array: O(N)
- Size/Rank array: O(N)
- Total: **O(N)**

```
Practical values:

N = 10^5: α(N) ≈ 3
N = 10^9: α(N) ≈ 4
N = 10^600: α(N) ≈ 5

For all realistic inputs: α(N) ≤ 4!

Union by Rank vs Union by Size:

Property              Union by Rank       Union by Size
─────────────────────────────────────────────────────────────
Metric                Rank (height)       Size (# nodes)
Initial value         0                   1
Update on union       Sometimes +1        Always updated
After compression     Distorted ✗         Accurate ✓
Intuitive             Less                More ✓
Practical use         Less common         More common ✓

Recommendation: Use Union by Size! ✓
→ More intuitive (actual node count)
→ Size remains accurate
→ Same time complexity
→ Easier to understand and debug
```

## Interview Questions

```
Q1: What is a Disjoint Set (Union-Find)?

A: It is a data structure that tracks which items are in the same
   group. Think of it like managing friend circles — it can quickly
   tell you if two people are in the same circle, and it can merge
   two circles when people become friends.

──────────────────────────────────────────────────────────────

Q2: What are the two main operations?

A: find(x) — tells you who the leader of x's group is.
   union(x, y) — merges the groups of x and y into one.

   Think of find as "who is the group leader?" and union as
   "these two groups are now friends, merge them."

──────────────────────────────────────────────────────────────

Q3: What is path compression?

A: When you look up the leader of a node, you make that node point
   directly to the leader instead of going through middlemen. Like
   giving everyone in a group the leader's phone number directly,
   instead of calling a friend who calls another friend.

   This makes future lookups incredibly fast — almost instant.

──────────────────────────────────────────────────────────────

Q4: What is union by size?

A: When merging two groups, always attach the smaller group to the
   larger group. Like a big company absorbing a small startup. This
   keeps the organization flat, so finding the CEO (leader) is fast
   for everyone.

──────────────────────────────────────────────────────────────

Q5: What is union by rank?

A: Similar to union by size, but uses "height" of the tree instead
   of number of nodes. It tries to keep the tree shallow. However,
   after path compression, the rank becomes inaccurate, so most
   people prefer union by size.

──────────────────────────────────────────────────────────────

Q6: Why not just use BFS/DFS to check connectivity?

A: BFS/DFS takes O(N+E) time each query. If you have thousands of
   queries, that is too slow. Union-Find answers each query in
   nearly O(1) time. It is built for dynamic graphs where edges are
   added over time.

──────────────────────────────────────────────────────────────

Q7: Where is Disjoint Set used?

A: Common uses include:
    → Kruskal's MST algorithm (checking if adding an edge creates
      a cycle)
    → Social networks (finding friend circles)
    → Image processing (grouping connected pixels)
    → Network routing (checking if two computers are connected)
    → Games (checking if two territories are connected)

──────────────────────────────────────────────────────────────

Q8: What does O(α(N)) mean?

A: α is the inverse Ackermann function. It grows so slowly that
   for any number you will ever use in real life, α(N) is at most
   4. So O(α(N)) is basically O(1) — constant time. It is one of
   the fastest data structures in computer science.

──────────────────────────────────────────────────────────────

Q9: Can Union-Find delete edges?

A: No, standard Union-Find only supports adding edges (union), not
    removing them. Once two groups are merged, they stay merged.
    There are advanced versions that support deletion, but they are
    much more complex and rarely needed in interviews.

──────────────────────────────────────────────────────────────

Q10: What happens if I union two nodes already in the same group?

A: Nothing. The code checks if they have the same leader and
    returns immediately. It is like trying to friend someone who is
    already in your friend circle — no change needed.

──────────────────────────────────────────────────────────────

Q11: Why initialize parent[i] = i?

A: At the start, every node is in its own group, and it is its own
    leader. Like everyone starting alone before making any friends.
    As unions happen, some nodes will point to other leaders.

──────────────────────────────────────────────────────────────

Q12: Why initialize size[i] = 1?

A: Each node starts as a group of exactly one person — itself. When
    two groups merge, their sizes add up. This helps us decide which
    group is bigger during union operations.

──────────────────────────────────────────────────────────────

Q13: Can Union-Find handle directed graphs?

A: Not directly. Union-Find is designed for undirected connections
    where "A is connected to B" means the same as "B is connected
    to A." For directed graphs, you need different algorithms.

──────────────────────────────────────────────────────────────

Q14: How do I know which node is the ultimate parent?

A: Follow the parent pointers until you find a node that points to
    itself. That node is the leader. The find() function does this
    automatically and also applies path compression along the way.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

    Mistake 1: Forgetting path compression in find()
    → Fix: Always write parent[node] = find(parent[node])

    Mistake 2: Not updating size after union
    → Fix: Add the smaller size to the larger size

    Mistake 3: Unioning nodes instead of their leaders
    → Fix: Always call find() first, then union the leaders

    Mistake 4: Using rank when you meant size
    → Fix: Be consistent — use one or the other

    Mistake 5: Initializing size to 0 instead of 1
    → Fix: Every node starts as a component of size 1

    Mistake 6: Forgetting to check if already connected
    → Fix: If find(u) == find(v), return immediately
```
