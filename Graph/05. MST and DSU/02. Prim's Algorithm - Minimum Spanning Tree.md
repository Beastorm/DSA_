## 1. Problem Statement

Given a **weighted, undirected, and connected graph** of `V` vertices and `E` edges, find the **sum of weights** of the edges of the Minimum Spanning Tree (MST).

Sometimes you may also need to find the actual MST edges, stored as pairs `{u, v}`.

**Input Format:**

- `V` = number of vertices
- `adj[]` = adjacency list where `adj[u]` contains `{v, weight}` pairs

**Output:**

- Sum of MST edge weights (and optionally the MST edges)

## Key Insight

```
Prim's Algorithm - Key Idea

Greedy Strategy: Grow the MST one edge at a time

Start from any node (typically node 0)

At each step:

  → Among all edges connecting MST to non-MST nodes
  → Pick the edge with MINIMUM weight
  → Add that edge and node to MST

Repeat until all nodes are in MST

Why does this work?

→ Cut Property: Minimum weight edge crossing any cut
  must be in the MST

→ At each step, we're finding minimum edge crossing
  the cut between {MST nodes} and {non-MST nodes}

→ Greedy choice is always optimal! ✓
```

---

## Intuition

```
Imagine you are building a road network to connect all villages:

→ Start at any village (say village 0)
→ Look at all roads from your current network to new villages
→ Build the CHEAPEST such road
→ Now you have two villages connected
→ Repeat: always build the cheapest road from your network
  to a village not yet connected
→ Stop when all villages are in your network

This is exactly how Prim's works!

Think of it like a spider web:
→ The spider sits at the center (starting node)
→ It sends out threads to nearby points
→ It always extends the shortest thread first
→ The web grows outward, always choosing the cheapest connection

Why does greediness work here?

At every step, you have two groups:
→ Group A: Villages already in your network
→ Group B: Villages not yet connected

The cheapest road between A and B MUST be in the MST.
This is the "cut property" — and Prim's uses it at every step!
```

## Algorithm

```
Data Structures Needed:

1. Priority Queue (Min-Heap):
   Stores: {weight, node} or {weight, node, parent}
   → Always gives us the minimum weight edge
   → Helps make greedy choice efficiently

2. Visited Array:
   Tracks which nodes are already in MST
   → Prevents cycles
   → Ensures we don't revisit nodes

3. Sum Variable:
   Accumulates total MST weight

4. MST Array (optional):
   Stores actual MST edges as {parent, node} pairs
   → Only needed if we want the tree structure
   → Not needed if only sum is required

Algorithm Steps:

1. Initialize:
   visited[V] = all false
   sum = 0
   pq.push({0, 0}) or {0, 0, -1}
   (weight=0, start node=0, parent=-1)

2. While priority queue is not empty:
   a. Pop {wt, node} with minimum weight
   
   b. If node is already visited: continue
   
   c. Mark node as visited
   
   d. Add wt to sum
   
   e. (Optional) Store {parent, node} in MST
   
   f. For each unvisited neighbor:
      → Push {edgeWeight, neighbor} to pq

3. Return sum (and MST if needed)
```

### Why Greedy Works

```
Proof using Cut Property:

Cut Property states:
"For any cut dividing vertices into two sets,
 the minimum weight edge crossing the cut
 must be in the MST."

Prim's algorithm:
At each step, MST nodes form one set,
non-MST nodes form another set.

We always pick minimum weight edge crossing this cut!

→ This edge MUST be in MST (by cut property)
→ Greedy choice is optimal! ✓

Example visualization:

Step 1:
MST set: {0}
Non-MST: {1, 2, 3, 4}
Cut edges: 0-1(2), 0-2(1)
Minimum: 0-2(1) ← Pick this! ✓

Step 2:
MST set: {0, 2}
Non-MST: {1, 3, 4}
Cut edges: 2-1(1), 2-3(2), 2-4(2), 0-1(2)
Minimum: 2-1(1) ← Pick this! ✓

And so on...
Each greedy choice is guaranteed optimal! ✓
```

### Priority Queue Duplicates

```
Why can same node appear multiple times in PQ?

Example:

       2
  0 ──── 1
   \    /
    1  1
     \/
      2

From node 0:
→ Add {2, 1} (via edge 0-1)
→ Add {1, 2} (via edge 0-2)

From node 2 (processed before node 1):
→ Add {1, 1} (via edge 2-1)

Priority queue now has:
{1, 1}, {2, 1}
 ↑       ↑
Same node 1, different weights!

When we pop {1, 1}:
→ Mark node 1 as visited
→ Add weight 1 to sum

When we pop {2, 1}:
→ Node 1 already visited
→ Skip! (continue) ✓

This is why we check vis[node] before processing!

Duplicates are OK:
→ We always process best (minimum) weight first
→ Later duplicates are skipped
→ No incorrect behavior ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    // Function to find the sum of weights of MST edges
    int spanningTree(int V, vector<vector<int>> adj[]) {
        
        // Min-heap: {weight, node}
        priority_queue<pair<int, int>, vector<pair<int, int>>, greater<pair<int, int>>> pq;
        
        // Visited array
        vector<int> vis(V, 0);
        
        // Start from node 0 with weight 0
        pq.push({0, 0});
        int sum = 0;
        
        while (!pq.empty()) {
            auto it = pq.top();
            pq.pop();
            
            int wt = it.first;
            int node = it.second;
            
            // Skip if already in MST
            if (vis[node] == 1) continue;
            
            // Add to MST
            vis[node] = 1;
            sum += wt;
            
            // Explore neighbors
            for (auto it : adj[node]) {
                int adjNode = it[0];
                int edW = it[1];
                
                if (!vis[adjNode]) {
                    pq.push({edW, adjNode});
                }
            }
        }
        
        return sum;
    }
};

int main() {
    int V = 5;
    vector<vector<int>> edges = {
        {0, 1, 2}, {0, 2, 1}, {1, 2, 1},
        {2, 3, 2}, {3, 4, 1}, {4, 2, 2}
    };
    
    // Build adjacency list
    vector<vector<int>> adj[V];
    for (auto it : edges) {
        vector<int> tmp(2);
        tmp[0] = it[1];
        tmp[1] = it[2];
        adj[it[0]].push_back(tmp);
        
        tmp[0] = it[0];
        tmp[1] = it[2];
        adj[it[1]].push_back(tmp);
    }
    
    Solution obj;
    int sum = obj.spanningTree(V, adj);
    
    cout << "Sum of MST edge weights: " << sum << endl;
    
    return 0;
}
```

**Output:** `Sum of MST edge weights: 5`

### Code with MST Edge Tracking

```cpp
#include <bits/stdc++.h>
using namespace std;

class Solution {
public:
    // Returns {sum, MST edges}
    pair<int, vector<pair<int, int>>> spanningTreeWithEdges(int V, vector<vector<int>> adj[]) {
        // Min-heap: {weight, node, parent}
        priority_queue<tuple<int, int, int>, vector<tuple<int, int, int>>, greater<tuple<int, int, int>>> pq;
        
        vector<int> vis(V, 0);
        vector<pair<int, int>> mstEdges;
        
        // Start: weight=0, node=0, parent=-1
        pq.push({0, 0, -1});
        int sum = 0;
        
        while (!pq.empty()) {
            auto [wt, node, parent] = pq.top();
            pq.pop();
            
            if (vis[node]) continue;
            
            vis[node] = 1;
            sum += wt;
            
            // Store edge (skip first node with parent=-1)
            if (parent != -1) {
                mstEdges.push_back({parent, node});
            }
            
            // Explore neighbors
            for (auto it : adj[node]) {
                int adjNode = it[0];
                int edW = it[1];
                
                if (!vis[adjNode]) {
                    pq.push({edW, adjNode, node});
                }
            }
        }
        
        return {sum, mstEdges};
    }
};

int main() {
    int V = 5;
    vector<vector<int>> edges = {
        {0, 1, 2}, {0, 2, 1}, {1, 2, 1},
        {2, 3, 2}, {3, 4, 1}, {4, 2, 2}
    };
    
    vector<vector<int>> adj[V];
    for (auto it : edges) {
        vector<int> tmp(2);
        tmp[0] = it[1];
        tmp[1] = it[2];
        adj[it[0]].push_back(tmp);
        
        tmp[0] = it[0];
        tmp[1] = it[2];
        adj[it[1]].push_back(tmp);
    }
    
    Solution obj;
    auto [sum, mstEdges] = obj.spanningTreeWithEdges(V, adj);
    
    cout << "MST sum: " << sum << endl;
    cout << "MST edges:\n";
    for (auto [u, v] : mstEdges) {
        cout << u << " - " << v << endl;
    }
    
    return 0;
}
```

**Output:**

```
MST sum: 5
MST edges:
0 - 2
1 - 2
2 - 3
3 - 4
```

### Starting Node Choice

```
Question: Can we start from any node?

Answer: YES! ✓

MST is unique (if all weights distinct)
regardless of starting point.

Example: Start from node 0 vs node 2

Start from 0:           Start from 2:

Step 1: Add 0-2 (1)    Step 1: Add 2-0 (1)
Step 2: Add 2-1 (1)    Step 2: Add 2-1 (1)
Step 3: Add 2-3 (2)    Step 3: Add 2-3 (2)
Step 4: Add 3-4 (1)    Step 4: Add 3-4 (1)

Same edges, same sum! ✓

Why does this work?
→ MST is a property of the graph, not the algorithm
→ Greedy choice (minimum edge) is independent of start
→ All starting points lead to same MST ✓

Convention:
→ Usually start from node 0 for simplicity
→ But any node works! ✓
```

## Edge Cases

```
Single node:              Disconnected graph:
V = 1, edges = []        Graph has multiple components
sum = 0                  MST doesn't exist!
MST = {}                 (Problem states "connected graph")

Complete graph:           All edges same weight:
n(n-1)/2 edges           Multiple valid MSTs
Pick any n-1 edges       Algorithm picks one arbitrarily
Sum is well-defined ✓    Sum is same for all ✓

Self-loops:              Parallel edges:
Edge u→u                 Multiple edges between u-v
Ignored (creates cycle)  Pick minimum weight edge ✓

Zero-weight edges:       Negative weights:
Allowed! ✓               Not typical for MST problems
MST includes them        But algorithm still works ✓
```

## Complexity Analysis

**Time Complexity:** `O(E log E)`

- Priority queue operations: O(E) insertions × O(log E) each
- Each edge added to PQ at most once (per endpoint)
- Maximum PQ size: O(E)
- Total: **O(E log E)**

**Space Complexity:** `O(V + E)`

- Adjacency list: O(E)
- Visited array: O(V)
- Priority queue: O(E) maximum
- MST edges (optional): O(V-1) = O(V)
- Total: **O(V + E)**

```
Detailed analysis:

E = number of edges
V = number of vertices

Best case: O(E log V)
  Sparse graph, few PQ operations

Worst case: O(E log E)
  Dense graph, many duplicates in PQ
  
Average case: O(E log E)
  Since E ≤ V² in connected graph
  log E ≈ log V² = 2 log V
  So O(E log E) ≈ O(E log V)

Space breakdown:
→ adj[]: O(2E) = O(E) for undirected
→ vis[]: O(V)
→ pq: O(E) worst case (duplicates)
→ Total: O(V + E)

Optimization:
Using Fibonacci heap: O(E + V log V)
But complex implementation, rarely used in practice.
```

## Interview Questions

```
Q1: What is Prim's algorithm used for?

A: Prim's finds the Minimum Spanning Tree — the cheapest set of
   roads that connects all villages. It starts at one village and
   greedily grows the network by always adding the cheapest road
   to a new village.

──────────────────────────────────────────────────────────────

Q2: How is Prim's different from Dijkstra?

A: Both use a priority queue and a visited set, but they solve
   different problems:

   → Dijkstra finds the cheapest path from ONE source to all
     other nodes. It cares about the total cost to reach each
     node from the start.

   → Prim's finds the cheapest set of edges that connects ALL
     nodes together. It cares about the cost of adding each new
     node to the growing tree.

   Think of it like this:
   → Dijkstra = "What is the cheapest way to get TO each place?"
   → Prim's = "What is the cheapest way to CONNECT all places?"

   In code, the difference is tiny:
   → Dijkstra: dist[neighbor] = dist[node] + edgeWeight
   → Prim's: just use edgeWeight directly (no accumulated cost)

──────────────────────────────────────────────────────────────

Q3: Why does Prim's greedy approach work?

A: Because of the "cut property." At every step, Prim's has two
   groups: villages in the MST and villages outside. The cheapest
   road between these two groups MUST be in the final MST. So
   picking it is always safe — you never make a mistake.

──────────────────────────────────────────────────────────────

Q4: What data structures does Prim's need?

A: Three main things:
   → A priority queue (min-heap) to always get the cheapest edge
   → A visited array to track which villages are already in the MST
   → A variable to add up the total cost

   The priority queue is the key. Without it, you would have to
   scan all edges every time, which is very slow.

──────────────────────────────────────────────────────────────

Q5: Why do we skip nodes already visited?

A: Because adding them again would create a loop. A spanning tree
   has no cycles. Once a village is in the MST, we never need to
   add it again. Skipping duplicates also handles the case where
   the same node appears multiple times in the priority queue with
   different edge weights.

──────────────────────────────────────────────────────────────

Q6: Can Prim's handle duplicate edges in the priority queue?

A: Yes, and it happens all the time! The same village might be
   reached from different neighbors with different costs. We push
   all of them into the queue. When we pop one, we mark the
   village as visited. Any later pops for the same village are
   simply skipped. The first pop is always the cheapest because
   the queue is a min-heap.

──────────────────────────────────────────────────────────────

Q7: Does the starting node matter?

A: No. You can start from any village and you will get the same
   MST (assuming all edge weights are different). The algorithm
   only cares about picking the cheapest edge at each step, not
   where it began. Starting from node 0 is just a convention.

──────────────────────────────────────────────────────────────

Q8: When should I use Prim's vs Kruskal's?

A: Use Prim's when the graph has many edges (dense) and is stored
   as an adjacency list or matrix. It grows one connected tree,
   which feels natural for dense graphs.

   Use Kruskal's when the graph has few edges (sparse) and is
   stored as a simple edge list. It sorts all edges first, which
   is efficient when there are not many.

   Both give the same answer. The choice is about convenience.

──────────────────────────────────────────────────────────────

Q9: What happens if the graph is disconnected?

A: Prim's will only find the MST for the connected component that
   contains the starting node. Villages on other islands will
   never be visited. Standard Prim's assumes the graph is fully
   connected. If it might not be, you need to check if all nodes
   were visited at the end.

──────────────────────────────────────────────────────────────

Q10: Can Prim's handle negative edge weights?

A: Yes, but it is unusual. Negative weights mean someone pays you
    to build that road. Prim's will happily include them because
    it lowers the total cost. However, most MST problems assume
    non-negative weights.

──────────────────────────────────────────────────────────────

Q11: Why is the time complexity O(E log E)?

A: Every edge gets pushed into the priority queue at most once
    from each endpoint, so O(E) pushes. Each push and pop takes
    O(log E) time because the queue can hold up to O(E) items.
    Multiplying these gives O(E log E).

──────────────────────────────────────────────────────────────

Q12: What is the space complexity?

A: O(V + E). You need:
    → The adjacency list: O(E)
    → The visited array: O(V)
    → The priority queue: up to O(E) in the worst case

──────────────────────────────────────────────────────────────

Q13: Can we use a normal queue instead of a priority queue?

A: No. A normal queue gives you elements in the order they were
    added, not the cheapest one. Without a priority queue, Prim's
    would lose its greedy property and no longer find the minimum
    spanning tree. It would just find some spanning tree, which
    could be much more expensive.

──────────────────────────────────────────────────────────────

Q14: How do we actually get the MST edges, not just the sum?

A: Store the parent of each node when you add it to the MST.
    When you push a neighbor into the priority queue, also record
    which node it came from. When you pop and mark a node as
    visited, save the edge (parent, current node). This gives you
    the exact set of edges in the MST.

──────────────────────────────────────────────────────────────

Q15: What are the most common mistakes?

A: Common mistakes and fixes:

    Mistake 1: Forgetting to check if node is already visited
    → Fix: Always check vis[node] right after popping from PQ

    Mistake 2: Using a normal queue instead of priority queue
    → Fix: Use min-heap so cheapest edge is always picked first

    Mistake 3: Adding edge weight to accumulated distance
    → Fix: Prim's just uses edgeWeight, not dist[node] + weight
           (that is Dijkstra, not Prim's!)

    Mistake 4: Not marking the starting node as visited
    → Fix: The first pop marks it, but make sure sum starts at 0

    Mistake 5: Pushing visited neighbors into the queue
    → Fix: Only push neighbors that are NOT yet visited

    Mistake 6: Confusing Prim's with Dijkstra
    → Fix: Dijkstra tracks total cost from source. Prim's only
           cares about the cost of the single edge being added.
```
