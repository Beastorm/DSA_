## Most Stones Removed with Same Row or Column - DSU

### Problem Statement:

There are `n` stones at integer coordinate points on a 2D plane. Each coordinate point may have at most one stone.

A stone can be **removed** if it shares either the **same row** or **same column** as another stone that has **not been removed**.

Given an array `stones` where `stones[i] = [xi, yi]` represents the location of the i-th stone, return the **maximum possible number of stones** that you can remove.

---

## Key Insight

```
Observation 1: Connectivity

Two stones are "connected" if they share:
→ Same row, OR
→ Same column

This forms groups of connected stones!

Example:
[0,0] -- row 0 -- [0,1] -- col 1 -- [2,1]
All three stones are in same group
(even though [0,0] and [2,1] don't directly share row/col)

Observation 2: Maximum Removal

From a group of K stones:
→ Can remove at most K-1 stones
→ Must keep 1 stone (can't remove last stone in group)

Why? Each removal needs another stone in same row/col
→ When only 1 stone left, can't remove it!

Observation 3: Formula

Total stones = N
Number of groups = G

Maximum removable = N - G

Proof:
→ Each group contributes (size - 1) removals
→ Sum of (size_i - 1) = Σsize_i - Σ1 = N - G ✓

So we just need to count number of groups!
```

## Why Use DSU?

```
Problem: Find connected components (groups)

Traditional approach (DFS/BFS):
→ Build graph: connect stones sharing row/col
→ Run DFS/BFS to find components
→ Time: O(N²) to check all pairs for edges

DSU approach:
→ Treat rows and columns as nodes!
→ Union stone's row with stone's column
→ Count unique components
→ Time: O(N × α(N)) ≈ O(N) ✓

Key trick: Row-Column as nodes

Instead of connecting stones directly,
connect their row number and column number!

Stone at [r, c]:
→ Unite row r with column c
→ All stones sharing row r or column c
  will be in same component!

Example:
Stone [0, 0]: unite(row 0, col 0)
Stone [0, 1]: unite(row 0, col 1)
Stone [2, 1]: unite(row 2, col 1)

After processing:
→ row 0 connected to col 0
→ row 0 connected to col 1
→ row 2 connected to col 1
→ So: row 0 ↔ col 0, col 1 ↔ row 2
→ All in same component! ✓
```

---

## Row-Column Collision Problem

```
Problem: Row and column numbers can overlap!

Example:
Row 0 and Column 0 → both represented as 0
→ Collision! Can't distinguish row from column

Solution: Offset column numbers

Add a large offset to column numbers:
→ Row r stays as r
→ Column c becomes c + 10001

Why 10001?
→ Max coordinates in problem: [-10000, 10000]
→ So max row/col value = 10000
→ Offset by 10001 ensures no collision! ✓

Example:
Stone [0, 0]: unite(0, 10001)
Stone [0, 1]: unite(0, 10002)
Stone [1, 0]: unite(1, 10001)

Now:
→ Row 0, Row 1, ... are 0, 1, ...
→ Col 0, Col 1, ... are 10001, 10002, ...
→ No collision! ✓
```

## Algorithm

```
Step 1: Initialize DSU
Create DSU with map-based parent storage
(dynamic size, creates parent on-demand)

Step 2: Process each stone
For stone at [row, col]:
  unite(row, col + OFFSET)
  
This connects:
→ All stones with same row
→ All stones with same column
→ Into same component!

Step 3: Count unique components
For each stone [row, col]:
  Add find(row) to set of components
  
Set size = number of unique components

Step 4: Calculate answer
return totalStones - numberOfComponents
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class DSU {
public:
    unordered_map<int, int> parent;
    
    // Find with path compression
    int find(int x) {
        // Create parent if doesn't exist
        if (parent.find(x) == parent.end()) {
            parent[x] = x;
        }
        
        // Path compression
        if (x != parent[x]) {
            parent[x] = find(parent[x]);
        }
        
        return parent[x];
    }
    
    // Union operation
    void unite(int x, int y) {
        parent[find(x)] = find(y);
    }
};

class Solution {
public:
    int removeStones(vector<vector<int>>& stones) {
        DSU dsu;
        
        // Offset to avoid row-column collision
        const int OFFSET = 10001;
        
        // Unite each stone's row and column
        for (auto& stone : stones) {
            int row = stone[0];
            int col = stone[1];
            dsu.unite(row, col + OFFSET);
        }
        
        // Count unique components
        unordered_set<int> components;
        for (auto& stone : stones) {
            components.insert(dsu.find(stone[0]));
        }
        
        // Max removable = total stones - components
        return stones.size() - components.size();
    }
};

int main() {
    Solution sol;
    
    // Example 1
    vector<vector<int>> stones1 = {
        {0, 0}, {0, 1}, {1, 0}, {1, 2}, {2, 1}, {2, 2}
    };
    cout << "Example 1: " << sol.removeStones(stones1) << endl;
    
    // Example 2
    vector<vector<int>> stones2 = {
        {0, 0}, {0, 2}, {1, 3}, {3, 1}, {3, 2}, {4, 3}
    };
    cout << "Example 2: " << sol.removeStones(stones2) << endl;
    
    return 0;
}
```

**Output:**
```
Example 1: 5
Example 2: 4
```

## Why This DSU Approach Works

```
Traditional Graph Approach:
→ Create edge between every pair of stones sharing row/col
→ Time: O(N²) to check all pairs
→ Space: O(N²) for adjacency list
→ DFS/BFS to find components: O(N + E)

DSU Row-Column Trick:
→ Only O(N) unions (one per stone)
→ No need to explicitly create stone-to-stone edges
→ Implicitly connects via row/col nodes
→ Time: O(N × α(N)) ≈ O(N)
→ Much faster! ✓

Why it's equivalent:

If stones A and B share row r:
→ A: unite(r, colA)
→ B: unite(r, colB)
→ Both connect to row r
→ So A and B are in same component! ✓

If stones B and C share column c:
→ B: unite(rowB, c)
→ C: unite(rowC, c)
→ Both connect to col c
→ So B and C are in same component! ✓

Transitively: A ↔ r ↔ B ↔ c ↔ C
→ A, B, C all in same component! ✓

This is why the trick works!
```

## Alternative: Count Components Differently

```cpp
class Solution {
public:
    int removeStones(vector<vector<int>>& stones) {
        DSU dsu;
        const int OFFSET = 10001;
        
        // Unite row and column for each stone
        for (auto& stone : stones) {
            dsu.unite(stone[0], stone[1] + OFFSET);
        }
        
        // Count components by counting unique ultimate parents
        // for all row and column nodes that were used
        unordered_set<int> components;
        
        // Check both rows and columns
        for (auto& stone : stones) {
            components.insert(dsu.find(stone[0]));
            components.insert(dsu.find(stone[1] + OFFSET));
        }
        
        // But this double counts! Each component has both row and col nodes
        // Better to stick with original approach:
        // Just check ultimate parent of rows (or columns)
        
        components.clear();
        for (auto& stone : stones) {
            components.insert(dsu.find(stone[0]));
        }
        
        return stones.size() - components.size();
    }
};
```

## Complexity Analysis

**Time Complexity:** `O(N × α(N)) ≈ O(N)`
- Process N stones: O(N)
- Each stone: 1 unite operation: O(α(N))
- Count components: O(N × α(N))
- Total: **O(N × α(N)) ≈ O(N)**

**Space Complexity:** `O(N)`
- Parent map: O(N) (at most 2N entries for rows+cols)
- Components set: O(N)
- Total: **O(N)**

```
Detailed breakdown:

Unite operations:
→ N stones × O(α(N)) per unite
→ O(N × α(N))

Find operations (counting):
→ N stones × O(α(N)) per find
→ O(N × α(N))

Total: O(N × α(N))

Since α(N) ≤ 4 for all practical N:
→ Effectively O(N) ✓

Space:
→ Parent map: max 2N entries (N rows + N cols used)
→ Components set: max N entries
→ Total: O(N)

Comparison with graph approach:
Graph (DFS/BFS): O(N²) time, O(N²) space
DSU: O(N) time, O(N) space
DSU wins! ✓
```

## Common Mistakes

```
❌ Mistake 1: Not using offset for columns
dsu.unite(stone[0], stone[1]);  // WRONG! Row-col collision

✓ Correct:
dsu.unite(stone[0], stone[1] + 10001);

❌ Mistake 2: Counting components incorrectly
for (auto& [node, par] : dsu.parent) {
    if (node == par) components++;  // WRONG! Includes both rows and cols
}

✓ Correct:
for (auto& stone : stones) {
    components.insert(dsu.find(stone[0]));
}

❌ Mistake 3: Wrong formula
return components.size();  // WRONG! This is number remaining

✓ Correct:
return stones.size() - components.size();

❌ Mistake 4: Not implementing path compression
int find(int x) {
    if (parent[x] != x)
        return find(parent[x]);  // WRONG! No compression
    return parent[x];
}

✓ Correct:
return parent[x] = find(parent[x]);  // Path compression!

❌ Mistake 5: Insufficient offset
dsu.unite(stone[0], stone[1] + 1000);  // WRONG! Max coord is 10000

✓ Correct:
dsu.unite(stone[0], stone[1] + 10001);  // Offset > max coordinate
```

## Why Formula Works: Mathematical Proof

```
Proof: Max removable = N - G (G = number of groups)

Claim: From group of size K, can remove K-1 stones

Proof by induction:
Base case: K = 1
  → Can't remove last stone
  → Remove 0 stones = 1-1 ✓

Inductive step: Assume true for K-1
  → Group of K stones
  → Remove 1 stone (has neighbor, so valid)
  → Left with K-1 stones (still connected)
  → By induction, can remove (K-1)-1 stones
  → Total: 1 + (K-2) = K-1 ✓

Therefore:
Total removable = Σ(size_i - 1) for all groups i
                = Σsize_i - Σ1
                = N - G ✓

So we just need to count groups (components)!
```

## Summary

```
Problem: Maximum stones removable (sharing row/col)

Key insights:
1. Stones sharing row/col form connected groups
2. From each group, can remove all but 1 stone
3. Answer = total stones - number of groups

DSU trick:
→ Unite row number with column number
→ All stones with same row/col implicitly connected
→ Count components efficiently

Algorithm:
1. For each stone [r,c]: unite(r, c+OFFSET)
2. Count unique components
3. Return N - components

Time: O(N)
Space: O(N)

Formula:
removable = totalStones - numberOfComponents

Simple, elegant, efficient! ✓
```
