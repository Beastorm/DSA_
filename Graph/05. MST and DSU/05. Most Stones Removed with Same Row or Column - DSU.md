## 1. Problem Statement

There are `n` stones at integer coordinate points on a 2D plane. Each coordinate point may have at most one stone.

A stone can be **removed** if it shares either the **same row** or **same column** as another stone that has **not been removed**.

Given an array `stones` where `stones[i] = [xi, yi]` represents the location of the i-th stone, return the **maximum possible number of stones** that you can remove.

```

n = 6
stones = [[0,0], [0,1], [1,0], [1,2], [2,1], [2,2]]

4. Remove [2,1] (shares row with [2,2])
```

## Key Insight

```
Observation 1: Connectivity

Two stones are "connected" if they share:
→ Same row, OR
→ Same column

This forms groups of connected stones!

Example chain:

   [0,0] ──row 0──→ [0,1] ──col 1──→ [2,1]

   Visual on grid:

     0 1 2
   0 S S .
   1 . . .
   2 . S .

   [0,0] and [2,1] do NOT directly share row or column.
   But [0,0] → [0,1] → [2,1] forms a chain.
   So all three are in the SAME group!

──────────────────────────────────────────────────────────────

Observation 2: Maximum Removal

From a group of K stones:
→ Can remove at most K-1 stones
→ Must keep 1 stone (can't remove last stone in group)

Why? Each removal needs another stone in same row/col.
When only 1 stone is left, it has no neighbor to "support"
its removal, so it must stay!

Visual (group of 4 stones):

   Before:              After removing 3:

     S ─ S                .   .
     │                    .   .
     S ─ S                    S  ← stuck! No neighbor

   Removed: 3 = 4 - 1 ✓

──────────────────────────────────────────────────────────────

Observation 3: Formula

Total stones = N
Number of groups = G
Maximum removable = N - G

Visual proof:

   Group 1 (size 4)          Group 2 (size 2)

      S ─ S                     S
      │                         │
      S ─ S                     S

   Remove 3                  Remove 1

   Total stones N = 4 + 2 = 6
   Total groups G = 2
   Removable = 3 + 1 = 4 = 6 - 2 ✓

   Math:
   → Each group contributes (size - 1) removals
   → Sum of (size_i - 1) = Σsize_i - Σ1 = N - G ✓

So we just need to count the number of groups!
```

### Why Use DSU?

```
Traditional approach (DFS/BFS):
→ Build graph: connect every pair of stones sharing row/col
→ Time: O(N²) to check all pairs for edges
→ Space: O(N²) for adjacency list

Visual (BFS/DFS — checking every pair):

   Stone 1 ──?──→ Stone 2   (check: same row? same col?)
   Stone 1 ──?──→ Stone 3   (check: same row? same col?)
   Stone 1 ──?──→ Stone 4   (check: same row? same col?)
   ...
   → N × N checks! Too slow!

DSU approach:
→ Treat rows and columns as nodes
→ Union each stone's row with its column
→ Count unique components
→ Time: O(N × α(N)) ≈ O(N) ✓

Visual (DSU — one union per stone):

   Stone [0,0]:  row 0 ──→ col 0
   Stone [0,1]:  row 0 ──→ col 1
   Stone [1,0]:  row 1 ──→ col 0
   ...
   → N unions only! Much faster!
```

### The Row-Column Trick

```
Instead of connecting stones directly, connect their row and column!

Stone at [r, c]:
→ Unite row r with column c
→ All stones sharing row r or column c will be in the same component

Visual (connecting stones directly — messy):

   [0,0] ──row 0──→ [0,1]
      \             /
       \   col 1   /
        \    |    /
         \   |   /
          \  |  /
           [2,1]

   3 stones → need to check all pairs → O(N²)

Visual (row-column trick — clean):

        col 0     col 1
          │         │
   row 0──┴──S──────┘
              [0,0] [0,1]
                    │
              row 2─┘
              [2,1]

   DSU unions:
   [0,0]: unite(row 0, col 0)
   [0,1]: unite(row 0, col 1)
   [2,1]: unite(row 2, col 1)

   After unions:

   row 0 ──→ col 0
    │
    └──────→ col 1 ←── row 2

   All rows and cols connect to one root!
   → All 3 stones in same component! ✓
```

### Row-Column Collision Problem

```
Problem: Row and column numbers can overlap!

Visual (WRONG — collision!):

   Stone [0, 0]:  unite(row 0, col 0)
   → unite(0, 0)

   DSU sees: 0 ──→ 0  (self loop!)

   Stone [0, 1]:  unite(row 0, col 1)
   → unite(0, 1)

   DSU sees: 0 ──→ 1

   But wait — is node "0" a row or a column?
   DSU gets confused! Row 0 and Col 0 look the same!

   Row 0 ──→ ???
   Col 0 ──→ ???
   Both are "0" → COLLISION! ✗

──────────────────────────────────────────────────────────────

Solution: Offset column numbers

Add an offset to column numbers:
→ Row r stays as r
→ Column c becomes c + offset

What offset to use?
→ Find the maximum row value: maxr
→ Set offset = maxr + 1
→ This guarantees columns start AFTER all rows!

Visual (CORRECT — no collision):

   Example: stones with rows 0,1,2 and cols 0,1,2
   maxr = 2, so offset = 3

   Stone [0, 0]:  unite(0, 0 + 3) = unite(0, 3)

   Row 0 ──→ Col 0 (3)

   Stone [0, 1]:  unite(0, 1 + 3) = unite(0, 4)

   Row 0 ──→ Col 1 (4)

   Stone [1, 0]:  unite(1, 0 + 3) = unite(1, 3)

   Row 1 ──→ Col 0 (3)

   Now:
   → Row 0, Row 1, Row 2 are 0, 1, 2
   → Col 0, Col 1, Col 2 are 3, 4, 5
   → No overlap! No collision! ✓

   Visual separation:

   Rows:    0    1    2
            │    │    │
            ↓    ↓    ↓
   Cols:    3    4    5

   Completely separate worlds! ✓

   DSU size needed:
   → Max row index = 2
   → Max col index = 2 + 3 = 5
   → Total nodes = 6 (indices 0 to 5)
   → Create DSU of size maxr + maxc + 2 = 2 + 2 + 2 = 6 ✓
```

## Intuition

```
Imagine a giant checkerboard with stones placed on some squares.

Visual:

     0 1 2
   0 S S .
   1 S . S
   2 . S S

You can pick up a stone ONLY if another stone sits somewhere in
the same row or the same column.

Visual (can remove [0,0] because [0,1] is in same row):

     0 1 2
   0 ✓ S .     ← [0,0] has neighbor [0,1] in row 0
   1 S . S
   2 . S S

Visual (cannot remove [2,2] if it is the last stone in its group):

     0 1 2
   0 . . .
   1 . . .
   2 . . S     ← [2,2] has NO neighbor → stuck! ✗

Think of it like a game of pick-up-sticks where each stick needs
a neighbor to lean on.

──────────────────────────────────────────────────────────────

Now, what if stones form a chain?

Visual (chain of 3 stones):

   Stone A at [0,0] ──row 0──→ Stone B at [0,1]
                                      │
                                      │ col 1
                                      ↓
                                Stone C at [2,1]

   On the grid:

     0 1 2
   0 A B .
   1 . . .
   2 . C .

   A and C do NOT share a row or column directly.
   But A → B → C forms a chain.
   So all three belong to the same "family" of stones!

──────────────────────────────────────────────────────────────

From any family of K stones, you can remove K-1 of them.

Visual (family of 4 stones):

   Before removals:

     S ─ S
     │
     S ─ S

   Step 1: Remove top-left (has neighbors) → 3 left
   Step 2: Remove top-right (has neighbors) → 2 left
   Step 3: Remove bottom-right (has neighbor) → 1 left
   Step 4: Last stone has NO neighbor → CANNOT remove!

   After:

     .   .
     .   .
         S   ← this one stays forever

   Removed: 3 = 4 - 1 ✓

   Why? Remove them one by one, always leaving at least one stone
   behind as a "pillar." The last stone has no neighbor left, so
   it stays.

──────────────────────────────────────────────────────────────

The DSU trick is like giving every row and every column a team badge.

Visual (team badges before any stones):

   Row 0: Badge R0    Col 0: Badge C0
   Row 1: Badge R1    Col 1: Badge C1
   Row 2: Badge R2    Col 2: Badge C2

   Each row and column starts in its own team.

When a stone sits at [r, c], it merges the team of row r with
the team of column c.

Visual (stone [0,0] merges R0 and C0):

   Before:              After:
   R0: {R0}             R0+C0: {R0, C0}
   C0: {C0}

Visual (stone [0,1] merges R0 and C1):

   Before:              After:
   R0+C0: {R0,C0}       R0+C0+C1: {R0,C0,C1}
   C1: {C1}

Visual (stone [2,1] merges R2 and C1):

   Before:              After:
   R0+C0+C1: {R0,C0,C1}  One big team: {R0,C0,C1,R2}
   R2: {R2}

After processing all stones, every connected family shares the
same team badge.

Count the number of unique team badges = number of families.
Answer = total stones - number of families.
```

## Algorithm

```
Step 1: Initialize DSU

Create DSU with map-based parent storage
(dynamic size, creates parent on-demand)

Visual:

   parent = {}  (empty map)
   No nodes yet!

──────────────────────────────────────────────────────────────

Step 2: Process each stone

For stone at [row, col]:
  unite(row, col + offset)

Visual (processing stones one by one):

   maxr = 2, so offset = 3

   Stone [0,0]: unite(0, 0+3) = unite(0, 3)

   parent: [0, 1, 2, 0, 4, 5]

        0 ←── 3

   Stone [0,1]: unite(0, 1+3) = unite(0, 4)

   parent: [0, 1, 2, 0, 0, 5]

        0 ←── 3
        ↑
        └── 4

   Stone [2,1]: unite(2, 1+3) = unite(2, 4)

   parent: [0, 1, 0, 0, 0, 5]

        0 ←── 3
        ↑
        ├── 4
        └── 2

This connects:
→ All stones with same row
→ All stones with same column
→ Into same component!

──────────────────────────────────────────────────────────────

Step 3: Count unique components

For each stone [row, col]:
  Add find(row) to set of components

Visual:

   Stone [0,0]: findUParent(0) → 0  ┐
   Stone [0,1]: findUParent(0) → 0  │ same root!
   Stone [2,1]: findUParent(2) → 0  ┘

   components = {0} → 1 unique component

Set size = number of unique components

──────────────────────────────────────────────────────────────

Step 4: Calculate answer

return totalStones - numberOfComponents

Visual:

   Total stones:     3
   Components:       1
   ─────────────────────
   Removable:        2 ✓
```

### Why the Trick Works

```
Case 1: Two stones share a row

Visual:

   Grid:
     0 1 2
   0 A B .     A at [0,0], B at [0,1]

   DSU operations:
   A: unite(row 0, col 0)
   B: unite(row 0, col 1)

   DSU tree:

        row 0
       /     \
   col 0     col 1

   Both A and B connect to row 0!
   → A and B are in same component! ✓

──────────────────────────────────────────────────────────────

Case 2: Two stones share a column

Visual:

   Grid:
     0 1 2
   0 . B .
   1 . . .
   2 . C .     B at [0,1], C at [2,1]

   DSU operations:
   B: unite(row 0, col 1)
   C: unite(row 2, col 1)

   DSU tree:

        col 1
       /     \
   row 0     row 2

   Both B and C connect to col 1!
   → B and C are in same component! ✓

──────────────────────────────────────────────────────────────

Case 3: Chain connection (transitive)

Visual:

   Grid:
     0 1 2
   0 A B .
   1 . . .
   2 . C .     A at [0,0], B at [0,1], C at [2,1]

   DSU operations:
   A: unite(row 0, col 0)
   B: unite(row 0, col 1)
   C: unite(row 2, col 1)

   DSU tree:

        row 0
       /     \
   col 0     col 1
               |
             row 2

   Path: A → row 0 → col 1 → row 2 → C

   A and C do NOT share row or column directly.
   But they are connected through the DSU tree!
   → A, B, C all in same component! ✓
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class DisjointSet {
private:
    vector<int> parent, rank;

public:
    DisjointSet(int n) {
        parent.resize(n);
        rank.resize(n, 0);
        for (int i = 0; i < n; i++) {
            parent[i] = i;
        }
    }

    int findUParent(int node) {
        if (parent[node] == node)
            return node;
        return parent[node] = findUParent(parent[node]);
    }

    void unionByRank(int u, int v) {
        int upu = findUParent(u);
        int upv = findUParent(v);

        if (upu == upv)
            return;

        if (rank[upu] > rank[upv]) {
            parent[upv] = upu;
        }
        else if (rank[upu] < rank[upv]) {
            parent[upu] = upv;
        }
        else {
            parent[upv] = upu;
            rank[upu]++;
        }
    }
};

int stoneRemove(vector<vector<int>> stones) {
    int maxr = INT_MIN, maxc = INT_MIN;
    for (auto&& i : stones) {
        maxr = max(maxr, i[0]);
        maxc = max(maxc, i[1]);
    }

    int offset = maxr + 1;
    DisjointSet ds(maxr + maxc + 2);

    for (auto&& i : stones) {
        ds.unionByRank(i[0], i[1] + offset);
    }

    unordered_set<int> components;
    for (auto&& i : stones) {
        components.insert(ds.findUParent(i[0]));
    }

    return stones.size() - components.size();
}

int main() {
    vector<vector<int>> stones = {
        {0, 0}, {0, 1}, {1, 0}, {1, 2}, {2, 1}, {2, 2}
    };

    int ans = stoneRemove(stones);
    cout << "Removed Stones: " << ans << endl;

    return 0;
}
```

**Output:**

```
Removed Stones: 5
```

```
Example 1: stones = [[0,0], [0,1], [1,0], [1,2], [2,1], [2,2]]

Step 0: Find max row and max column
  maxr = 2, maxc = 2
  offset = maxr + 1 = 3
  DSU size = maxr + maxc + 2 = 6 (indices 0 to 5)

── Initialization ──

parent: [0, 1, 2, 3, 4, 5]
rank:   [0, 0, 0, 0, 0, 0]

Nodes 0,1,2 = rows 0,1,2
Nodes 3,4,5 = cols 0,1,2 (offset by 3)

Stone [0, 0]:
  unionByRank(0, 0+3) = unionByRank(0, 3)
  upu = findUParent(0) = 0, upv = findUParent(3) = 3
  rank[0] == rank[3] == 0
  → parent[3] = 0, rank[0] = 1

        ↑
```

## Edge Cases

```
Single stone:
stones = [[0,0]]

   Grid:
     0
   0 S

   Groups = 1
   Removable = 1 - 1 = 0 ✓
   (No neighbor to support removal)

──────────────────────────────────────────────────────────────

All isolated stones:
stones = [[0,0],[1,1],[2,2]]

   Grid:
     0 1 2
   0 S . .
   1 . S .
   2 . . S

   Each stone in different row & col
   Groups = 3
   Removable = 3 - 3 = 0 ✓
   (Three separate islands, no bridges)

──────────────────────────────────────────────────────────────

All stones in same row:
stones = [[0,0],[0,1],[0,2]]

   Grid:
     0 1 2
   0 S S S

   Groups = 1
   Removable = 3 - 1 = 2 ✓
   (One straight line, keep one end)

──────────────────────────────────────────────────────────────

All stones in same column:
stones = [[0,0],[1,0],[2,0]]

   Grid:
     0
   0 S
   1 S
   2 S

   Groups = 1
   Removable = 3 - 1 = 2 ✓
   (One straight column, keep one end)

──────────────────────────────────────────────────────────────

Two separate lines:
Row line: [[0,0],[0,1]]
Col line: [[5,5],[6,5]]

   Grid 1:              Grid 2:
     0 1                  5
   0 S S                5 S
   1 . .                6 S

   Groups = 2
   Removable = 4 - 2 = 2 ✓
   (Two separate groups, each keeps one)

──────────────────────────────────────────────────────────────

L-shaped group:
[[0,0],[0,1],[1,1]]

   Grid:
     0 1
   0 S S
   1 . S

   All connected via [0,1]
   Groups = 1
   Removable = 3 - 1 = 2 ✓

──────────────────────────────────────────────────────────────

Negative coordinates:
stones = [[-1,0],[0,-1]]

   Grid:
      -1  0
   -1   S  .
    0   .  S

   [-1,0] and [0,-1] share neither row nor column
   Groups = 2
   Removable = 2 - 2 = 0 ✓
   (Vector DSU needs shifting for negatives; map-based DSU handles them directly)

──────────────────────────────────────────────────────────────

Large grid, sparse stones:

   Grid (10000 × 10000, only 3 stones):
     0     5000    9999
   0 S . . . . . . . .
   . . . . . . . . . .
   5000 . . S . . . . .
   . . . . . . . . . .
   9999 . . . . . . . S

   Many groups possible
   DSU handles efficiently ✓
   (Only 3 unions needed!)
```

### Common Mistakes

```
❌ Mistake 1: Not using offset for columns
  ds.unionByRank(row, col);  // WRONG! Row-col collision

  Visual:
    Row 0 ──→ 0
    Col 0 ──→ 0   ← same node! DSU crashes!

✓ Correct:
  ds.unionByRank(row, col + offset);

  Visual:
    Row 0 ──→ 0
    Col 0 ──→ 3   ← different node! ✓

──────────────────────────────────────────────────────────────

❌ Mistake 2: DSU too small
  DisjointSet ds(maxc + maxr);  // WRONG!

  Example: maxr=2, maxc=2, size=4
  But col 2 needs index 2+3=5 → OUT OF BOUNDS!

✓ Correct:
  DisjointSet ds(maxr + maxc + 2);

  Visual:
    Rows need: 0 to maxr
    Cols need: offset to offset+maxc
    Max index = maxr + 1 + maxc
    Size = maxr + maxc + 2 ✓

──────────────────────────────────────────────────────────────

❌ Mistake 3: Counting components incorrectly
  for (int i = 0; i < ds.parent.size(); i++) {
      if (ds.findUParent(i) == i) cnt++;  // WRONG!
  }

  Visual:
    Unused nodes: 0←0  1←1  2←2 ...
    You count EVERYTHING, even unused rows/cols!

✓ Correct:
  unordered_set<int> components;
  for (auto& stone : stones) {
      components.insert(ds.findUParent(stone[0]));
  }

  Visual:
    Only check rows that actually have stones:
    row 0 → root A
    row 1 → root A
    → 1 component! ✓

──────────────────────────────────────────────────────────────

❌ Mistake 4: Wrong formula
  return components.size();  // WRONG! This is stones remaining

✓ Correct:
  return stones.size() - components.size();

  Visual:
    Total stones: 6
    Components:   1
    ───────────────
    Removable:    5 ✓

──────────────────────────────────────────────────────────────

❌ Mistake 5: Not implementing path compression
  int findUParent(int node) {
      if (parent[node] == node)
          return node;
      return findUParent(parent[node]);  // WRONG! No compression
  }

✓ Correct:
  return parent[node] = findUParent(parent[node]);

  Visual (without compression):
    0 ← 1 ← 2 ← 3   find(3) → 3 steps

  Visual (with compression):
    0 ← 1   find(3) → 1 step, then:
    0 ← 2
    0 ← 3

──────────────────────────────────────────────────────────────

❌ Mistake 6: Wrong comparison in unionByRank
  else if (rank[upu < rank[upv]])  // WRONG! Syntax error!

✓ Correct:
  else if (rank[upu] < rank[upv])  // Compare ranks properly
```

## Complexity Analysis

**Time Complexity:** `O(N × α(N)) ≈ O(N)`

- Process N stones: O(N)
- Each stone: 1 unite operation: O(α(N))
- Count components: O(N × α(N))
- Total: **O(N × α(N)) ≈ O(N)**

**Space Complexity:** `O(N)`

- Parent map: O(N) (at most 2N entries for rows + cols)
- Components set: O(N)
- Total: **O(N)**

```
Visual comparison:

BFS/DFS approach:

   Stones:  S1  S2  S3  S4  S5
            │╲ │╲ │╲ │╲ │
            │ ╲│ ╲│ ╲│ ╲│
            Check EVERY pair!

   Time:  O(N²)  →  1000 stones = 1,000,000 checks!
   Space: O(N²)  →  Huge adjacency list!

DSU approach:

   S1: row 0 ──→ col 1
   S2: row 0 ──→ col 2
   S3: row 1 ──→ col 2
   ...

   One union per stone only!

   Time:  O(N)   →  1000 stones = 1000 unions!
   Space: O(N)   →  Small parent map!

Detailed breakdown:

Unite operations:
→ N stones × O(α(N)) per unite
→ O(N × α(N))

Find operations (counting):
→ N stones × O(α(N)) per find
→ O(N × α(N))

Total: O(N × α(N))
Since α(N) ≤ 4 for all practical N:
→ Effectively O(N) ✓

Space:
→ Parent map: max 2N entries (N rows + N cols used)
→ Components set: max N entries
→ Total: O(N)

Comparison with graph approach:
Graph (DFS/BFS): O(N²) time, O(N²) space
DSU:             O(N) time,  O(N) space
DSU wins! ✓
```

## Interview Questions

```
Q1: What is the main idea of this problem?

A: Imagine stones on a checkerboard. You can only pick up a stone
   if another stone sits in the same row or column.

   Visual:

     0 1 2
   0 S S .     Can remove [0,0] because [0,1]
   1 S . S       shares the same row!
   2 . S S
       ↑
     Can remove [2,1] because [2,2]
       shares the same row!

   Think of it like removing chairs from a room — you can only
   remove a chair if another chair shares the same row or column
   of tiles.

──────────────────────────────────────────────────────────────

Q2: Why does the answer equal total stones minus groups?

A: Think of each connected group as a team. In each team, only one
   person can be left standing — everyone else can leave.

   Visual:

   Group of 5 stones:        Group of 3 stones:

      S ─ S                    S
      │                        │
      S ─ S                    S
      │
      S

   Remove 4 stones            Remove 2 stones
   Keep 1 stone               Keep 1 stone

   Total stones = 5 + 3 = 8
   Total groups = 2
   Removable = 4 + 2 = 6 = 8 - 2 ✓

   If you have 5 people on a team, 4 can leave. So total leaving
   equals total people minus number of teams.

──────────────────────────────────────────────────────────────

Q3: Why do we use DSU instead of BFS or DFS?

A: BFS or DFS would check every pair of stones to see if they
   share a row or column.

   Visual (BFS/DFS — slow):

   Stone 1: "Do you share row with Stone 2? Stone 3? Stone 4?"
   Stone 2: "Do you share row with Stone 3? Stone 4?"
   ...
   → O(N²) checks!

   Visual (DSU — fast):

   Row 0 ──→ Col 1     (one union per stone)
   Row 1 ──→ Col 2
   ...
   → O(N) unions!

   That is like asking every person in a stadium if they know each
   other — it takes forever. DSU is like giving everyone a team
   badge instantly. It runs in nearly O(N) time instead of O(N²).

──────────────────────────────────────────────────────────────

Q4: Why do we add an offset to column numbers?

A: Row 0 and Column 0 would both be called "0".

   Visual (WRONG — collision!):

   Row 0 ──→ ???
   Col 0 ──→ ???
   Both are "0" — DSU gets confused!

   Visual (CORRECT — offset fixes it):

   Find max row first: maxr = 2
   Offset = maxr + 1 = 3

   Row 0 stays as 0
   Col 0 becomes 0 + 3 = 3

   Row 0 ──→ 0
   Col 0 ──→ 3
   No collision! ✓

   It is like two people with the same name in a classroom. We
   give columns a different last name (the offset) so we can tell
   them apart in the DSU.

──────────────────────────────────────────────────────────────

Q5: Can two stones in different rows and columns be in the same group?

A: Yes! Like a chain of friends.

   Visual:

   Stone A at [0,0] ──row 0──→ Stone B at [0,1]
                                    │
                                    │ col 1
                                    ↓
                              Stone C at [2,1]

   A and C do NOT share a row or column directly.
   But A → B → C forms a chain.
   So A, B, and C are all in the same group!

   DSU catches these hidden connections automatically by connecting
   row 0 to col 1, and row 2 to col 1 — all in one family.

──────────────────────────────────────────────────────────────

Q6: What happens if all stones are alone (no shared rows or columns)?

A: Then every stone is its own group.

   Visual:

     0 1 2
   0 S . .
   1 . S .
   2 . . S

   Each stone lives on its own island:

   Island 1    Island 2    Island 3
      S           S           S

   You cannot remove any stone because no stone has a neighbor
   to lean on. The answer is 0.

   It is like having separate islands with no bridges between them.

──────────────────────────────────────────────────────────────

Q7: Can we remove ALL stones?

A: No. Every group must keep at least one stone.

   Visual:

   Before removals:
      S ─ S
      │
      S ─ S

   After removing 3 stones:
      .   .
      .   .
          S   ← last stone must stay!

   Why? The last stone has no neighbor in its row or column,
   so there is no one to "support" its removal.

   It is like musical chairs — when the music stops, one chair
   per group must still have someone sitting on it.

──────────────────────────────────────────────────────────────

Q8: Why does uniting row and column work?

A: Every stone is like a bridge connecting its row street and its
   column avenue.

   Visual:

        col 1
          │
   row 0──S──[0,1]
          │
        col 0

   Stone at [0,1] connects row 0 and col 1.

   Two stones sharing a row:

   row 0──S──S──  (both connect to row 0)
         [0,0] [0,1]

   Two stones sharing a column:

         S  ← [0,1]
         │
   col 1──S  ← [2,1]

   If two stones share a row, they both connect to the same
   street, so they are in the same neighborhood. DSU tracks
   these neighborhoods by merging row and column teams.

──────────────────────────────────────────────────────────────

Q9: What if coordinates are negative?

A: The vector-based DSU in our code assumes non-negative rows
   because rows are used as array indices.

   Visual (problem):

   Stone [-1, 0]: row = -1
   Vector index -1 → OUT OF BOUNDS! ✗

   Solutions:

   1. Shift all coordinates to positive first:
      Add 10000 to every row and column
      [-1, 0] becomes [9999, 10000]

   2. Use map-based DSU (creates nodes on demand):
      parent[-1] = -1      ← works!
      parent[10001] = 10001

   Visual (map-based fix):

   parent: { -1→-1, 0→0, 10001→10001 }
   No array indexing needed! ✓

   For this problem, coordinates are usually non-negative.
   If negatives appear, use the map-based approach or shift them.

──────────────────────────────────────────────────────────────

Q10: What is the most common mistake in this problem?

A: Mistake 1: Forgetting the offset.

   Visual (WRONG):

   Row 0 ──→ 0
   Col 0 ──→ 0   ← same node! DSU thinks they are the same!

   Visual (CORRECT):

   offset = maxr + 1 = 3
   Row 0 ──→ 0
   Col 0 ──→ 3   ← different node! ✓

   Mistake 2: DSU size too small.

   Visual (WRONG):

   maxr = 2, maxc = 2
   DSU size = 4
   But col 2 needs index 2+3 = 5 → CRASH!

   Visual (CORRECT):

   DSU size = maxr + maxc + 2 = 6
   Indices 0 to 5 all valid! ✓

   Mistake 3: Counting all DSU nodes as components.

   Visual (WRONG):

   DSU has 6 nodes: 0,1,2,3,4,5
   You check all 6 → count unused nodes too → WRONG!

   Visual (CORRECT):

   Only check rows of actual stones:
   row 0 → root A
   row 1 → root A
   row 2 → root A
   → 1 component! ✓

   It is like putting two students with the same ID number in the
   database — everything gets confused.

──────────────────────────────────────────────────────────────

Q11: How do we know the formula N - G is the MAXIMUM removable?

A: From any group of K stones, you can NEVER remove more than K-1.

   Visual (why K-1 is the limit):

   Group of 4 stones:
      S ─ S
      │
      S ─ S

   Step 1: Remove one stone (has neighbor) → 3 left
   Step 2: Remove one stone (has neighbor) → 2 left
   Step 3: Remove one stone (has neighbor) → 1 left
   Step 4: Last stone has NO neighbor → CANNOT remove!

   Maximum removed = 3 = 4 - 1 ✓

   And you can ALWAYS achieve K-1 by removing carefully.
   So K-1 is both the upper bound and achievable — it is the
   maximum.

──────────────────────────────────────────────────────────────

Q12: Can we solve this with a map-based DSU instead?

A: Yes! The vector-based DSU is fast but needs careful sizing.

   Visual (vector-based — needs sizing):

   maxr = 2, maxc = 2
   Need size = 2 + 2 + 2 = 6
   parent: [0, 1, 2, 3, 4, 5]
   Must calculate size BEFORE creating DSU!

   Visual (map-based — easier):

   parent: {}  (empty, creates on demand)

   unite(0, 3):  parent[0]=0, parent[3]=3 → parent[0]=3
   unite(0, 4):  parent[4]=4 → parent[3]=4
   No size calculation needed! ✓

   Trade-offs:
   → Vector: Faster, but needs max size upfront
   → Map:    Simpler, but slightly slower

   Both work perfectly for this problem!

──────────────────────────────────────────────────────────────

Q13: What does it mean when find(row) gives the same result for two stones?

A: It means those two stones are in the same connected group.

   Visual:

   Stone [0,0]: findUParent(row 0) → root 0
   Stone [2,1]: findUParent(row 2) → root 0

   Same root → same family!

   DSU family tree (offset = 3):

        root 0
       /  |   \
   row 0 row 1 row 2
    │      │      │
   col 0  col 2  col 1
   (3)    (5)    (4)

   They might be connected directly (same row or column) or
   indirectly through a chain of other stones. Same DSU root
   means same family.

──────────────────────────────────────────────────────────────

Q14: Why check findUParent(stone[0]) and not findUParent(stone[1] + offset)?

A: Either works! Every stone connects its row to its column, so
   both lead to the same ultimate parent.

   Visual (offset = 3):

   Stone [0, 1]:

   row 0 ──→ col 1 (4)
      \       /
       \     /
        \   /
       same root!

   findUParent(row 0)    → root 0
   findUParent(col 1)    → root 0
   Both give the same answer! ✓

   We pick rows for convenience, but picking columns would give
   the exact same count.

──────────────────────────────────────────────────────────────

Q15: What are real-world uses of this idea?

A: The idea of finding connected components through shared
   attributes shows up everywhere:

   Social networks:

   Alice ──friend──→ Bob ──friend──→ Carol
   (connected through shared friends)

   Network routing:

   PC 1 ──switch──→ PC 2 ──router──→ PC 3
   (connected through shared hardware)

   Logistics:

   Package A ──hub──→ Package B ──hub──→ Package C
   (connected through shared transit hubs)

   Biology:

   Gene A ──function──→ Gene B ──function──→ Gene C
   (connected through shared functions)

   DSU helps find these hidden connections quickly!
```
