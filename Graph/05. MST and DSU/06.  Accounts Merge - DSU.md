## 1. Problem Statement

Given a list of `accounts` where each element `accounts[i]` is a list of strings:
- The first element `accounts[i][0]` is a **name**
- The rest are **emails** belonging to that account

**Merge** accounts that share at least one common email.

Return each user's name followed by their **sorted, unique emails**.

> **Note:** Two accounts can have the same name but belong to different people if they don't share any common email.

## Key Insight

```
Observation 1: Email sharing creates connectivity

If two accounts share even ONE email,
they belong to the SAME person.

Visual:

  Account A          Account B
  ├─ e1@             ├─ e1@  ← shared!
  ├─ e2@             └─ e3@
  └─ e4@

  A and B share e1@ → SAME PERSON → Merge!

──────────────────────────────────────────────────────────────

Observation 2: Sharing is transitive

If A shares with B, and B shares with C,
then A, B, and C are ALL the same person!

Visual (chain):

  Account A          Account B          Account C
  ├─ e1@             ├─ e1@             ├─ e3@
  └─ e2@             └─ e3@             └─ e4@
       ↑ shared           ↑ shared

  A ↔ B ↔ C  →  All three = SAME PERSON!

  Even though A and C share NO direct email,
  they connect through B.

──────────────────────────────────────────────────────────────

Observation 3: DSU tracks these groups perfectly

Each account starts as its own group.
When two accounts share an email → union them.
After processing all accounts:
→ Accounts with same DSU root = same person
→ Collect all emails under that root
→ Sort and return!
```

### Why Use DSU?

```
Traditional approach (DFS/BFS):
→ Build graph: account → accounts sharing emails
→ Run DFS from each unvisited account
→ Time: O(N² × K) to find shared emails
→ Space: O(N²) for adjacency list

Visual (DFS — messy):

  Account 0 ──?──→ Account 1  (check all emails)
  Account 0 ──?──→ Account 2  (check all emails)
  ...
  → Too many checks!

DSU approach:
→ One pass through all emails
→ When email seen before → union accounts
→ Time: O(N × K × α(N)) ≈ O(N × K)
→ Much faster!

Visual (DSU — clean):

  j1@ appears in Account 0 and 3
  → union(0, 3)  ← one operation!

  r2@ appears in Account 2 and 4
  → union(2, 4)  ← one operation!
```

## Intuition

```
Imagine you are organizing a big family reunion.

Each account is like a person who wrote down their phone numbers.
If two people have the SAME phone number, they must be the same
person (or at least family members who should sit together).

Account 0: John wrote [111, 222, 333]
Account 1: John wrote [111, 444]

They both have phone 111 → SAME JOHN!
→ Merge their phone lists: [111, 222, 333, 444]

Account 2: Mary wrote [555]
Account 3: John wrote [666]

No matching phones → Different people → Keep separate!

──────────────────────────────────────────────────────────────

The DSU is like a seating chart organizer:

Start: Everyone sits at their own table.

Table 0: John (account 0)
Table 1: John (account 1)
Table 2: Mary (account 2)
Table 3: John (account 3)

When we see phone 111 in both Table 0 and Table 1:
→ "You two sit together!" → Merge tables!

After merging:

Table 0+1: John ── 111, 222, 333, 444
Table 2:   Mary ── 555
Table 3:   John ── 666

The organizer (DSU) remembers who sits where.
We just ask: "Who is at the head of Table 0's group?"
→ Everyone at that table belongs together!
```

## Algorithm

```
Step 1: Build email-to-account mapping

For each account i:
  For each email in account i (skip name):
    If email not seen before:
      map[email] = i
    Else:
      union(i, map[email])  ← same email = same person!

Visual:

  Process Account 0: John, j1@, j2@, j3@
    j1@ → map[j1@] = 0
    j2@ → map[j2@] = 0
    j3@ → map[j3@] = 0

  Process Account 3: John, j1@, j5@
    j1@ → already in map! value = 0
    → union(3, 0)  ← accounts 3 and 0 connected!
    j5@ → map[j5@] = 3

  DSU after unions:

    0 ←── 3
    1
    2
    4
    5

──────────────────────────────────────────────────────────────

Step 2: Group emails by ultimate parent

For each (email, accountIndex) in map:
  root = find(accountIndex)
  mergedEmails[root].push_back(email)

Visual:

  j1@ (account 0): find(0) → 0  ┐
  j2@ (account 0): find(0) → 0  │ same group
  j3@ (account 0): find(0) → 0  │
  j5@ (account 3): find(3) → 0  ┘

  mergedEmails[0] = [j1@, j2@, j3@, j5@]

  j4@ (account 1): find(1) → 1
  mergedEmails[1] = [j4@]

  r1@ (account 2): find(2) → 2  ┐
  r2@ (account 2): find(2) → 2  │ same group
  r3@ (account 4): find(4) → 2  ┘

  mergedEmails[2] = [r1@, r2@, r3@]

  m1@ (account 5): find(5) → 5
  mergedEmails[5] = [m1@]

──────────────────────────────────────────────────────────────

Step 3: Build final result

For each account i:
  If mergedEmails[i] is not empty:
    Sort mergedEmails[i]
    Create [name] + sorted emails
    Add to answer

Visual:

  Account 0: mergedEmails[0] = [j1@, j2@, j3@, j5@]
    Sort → [j1@, j2@, j3@, j5@]
    Result → ["John", "j1@", "j2@", "j3@", "j5@"]

  Account 1: mergedEmails[1] = [j4@]
    Result → ["John", "j4@"]

  Account 2: mergedEmails[2] = [r1@, r2@, r3@]
    Result → ["Raj", "r1@", "r2@", "r3@"]

  Account 3: mergedEmails[3] = [] → skip!
  Account 4: mergedEmails[4] = [] → skip!

  Account 5: mergedEmails[5] = [m1@]
    Result → ["Mary", "m1@"]
```

## Code

```cpp
#include <bits/stdc++.h>
using namespace std;

class DisjointSet {
    vector<int> rank, parent, size;

public:
    DisjointSet(int n) {
        rank.resize(n + 1, 0);
        parent.resize(n + 1);
        size.resize(n + 1, 1);
        for (int i = 0; i <= n; i++) {
            parent[i] = i;
        }
    }

    int findUPar(int node) {
        if (node == parent[node])
            return node;
        return parent[node] = findUPar(parent[node]);
    }

    void unionBySize(int u, int v) {
        int ulp_u = findUPar(u);
        int ulp_v = findUPar(v);
        if (ulp_u == ulp_v) return;
        if (size[ulp_u] < size[ulp_v]) {
            parent[ulp_u] = ulp_v;
            size[ulp_v] += size[ulp_u];
        }
        else {
            parent[ulp_v] = ulp_u;
            size[ulp_u] += size[ulp_v];
        }
    }
};

class Solution {
public:
    vector<vector<string>> accountsMerge(vector<vector<string>>& accounts) {
        int n = accounts.size();
        DisjointSet ds(n);

        // Map: email -> account index
        unordered_map<string, int> emailToNode;

        // Step 1: Union accounts with common emails
        for (int i = 0; i < n; i++) {
            for (int j = 1; j < accounts[i].size(); j++) {
                string email = accounts[i][j];
                if (emailToNode.find(email) == emailToNode.end()) {
                    emailToNode[email] = i;
                }
                else {
                    ds.unionBySize(i, emailToNode[email]);
                }
            }
        }

        // Step 2: Group emails by ultimate parent
        vector<string> mergedMails[n];
        for (auto& [email, node] : emailToNode) {
            int ultimateParent = ds.findUPar(node);
            mergedMails[ultimateParent].push_back(email);
        }

        // Step 3: Build final result
        vector<vector<string>> ans;
        for (int i = 0; i < n; i++) {
            if (mergedMails[i].empty()) continue;
            sort(mergedMails[i].begin(), mergedMails[i].end());
            vector<string> temp;
            temp.push_back(accounts[i][0]);  // Name
            for (string& email : mergedMails[i]) {
                temp.push_back(email);
            }
            ans.push_back(temp);
        }

        return ans;
    }
};

int main() {
    vector<vector<string>> accounts = {
        {"John", "j1@com", "j2@com", "j3@com"},
        {"John", "j4@com"},
        {"Raj", "r1@com", "r2@com"},
        {"John", "j1@com", "j5@com"},
        {"Raj", "r2@com", "r3@com"},
        {"Mary", "m1@com"}
    };

    Solution obj;
    vector<vector<string>> result = obj.accountsMerge(accounts);

    for (auto& acc : result) {
        cout << acc[0] << ": ";
        for (int i = 1; i < acc.size(); i++) {
            cout << acc[i] << " ";
        }
        cout << endl;
    }

    return 0;
}
```

**Output:**

```
John: j1@com j2@com j3@com j5@com 
John: j4@com 
Mary: m1@com 
Raj: r1@com r2@com r3@com 
```

## Edge Cases

```
Single account:
[["John", "j1@", "j2@"]]

→ Only one account → no merging needed
→ Output: [["John", "j1@", "j2@"]] ✓

──────────────────────────────────────────────────────────────

No common emails:
[["A", "a@"], ["B", "b@"], ["C", "c@"]]

→ Three separate groups
→ Output as three separate accounts ✓

Visual:

  A ── a@     B ── b@     C ── c@

  No connections → 3 separate groups

──────────────────────────────────────────────────────────────

Same name, different people:
[["John", "j1@"], ["John", "j2@"]]

→ No shared emails → different Johns!
→ Output: Two separate John accounts ✓

Visual:

  John ── j1@     John ── j2@

  No shared email → NOT the same person!

──────────────────────────────────────────────────────────────

All accounts merge into one:
[["J", "e1@"],
 ["J", "e1@", "e2@"],
 ["J", "e2@", "e3@"]]

→ All connected transitively
→ One merged account with all emails ✓

Visual:

  Account 0 ── e1@ ── Account 1 ── e2@ ── Account 2

  0 ↔ 1 ↔ 2 → All same root!

──────────────────────────────────────────────────────────────

Duplicate emails in same account:
[["J", "e1@", "e1@", "e2@"]]

→ Map stores last occurrence
→ DSU handles correctly
→ After sorting: unique emails only ✓

Note: map overwrites, but same account → no union needed

──────────────────────────────────────────────────────────────

One account with many emails:
[["J", "e1@", "e2@", ..., "e100@"]]

→ No other accounts → stays as-is
→ Sorting 100 emails: O(100 log 100) ✓

──────────────────────────────────────────────────────────────

Long chain of connections:
A→B→C→D→E→F→G→H→I→J

→ All merge into one group
→ DSU handles efficiently with path compression ✓
```

### Common Mistakes

```
❌ Mistake 1: Starting from index 0 (name)
for (int j = 0; j < accounts[i].size(); j++) {
    string email = accounts[i][j];  // WRONG! Index 0 is name!
}

✓ Correct: Start from index 1 (skip name)
for (int j = 1; j < accounts[i].size(); j++) {
    string email = accounts[i][j];
}

Visual:
  accounts[i] = ["John", "j1@", "j2@"]
                ↑ name    ↑ emails start here
                index 0   index 1

──────────────────────────────────────────────────────────────

❌ Mistake 2: Not sorting emails
mergedMails[ultimateParent].push_back(email);
// ... later ...
ans.push_back(temp);  // WRONG! Emails not sorted!

✓ Correct:
sort(mergedMails[i].begin(), mergedMails[i].end());

Visual (WRONG):
  ["John", "j3@", "j1@", "j2@"]  ← not sorted!

Visual (CORRECT):
  ["John", "j1@", "j2@", "j3@"]  ← sorted! ✓

──────────────────────────────────────────────────────────────

❌ Mistake 3: Using wrong account name
temp.push_back(accounts[ultimateParent][0]);  // WRONG!

Why wrong? ultimateParent might be account 0,
but account 3's name is also "John".
It happens to work here, but using index i is safer!

✓ Correct:
temp.push_back(accounts[i][0]);

Visual:
  mergedMails[0] contains emails from accounts 0 and 3
  Both have name "John" → same result
  But always use accounts[i][0] for correctness!

──────────────────────────────────────────────────────────────

❌ Mistake 4: Not checking if mergedMails is empty
for (int i = 0; i < n; i++) {
    sort(mergedMails[i].begin(), mergedMails[i].end());  // WRONG!
    // merged child accounts have empty lists!
}

✓ Correct:
if (mergedMails[i].empty()) continue;

Visual:
  Account 3 was merged into account 0
  mergedMails[3] = []  ← empty!
  Skip it! ✓

──────────────────────────────────────────────────────────────

❌ Mistake 5: Not using path compression
int findUPar(int node) {
    if (node == parent[node]) return node;
    return findUPar(parent[node]);  // WRONG! No compression
}

✓ Correct:
return parent[node] = findUPar(parent[node]);

Visual (without compression):
  0 ← 1 ← 2 ← 3   find(3) → 3 steps

Visual (with compression):
  0 ← 1   find(3) → 1 step, then:
  0 ← 2
  0 ← 3
```

## Complexity Analysis

**Time Complexity:** `O(N × K × α(N) + E log E)`

- N = number of accounts
- K = average emails per account
- E = total number of unique emails
- Process all emails: O(N × K × α(N))
- Sort emails for each group: O(E log E)
- Total: **O(N × K + E log E)**

**Space Complexity:** `O(N + E)`

- DSU arrays: O(N)
- Email map: O(E)
- Merged mails: O(E)
- Answer: O(E)
- Total: **O(N + E)**

```
Visual comparison:

DFS/BFS approach:

  Build graph: email → list of accounts
  For each account, BFS to find all connected accounts

  Account 0: check emails → find accounts → visit neighbors
  Account 1: check emails → find accounts → visit neighbors
  ...

  Time: O(N² × K)  → slow for many accounts!
  Space: O(N²)     → large adjacency list!

DSU approach:

  One pass through all emails:
  j1@ in account 0 and 3 → union(0, 3)
  r2@ in account 2 and 4 → union(2, 4)

  Time: O(N × K)   → one pass, much faster!
  Space: O(N + E)  → small and clean!

Detailed breakdown:

Processing accounts:
→ N accounts
→ K emails per account (average)
→ Each email: map lookup/insert + union
→ Union: O(α(N)) ≈ O(1)
→ Total: O(N × K × α(N)) ≈ O(N × K)

Grouping emails:
→ Iterate E emails
→ Each find(): O(α(N)) ≈ O(1)
→ Total: O(E)

Sorting emails:
→ For each group, sort emails
→ Total emails = E
→ Worst case: all in one group
→ Sort: O(E log E)

Building result:
→ O(N + E) to construct answer

Total: O(N × K + E log E)

Space:
→ parent, rank, size: O(N)
→ emailToNode map: O(E)
→ mergedMails: O(E)
→ Answer: O(E)
→ Total: O(N + E)
```

## Interview Questions

```
Q1: What is the main idea of this problem?

A: Imagine you found a pile of business cards. Some cards have the
   same phone number — those must belong to the same person! Your
   job is to group all cards by person and list their unique phone
   numbers in order.

   Visual:

   Card 0: John ── 111 ── 222
   Card 1: John ── 111 ── 333  ← shares 111 with Card 0!
   Card 2: Mary ── 444

   Group 1: John ── 111, 222, 333
   Group 2: Mary ── 444

──────────────────────────────────────────────────────────────

Q2: Why do we use DSU instead of DFS or BFS?

A: Think of DSU like a smart organizer who instantly says "sit
   together" when two people share a phone number. DFS is like
   walking around the room asking everyone if they know each other
   — it works but takes longer.

   Visual (DSU — instant):

   j1@ in account 0 and 3
   → union(0, 3)  ← done in one step!

   Visual (DFS — slower):

   From account 0, check all other accounts for shared emails
   From account 1, check all other accounts...
   → Many repeated checks!

   DSU is cleaner and faster for this problem.

──────────────────────────────────────────────────────────────

Q3: Why does the same name NOT always mean the same person?

A: There are many people named "John" in the world! Just because
   two business cards say "John" doesn't mean they are the same
   John. We need PROOF — a shared email — to merge them.

   Visual:

   John ── j1@mail.com     John ── j2@mail.com

   Same name, but no shared email → DIFFERENT PEOPLE! ✓

   John ── j1@mail.com     John ── j1@mail.com

   Same email → SAME PERSON! ✓

──────────────────────────────────────────────────────────────

Q4: What is the email-to-account map used for?

A: It is like a phone book. When you see a phone number, you look
   it up to find who had it before. If someone already has that
   number, you know those two people are the same person.

   Visual (phone book):

   j1@ → Account 0
   j2@ → Account 0
   j3@ → Account 0

   Later see j1@ again in Account 3:
   → Look up j1@ → found Account 0
   → union(3, 0) → same person!

──────────────────────────────────────────────────────────────

Q5: How does transitivity work in this problem?

A: Like a friend chain. If Alice knows Bob, and Bob knows Carol,
   then Alice, Bob, and Carol are all in the same friend circle.

   Visual:

   Account A: e1@, e2@
   Account B: e2@, e3@      ← shares e2@ with A
   Account C: e3@, e4@      ← shares e3@ with B

   A ── e2@ ── B ── e3@ ── C

   A and C never directly shared an email,
   but they are connected through B!
   → All three = same person! ✓

──────────────────────────────────────────────────────────────

Q6: Why do we skip empty mergedMails when building the result?

A: When account 3 merges into account 0, all its emails move to
   account 0's list. Account 3's list becomes empty. We skip it
   because we already counted those emails under account 0.

   Visual:

   Before skip check:
     Account 0: [j1@, j2@, j3@, j5@]  ← keep
     Account 3: []                     ← empty, skip!

   Without skip → we would output an empty John account! ✗

──────────────────────────────────────────────────────────────

Q7: Why do we sort the emails?

A: The problem asks for sorted emails. It is like alphabetizing
   a contact list so it is easy to read.

   Visual (unsorted):
     ["John", "zebra@", "apple@", "mango@"]

   Visual (sorted):
     ["John", "apple@", "mango@", "zebra@"] ✓

──────────────────────────────────────────────────────────────

Q8: Can we use union by rank instead of union by size?

A: Yes! Both work. Union by size attaches the smaller tree to the
   bigger tree using node count. Union by rank uses tree height.
   Both keep the DSU tree flat and fast.

   Visual (union by size):

   Size 3 tree ←── Size 1 tree
   (bigger absorbs smaller)

   Visual (union by rank):

   Rank 2 tree ←── Rank 1 tree
   (taller stays as root)

   For this problem, either is fine!

──────────────────────────────────────────────────────────────

Q9: What happens if an account has duplicate emails?

A: The map stores the LAST account index it saw for that email.
   But since both occurrences are in the SAME account, no union
   happens. The email appears once in the final list because we
   iterate over the map keys (which are unique).

   Visual:

   Account 0: ["J", "e1@", "e1@", "e2@"]

   Map stores:
     e1@ → 0 (overwritten, still 0)
     e2@ → 0

   mergedMails[0] = [e1@, e2@]  ← e1@ appears once! ✓

──────────────────────────────────────────────────────────────

Q10: What is the most common mistake?

A: Forgetting that index 0 is the NAME, not an email!

   Visual (WRONG):

   for (int j = 0; j < accounts[i].size(); j++)
       email = accounts[i][j]

   accounts[i][0] = "John" → treated as email! ✗

   Visual (CORRECT):

   for (int j = 1; j < accounts[i].size(); j++)
       email = accounts[i][j]

   accounts[i][1] = "j1@" → first email! ✓

   Another common mistake: forgetting to sort emails or skipping
   empty merged mail lists.

──────────────────────────────────────────────────────────────

Q11: Why use account index as DSU node instead of email?

A: Account indices are simple numbers (0, 1, 2, ...). Emails are
   long strings. Numbers are faster to compare and store. Also,
   we need the account name later, and keeping account indices
   makes that easy.

   Visual (account index):

   parent: [0, 1, 2, 0, 2, 5]
   Simple integers! ✓

   Visual (email as node):

   parent: {"j1@"→"j1@", "j2@"→"j1@", ...}
   String comparisons → slower! ✗

──────────────────────────────────────────────────────────────

Q12: How does path compression help?

A: After finding the root, path compression makes every node on
   the path point directly to the root. Next time, those nodes
   find the root in one step.

   Visual (before compression):

   0 ← 1 ← 2 ← 3

   find(3): 3→2→1→0 (3 steps)

   Visual (after compression):

   0 ← 1
   0 ← 2
   0 ← 3

   find(3): 3→0 (1 step!)

   This makes the algorithm nearly O(1) per operation.

──────────────────────────────────────────────────────────────

Q13: What if there are thousands of accounts?

A: DSU handles it easily! With path compression and union by size,
   each operation is almost O(1). Even with 10,000 accounts, it
   runs in milliseconds.

   Visual:

   10,000 accounts × 10 emails each = 100,000 operations
   Each operation ≈ O(1)
   Total time ≈ 100,000 steps → instant! ✓

──────────────────────────────────────────────────────────────

Q14: Can two different people have the exact same set of emails?

A: The problem says each email belongs to exactly one person. So
   if two accounts have the same emails, they ARE the same person
   and will be merged into one.

   Visual:

   Account 0: ["J", "e1@", "e2@"]
   Account 1: ["J", "e1@", "e2@"]

   e1@ and e2@ appear in both → union(0, 1)
   → One merged account! ✓

Q15: What are real-world uses of this idea?

A: The idea of merging connected groups shows up everywhere:

   Social media:

   Account A ── phone 123 ── Account B
   → Same person with multiple profiles → merge!

   Customer databases:

   Record 1: Jane ── jane@work.com
   Record 2: Jane ── jane@work.com, jane@home.com
   → Same customer → merge records!

   DSU helps find these hidden connections quickly!
```
